# 常见错误对照表

## 全流程

| 错误 | 正确做法 |
|------|----------|
| 跳过关卡直接执行 | 严格等待用户确认（参见 SKILL.md 关卡总览） |

## 数据查询

| 错误 | 正确做法 |
|------|----------|
| 探查型 SQL（SELECT *、抽样查看）不加 LIMIT | 探查型查询必须先 LIMIT 100 验证字段和数据格式 |
| 聚合型 SQL（GROUP BY + 聚合函数）盲目加 LIMIT | 聚合查询不应加 LIMIT（会截断分组结果），改用 WHERE 条件控制数据量 |
| 找不到字段就说"不存在" | 必须先解析 data/properties 等嵌套字段 |
| 同名字段在顶层和嵌套中都有值时直接取一个 | 必须用 AskUserQuestion 让用户选择，说明两者来源差异 |
| JOIN 关联字段大小写不一致 | JOIN 前统一做 `LOWER(a.key) = LOWER(b.key)` |
| Windows 终端中文乱码，从乱码猜测数据内容 | Python 脚本开头加 `sys.stdout.reconfigure(encoding='utf-8')` |

## 数据分析（阶段五）

| 错误 | 正确做法 |
|------|----------|
| 时间趋势图未标注季节性因素 | 必须用 `axvspan()` 标注寒暑假，`axvline()` 标注重大节假日 |
| 将活跃波动直接归因产品问题 | 先排查季节性因素（寒暑假、开学季、节假日），再归因产品/运营 |

## 报告写作（阶段六）

| 错误 | 正确做法 |
|------|----------|
| 只罗列数据不给结论 | 每段必须回答"所以呢" |
| 把 AI 分析单独列一章 | 深度思考融入每个分析模块 |
| 策略建议太泛 | 具体到人群、时间、动作 |
| 策略建议放在报告末尾 | 前置到「核心结论与战略建议」模块 |

## 图表交付（阶段七）

| 错误 | 正确做法 |
|------|----------|
| 一次性生成整篇飞书文档，图片集中插入末尾 | 逐章循环：每章 append 文字 → upload-image 图片 |
| 图表未校验就交付 | `ls -lh dar-charts/` 逐项检查 PNG 存在且 > 0 字节 |
| *（仅 HTML 报告模式）* HTML 报告只有图表没有文字 | 每个图表必须有 analysis-text（数据解读+观点） |
| *（仅 HTML 报告模式）* ECharts 图表未等待渲染完成就截图 | 截图前 `waitUntil: 'networkidle0'` + 额外 2s delay |
