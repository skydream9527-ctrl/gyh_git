## 核心原则

### 四大规则与检验方法

**① 结论先行**
- 规则：文档第一句话必须是核心结论/建议，不是背景铺垫。
- 检验：把文档第一段遮住，只看第一句话。读者能否立刻知道"这篇文档的答案是什么"？
- ❌ 失败："本报告分析了10款车型的电芯散差数据……"
- ✅ 通过："电芯散差整体可控，但269辆高散差车辆需立即介入，建议分级管理。"

**② 以上统下**
- 规则：每个章节用"简洁标题 + callout 分论点结论句"组合。标题是话题词用于导航，callout 是该层级的完整判断句。
- 检验：遮住正文，只看 callout。能否独立传达"这一章节得出了什么结论"？如果不能，callout 写的是描述而非判断。
- ❌ 失败：标题"车型差异分析" + 无 callout，或 callout 写"本节分析各车型散差数据"（描述，非判断）
- ✅ 通过：标题"车型差异" + callout "Modena系列散差系统性高于LeMans系列，差距达0.97%，需针对性优化BMS策略"

**③ 归类分组（MECE）**
- 规则：同一层级的论点相互独立（ME）且合起来覆盖全部情况（CE）。
- 检验ME：任意两个论点之间是否有内容重叠？如果有，合并或重新切分。
- 检验CE：把所有论点加起来，是否能完整支撑顶层结论？是否有明显遗漏的维度？
- ❌ 失败：论点A"散差整体可控" + 论点B"高散差车辆需关注" + 论点C"Modena-5问题严重"（B和C重叠，C是B的子集）
- ✅ 通过：论点A"整体可控" + 论点B"局部风险集中" + 论点C"车型差异显著"（独立且完整）

**④ 逻辑递进**
- 规则：同一层级的论点必须有明确的排列依据，且全程只用一种排序逻辑。
- 检验：能否用一句话说明这几个论点为什么是这个顺序？说不出来就是随意排列。
- 四种合法排序：重要性（最重要→次重要）、时间（过去→现在→未来）、结构（组成部分A→B→C）、逻辑因果（原因→影响→对策）
- ❌ 失败：论点按"数据量大→问题严重→建议优化"排列（混用了结构顺序和逻辑因果）
- ✅ 通过：按重要性排列，且骨架中声明"按风险优先级从高到低排列"

### 三层结构
```
结论/建议（1句话）
├── 支撑论点 A（一句话概括）
│   ├── 证据/数据 A1
│   └── 证据/数据 A2
├── 支撑论点 B
│   ├── 证据/数据 B1
│   └── 证据/数据 B2
└── 支撑论点 C
    └── ...
```

## SCQA 开场框架

适用于需要说服受众、提出建议的文档（汇报、方案、分析报告）。

| 元素 | 说明 | 示例 |
|------|------|------|
| **S**ituation（情境） | 读者已知的背景事实，建立共识 | "当前电池健康度分析每周需要人工处理 8 小时" |
| **C**omplication（冲突） | 打破平衡的变化或问题，制造张力 | "随着车型增加，人工处理已无法满足时效要求" |
| **Q**uestion（疑问） | 读者心中自然产生的问题（可不写出） | "我们该怎么办？" |
| **A**nswer（答案） | 直接给出结论/建议 | "建议引入自动化 SOH 分析流程，预计节省 90% 人工" |

**SCQA 变体：**
- **标准式**：S → C → Q → A（适合说服型文档）
- **直入式**：A → S → C（适合内部技术文档，读者关心结论）
- **突出忧虑式**：C → S → Q → A（适合危机/紧急场景）

## MECE 原则

**Mutually Exclusive, Collectively Exhaustive**（相互独立，完全穷尽）

### 相互独立（ME）
- 每个分类之间不重叠
- ❌ 错误：按「用户类型」分为「新用户」「活跃用户」「付费用户」（付费用户可能是活跃用户）
- ✅ 正确：按「注册时间」分为「30天内」「31-90天」「90天以上」

### 完全穷尽（CE）
- 所有分类加起来覆盖全部情况
- 常用穷尽方式：
  - **公式拆解**：利润 = 收入 - 成本
  - **流程拆解**：获客 → 激活 → 留存 → 变现
  - **矩阵拆解**：2×2矩阵（重要/不重要 × 紧急/不紧急）
  - **时间拆解**：短期/中期/长期

### 常用 MECE 框架
| 框架 | 适用场景 |
|------|---------|
| 时间轴（过去/现在/未来） | 复盘、规划类文档 |
| 内外部因素 | 问题分析、风险评估 |
| 人/流程/技术 | 系统性问题 |
| 战略/战术/执行 | 方案规划 |
| 问题/影响/解决方案 | 故障报告、问题分析 |

## 纵向逻辑

### 演绎推理（Deductive）
**适用**：论证一个观点、说服不确定的读者  
**结构**：大前提 → 小前提 → 结论

```
大前提：提高自动化程度可降低人工成本
小前提：当前 SOH 分析 80% 依赖人工
结论：  推进 SOH 自动化可显著降低成本
```

### 归纳推理（Inductive）
**适用**：列举多个并列证据支持同一结论  
**结构**：结论 ← [证据1, 证据2, 证据3]

```
结论：自动化 SOH 分析方案可行
  ├── 证据1：技术方案已验证（POC 完成）
  ├── 证据2：成本节约可达 90%
  └── 证据3：行业内已有成功案例
```

**选择原则**：
- 证据类型相似 → 用归纳
- 需要推导过程 → 用演绎
- 汇报类文档 → 优先归纳（更清晰）

## 横向逻辑

同一层级的观点排列顺序：

| 排序方式 | 适用场景 | 示例 |
|---------|---------|------|
| **时间顺序** | 步骤、流程、历史 | 第一步→第二步→第三步 |
| **结构顺序** | 系统组成、空间划分 | 前端→后端→数据库 |
| **重要性顺序** | 建议、问题列举 | 最重要→次重要→一般 |
| **逻辑顺序** | 因果关系 | 原因→影响→对策 |

**原则**：同一层级只用一种排序方式，不要混用。

## 常见错误与修正

| 错误类型 | 错误示例 | 修正方式 |
|---------|---------|---------|
| **结论在末尾** | "…因此我们建议采用方案A" | 开头直接写"建议采用方案A，原因如下" |
| **上层是引言而非总结** | "本文将从三个方面分析…" | 改为"分析表明，有三个核心问题需解决：…" |
| **分组不MECE** | 同类内容分散在不同层级 | 重新归类，确保同质性 |
| **过渡句代替论点** | "接下来我们看第二点" | 删除过渡句，直接写实质论点 |
| **数据堆砌无结论** | 列出一堆数字但不给结论 | 先写数据说明了什么，再附数据 |

## 写作检查清单

完成文档后逐项检查：

- [ ] 第一段是否直接给出核心结论/建议？
- [ ] 每个一级标题下方是否有 callout 分论点结论句？二级标题下是否用加粗首句代替？
- [ ] 二级标题是否用"角色：对象"格式，不是纯名词？
- [ ] 每层的分组是否满足 MECE（独立且完整）？
- [ ] 同一层级的内容类型是否一致？
- [ ] 支撑论点的顺序是否有明确依据（时间/重要性/逻辑），且全程只用一种？
- [ ] 是否有不必要的过渡句/废话？
