# 飞书金字塔写作

按金字塔原理组织内容，输出符合飞书排版规范的结构化文档。

**核心约束：在用户确认骨架之前，禁止输出任何正文内容。每个阶段必须等用户回复后才能进入下一阶段。**

---

## 阶段一：逐步收集三个关键信息

每次只问一个问题，等用户回答后再问下一个。不得将多个问题打包成列表一次抛出。

**问题 1 — 受众**

问法示例：
> 这篇文档主要给谁看？是需要做决策的（比如老板、评审委员会），还是需要执行的（比如工程师、分析师），还是两者都有？

根据答案调整后续问法：
- 决策层 → 问题2 问"你希望他们看完之后做什么决定或行动"
- 执行层 → 问题2 问"你希望他们看完之后理解什么、改变什么行为"

**问题 2 — 核心结论（顶层答案）**

这是整篇文档最重要的一句话。根据受众类型调整问法，引导用户说出一个完整的判断句，而不是话题词。

如果用户给出的是话题词（如"分析散差情况"），追问：
> 你的结论是什么？散差是好是坏？需要做什么？

**问题 3 — 写作目的（按需，可跳过）**

如果前两个答案已经足够清晰，跳过此问题。否则问：
> 这篇文档是为了说服对方接受一个结论，还是传递分析过程让他自己判断，还是推动一个具体决策？

---

## 阶段二：构建骨架并强制验证四大原则

读取 `references/pyramid-principle.md` 了解完整方法论。

基于用户提供的信息，构建逻辑骨架，并在骨架中**显式声明**每一条原则的执行情况：

```
顶层结论：[用户确认的核心结论，一句完整判断句]

支撑论点（归纳/演绎）：
├── 论点A：[一句话论点]
├── 论点B：[一句话论点]
└── 论点C：[一句话论点]

四大原则检验：
① 结论先行：顶层结论是否直接回答"我们该怎么办/这说明了什么"？[是/否，说明]
② 以上统下：每个论点是否是其下层内容的总结句而非引言句？[是/否，指出问题点]
③ 归类分组（MECE）：论点之间是否相互独立？合起来是否覆盖全部情况？[是/否，说明]
④ 逻辑递进：论点排列采用什么顺序（重要性/时间/逻辑因果）？依据是什么？[说明]
```

**禁止跳过检验步骤。** 如果某条原则未达标，必须先修正骨架，再呈现给用户。

将骨架（不含正文）呈现给用户，明确询问：
> 这个结构符合你的意图吗？有需要调整的地方吗？

**在用户明确确认骨架之前，禁止进入阶段三。**

---

## 阶段三：生成文档内容

用户确认骨架后，读取 `references/feishu-doc-structure.md` 套用飞书排版规范。

写作时逐条执行：

- **结论先行**：顶层结论用 callout 块放在文档最开头，不得用背景铺垫开场
- **以上统下**：callout 只出现在两处——文档顶部总结论、每个一级标题（`#`）下方的分论点结论句。二级标题下用**加粗第一句话**代替 callout，三级及以下不需要结论句。标准：只看各一级 callout，能否串联出完整的论证链
- **归类分组**：同一层级的内容类型必须一致（全是原因、全是证据、或全是建议），不得混放
- **逻辑递进**：同一层级的论点按骨架中声明的排序方式排列，不得随意调整顺序
- SCQA 背景：在 callout 之后、第一个论点之前，用 1-2 句话写出情境（S）和冲突（C），建立"为什么要看这篇文档"的张力

写完后逐项自查：

- [ ] 第一段是核心结论，不是背景铺垫
- [ ] 每个一级标题下方有 callout 分论点结论句；二级标题下用加粗首句，无 callout
- [ ] 同层论点满足 MECE（独立且完整）
- [ ] 同层论点排列顺序有明确依据，且全程只用一种排序逻辑
- [ ] SCQA 中有明确的冲突（C），不只是背景陈述
