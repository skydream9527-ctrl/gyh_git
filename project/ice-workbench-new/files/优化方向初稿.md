## 1. 架构层面优化

### 1.1 数据库演进

**现状**：使用 SQLite + aiosqlite，单文件存储，适合单机小团队。

**问题**：
- 多 worker 下写入冲突（GlobalRateLimit 内存计数，无跨 worker 协调）
- 无法支撑多实例水平扩展
- 全文搜索 FTS 功能在 SQLite 下能力有限

**优化方向**：
- [ ] **Phase 1**：保留 SQLite 作为默认开发/单机部署选项，抽象 Repository 层
- [ ] **Phase 2**：支持 PostgreSQL 作为生产级数据库选项（SQLAlchemy 已支持，需补充迁移脚本）
- [ ] **Phase 3**：对于高频计数器、inflight 锁、rate limit 引入 Redis
- [ ] FTS 从 SQLite FTS5 迁移到 Elasticsearch/Meilisearch（当数据量增长时）

### 1.2 服务解耦与消息队列

**现状**：所有逻辑在单一 FastAPI 进程内，后台任务通过 `asyncio.create_task` fire-and-forget。

**问题**：
- auto_memory 提取、scheduler 任务与 API 请求共进程，资源争抢
- 进程重启导致后台任务丢失
- 无法独立扩展计算密集型任务

**优化方向**：
- [ ] 引入轻量级任务队列（推荐 `arq` 基于 Redis，或 `Celery`）
- [ ] 将 auto_memory_extract、memory_sweep、scheduler 任务移到独立 Worker 进程
- [ ] 支持多 Worker 实例时的任务分发
- [ ] 考虑 LLM 调用和工具执行的异步队列化（长任务不阻塞 WS）

### 1.3 部署架构升级

**现状**：单端口同服前后端，单 uvicorn 进程。

**优化方向**：
- [ ] 生产环境支持 gunicorn + uvicorn workers 多进程
- [ ] 静态资源 CDN 分离（froentend/dist 独立部署）
- [ ] WebSocket 连接与 HTTP API 拆分部署（WS 长连接单独扩容）
- [ ] 提供 Docker Compose / Helm Chart 一键部署
- [ ] GitLab CI 已配置，补充自动测试和镜像构建流水线

### 1.4 多租户与权限

**现状**：基础用户系统 + 邀请协作，权限粒度较粗。

**优化方向**：
- [ ] 完善 RBAC 角色权限体系（viewer/editor/admin/owner）
- [ ] 工作空间（Workspace）概念隔离多团队数据
- [ ] API Key 管理，支持第三方集成
- [ ] 数据隔离审计日志

## 2. Agent Kernel 增强

### 2.1 ReAct 循环能力增强

**现状**：已实现统一 Kernel，支持并行工具、计划审批、人工介入。

**优化方向**：
- [ ] **思维链（CoT）结构化**：要求 LLM 输出 `<thought>` 标签分离思考过程，不暴露给用户但用于调试
- [ ] **子 Agent 路由增强**：spawn_routing 目前是 prompt 驱动，增加基于意图分类的硬路由规则
  - 意图分类器（轻量模型或规则）预判断应该 spawn 哪个子 Agent
  - 支持 Agent 能力注册和自动发现
- [ ] **工具调用前置校验**：参数在执行前做 JSON Schema 校验，减少 LLM 幻觉导致的工具错误
- [ ] **部分失败恢复**：多工具并行时，单个失败不影响其他工具结果，支持只重试失败的工具
- [ ] **上下文压缩（Compaction）增强**：
  - 现有 compaction_svc 可进一步优化
  - 长对话自动摘要，保留关键结论和工具结果，丢弃冗余交互
  - 支持 token 水位触发自动压缩（如上下文达到 70% 时触发）

### 2.2 工具系统升级

**现状**：9个工具模块，支持元数据标记（parallel_safe, side_effect, timeout）。

**问题**：
- 工具定义目前硬编码在 tools/ 目录下，动态扩展性弱
- 没有工具调用结果的缓存机制，重复查询浪费 token 和时间

**优化方向**：
- [ ] **工具注册机制**：动态工具注册，支持技能（skills）按需加载工具
- [ ] **工具结果缓存**：幂等性读操作（如 SQL 查询、文件读取）结果缓存，缓存 key 基于参数 hash
- [ ] **工具调用链追踪**：每个工具调用分配 trace_id，跨子 Agent 链路追踪
- [ ] **工具权限分级**：危险工具（写文件、执行代码、发飞书消息）需要审批才能执行
- [ ] **MCP（Model Context Protocol）支持**：兼容 MCP 标准工具协议，可接入外部工具服务器
- [ ] **工具使用统计与优化**：统计各工具成功率、平均耗时、错误类型，指导 prompt 优化

### 2.3 错误处理与鲁棒性

**现状**：error_classifier 结构化分类，支持重试。

**优化方向**：
- [ ] **自动恢复策略**：
  - Kyuubi/SQL 错误 → 自动修正 SQL 语法重试（最多1次）
  - LLM 格式错误 → 自动格式修正重试
  - 网络超时 → 指数退避重试（已有，可细化）
- [ ] **降级策略**：主模型不可用时自动降级到备用模型
- [ ] **错误归因分析**：统计各类错误占比，自动发现高频失败点
- [ ] **对话恢复**：进程崩溃/重启后，能从断点恢复对话状态（持久化中间状态）

### 2.4 计划与执行分离增强

**现状**：已支持 exit_plan_mode 提出计划，PlanApprovalPanel 用户审批。

**优化方向**：
- [ ] **可编辑计划**：用户审批时可以修改计划步骤，不仅是同意/拒绝
- [ ] **分步执行**：计划按步骤执行，每步确认，执行中可随时暂停调整
- [ ] **执行进度可视化**：Todo 列表更细粒度展示当前步骤、进度百分比、预计剩余时间
- [ ] **执行回放**：任务完成后可回看每一步做了什么，支持结果修正重跑

## 3. 记忆系统升级

记忆系统是当前架构中已做得相当不错的部分（BM25+生命周期+自动沉淀），但仍有很大提升空间。

### 3.1 检索能力增强

**现状**：BM25 + recency + confidence 复合评分，纯关键词匹配。

**问题**：
- 无法理解语义相似性（"用户偏好" 和 "喜欢的设置" 关键词不匹配）
- 中英文混合检索效果一般
- 记忆条目body全文没有参与BM25评分（目前只用slug+title+hook）

**优化方向**：
- [ ] **混合检索**：BM25 关键词检索 + 向量语义检索（Embedding）
  - 本地轻量向量库：推荐 `chromadb` 或 `lance-db`，无需额外服务
  - Embedding 模型：使用 Mify 网关的 embedding 接口或本地小模型
  - RRF（Reciprocal Rank Fusion）合并两路结果
- [ ] **检索范围扩展**：BM25  corpus 从 `slug+title+hook` 扩展到包含 body 文本
- [ ] **查询改写**：用户查询先用 LLM 做查询扩展（同义词、相关术语），提升召回率
- [ ] **记忆类型权重**：user偏好 > project知识 > feedback反馈，检索时加权
- [ ] **跨 Agent 记忆共享**：允许配置某些 Agent 可以访问其他 Agent 的记忆

### 3.2 自动记忆质量提升

**现状**：基于正则匹配结论章节，规则化提取。

**问题**：
- 正则提取召回率有限，很多有价值的结论没有明确标题
- 提取的 insight 质量参差不齐，没有质量过滤
- 没有对重复/矛盾记忆的处理

**优化方向**：
- [ ] **LLM 辅助记忆提取**：对话结束后，用小模型判断是否有值得沉淀的知识，并生成结构化摘要
  - 不是每轮都调用，而是对话结束/任务完成时后台异步提取
  - 控制成本：用小参数模型或只在长任务后提取
- [ ] **记忆去重与合并**：
  - 新记忆入库前，语义相似度检测已有相似记忆
  - 相似记忆合并更新，而不是新增（更新usage_count，保留最新信息）
- [ ] **矛盾检测**：新记忆与已有记忆矛盾时，标记冲突，提示用户确认哪条正确
- [ ] **置信度自动调整**：
  - 用户采纳了 Agent 基于某条记忆给出的建议 → 提高 confidence
  - 用户修正了基于某条记忆的结论 → 降低 confidence
  - 多次命中且未被纠正 → 逐步提升 confidence

### 3.3 记忆生命周期优化

**现状**：固定时间窗口（30天stale，7天archive，3次使用+2用户晋升）。

**优化方向**：
- [ ] **自适应衰减**：不同类型记忆衰减速率不同
  - user 偏好：衰减很慢（180天半衰期）
  - project 知识：中等衰减（30天半衰期，当前值）
  - feedback 反馈：较快衰减（7天半衰期）
- [ ] **访问频率加速遗忘/巩固**：
  - 高频访问记忆延长生命周期
  - 长期不访问即使未到30天也逐步降权
- [ ] **记忆归档检索**：归档记忆不是删除，检索时可选择"包含归档"
- [ ] **人工记忆管理UI**：用户可以查看、编辑、删除、 pin（置顶永不衰减）自己的记忆
- [ ] **记忆导出/导入**：支持记忆的导出备份和跨实例迁移

### 3.4 团队记忆增强

**现状**：支持自动晋升，简单的共享池。

**优化方向**：
- [ ] **团队记忆审核**：用户 Agent 记忆自动晋升到团队前，需要管理员审核
- [ ] **记忆贡献者可见**：显示记忆来源用户，支持评价（有用/无用）
- [ ] **团队记忆分层**：公共知识 / 项目组私有 / 个人，不同可见范围
- [ ] **组织级记忆**：比团队更大范围的公共知识库

### 3.5 工作记忆（Working Memory）

**现状**：messages 列表就是上下文，没有专门的工作记忆层。

**优化方向**：
- [ ] **任务级工作记忆**：单个任务执行期间维护结构化工作记忆
  - 已验证的事实
  - 待办事项
  - 已完成步骤
  - 发现的问题
- [ ] **工作记忆持久化**：任务中断恢复时，工作记忆作为上下文恢复点

## 4. 性能优化

### 4.1 后端性能

**问题观察**：
- 记忆检索每次都要读文件解析 MEMORY.md 和所有 md 文件，IO 和解析开销大
- LLM 调用是主要瓶颈，没有流式缓存
- 全局 rate limit 是进程内存级，多 worker 不准确

**优化方向**：
- [ ] **记忆索引缓存**：启动时加载索引到内存，文件变更时增量更新（inotify）
- [ ] **常用查询结果内存缓存**：系统配置、Agent 配置等热点数据 LRU 缓存
- [ ] **LLM 流式响应优化**：确保 SSE/WS 不做不必要的缓冲，真正的 token-by-token 流式
- [ ] **数据库连接池优化**：aiosqlite 实际上是串行执行，迁移 PostgreSQL 后使用 asyncpg 连接池
- [ ] **API 响应压缩**：启用 gzip/brotli 压缩
- [ ] **慢查询日志**：记录耗时 >500ms 的 API 和 >1s 的工具调用

### 4.2 前端性能

**现状**：已做 ECharts 懒加载、代码分割。

**问题**：
- 长对话（>100条消息）渲染可能卡顿
- Markdown 渲染每条消息都重新高亮
- 首屏加载资源体积可优化

**优化方向**：
- [ ] **虚拟滚动**：MessageList 使用虚拟滚动（react-virtuoso），只渲染可视区域消息
- [ ] **Markdown 渲染优化**：
  - 代码高亮按需懒加载语言包
  - 已完成的消息 memoize，不重复渲染
  - 长内容折叠展开
- [ ] **首屏资源优化**：
  - 路由级代码分割（已做基础版，进一步细化）
  - 大依赖（ECharts、代码高亮）异步加载
  - 分析 bundle 体积，拆分大 chunk
- [ ] **WebSocket 消息处理**：高频 delta 消息做 requestAnimationFrame 节流，避免过度重渲染
- [ ] **状态管理优化**：Zustand selectors 精细化订阅，避免无关组件重渲染
- [ ] Service Worker 缓存静态资源，离线可用基础界面

## 附录：当前架构优势（值得保留和强化）

在提出优化方向的同时，需要肯定现有架构中已经做得很好的部分：

1. ✅ **Agent Kernel 统一抽象**：EventSink 设计优雅，流式和后台调用统一，扩展性好
2. ✅ **错误分类体系**：error_classifier 结构化错误，支持不同恢复策略，基础已经打好
3. ✅ **记忆系统基础**：分层记忆 + BM25排序 + 生命周期管理 + 自动沉淀，架构领先
4. ✅ **服务层模块化拆分**：backend 按领域拆分模块，代码组织清晰
5. ✅ **WebSocket 重连与心跳**：前端 WS 层考虑周全（指数退避、token刷新重连、离线队列）
6. ✅ **移动端适配起步**：hooks + mobile.css，响应式基础好
7. ✅ **安全考虑周全**：CSP安全头、密钥校验、速率限制、路径遍历防护
8. ✅ **前后端同构部署**：单端口部署简单，适合快速交付

优化应该在这些优势基础上迭代，不要推倒重来。
