## 0. 结论摘要

1. **有两件事已经做完，不要再当成「待建设」**
   结构化工具错误（`error_type / recoverable / suggested_action`）与条件重试**已实现并接线**：
   - [`backend/app/services/llm/error_classifier.py`](../backend/app/services/llm/error_classifier.py) 做确定性分类；
   - [`llm_gateway.py:709`](../backend/app/services/llm/llm_gateway.py#L709) 在 `normalize_tool_outcome` 里挂上三字段；
   - [`agent_kernel.py`](../backend/app/services/agent/agent_kernel.py) 用 `run_tool_with_retry`，并在失败时向父 Agent 暴露这三个字段（R6.3）。
   即「讨论稿」里的议题八/九已落地，后续只剩 timeout/exception 路径补结构化错误（初稿2 P0-5）。

2. **主动性 / 自进化 已被「初稿2」系统梳理过**
   初稿2 已经给出 ProactiveOpportunityDetector、风险分级审批、后台结果回流、监控状态闭环、Memory 生命周期接线、Memory policy、经验卡片自动 draft、AgentEvalSuite 等较完整的 P0/P1/P2 路线。本文档不重复，只做两件事：**核对其关键结论在当前代码里是否仍成立**，并补充它没强调的点。

3. **真正缺口集中在两个维度：安全性 + Agent-to-Agent 对话协作**
   - 安全性：限流多 worker 失效、SQL 仅首词判断、提示注入/工具输出不可信、无出站白名单、无人值守下 `confirm_*` 语义未定义、沙盒强度等，基本不在前两份文档内。
   - 协作：当前是**严格单向派单**（父→子，只回 `final_text`），没有子→父反向提问、共享黑板、结构化结果契约、评审协作、声明式编排——这正是用户问的「agent to agent 对话协作」，目前几乎为空白。

4. **一句话**：先补「安全底座」和「协作契约」，再把初稿2 的主动性/自进化闭环接线，收益最大且互不冲突。

## 2. 安全性（重点补全）

### 2.1 现状（已较扎实）

- 双认证（Aegis RSA 验签 / JWT）+ 三级角色 + 任务级 RBAC：[`backend/app/core/deps.py`](../backend/app/core/deps.py)、[`core/aegis.py`](../backend/app/core/aegis.py)。
- 单一工具权限闸门（`disallowed → pre_tool hooks → permission_mode → plan/subagent gate`）：[`tool_runner.execute_tool`](../backend/app/services/llm/tool_runner.py#L1241)。
- Python 沙盒：RLIMIT（CPU/内存/fsize/nproc/nofile）+ 环境变量去密钥 + 网络阻断：[`python_sandbox.py`](../backend/app/services/sandbox/python_sandbox.py)。
- SQL 三级分类（allow/warn/block）+ 审计落库：[`sql_audit_svc.py`](../backend/app/services/admin/sql_audit_svc.py)。
- 登录/注册限流、操作审计、调度转写脱敏（`_redact_sensitive`）。
- Dev bypass 受 loopback + 显式 header 双重限制。

### 2.2 短板与改进项

| 优先级 | 短板 | 现状/根因 | 改进方向 | 主要落点 |
|---|---|---|---|---|
| **P0** | 限流在多 worker 下失效 | [`rate_limit_svc.py`](../backend/app/services/auth/rate_limit_svc.py) 为进程内计数；生产默认 `PROD_WORKERS=4`（Makefile）。各 worker 各算各的，暴力破解防护与锁定约 4× 削弱 | 计数迁移到共享存储。沿文件优先架构可用 `portalocker` 守护的 sqlite/jsonl，或 Redis | `rate_limit_svc.py` |
| **P0** | `memory_save` 敏感信息扫描未接入 | config 已有 `ICE_MEMORY_SENSITIVE_PATTERNS` + `_scan_sensitive`，但 [`save_memory`](../backend/app/services/task/context_svc.py) 未调用 | 写入前扫描 `title/hook/body`，命中只返回类别不返回原文；自动提取命中疑似 token 直接丢弃（同初稿2 P0-2） | `context_svc.py`、`config.py` |
| **P0/P1** | SQL 防护只看首词关键字 | [`classify()`](../backend/app/services/admin/sql_audit_svc.py) 去注释后只判断第一个 token。`SELECT 1; DELETE …`、`WITH x AS (…) DELETE` 会被判为 allow/warn | ① 先确认 kyuubi CLI 是否执行多语句（若拒则风险降）；② 用真正解析（如 `sqlglot`）拒绝多语句、拒绝非 SELECT/CTE 节点、强制 `LIMIT` | `sql_audit_svc.py`、[`tools/kyuubi.py`](../backend/app/services/llm/tools/kyuubi.py) |
| **P1** | 提示注入 / 工具输出不可信（新兴重点） | SQL 结果、文件内容、飞书文档、未来网页内容都会回灌 LLM；而 `write_file`/`feishu_publish`/`spawn_subagent`/`data_platform_call` 都是有副作用工具，注入指令可触发真实动作 | ① prompt 中用明确「不可信数据」分隔标注工具输出；② 对消费外部内容的 Agent 保留 `confirm_network`/`confirm_write`；③ 配合下方出站白名单 | prompt builder、`tool_runner.py` |
| **P1** | 无出站域名白名单（SSRF/外泄） | [`_tool_data_platform_call`](../backend/app/services/llm/tools/workflow.py) 打向 `DATA_PLATFORM_MCP_URL`（env + 硬编码默认），无 host 白名单；沙盒 `allow_cli=True` 完全放开网络 | 在 HTTP client 层强制出站 host 白名单；限定哪些 Agent 可用 `allow_cli` | `workflow.py`、`python_sandbox.py` |
| **P1** | 无人值守下 `confirm_*` 语义未定义（安全×正确性） | 调度/后台/子 Agent 路径调用 `run_agent_turn` 时不会填充 `ctx.approved_side_effects`。若 Agent `permission_mode=confirm_write`，无人值守运行会对每个写工具返回 `PERMISSION_REQUIRED`，无人审批 → 任务静默卡死 | 明确策略：无人值守要么把 `confirm_*` 降级为 `read_only`，要么从任务配置携带预授权 scope | `scheduler_svc.py`、`bg_task_svc.py`、`tool_runner._check_agent_permission` |
| **P2** | 沙盒强度上限 | [`python_sandbox.py`](../backend/app/services/sandbox/python_sandbox.py) 自述：`RLIMIT_AS`（内存）仅 Linux 生效（macOS 跳过）；nproc「非真正 fork-bomb 防护，需 cgroups/pid namespace」；去密钥是子串匹配（可能漏掉自定义名密钥或误删） | 对外自托管/多租户：容器级隔离（gVisor/firecracker）或 seccomp；环境变量改「白名单」而非「黑名单」 | `python_sandbox.py`、`sandbox/preexec.py` |
| **P2** | 收尾项 | ① WS `?token=` 兜底会经日志/代理泄露（已知 D46）；② `AEGIS_DEV_BYPASS_EMAIL` 留在 .env 是配置脚枪；③ 审计/对话/工具转写为明文 jsonl，无防篡改、可能含 PII | ① 完成 subprotocol-only 升级并去掉 `?token=`；② 启动时若非 DEBUG 模式而 dev bypass 开启则拒绝启动/强告警；③ 敏感部署上 hash 链式只追加审计 + 转写 PII 脱敏（复用 `_redact_sensitive`） | [`deps.py`](../backend/app/core/deps.py)、审计/转写写入处 |

## 3. Agent-to-Agent 对话协作（重点补全）

### 3.1 现状：严格单向派单

- [`_tool_spawn_subagent`](../backend/app/services/llm/tools/workflow.py) 只回 `{final_text, run_id, tool_count, duration_ms}`；`spawn_parallel` 是 fan-out（最多 5）。
- 子 Agent **不共享对话历史**（设计如此）、**不能再 spawn**（depth=1）、**不能进 plan mode**。
- [`spawn_routing.md`](../agents/_shared/_partials/spawn_routing.md) 明确「不要派澄清需求」「不要无脑透传」。
- 反馈仅单向沉淀：[`subagent_feedback.py`](../backend/app/services/task/subagent_feedback.py) 把子 Agent 结论写进**父** Agent 记忆。
- 代码核对：后端**不存在**任何 子→父反向提问 / blackboard / scratchpad / A2A 消息总线原语（这些只在被 vendored 的 `cc-recovered-main/` 里，非本项目）。

### 3.2 短板与改进项

| 优先级 | 短板 | 改进方向 | 主要落点 |
|---|---|---|---|
| **P0/P1** | 子 Agent 只回自由文本，父 Agent 要二次 LLM 解析 | 既然运行时已产出 `error_type/recoverable/suggested_action`，把子运行返回升级为**结构化结果契约**：`{summary, files_written, data_refs, confidence, open_questions, error?}`。同时补 `files_written`（含超时也尽量从 transcript 提取，避免部分产物丢失）——衔接初稿2 P0-3 | `workflow.py`、`agent_runtime.py` |
| **P1** | 子 Agent 无法向父 Agent 提问（真正的「对话」缺失） | 给子 Agent 一个**有界的子→父提问**通道，复刻现有 HITL 模式（[`hitl_svc.py`](../backend/app/services/notification/hitl_svc.py) 暂停 → resolve → 注入消息），但指向父 Agent（如 `request_parent_input`）。限制 depth + 1~2 轮，避免循环与成本失控 | 新增工具 + `agent_kernel`/`agent_runtime` 暂停-恢复 |
| **P1** | 并行子 Agent 互相看不到对方产物（重复查数） | 任务作用域的**共享黑板**（只追加 `{agent, key, value, ts}`），并行 Agent 可读写协调，无需共享完整历史。文件虽共享但无结构化协调层 | 新增 blackboard service + `paths.py` |
| **P1** | 交接全靠父 Agent 在 prompt 里手抄 OKR/口径/表名 | 自动组装**交接包**（任务 OKR + 绑定 skill + 相关 memory + 既有结论）注入子 Agent，减少 prompt 工程负担与重复查数 | `experience_card_svc.merged_system_prompt`、`context_svc.py` |
| **P1** | 无评审/批判协作（质量杠杆） | 新增**评审型派单**：Agent B 批判 Agent A 的交付（SQL 口径 / 结论自查），返回结构化意见，A 再修订。对数据分析尤其有效，且是真正的 A2A 对话。可借力现有 `ab-experiment` agent 与语义评审能力 dogfood | `workflow.py` + 新协作模式 |
| **P2** | 无多级 Agent 树 + 编排靠 general prompt 临场路由 | ① 允许 depth=2（更严 token/轮数预算 + 循环检测）；② 把选定的 `agents/*/workflows/` 从 prompt hint 升级为运行时可执行的 **DAG 编排**（如日报 = 查→析→写→发跨 Agent 流水线），比每次靠 prompt 路由更可靠可观测 | `agent_runtime.py`、新增 orchestration |

## 4. Agent 主动性（校验 + 补充）

初稿2 §1 已较完整覆盖机会探测、风险分级审批、后台结果回流（P0-4）、监控状态回写（P1-3）。以下为**校验后补充**项：

| 优先级 | 项 | 说明 | 落点 |
|---|---|---|---|
| **P0/P1** | data-monitor 阈值判断交给了 LLM —— 应改为确定性代码 | [`_build_monitor_prompt`](../backend/app/services/task/monitor_svc.py) 把 `pct_change/absolute/consecutive_decline` 写成自然语言由 LLM 判断是否越界，告警可靠性差。应：跑 SQL → Python 解析行 → 确定性比较阈值 → LLM 只负责生成告警文案。**这是主动性最大的可靠性缺口，前两份文档未强调** | `monitor_svc.py`、`scheduler_svc.py` |
| **P1/P2** | 缺事件驱动触发 | 当前主动性都是时间驱动（cron）或监控（cron 背书）。无「新文件上传→自动摘要」「上游表更新→刷新报告」「监控告警→自动派归因子 Agent」等数据/事件触发。内部事件总线 → Agent 触发，可让主动性变「响应式」，并与协作打通 | 新增 event bus |
| **P1** | 调度器只在重启时故障转移 | [`scheduler_svc.py`](../backend/app/services/storage/scheduler_svc.py) leader 锁持有整个进程生命周期；leader **挂起（非崩溃）**时定时任务静默停摆，无接管 | 加心跳/租约（续期时间戳，过期则他人接管） |
| **P2** | HITL 无超时/升级 | `request_human_input` 会让任务无限期 paused，无人回应则永久卡住 | 加 TTL + 升级/自动取消/通知 |

## 5. Agent 自我进化（校验 + 补充）

初稿2 §2 已覆盖生命周期接线（P0-1）、memory policy（P0-2）、经验卡片自动 draft（P1-2）、去重（P1-4）、AgentEvalSuite（P1-5）、反馈驱动 confidence（P2-2）等。以下为**校验结论 + 新增视角**：

### 5.1 校验结论：记忆生命周期引擎「已建好但未接线」——当前仍成立

- 引擎齐备且有测试：[`memory_lifecycle.py`](../backend/app/services/task/memory_lifecycle.py)（`bump_usage`/`enrich_entries`）、[`memory_manager.py`](../backend/app/services/task/memory_manager.py)（`run_sweep`：decay/archive/auto-promote）、[`memory_ranking.py`](../backend/app/services/task/memory_ranking.py)（BM25 + recency + confidence）。
- 但断点仍在：[`save_memory`](../backend/app/services/task/context_svc.py) 仍走 legacy `_memory_file`，不写 lifecycle metadata；检索路径不 `enrich_entries`、命中不 `bump_entries` → `usage_count/last_hit_at` 不增长 → `auto_promote`（需 ≥3 命中）永不触发 → confidence/recency 因子实际失效。
- 结论：**初稿2 P0-1 准确且至今未做，应作为自进化第一优先**。

### 5.2 新增视角

| 优先级 | 项 | 说明 | 落点 |
|---|---|---|---|
| **P1** | 捕获用户纠正信号（新增视角） | 用户纠正（「不是 X 是 Y」）是最高信号的学习数据，目前只躺在对话历史里。检测纠正（启发式或低成本 LLM）→ 生成经验卡片/记忆 draft，与初稿2 的「结论式自动提取」互补 | `auto_memory_svc.py`、`experience_card_svc.py` |
| **P2** | 跨 Agent / 组织级学习 | [`subagent_feedback.py`](../backend/app/services/task/subagent_feedback.py) 只写**父** Agent；产出洞察的子 Agent 不学习，也无组织级提升。memory 已有 private/team/org scope + `team_memory` + `auto_promote_candidates`，可串通跨用户/Agent 的（带审核）晋升 | `memory_manager.py`、`team_memory.py` |
| **P2** | Prompt 版本 A/B（dogfooding） | 已有 prompt history + rollback 和 `ab-experiment` agent，但无对自身 Agent 两版 prompt 的线上 A/B。作为实验平台，A/B 自家 Agent（由初稿2 P1-5 的 AgentEvalSuite 把关）可闭合初稿2 自己提的「线上指标观测 → 保留或回滚」 | `admin_svc.py` + eval/AB service |
| **P1** | 经验卡片无效果追踪 / 废弃 | [`experience_card_svc.py`](../backend/app/services/admin/experience_card_svc.py) approved 为终态、只追加，`cards.md` 无限膨胀、过期规则持续注入。需命中追踪 + `deprecated`/`superseded_by`（初稿2 已提，此处校验确实存在） | `experience_card_svc.py` |

## 7. 建议先改的 5 个具体点

1. **限流迁共享存储**：`rate_limit_svc` 计数与锁定状态用 `portalocker`+sqlite/jsonl 或 Redis 跨 worker 共享。
2. **`save_memory` 调 `_scan_sensitive` + 改用 `render_memory_file`**：一次同时补「敏感扫描」（安全）与「生命周期 metadata」（自进化）两件事。
3. **子 Agent 返回结构化契约**：`{summary, files_written, data_refs, confidence, open_questions, error?}`，并在超时分支也尽量回填 `files_written`。
4. **监控阈值确定性化**：在 `monitor_svc` / 调度执行链路里用代码判定阈值，LLM 仅负责告警文案与汇总。
5. **SQL 改真解析**：用 `sqlglot` 替换首词关键字判断，拒绝多语句/非 SELECT 节点并强制 `LIMIT`。

---

## 8. 不建议近期优先做

1. 不要先上「完整 A2A 消息总线/多智能体框架」——先做「结构化结果契约 + 子→父有界提问 + 共享黑板」三件低成本原语即可覆盖绝大多数协作场景。
2. 不要先上 embedding 全家桶——记忆量不大时，先让生命周期生效、body 参与检索、去重合并收益更直接（同初稿2）。
3. 不要让 Agent 自动执行高风险主动动作——建监控/飞书发送/公开发布/删除回滚必须审批（同初稿2）。
4. 不要先做重事务回滚——先把有副作用工具的 `side_effects` 清单与 append/overwrite 语义做清楚（同讨论稿 §4）。
5. 不要把安全只写进 prompt 约束——限流、SQL 解析、出站白名单、沙盒隔离必须在 service/基础设施层兜底。

## 9. 一句话路线

> 前两份文档已经把「主动性 + 自进化」的闭环讲清楚；这一版补上两块地基——**安全底座**（限流/SQL 解析/出站白名单/沙盒/无人值守权限语义）与**协作契约**（结构化结果 + 子→父对话 + 共享黑板 + 评审）。地基稳了，再把记忆生命周期接线、主动机会探测跑通，ICE Workbench 的 Agent 才能既「会持续变好」又「不出事」。
