# ICE Workbench Agent 主动性与自我进化优化档案

> 版本：v2.0  
> 日期：2026-06-29  
> 范围：基于当前目录下 ICE Workbench v3 代码、`files/优化方向初稿.md`、`files/ICE Workbench Agent 改进方向讨论稿.md` 的补充分析。  
> 聚焦：Agent 主动性、Agent 自我进化、两者之间的闭环建设。

---

## 0. 结论摘要

当前 ICE Workbench 已经不是“被动问答型 Agent”：它具备自动选 Agent、计划审批、Todo、子 Agent、并行派单、后台任务、定时任务、数据监控、自动记忆、经验卡片、Prompt 历史等基础设施。

但从“主动性”和“自我进化”角度看，现状更像是**能力点已经散落存在，但闭环还没完全打通**：

1. **主动性有执行能力，缺机会识别层**  
   Agent 能创建定时任务、跑后台任务、派子 Agent，但多数仍要用户明确要求；系统缺少一层在每轮结束后自动判断“下一步最有价值动作”的机制。

2. **自我进化有存储和注入，缺学习闭环**  
   已有 memory、auto_memory、sub-agent feedback、experience cards、prompt history，但 usage/confidence/lifecycle/审核/评估没有完全串起来。

3. **若只做一个方向，优先做“闭环”而不是新增能力**  
   近期最值得补的是：Memory 生命周期真实生效、子 Agent 产物回灌、主动机会探测、后台结果回流、Prompt/经验卡片回归评估。

## 1. 当前主动性能力现状

### 1.1 已具备的主动执行能力

| 能力 | 当前实现 | 说明 |
|---|---|---|
| 自动选 Agent | `backend/app/services/task/task_intent_svc.py`、`backend/app/services/task/task_svc.py` | 创建任务时可根据广告/财收类关键词自动路由到 `data-analysis`。 |
| 运行时提示 | `backend/app/api/v1/ws_stream.py` | 每轮对话可注入 intent hint，减少业务标识误判。 |
| 计划审批 | `exit_plan_mode`、Plan Mode banner | 复杂/高风险任务可先调研、出计划、等用户批准。 |
| Todo 进度 | `todo_write` | 多步骤任务可持续展示当前进展。 |
| 子 Agent 派单 | `spawn_subagent` | 父 Agent 可把独立任务交给专业 Agent。 |
| 并行子 Agent | `spawn_parallel` | 已支持最多 5 个子任务并行执行。 |
| 后台任务 | `run_background`、`bg_task_svc.py` | 长任务可不阻塞当前对话，完成后发通知。 |
| 定时任务 | `scheduler_svc.py` | cron 调度，可无人值守执行 Agent。 |
| 数据监控 | `monitor_svc.py`、`data-monitor` Agent | 可配置 SQL + 阈值 + 飞书/通知告警。 |
| HITL 人工介入 | `request_human_input` | Agent 可在关键节点主动请求用户输入或审批。 |
| 工具错误恢复 | `error_classifier.py`、`run_tool_with_retry` | 已有 `error_type/recoverable/suggested_action` 与条件重试能力。 |

### 1.2 当前主动性短板

#### 短板 A：缺少“主动机会探测器”

现在 Agent 可以执行很多主动动作，但缺少统一机制判断：

- 这轮分析结束后是否应该建议建立监控？
- 是否应该把本次结论沉淀为经验卡片？
- 是否应该开启后台任务继续跑长分析？
- 是否应该追加一轮归因或下钻？
- 是否应该提醒用户某个定时任务连续失败？

目前这些判断主要靠各 Agent SOP 或 LLM 临场发挥，稳定性不足。

**建议新增：`ProactiveOpportunityDetector`**

每轮对话结束后，基于以下输入生成候选动作：

```text
final_text
tool_uses_log
files_written
Task State
Memory hit list
scheduled/monitor/bg job history
last_error / structured_error
```

输出一组结构化建议：

```json
{
  "suggestions": [
    {
      "type": "create_monitor",
      "confidence": 0.82,
      "title": "为 CC 消费 UV 建立日报监控",
      "reason": "本轮发现核心指标环比下跌 12%，用户持续关注该指标",
      "requires_approval": true,
      "action_payload": {...}
    }
  ]
}
```

第一阶段只展示建议，不自动执行有副作用动作。

推荐接入点：

- `backend/app/api/v1/ws_stream.py`：在 agent turn completed 后触发。
- `backend/app/services/storage/scheduler_svc.py`：定时任务结束后也可触发。
- `backend/app/services/task/bg_task_svc.py`：后台任务结束后触发。

#### 短板 B：主动动作没有统一审批策略

不同主动动作风险不同：

| 动作 | 风险 | 建议策略 |
|---|---|---|
| 展示后续分析建议 | 低 | 可直接展示 |
| 写 Task State | 低 | 可自动执行 |
| 保存个人 memory | 中 | 用户明确偏好可直接写；自动提取建议走草稿 |
| 创建经验卡片 draft | 中 | 可自动建草稿，管理员审核后注入 |
| 创建定时任务/监控 | 中高 | 必须用户确认 |
| 飞书发送/发布 | 高 | 必须显式确认或已有任务配置授权 |
| 删除/回滚/放量决策 | 高 | 只给建议，不自动执行 |

建议将该策略做成共享模块和共享 prompt partial，而不是散落在每个 Agent SOP。

#### 短板 C：后台任务结果没有回流到父上下文

`run_background` 完成后会写 `bg_jobs.jsonl` 并发通知，但父 Agent 后续对话并不会自然知道后台任务的最终结果。

建议：

1. 后台任务完成后追加写入 `Task State` 的“后台结果”段落。
2. 将 `job_id/final_text_excerpt/files_written/error` 写入结构化记录。
3. 下一轮 prompt 注入时，父 Agent 可直接使用后台结果继续推进。

推荐落点：

- `backend/app/services/task/bg_task_svc.py`
- `backend/app/services/task/context_svc.py`

#### 短板 D：数据监控状态闭环不完整

`monitor_svc.py` 里已有 `last_check_at`、`last_alert_at`、`alert_count` 字段，但当前调度执行完成后没有看到反向更新监控规则状态的闭环。

建议：

- scheduler run 完成后，根据工具调用结果更新 monitor 元数据。
- 告警发送成功后更新 `last_alert_at/alert_count`。
- SQL 失败、空结果、连续失败要进入监控健康状态。
- 前端展示“监控最近一次检测结果 / 连续失败次数 / 最近一次告警”。

推荐落点：

- `backend/app/services/task/monitor_svc.py`
- `backend/app/services/storage/scheduler_svc.py`

## 2. 当前自我进化能力现状

### 2.1 已具备的自我进化基础设施

| 能力 | 当前实现 | 说明 |
|---|---|---|
| User / Agent / Task 三层上下文 | `context_svc.py`、`context-protocol.md` | 每轮自动注入用户记忆、Agent 记忆、Task State。 |
| Memory 检索 | `memory_ranking.py` | 已有 BM25 + recency + confidence 的排序模块。 |
| Memory 生命周期 | `memory_lifecycle.py`、`memory_manager.py` | 已有 metadata、usage bump、stale/archive、auto-promote 代码。 |
| Team Memory | `team_memory.py` | 已支持团队级记忆层注入。 |
| 自动记忆提取 | `auto_memory_svc.py` | 对话完成后基于正则提取结论并写 Agent Memory。 |
| 子 Agent 反馈提取 | `subagent_feedback.py` | 子 Agent 成功返回后提取可复用 insight。 |
| 经验卡片 | `experience_card_svc.py` | 用户/管理员审核后注入 Agent prompt。 |
| Prompt 历史 | `admin_svc.py` | Prompt 修改有 history 和 rollback。 |
| 测试沙盒 | `/admin/agents/{aid}/test-chat` | 管理员可测试 Agent prompt。 |

### 2.2 当前自我进化短板

#### 短板 A：Memory 生命周期“代码存在，但链路未完全打通”

`memory_lifecycle.py` 已提供：

- `render_memory_file`
- `enrich_entry/enrich_entries`
- `bump_usage/bump_entries`
- `created_at/updated_at/confidence/usage_count/last_hit_at/visibility`

但当前写入和读取路径仍有断点：

1. `MemoryWriter.save_memory` 仍使用 legacy `_memory_file`，没有默认写入 lifecycle metadata。
2. `ContextLoader._load_memory_dir` 解析 `MEMORY.md` 后直接 rank，没有先 `enrich_entries`。
3. 检索命中后没有调用 `bump_entries`，`usage_count/last_hit_at` 不会自然增长。
4. `auto_promote_candidates` 依赖 `usage_count >= 3`，如果 usage 不增长，自动晋升很难真实触发。

推荐优先修复：

```text
MemoryWriter.save_memory
  -> memory_lifecycle.render_memory_file

ContextLoader._load_memory_dir
  -> parse index
  -> enrich_entries
  -> rank_entries
  -> bump_entries(best-effort)
  -> load selected bodies
```

推荐落点：

- `backend/app/services/task/context_svc.py`
- `backend/app/services/task/memory_lifecycle.py`
- `backend/app/services/task/memory_manager.py`

#### 短板 B：Memory 写入 policy 仍偏软

配置里已有敏感信息正则：

- `backend/app/core/config.py` 中的 `ICE_MEMORY_SENSITIVE_PATTERNS`
- `context_svc.py` 中的 `_scan_sensitive`

但 `MemoryWriter.save_memory` 当前没有调用 `_scan_sensitive`。

建议：

1. `memory_save` 写入前扫描 `title/hook/body`。
2. 命中敏感模式时返回 `VALIDATION_ERROR`，只返回类别，不返回原文。
3. 对自动提取 memory 使用更严格策略：命中疑似 token/password/key 直接丢弃。
4. 加入 body 长度限制、SOP 重复检测、同 slug 冲突检测。

#### 短板 C：自动记忆提取质量有限

`auto_memory_svc.py` 和 `subagent_feedback.py` 主要依赖正则识别“结论/总结/关键发现”等标题，优点是便宜稳定，缺点是：

- 无标题但有价值的经验容易漏掉。
- 提取出的 body 可能只是报告片段，不一定是可复用规则。
- 缺少冲突检测和更新合并。
- 自动提取内容直接写 Agent Memory，缺少“草稿/审核”中间态。

建议分两阶段：

**阶段 1：规则增强，不引入额外 LLM 成本**

- 从 `tool_uses_log` 提取关键事实：表名、SQL 失败类型、文件产物、最终结论。
- 自动提取只保存“稳定偏好/稳定项目背景/可复用方法”，不保存单次指标数值。
- 新增相似 hook 检测，避免同类记忆堆积。

**阶段 2：LLM 辅助提取**

- 只在长任务、工具数 >= N、用户正反馈、任务完成后触发。
- 输出结构化候选：`type/title/hook/body/confidence/conflict_candidates`。
- 高置信且低风险的写 memory；其他进入 draft queue。

#### 短板 D：经验卡片缺自动提案和效果评估

经验卡片当前流程是：创建 draft → 管理员审核 → rebuild `cards.md` → runtime 注入。

短板：

- draft 主要靠人手动创建。
- approved 后缺效果追踪，不知道是否改善 Agent 表现。
- approved 卡片不可回滚到 rejected，虽然保证稳定，但缺“废弃/降权/替换”机制。

建议：

1. `auto_memory` 或 `ProactiveOpportunityDetector` 生成 experience card draft。
2. 管理后台展示来源任务、原消息、命中次数、相似失败案例。
3. approved card 注入后记录使用命中情况。
4. 支持 `deprecated` 或 `superseded_by`，而不是直接删除历史规则。
5. 卡片上线前跑 golden tasks 回归集。

推荐落点：

- `backend/app/services/admin/experience_card_svc.py`
- `backend/app/api/v1/admin_review.py`
- `frontend/src/pages/admin/*Experience*`

#### 短板 E：Prompt 进化没有回归测试闭环

已有能力：

- Prompt 保存历史
- Prompt rollback
- `/admin/agents/{aid}/test-chat`

但当前 test-chat 是单轮 `complete_once`，不走工具、不走真实任务上下文、不计成功率。因此它更像“Prompt 文案预览”，不是 Agent 能力评估。

建议新增：`AgentEvalSuite`

每个 Agent 建一组 golden tasks：

```json
{
  "agent_id": "data-analysis",
  "cases": [
    {
      "id": "cc_dau_drop_basic",
      "input": "分析 CC DAU 近 14 天下跌原因",
      "expected_tools": ["kyuubi_query", "execute_python"],
      "must_include": ["时间窗", "口径", "下钻", "结论"],
      "must_not_include": ["无法查询但猜测", "虚构表名"]
    }
  ]
}
```

评估指标：

- 工具调用成功率
- 是否调用必要工具
- 是否违反工具/安全约束
- 是否完成核心产物
- token 成本
- 总耗时
- 用户反馈分

Prompt/经验卡片上线前先跑评估，效果下降则提示不要保存或建议回滚。

## 3. 主动性 × 自我进化的关键闭环

建议把 ICE Workbench 的 Agent 演进目标定义为四个闭环。

### 3.1 任务执行闭环

```text
用户目标
  -> Agent 计划 / Todo
  -> 工具执行 / 子 Agent / 后台任务
  -> 产物与结论
  -> Task State 更新
  -> 下一轮继续使用
```

当前已有大部分能力，短板是后台任务和子 Agent 产物没有充分回流。

### 3.2 经验沉淀闭环

```text
任务完成
  -> 自动识别可复用经验
  -> Memory / Experience Card draft
  -> 审核或用户确认
  -> 注入未来 Agent
  -> 命中统计与效果反馈
```

当前已有 auto_memory 和 experience cards，短板是自动提案、审核前质量过滤、命中统计、效果评价。

### 3.3 主动建议闭环

```text
分析结果 / 异常 / 用户偏好
  -> 机会探测
  -> 主动建议
  -> 用户批准
  -> 创建监控 / 定时任务 / 后台任务 / 经验卡片
  -> 后续结果回流
```

这是当前最明显的产品化增量。

### 3.4 Prompt 进化闭环

```text
失败案例 / 用户反馈 / 管理员观察
  -> 生成修改建议
  -> 沙盒测试
  -> golden tasks 回归
  -> 保存 prompt / card
  -> 线上指标观测
  -> 自动建议保留或回滚
```

当前只有 history + rollback + test-chat，缺评估和线上指标关联。

## 5. 建议先改的 5 个具体点

### 5.1 MemoryWriter 改用 lifecycle renderer

当前：

```text
MemoryWriter.save_memory -> _memory_file
```

建议：

```text
MemoryWriter.save_memory -> render_memory_file(
  slug,
  hook,
  type_,
  body,
  confidence=1.0,
  usage_count=0,
  visibility=args.visibility or private
)
```

同时可让自动提取 memory 默认 `confidence=0.7`。

### 5.2 ContextLoader 检索后 bump usage

建议读取流程：

```text
entries = _parse_memory_index(index_text)
entries = enrich_entries(entries, base)
entries = rank_entries(entries, query)
bump_entries(entries, base)  # best-effort
load selected memory bodies
```

这样 auto-promote 才有真实依据。

### 5.3 spawn_subagent 透传子运行结构化信息

当前最终返回：

```json
{
  "final_text": "...",
  "run_id": "...",
  "tool_count": 3,
  "duration_ms": 12000
}
```

建议补充：

```json
{
  "files_written": [...],
  "stop_reason": "...",
  "error_type": "transient",
  "recoverable": true,
  "suggested_action": "retry_once"
}
```

失败/超时时也尽量从 transcript 或已记录 state 提取 `files_written`。

### 5.4 新增主动建议事件

建议新增 WS 事件：

```json
{
  "type": "proactive_suggestions",
  "items": [
    {
      "id": "...",
      "type": "create_scheduled_task",
      "title": "每天 10 点自动生成 CC 指标日报",
      "reason": "用户连续多次查询相同指标",
      "requires_approval": true
    }
  ]
}
```

前端先以卡片形式展示，用户点击确认后再调用对应 API。

### 5.5 Prompt 变更前跑回归集

在 admin prompt 保存流程中增加可选步骤：

```text
编辑 prompt
  -> 运行 EvalSuite
  -> 展示 pass/fail/cost/diff
  -> 管理员确认保存
  -> 保存 history
```

短期可先只做离线 CLI/API，不必一开始完整前端化。

## 6. 不建议近期优先做的事项

1. **不要把“思维链展示”作为主线**  
   更建议做 execution trace / tool trace / decision summary，而不是要求模型输出隐藏 CoT。

2. **不要先上复杂事务回滚**  
   对 Agent 工具调用来说，先做 side_effects 清单、文件版本/append/overwrite 明确化，性价比更高。

3. **不要一开始就上 embedding 全家桶**  
   Memory 数量还不大时，先让 body 参与检索、metadata 生效、去重合并，收益更直接。

4. **不要让 Agent 自动执行高风险主动动作**  
   主动建议可以自动出现，但创建监控、飞书发送、公开发布、删除/回滚必须审批。

5. **不要继续把能力只写进 SOP**  
   SOP 有用，但主动性和进化闭环需要 runtime/service 层承接，否则不稳定、不可观测、不可评估。

## 8. 一句话路线

短期不要再堆新的 Agent 能力点，而是优先把现有能力串成闭环：

```text
能发现机会 -> 能提出建议 -> 能获批执行 -> 能记录结果 -> 能沉淀经验 -> 能评估收益 -> 能影响下一次表现
```

这条闭环跑通后，ICE Workbench 的 Agent 才会从“会调用工具的助手”升级为“会持续变好的工作流伙伴”。
