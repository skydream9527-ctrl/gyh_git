# ICE Workbench v6: 前端组件库与状态管理架构设计

在确认了 v6 的视觉和交互形态（三栏布局、HITL 卡片、Dashboard 大盘）之后，前端工程化落地是下一步的关键。本篇文档探讨如何在现有的 React 18 + Vite + Zustand 技术栈下，实现这些复杂界面的组件拆解与状态管理。

## 一、 组件库架构设计 (Component Architecture)

为了支撑 v6 的高信息密度和多变卡片，我们需要重构 `frontend/src/components` 目录，将其划分为三个层次：

### 1. 基础 UI 组件 (Base UI / Atoms)
这些是无状态的、纯展示的组件，基于 Tailwind CSS 构建，构成整个系统的视觉基础。
- `Button`: 支持 `variant="primary|secondary|danger|ghost"`，支持 `isLoading` 状态。
- `Card`: 支持 `elevation="sm|md|lg"` 和 `border="default|warning|danger"`。
- `Badge/Tag`: 用于状态展示（如 Running, Paused, Failed），支持不同的配色主题。
- `Icon`: 统一封装 Phosphor Icons。
- `Switch/Toggle`: 用于 Scheduled 页面控制定时任务的启停。

### 2. 复合领域组件 (Domain Components / Molecules)
结合具体业务逻辑的组件。
- `HITLRichCard`: **核心组件**。接收符合 `RichCardData` Schema 的 JSON props，负责渲染表单、表格和按钮，并将用户输入向上抛出。
- `TaskStreamItem`: 用于 Dashboard 底部展示近期任务的单行记录组件。
- `ExecutionTreeNode`: 右侧执行计划树的单个节点，支持动态显示进度动画或卡点状态。

### 3. 布局与页面容器 (Layout & Page Containers / Organisms)
- `ThreeColumnLayout`: v6 Task Workspace 的核心布局容器，负责管理左右边栏的收起/展开状态（响应式设计必须考虑小屏幕下侧边栏的折叠）。
- `MissionControlLayout`: 带有顶部导航和左侧菜单的 Dashboard 布局。

## 二、 状态管理重构 (Zustand Stores)

v6 中，前端需要管理大量异步状态（特别是后台长耗时任务的状态更新）。我们需要在 `frontend/src/stores/` 中建立清晰的 Store 划分。

### 1. `taskStore` (任务执行流状态)
负责管理单个任务的运行生命周期。

```typescript
interface TaskStore {
  // 当前正在查看的任务 ID
  activeTaskId: string | null;
  
  // 任务状态树 (右栏数据)
  executionPlan: ExecutionNode[];
  
  // 对话流数据 (中栏数据)，包含普通消息和 HITL 卡片
  messages: Array<ChatMessage | HITLMessage>;
  
  // 更新执行树状态 (由 WebSocket 触发)
  updateExecutionNode: (nodeId: string, status: NodeStatus) => void;
  
  // 提交 HITL 表单
  submitHITLResponse: (messageId: string, payload: HITLResponsePayload) => Promise<void>;
}
```

### 2. `dashboardStore` (全局大盘状态)
负责拉取和缓存 Dashboard 所需的概览数据，减少页面切换时的 loading 闪烁。

```typescript
interface DashboardStore {
  // 需关注的高危/挂起任务
  attentionTasks: TaskSummary[];
  
  // 正在运行的后台任务
  runningTasks: TaskSummary[];
  
  // 今日即将执行的定时任务
  upcomingCronJobs: CronJobSummary[];
  
  fetchDashboardData: () => Promise<void>;
}
```

## 三、 WebSocket 协议层改造 (重点)

v6 核心要求前端解绑单线程等待。即使关闭浏览器，Agent 依然在后端跑，当重新打开时，需要能**回捞**最新状态。

1. **连接建立**：
   - 过去：进入 Task 页面才建立 WebSocket。
   - v6：**在 App 顶层建立一个全局 WebSocket**，用于接收所有正在运行任务的广播更新（支撑 Dashboard 的实时进度和新出现的 Attention Card 报警红点）。

2. **断线重连与状态同步 (Sync)**：
   - 当用户重新进入一个正在运行的 Task 时，前端需先发送一次 `sync_request`，要求后端推送完整的最新 `messages` 和最新的 `executionPlan`。

## 四、 样式系统 (CSS/Tailwind) 规范建议

为了保持 v6 的“冰橙色 (Ice Orange)”主题一致性，建议在 `tailwind.config.ts` 中固定以下核心色：

```javascript
theme: {
  extend: {
    colors: {
      brand: { // 替换原来的 primary 颜色
        50: '#fff7ed',  // 极浅背景
        100: '#ffedd5', // 浅色高亮/边框
        500: '#f97316', // 主按钮/主品牌色 (Copper Orange)
        600: '#ea580c', // Hover 态
        900: '#7c2d12', // 深色文字
      },
      // 专门为状态定义语义色
      status: {
        info: '#3b82f6',     // 蓝
        success: '#10b981',  // 绿
        warning: '#f59e0b',  // 黄/琥珀
        error: '#ef4444',    // 红
      }
    }
  }
}
```

这套架构能确保我们在把 `prototype_v6_spa.html` 转换为真实的 React 代码时，组件层级清晰，数据流向单向且可预测。
