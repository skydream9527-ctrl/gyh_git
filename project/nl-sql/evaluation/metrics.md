# 评估指标 — Text-to-SQL 的度量体系

> 定义清晰的评估指标，量化 nl-sql 的进展和差距。

## 核心指标

### 1. 执行准确率 (Execution Accuracy, EX)

**定义**：生成的 SQL 执行结果与标准答案执行结果完全一致的比例。

- 学术界标准指标（Spider、BIRD benchmark 都用）
- 优点：最终用户关心的就是结果对不对
- 局限：不同 SQL 可能产生相同结果（假阳性）

### 2. 逻辑准确率 (Exact Match, EM)

**定义**：生成的 SQL 在逻辑结构上与标准答案完全匹配（标准化后比较）。

- 比 EX 更严格
- 对我们用处有限：业务场景中同一问题可以有多种正确写法

### 3. 覆盖率

**定义**：nl-sql 能处理的查询类型占日常取数需求的比例。

- 我们的核心业务指标
- 计算方式：统计一周/月内所有取数需求，标记 nl-sql 能处理的比例
- 目标：80%

### 4. 首次成功率

**定义**：用户提问后无需修正即可直接使用结果的比例。

- 反映实际用户体验
- 与执行准确率的区别：有些 SQL 技术上正确但不符合用户意图

### 5. 平均延迟

**定义**：从用户提交问题到返回结果的平均时间。

- 目标：< 30s（含 SQL 执行时间）

## Badcase 分类体系

| 类型 | 说明 | 占比目标 |
|---|---|---|
| Schema 错误 | 选错表或列 | 通过 schema linking 改善 |
| 条件错误 | WHERE 条件写错（日期格式、枚举值） | 通过口径知识改善 |
| 聚合错误 | GROUP BY / 聚合函数错误 | 通过范式模板改善 |
| JOIN 错误 | 表关联条件错误 | 通过预置 JOIN 关系改善 |
| 语义错误 | SQL 语法正确但语义不符合用户意图 | 通过多轮澄清改善 |
| 执行错误 | SQL 语法错误无法执行 | 通过 self-correction 改善 |

## 评估流程

1. **收集测试集**：从真实取数需求中抽取，覆盖各种查询类型
2. **标注标准答案**：人工编写标准 SQL 和预期结果
3. **自动化评测**：batch 跑测试集，计算各项指标
4. **Badcase 分析**：对失败样例分类归因，指导优化方向
