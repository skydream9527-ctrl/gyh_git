# Schema Linking — 表/列匹配

> Schema Linking 是 Text-to-SQL 中准确率影响最大的环节：将用户自然语言中的实体映射到数据库 schema 的表名和列名。

## 为什么重要

大部分 Text-to-SQL 错误的根因是 schema linking 失败：模型选错了表或列。当 schema 规模大时（数十张表、数百个列），这个问题更突出。

## 核心方法

### 1. 基于字符串匹配

最简单的方式：将问题中的关键词与表/列名做模糊匹配（编辑距离、n-gram）。

- 优点：快、可解释
- 局限：无法处理同义词（"日活" → `dau`）

### 2. 基于语义相似度

用 embedding 模型将问题和 schema 元素编码到同一向量空间，按余弦相似度匹配。

- 优点：能处理同义词和缩写
- 局限：需要高质量 embedding，schema 描述质量影响大

### 3. 基于 LLM 的 schema 筛选

让 LLM 先根据问题从全量 schema 中筛选相关表/列，再基于筛选后的 schema 生成 SQL。

- 代表：CHESS 的 schema 筛选模块
- 优点：利用 LLM 的语义理解能力
- 局限：增加一轮 LLM 调用延迟

### 4. 业务知识增强

在 schema 信息中附加业务描述：`dau -- 日活跃用户数（app_open 事件去重）`。

- 对我们最关键：内容生态的表/列命名不总是自解释的
- 指标口径字典就是在做这件事

## 我们的实践方向

1. 为核心表的每个列补充中文业务描述（列注释）
2. 将指标口径字典结构化，与 schema 描述关联
3. 按业务域预分组（流量、内容、用户、实验），问题先分类再匹配对应 schema 子集
