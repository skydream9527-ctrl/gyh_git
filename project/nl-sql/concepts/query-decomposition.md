# 复杂查询拆解策略

> 将复杂的自然语言问题分解为多个简单的 SQL 子查询，再组合得到最终结果。

## 为什么需要拆解

LLM 直接生成复杂 SQL（多表 JOIN + 嵌套 + 窗口函数）的错误率显著高于简单查询。拆解后每步更简单，准确率更高。

## 拆解方法

### 1. 问题拆解（Question Decomposition）

将复杂问题拆成多个子问题，每个子问题生成一个简单 SQL。

```
原始问题："过去7天各渠道的新用户次日留存率是多少？"
拆解：
  Q1: "过去7天各渠道的新用户有哪些？" → SQL1
  Q2: "这些新用户中次日还活跃的有哪些？" → SQL2
  Q3: "按渠道计算留存率" → 组合 SQL1 和 SQL2
```

### 2. SQL 骨架拆解

先生成 SQL 结构骨架（SELECT-FROM-WHERE-GROUP BY），再逐步填充具体的表/列/条件。

### 3. Chain-of-Thought 拆解

在 prompt 中引导模型先推理再生成：

```
让我一步步思考：
1. 需要哪些表？→ ...
2. JOIN 条件是什么？→ ...
3. 筛选条件是什么？→ ...
4. 聚合方式是什么？→ ...
最终 SQL：...
```

## 我们的应用

对于高频复杂查询（如留存分析、漏斗分析），直接在范式库中预置拆解模板比让模型现场拆解更可靠。
