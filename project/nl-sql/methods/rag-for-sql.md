# RAG 增强取数

> 通过检索增强生成（RAG）为 Text-to-SQL 提供业务上下文，提升准确率。

## 为什么需要 RAG

LLM 不知道我们的业务语义（"有效 DAU" 的口径、"小说频道" 对应的 channel_id）。RAG 将这些知识在推理时检索注入。

## 检索什么

| 检索对象 | 说明 | 索引方式 |
|---|---|---|
| 相似 SQL 样例 | 历史成功的 (问题, SQL) 对 | 问题 embedding 相似度 |
| 指标口径 | 指标定义、计算公式、数据源 | 关键词 + embedding 混合 |
| 表/列描述 | 带业务注释的 schema | 表名 + 列名匹配 |
| 业务规则 | 如"7日留存不含注册当日" | 规则关键词匹配 |

## 检索策略

### 两阶段检索

1. **粗筛**：用 BM25/关键词匹配缩小候选范围
2. **精排**：用 embedding 模型重排序，取 top-k

### 上下文窗口管理

检索到的内容需要与 schema 和 few-shot 一起放入 prompt，要控制总量：
- Schema: ~2000 tokens
- 检索知识: ~1000 tokens
- Few-shot: ~2000 tokens
- 用户问题 + 系统指令: ~500 tokens

## 与范式库的关系

范式库（`patterns/`）是我们最核心的 RAG 语料：
- 每条范式 = (问题模板, SQL 模板, 参数说明, 适用场景)
- 检索时优先匹配范式库，命中率高则准确率极高
- 未命中时 fallback 到通用生成

## 落地步骤

1. 将 `data-product-workspace/00-知识库/数据资产/SQL模板库/` 中的模板结构化
2. 构建 embedding 索引（问题维度）
3. 集成到 nl-sql Skill 的 pipeline 中
