# AI基础设施 — 开篇导读

> 这是 [ai-infra/](.) 目录的总入口。读完这一篇，你应该能回答：大模型从训练到推理到落地到底需要什么基础设施、PM/产品工程师需要懂到什么程度、成本/延迟/效果三者怎么权衡、怎么支撑AI产品稳定落地。

---

## 一、为什么AI产品经常"看起来很美，用起来崩"（一段扎心的论述）

AI产品Demo和生产环境之间隔着一条巨大的鸿沟：

> "Demo上响应如飞，上线了慢得要死，等10秒才出结果"
> "测试的时候效果很好，上线了成本爆炸，一天烧几万块"
> "偶尔崩一次，不知道为什么，也查不到问题出在哪"
> "用的开源模型，换个版本效果就变了，也不知道为什么"

这些问题本质上都不是模型算法问题，而是**AI基础设施没做好**——就像你造了一辆发动机很强的跑车，但是传动系统、刹车、散热、油箱都没做好，根本开不上路。

很多人以为AI产品的核心是模型，但现实是：**90%的AI产品问题出在工程落地，90%的成本在推理，90%的用户流失来自慢和不稳定。**

AI基础设施要解决的就是：**怎么把大模型从实验室里的Demo，变成稳定、快速、低成本、可监控、可迭代的生产级服务，支撑AI产品规模化落地。**

> 如果你只是做个Demo玩，可以不用关心这些；但如果你要做真正能给用户用的AI产品，这一层是绕不过去的基础。详见 [../agent-llm/](../agent-llm/) 里的产品工程化部分。

## 二、AI基础设施三层架构

从训练到落地的完整链路：

```
┌───────────────────────────────────────────────────────────┐
│  3. MLOps/平台层    怎么管模型、管服务、管成本、管效果      │ ← 网关、监控、A/B、版本、成本
│  2. 推理服务层      怎么跑得快、跑得稳、跑得便宜            │ ← 量化、vLLM、KV Cache、批处理
│  1. 训练/模型层     怎么得到一个能用的模型                  │ ← 分布式训练、微调、对齐
└───────────────────────────────────────────────────────────┘
            离产品越近↑                           产品人员需要懂的程度↑
```

**重点**：
- 第1层训练层是算法工程师的主战场，产品人员不需要懂细节，知道大概原理和成本即可
- 第2层推理层和产品体验/成本直接相关，延迟高一点、成本贵几倍，产品就死了，必须懂核心概念和trade-off
- 第3层MLOps层是规模化落地的关键，没有这一层模型服务就是裸奔，出问题查不到，迭代没效率

**记一条铁律**：产品可以不懂怎么训模型，但必须懂推理层的成本/延迟/效果权衡——这直接决定你的产品能不能活下来。

## 三、核心权衡三角（所有决策都围绕这个）

AI基础设施所有决策本质上都是这三者的权衡，没有银弹：

```
          效果好（Effectiveness）
             △
            / \
           /   \
          /     \
         /   你  \
        /  在这儿 \
       /           \
      /_____________\
延迟低（Latency）   成本低（Cost）
```

| 优化方向 | 常用手段 | 代价 |
|---|---|---|
| **降延迟** | 量化（4/8bit）、小模型、投机解码、KV Cache优化、Flash Attention | 可能损失一点效果 |
| **降成本** | 量化、批处理、模型蒸馏、请求路由、缓存、小模型处理简单任务 | 增加系统复杂度，可能影响长尾效果 |
| **提效果** | 大模型、长上下文、RAG、多模型融合 | 成本和延迟都会上升 |

**产品经理必须会算账**：
- 一次请求多少token？
- 用什么模型？单价多少？
- 一天多少请求？一个月多少钱？
- p95延迟是多少？用户能接受吗？
- 并发量多少？需要多少张卡？

不算清楚这笔账，产品做出来也活不下去。

## 四、关键技术概念（产品/工程师都要懂）

不用懂怎么实现，但必须懂这些概念是干嘛的、对产品有什么影响：

| 概念 | 一句话理解 | 对产品的影响 |
|---|---|---|
| **量化（Quantization）** | 把模型参数从16位浮点数压成8位/4位甚至更低 | 推理速度提升2-4倍，成本降一半，效果掉得不多（4bit通常能接受） |
| **KV Cache** | 缓存之前算过的Key/Value，不用每次都重新算 | 长上下文速度提升很多，但占显存，是显存占用大头 |
| **PagedAttention (vLLM)** | 像操作系统管理内存一样管理KV Cache | 吞吐量提升2-4倍，现在开源推理服务标配 |
| **Flash Attention** | 优化Attention计算的IO，减少显存读写 | 速度提升2-4倍，显存占用更少，长文本必备 |
| **投机解码（Speculative Decoding）** | 用小模型先猜几个token，大模型验证 | 速度提升2-3倍，效果完全不变，需要配一个小模型 |
| **模型网关（Gateway）** | 统一入口，路由请求到不同模型，做缓存、限流、计费 | 多个模型统一接入，根据请求复杂度自动选模型，能省一半以上成本 |
| **RAG（检索增强生成）** | 把知识库存在外面，回答问题前先检索相关内容 | 解决大模型幻觉、知识过时问题，不用重新训练模型，是企业级AI应用标配 |

详见各对应文档。

## 五、生产环境最常见的五个坑

1. **成本爆炸**
   - 现象：上线前估的成本和上线后差10倍，一天烧几千几万块
   - 原因：没做缓存、没用量化、没做路由简单问题也用大模型、没设并发限制
   - 解法：prompt缓存、模型路由（小模型做简单任务）、量化、限流、监控token用量

2. **延迟太高**
   - 现象：p95延迟5秒+，用户等不及直接走了
   - 原因：没做推理优化、长上下文没优化、批处理设置不合理、冷启动
   - 解法：量化、vLLM/TensorRT-LLM、KV Cache优化、长上下文切片、模型预热

3. **效果不稳定**
   - 现象：有时候回答好有时候回答差，换个prompt结果差很多，升级模型版本效果就变了
   - 原因：大模型本身有随机性、prompt没做好约束、没有版本管理
   - 解法：temperature设低、prompt结构化、加校验、模型版本灰度发布、做回归测试集

4. **没有监控，出问题不知道**
   - 现象：用户来投诉了才知道服务挂了/变慢了/效果变差了
   - 原因：只监控服务存活性，没监控延迟、token成本、输出质量、用户反馈
   - 解法：四大监控：系统指标（CPU/内存/显存/延迟/错误率）、业务指标（token量/成本/请求量）、质量指标（回答合格率/幻觉率/用户满意度）、用户反馈（点赞/点踩/投诉）

5. **RAG效果不好**
   - 现象：接了知识库还是答不对，胡说八道，检索不到相关内容
   - 原因：分块策略不对、嵌入模型没选对、检索TopK不对、没做重排序、文档质量差
   - 解法：优化分块（按语义分，不要按固定长度）、选好的嵌入模型、加Rerank、先清洗文档

## 六、MLOps落地最小闭环

不需要一开始就搞复杂的平台，先把这六件事做好就能支撑中小规模AI产品：

```
1. 模型管理
   ├─ 模型版本管理（哪个模型、哪个版本、什么时候上线的）
   ├─ Prompt版本管理（Prompt改了要记录，能回滚）
   └─ 灰度发布（新模型/Prompt先放5%流量，看效果再全量）

2. 网关层
   ├─ 统一API接入，不用业务方直接调模型API
   ├─ 限流、降级、熔断（防止服务被打挂）
   ├─ 请求路由（简单任务走小模型，复杂任务走大模型）
   └─ Prompt缓存（相同问题直接返回缓存结果）

3. 监控告警
   ├─ 系统监控：延迟、错误率、QPS、GPU利用率
   ├─ 成本监控：token用量、费用统计、异常告警
   ├─ 质量监控：抽样检查回答质量、用户反馈统计
   └─ 告警：出问题第一时间通知，不要等用户投诉

4. 评测体系
   ├─ 黄金测试集：30-50个典型问题，每次发版前跑一遍
   ├─ 自动评测：用大模型做LLM-as-Judge，判断回答好坏
   └─ 人工评测：重要场景定期人工抽检

5. 日志链路
   ├─ 每个请求的完整日志：输入、输出、Prompt、模型、耗时、token数
   └─ 能根据用户反馈trace到具体请求，方便排查问题

6. 迭代闭环
   └─ 用户点踩/投诉的bad case → 定期分析 → 优化Prompt/RAG/模型 → 发版
```

## 九、心法

如果只能记三句话：

1. **Demo到生产差100倍**——Demo能跑和能稳定给用户用是完全两回事，永远预留足够的工程化时间。
2. **成本是设计出来的不是优化出来的**——产品设计阶段就算清楚成本账，选对模型、做好缓存、用好路由，比上线了再砍成本有效10倍。
3. **可观测性比优化更重要**——先把监控、日志、告警做好，知道问题在哪，再谈优化；不然出了问题都不知道怎么查。

下次做AI产品上线前，Checklist：
1. 单请求成本是多少？月活1万/10万/100万时成本是多少？算过吗？
2. p50/p95/p99延迟是多少？用户能接受吗？
3. 监控有没有做好？延迟、错误率、成本、质量都能看到吗？出问题会告警吗？
4. 有没有灰度发布机制？新版本出问题能快速回滚吗？
5. Bad case怎么收集？怎么迭代优化？闭环有没有跑通？

回答不出来，先别全量上线。
