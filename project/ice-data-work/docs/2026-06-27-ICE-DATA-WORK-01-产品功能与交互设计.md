# ICE-DATA-WORK 材料一 · 产品功能与交互设计

- 日期：2026-06-27
- 上级文档：`2026-06-27-ICE-DATA-WORK-设计方案-v1.md`（总纲）
- 本文范围：信息架构、角色权限（产品视角）、全局布局、核心用户旅程、各页面交互规格、多 Agent 协作区、关键交互细节、v1 页面清单与验收
- 决策依据：总纲 D-01~D-11

## 1. 产品设计原则

1. **看板是入口，Workspace 是现场，Twin 是常驻陪伴**：用户在看板看见"在运行/待推进"的工作，进入任务后在 Workspace 里和多个 Agent 协作，右侧 Twin 始终在场。
2. **Agent 是同事，不是工具栏**：工具 Agent、Twin 在看板上是一等 assignee，在对话里是一等发言者，有头像、有身份、有记忆。
3. **核心区只放产出物与对话**：Workspace 主区域只展示工作产出物（报告/文档/图表）与对话窗口；状态、文件、Agent、历史、时间线一律抽屉隐藏（继承 ICE/TokenWisdom）。
4. **空间边界对用户可见**：当前处在哪个 Team/Project、任务带了谁的记忆、产物会沉淀到哪一层，都要在界面上显性化。
5. **高风险动作显性确认**：写文件、跑命令、发消息、付费调用、固化记忆、跨空间读取——一律走确认卡，绝不静默执行。
6. **每次工作都能沉淀**：任务结束有"沉淀经验/产物"的明确出口，且能选择沉淀到个人 / 项目 / 团队 / Agent 团队经验。

## 3. 角色与权限（产品视角）

三条正交权限轴（D-10），界面上分别有不同入口与表现：

| 轴 | 取值 | 管什么 | 在哪配置 |
|---|---|---|---|
| 平台角色 | super_admin / admin / user | 平台级管理能力 | /admin/users |
| 成员角色 | Team(owner/admin/member) · Project(member) | 能进哪些团队/项目、能否管理成员与共享空间 | /team/:id · /project/:id |
| Twin 权限 | L0 Observe ~ L5 Sovereign | 自己的分身能自主到什么程度 | /twin |

**有效权限 = 平台角色 ∩ 成员资格 ∩ Twin 权限**。例如：一个 user 在 ProjectA 是 member、其 Twin 为 L3，则其 Twin 在 ProjectA 的任务里可"请求执行"，但写文件仍需用户确认；该用户看不到 ProjectB 的任何任务与共享空间。

界面体现：
- 进入任何任务/项目前，导航和列表只展示用户有成员资格的范围。
- Twin 的每个动作旁标注其权限等级徽标（如 `L3 Request`）。
- 高于当前等级的动作渲染为"请求确认"而非"执行"。

## 5. 核心用户旅程

### 5.1 冷启动（首次登录）

```text
飞书/账号登录 → 自动建 user 账号
  → 引导创建/确认 Twin（命名、工作风格、初始权限等级 L1）
  → 加入或创建 Team / Project（无则进入个人项目）
  → （可选）连接第一个工具 Agent，Twin 冷启动采访生成 Agent Card
  → 进入 Workbench
```

### 5.2 从看板进入任务并多方协作（主场景）

```text
Workbench 看见"6.20 DAU 下滑归因" → 点击进入 /task/:id（数据分析工作台）
  → Workspace 顶部"参与者"区显示：我 + Twin（已默认在场）
  → 点"+邀请" → 选择 Data Agent 加入
  → 此时空间上下文 = 任务文件 + Data Agent 记忆(本人分区) + Twin 记忆（见材料三）
  → 对话区出现三方：我 / Twin / Data Agent
  → Data Agent 发起 kyuubi_query（只读，低风险）→ 直接执行，结果回填报告面板
  → Twin 建议"补查 ANR"，但这是新查询 → 生成确认卡（Twin Dock + 内联）
  → 用户确认 → 执行 → 报告更新
  → 任务完成 → Twin 生成 2 条 Memory Candidate（待确认）
```

### 5.3 记忆沉淀与"贡献给团队"

```text
Twin Dock"待确认"出现 Memory Candidate 卡片
  → 用户点开，选择 scope：
       · 个人偏好（进我的 Twin 记忆）
       · 项目共享（进当前 Project）
       · 团队共享（进 Team）
       · 贡献给该 Agent 的团队经验（agent_team，需审核）
  → 确认 → file_transaction 原子晋升到目标空间
```

### 5.4 二期预览：邀请他人 Twin 协作

```text
"+邀请" → 选择"他人的数字分身" → 选中同事的 Twin
  → 向该 Twin 的管理员发起授权请求（scoped grant：本任务/只读宿主/有效期）
  → 管理员批准 → 该 Twin 作为新参与者进入对话
  → 空间内出现多个 Twin + 工具 Agent 同台对话（P2-02）
```

## 6. 页面级交互规格

> 每页统一模板：**目的 / 布局 / 关键元素 / 关键交互 / 权限可见性**。

### 6.1 Workbench 看板首页 `/workbench`

- **目的**：总控台。看见正在运行的内容、待我确认的事、最近任务、核心产物、今日总结。
- **布局**：顶部 4 个状态卡（运行中任务 / 待我确认 / 本周核心产物 / 当前 Twin 权限等级）→ 下方 2×N 卡片网格。
- **关键元素**：
  - 定时任务卡（Autopilot，二期 P2-04 接入触发）
  - 当前一次对话任务卡（进行中、参与 Agent、待确认项）
  - 最近任务卡（进 Workspace）
  - 核心产物卡（报告/文档/Agent Card，点击直达）
  - 今日总结（Twin 自动生成进展/风险/下一步）
- **关键交互**：每张卡片有主操作"进入/打开"；卡片显示所属 Project 标签。
- **权限**：只显示用户有成员资格的 Project 下的任务与产物。

### 6.2 任务看板 `/board`

- **目的**：全量任务的状态泳道视图（Multica 式 agents-as-teammates）。
- **布局**：按状态分列 —— `待办 / 进行中 / 待确认 / 已完成`；可按 Project 过滤。
- **关键元素**：任务卡显示标题、类型(数据分析/通用)、assignee 头像（人/Twin/工具Agent）、待确认徽标。
- **关键交互**：
  - 指派：点 assignee → 选择"人 / 我的 Twin / 工具 Agent"（Squad 二期）
  - 状态流转：v1 用按钮/下拉切换状态（拖拽美化留后，D-09）
  - 点击卡片进入 Workspace
- **权限**：assignee 可选范围受成员资格限制。

### 6.3 新建任务 `/new-mission`

- **目的**：把目标交给 Twin，由 Twin 规划路线、推荐 Agent、预估权限与成本。
- **布局**：左"任务输入" + 右"Twin 推荐路线"两栏。
- **关键交互**：
  1. 输入目标（可附文件、选历史产物、指定 Project）
  2. "让分身规划" → Twin 输出分步路线，每步标 Agent、权限、风险、Token 预估
  3. 用户勾选/调整 → "确认执行"创建任务并进入 Workspace
- **权限**：高风险步骤标注"需确认"，不在规划阶段执行。

### 6.4 任务 Workspace `/task/:taskId`（数据分析 / 通用）

- **目的**：协作现场。核心 = 产出物面板 + 对话面板。
- **布局**：左右双面板。
  - 数据分析：左=报告草稿（指标/图表/发现），右=对话（User+Twin+Data Agent）
  - 通用：左=文档正文，右=对话（User+Twin+Codex 等）
- **顶部条**：任务标题 + **参与者区**（头像 chips + "+邀请"）+ 抽屉按钮组（状态/Agent/文件/对话/时间线）+ 主操作（请求确认/保存为产物）。
- **关键交互**：
  - **参与者管理**：chips 显示当前参与者及其权限徽标；"+邀请"打开 picker（工具 Agent / 我的 Twin / 二期他人 Twin）。邀请即触发上下文重组（带入该参与者记忆）。
  - **抽屉**：点按钮右侧滑出，展示非核心信息，ESC/点遮罩关闭（继承 ICE 抽屉）。
  - **产出物面板**：可编辑、版本号、"保存为产物"（选择沉淀层级）。
- **权限**：参与者可邀请范围、产物沉淀层级受成员资格 + Twin 权限约束。

### 6.5 多 Agent 对话区（Workspace 右面板，产品签名）

- **消息结构**：每条消息含角色徽标 + 头像 + 名称。角色：`我(User)` / `Twin` / `工具Agent名`（二期 `他人Twin`）。不同主体不同头像色。
- **工具调用卡**：Agent 发起工具调用时，内联渲染工具卡（工具名/参数摘要/状态/结果摘要），成功绿、失败红、待确认黄。
- **定向发言**：输入框支持 `@参与者` 把消息定向给某个 Agent/Twin。
- **Twin 主动发言**：Twin 可主动插入建议/确认请求（与普通消息区分样式）。
- **HITL 内联确认**：高风险动作在对话流里以确认卡出现，同时镜像到 Twin Dock"待确认"。
- **流式**：消息逐字流式（继承 ICE WebSocket）；多轮 Tool Calling 可见过程。

### 6.6 Agent Hub `/agents` 与 Agent 详情 `/agent/:agentId`

- **Hub**：能力卡网格。卡片含：图标、名称、类别、连接状态、权限级别、风险、最佳用法摘要。包含"我的 Twin"卡。
- **详情**：标签页 —— 能力/输入输出、**记忆（本人情景记忆 + 团队共享经验，分区展示）**、使用经验、权限边界、历史表现。admin 可编辑 Prompt/版本/沙盒（继承 ICE 4-Tab）。
- **关键交互**：连接 Agent、Twin 冷启动采访生成 Agent Card、查看/编辑记忆（贡献给团队的显式开关）。

### 6.7 项目空间 `/project/:projectId` 与团队空间 `/team/:teamId`

- **目的**：承载团队/项目级共享文件、知识、产物（D-07）。
- **布局**：成员区 + 共享文件 + 共享知识 + 共享产物 + 项目看板（项目页）。
- **关键交互**：上传/置顶共享文件（按项目或团队隔离）、管理成员（owner/admin）、从这里发起新任务（自动归属本项目）。
- **权限**：仅成员可见；owner/admin 可管理成员与共享内容。

### 6.8 知识与产物 `/knowledge`

- **目的**：长期资产库 —— 文档、报告、总结、Memory Candidate、Agent MD、模板。
- **关键交互**：筛选（按层级：个人/项目/团队）、打开、复用、确认记忆。

### 6.9 数字分身管理 `/twin`

- **目的**：管理"我的 Twin"——不做具体任务，只管分身如何理解我、如何调用工具、能花多少。
- **板块**：分身画像（目标/风格/边界/成熟度）、记忆文件（偏好/决策/项目上下文/pinned，可确认/编辑/冻结）、Agent 使用经验（我对各 Agent 的私有经验）、Token 与成本、权限策略（L0-L5 + 高风险例外）。
- **关键交互**：编辑画像、管理记忆、调权限等级、查看消耗、暂停我的 Twin。

### 6.10 审批与审计 `/approvals`

- **目的**：所有高风险动作进确认队列；所有执行可追溯。
- **板块**：待我确认（含 Memory Candidate）、权限日志、工具调用日志、成本与风险审计。
- **关键交互**：确认/修改/拒绝；**一键暂停全部执行**；失败复盘（error_type/recoverable/suggested_action）。

### 6.11 管理后台 `/admin/*`（admin+）

- 继承 ICE：用户管理、Agent/Skill 管理、审核中心、用量与成本、SQL 审计、系统设置、诊断。
- 新增：**团队/项目管理**（创建团队、分配 owner、项目归属）。

## 7. Twin Dock 常驻交互规格（右栏）

```text
┌─ Wisdom Twin ──────────── [L3 Request] ┐
│ 当前页面：<动态> · 当前项目：<动态>       │
├──────────────────────────────────────┤
│ 对话区（与"我的 Twin"私聊，独立于任务对话）│
│   me / twin 气泡                         │
│   [输入框] [Send]                        │
├──────────────────────────────────────┤
│ 数字分身建议                              │
│   建议卡（继续计划 / 稍后 / 生成任务）     │
├──────────────────────────────────────┤
│ 待用户确认                                │
│   确认卡（确认 / 修改 / 拒绝）            │
│   记忆候选卡（选层级 + 贡献给团队开关）    │
└──────────────────────────────────────┘
```

- **常驻**：页面切换不消失，顶部反映"当前页面 + 当前项目"上下文。
- **两类对话区分**：Twin Dock 是"我和我的 Twin 私聊"；Workspace 对话是"任务现场多方协作"。Twin 同时在两处出现但身份一致。
- **建议/确认**：与任务级动作联动（任务里产生的确认会镜像到这里）。
- **导航感知与控制（D-12）**：Dock 顶部实时显示"当前页面"，左侧 tab 一切换 Twin 即感知（页面上下文进入其状态）；反过来，在对话框对 Twin 说"带我去看板 / 打开数据分析工作台 / 回 Workbench"，Twin 解析意图并直接切换左侧 tab。Dock 提供常用目的地的快捷指令 chips，Twin 也会按当前页给出相关下一步建议。

## 8. 关键交互细节

| 交互 | 规格 |
|---|---|
| 邀请参与者 | chips + picker；邀请即重组上下文并在对话区广播"X 已加入，已带入其记忆" |
| HITL 确认卡 | 标题=动作；正文=影响/范围/成本；按钮=确认/修改范围/拒绝；超时不自动执行 |
| 记忆候选卡 | 展示候选内容 + 来源；scope 单选（个人/项目/团队/Agent团队经验）；"贡献给团队"为显式开关（D-06） |
| 跨空间读取 | 任何越过当前任务/用户边界的读取，需确认卡，并写审计 |
| 一键暂停 | 顶部/审批页常驻；点击后冻结所有进行中的工具调用与外部调用 |
| 空间标识 | 任务/产物/文件卡片上始终带所属 Project/Team 徽标，避免误存错层 |
| 权限徽标 | Twin 与动作旁显示 L0-L5 徽标；超限动作渲染为"请求"而非"执行" |
| Twin 导航感知与控制（D-12） | Dock 顶部实时显示当前页面（左侧 tab 切换即感知）；对 Twin 说"带我去 X"即切换左侧 tab；由 Twin 调 `navigate` 实现，支持快捷指令 chips |

## 9. v1 页面清单与验收

| 页面 | 必含 | 验收要点 |
|---|---|---|
| Workbench | 状态卡 + 运行任务 + 待确认 + 最近 + 核心产物 + 今日总结 + Twin Dock | 默认页；只显示有权限的 Project 内容 |
| Board | 状态泳道 + assignee(人/Twin/Agent) + 过滤 | 可指派给 Agent；可切状态；进 Workspace |
| New Mission | 输入 + Twin 规划 + Agent/权限/成本预估 + 确认 | 高风险步骤标注需确认，不在规划期执行 |
| 数据分析 Workspace | 报告面板 + 多方对话 + 参与者区 + 抽屉 + Twin Dock | 三方对话；工具卡内联；邀请重组上下文 |
| 通用 Workspace | 文档面板 + 多方对话 + 参与者区 + 抽屉 + Twin Dock | 同上；产物可保存并选层级 |
| Agent Hub/详情 | 能力卡 + 记忆分区(本人/团队) + 权限 + 历史 | 记忆按 D-06 分区展示；贡献给团队显式 |
| 项目/团队空间 | 成员 + 共享文件/知识/产物（按层隔离） | 仅成员可见；owner/admin 可管 |
| 知识与产物 | 文档/报告/总结/记忆候选/模板 + 层级筛选 | 可复用、可确认记忆 |
| Twin Manager | 画像/记忆/Agent经验/Token/权限 | 可调 L0-L5；可暂停我的 Twin |
| Approvals & Audit | 待确认 + 记忆候选 + 权限日志 + 工具日志 | 一键暂停；失败复盘结构化 |

> 移动端 v1 优先：Workbench、Workspace（只读+确认）、Approvals（审批日报），继承 ICE 移动端适配。

## 10. 用户贡献与 Skill / Agent 自演进（D-13，特别重点）

让平台"越用越强"的核心闭环：当 Agent 缺某业务能力时，用户在任务里把代码/内容给它，Agent 沙盒里跑通后，把这次经验蒸馏成可复用的 **Skill** 并绑进 **Agent**，下次自动会用。

### 10.1 一次贡献 → 三个沉淀目标

| 目标 | 是什么 | 例子 | 落点（沿用 D-06） |
|---|---|---|---|
| **Memory** | 陈述性经验/口径 | "查 XX 业务先按这个口径" | 个人偏好 / 项目 / 团队 |
| **Skill** | 可复用可执行能力（代码/SQL + 入参 schema + 版本） | `biz_xx_dau.sql` / `analyze_xx.py` | 个人草稿 Skill → 贡献给团队 |
| **Agent 更新** | 把 Skill 绑进 Agent + knowledge（何时用）+ 版本+1 | Data Agent 学会"XX 业务 DAU 查询" | 个人绑定 → 团队绑定 |

### 10.2 任务内贡献闭环（核心场景）

```text
用户："数据分析 Agent 没这个业务的代码，我把 SQL/代码给你，帮我跑一下"
  → 在 Workspace 对话里粘贴代码 / 拖入脚本文件
  → Agent 调 run_user_code 在沙盒执行（Python 沙盒：无网络/限资源/无凭证；SQL：kyuubi 只读 + SQL 审计）
  → 结果回填产出物面板（成功/失败 + 输出）
  ↓ 跑通后
  → Twin/Agent 主动提议："要把它沉淀为 Skill 并让 Data Agent 学会吗？"
  → 弹出「贡献为 Skill」表单（见 10.3）
  → 用户确认范围 → 晋升（写 Skill + 更新 Agent + 版本+1） → 审计
下次同业务：Agent 已学会，直接调用该 Skill，无需再贴代码
```

### 10.3 「贡献为 Skill」表单（Skill Candidate）

由 `run_user_code` 跑通后触发，字段：

- **名称 / 描述**：如"XX 业务 DAU 查询"
- **代码**：用户贡献的 SQL/Python（只读展示 + 可编辑）
- **入参 schema**：从代码抽取的参数（如 `date_range`、`channel`），可标注类型/默认值 → 决定它能否被复用
- **运行时**：sql(kyuubi) / python(沙盒)
- **适用业务 / 触发条件**：写进 Agent knowledge（"何时该用这个 Skill"）
- **沉淀范围（沿用 D-06）**：
  - ◉ 个人草稿 Skill（`by-user`，**立即可用**，仅我可见，低风险）
  - ○ 贡献给团队 Skill（`by-team`，需**沙盒测试通过 + 审核**）
- **绑定到 Agent**：☑ 让 Data Agent 学会（个人绑定）/（贡献团队时）团队绑定
- 同时可勾选"生成一条 Memory（口径/经验）"

### 10.4 Skill 管理 UX（Agent 详情 · Skills 标签）

在 `agent/:agentId` 的 **Skills 标签**（与"记忆分区"并列）：

- **已绑定 Skills**：内置/团队共享（如 kyuubi、nl-python）+ **我贡献的草稿 Skill**（分区展示，呼应 D-06）
- 每个 Skill：版本号、运行时、入参 schema、来源任务、**沙盒测试**按钮、**贡献给团队**开关、**回滚**到历史版本
- **草稿 → 团队**：点"贡献给团队"→ 进审核中心，需沙盒 test-run 通过 + owner/admin 审核
- Agent 的 **knowledge** 同步展示："对于 XX 业务，使用 Skill YY"

### 10.5 版本、审核与安全

| 关注点 | 设计 |
|---|---|
| 版本化 | Skill 与 Agent 均版本化（复用 ICE 的 Prompt 版本/快照/回滚）；每次更新生成新版本，可回滚 |
| 审核 | 团队级晋升进**审核中心**：强制"沙盒 test-run 通过 + owner/admin 审核"双门 |
| 安全 | 用户上传代码**一律先进沙盒**（不可信）：Python 无网络/限 CPU 内存/无凭证；SQL 只读 + SQL 审计；禁止写本机/外发 |
| 隔离/复用 | 个人草稿 Skill 不泄漏给他人；显式"贡献给团队"才共享；与记忆模型 D-06 完全一致 |
| 审计 | 贡献、执行、晋升、回滚全程进审计；失败结构化复盘 |

### 10.6 交互细节补充（接 §8）

| 交互 | 规格 |
|---|---|
| 贡献代码 | Workspace 对话支持粘贴代码块 / 拖入脚本文件；Agent 以 `run_user_code` 工具卡内联展示执行（成功绿/失败红） |
| 贡献为 Skill 卡 | 跑通后由 Twin/Agent 主动弹出；含名称/描述/入参 schema/范围/绑定 Agent；"贡献给团队"为显式开关 |
| 沙盒测试 | Skill 详情可一键 test-run（样例入参），通过才允许团队晋升 |
| 回滚 | Skill/Agent 版本列表可回滚；回滚即新版本，不删历史 |

## 11. 展示模型与状态机（D-14，卡片"状态优先"）

原则：Workbench/看板/工作台上，**卡片的主角是"状态"**——一眼看到任务在执行中/已完成/报错、Agent 在忙还是空闲。详情（产物内容、能力全清单）一律收到点击之后（渐进展开）。

### 11.1 任务状态机（6 态）

| 状态 | 颜色 | 卡片副行（主角之外的一行） | 可用动作 |
|---|---|---|---|
| 待办 Todo | 灰 | — | 开始 / 指派 |
| 执行中 Running | 蓝 | **当前动作/步骤**："Data Agent 执行 kyuubi_query（2/3）" | 进入 / 暂停 |
| 待确认 Awaiting | 琥珀 | **待确认事项**："是否补查 ANR" | 确认 / 修改 / 拒绝 |
| 已完成 Done | 绿 | 完成时间 + 一句结论 | 进入 / 复用产物 |
| 报错 Error | 红 | **错误摘要 + 建议**（error_type / recoverable / suggested_action） | **重试 / 查看日志** |
| 已暂停 Paused | 灰 | 暂停原因（如"一键暂停"） | 恢复 |

- 状态来源：执行中/报错来自运行时与工具调用；待确认来自 HITL 队列；已暂停来自一键暂停。报错复用 ICE `error_classifier` 的结构化字段。
- **任务卡结构**：
  ```text
  [● 执行中]  浏览器 6.20 DAU 下滑归因
  现在：Data Agent 执行 kyuubi_query（2/3）        ← 状态副行
  负责人 我·分身·DA            产物 ▸ 3            ← 产物只给入口
  ```
- **看板映射**：列按生命周期分 待办 / 执行中 / 待确认 / 已完成（待确认单独成列，因为需要你介入）；**报错 / 已暂停以红/灰徽标在其所处列高亮**，不单独占列。

### 11.2 产物：渐进展开 + 3 态

产物在任何卡片上**只显示数量入口**（`产物 ▸ N`），不铺开内容：

```text
产物 ▸ 3  →  产物列表  →  单条内容
            每条：类型(报告/文档/图表/代码) · 标题 · 版本 · 所属层级(任务/项目/团队) · 更新时间 · 状态
```

产物状态（3 态）：

| 状态 | 颜色 | 含义 |
|---|---|---|
| 草稿 Draft | 琥珀 | 生成中/待完善（如报告 v0.7） |
| 已确认 Confirmed | 绿 | 用户确认采纳 |
| 已发布 Published | 蓝 | 已对外/沉淀到项目或团队 |

### 11.3 Agent 展示三件套（5 态）

每个 Agent 卡固定展示三件套，不堆全量能力清单：

| 展示项 | 取自 | 例 |
|---|---|---|
| **1. 当前状态** | 运行时 / inflight | 空闲 / 工作中 / 等待确认 / 报错 / 未连接（本机 Agent 二期可能未连接） |
| **2. 核心能力** | 能力 headline（一句） | "SQL 查询与归因分析" |
| **3. 正在执行的工作** | 当前 task + 当前 tool 调用 | "为「DAU 下滑归因」执行 kyuubi_query"；空闲时显示"空闲 / 上次：…" |

- **Agent 卡结构**：
  ```text
  [DA] Data Agent                         [● 工作中]
  核心能力：SQL 查询与归因分析
  正在做：为「DAU 下滑归因」执行 kyuubi_query
  ```
- Agent 状态色：工作中=蓝 · 等待确认=琥珀 · 报错=红 · 空闲=灰 · 未连接=灰描边。
- **展示位置**：Agent Hub 卡、工作台「参与者状态」抽屉（每个参与者一行状态 + 正在做）、Workbench"在工作的 Agent"小卡。

### 11.4 对既有页面规格的收敛（覆盖 §6 相关处）

- **Workbench**：当前对话任务卡 = 状态徽标 + 状态副行 + `产物 ▸ N`；新增"在工作的 Agent"小卡（三件套）。
- **看板**：卡片状态徽标为主 + 副行；报错/已暂停以徽标高亮，报错卡带 重试/查看日志。
- **工作台头部**：显著展示任务状态徽标 + "现在：…"；工具条加 `产物 (N)`（开产物列表抽屉）与 `参与者状态`。
- **产出物面板**：报告等产物带 3 态徽标（草稿/已确认/已发布）。
