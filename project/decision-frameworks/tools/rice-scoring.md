# RICE优先级评分框架

> RICE是一种简单但有效的优先级评分方法，帮助团队客观比较不同功能或项目的优先级。

## 什么是RICE？

RICE是四个因素的首字母缩写：

- **R**each（覆盖范围）：影响多少用户？
- **I**mpact（影响程度）：对用户的影响有多大？
- **C**onfidence（信心指数）：你对估算有多大把握？
- **E**ffort（投入成本）：需要多少工作量？

## 计算公式

```
RICE Score = (Reach × Impact × Confidence) / Effort
```

## 详细说明

### Reach（覆盖范围）

**定义**：在特定时间段内，有多少用户会受到影响？

**测量方法**：
- 使用历史数据估算
- 考虑目标用户群体
- 设定明确的时间范围（如：每季度）

**示例**：
| 功能 | 预计影响用户数 | 时间范围 |
|------|----------------|----------|
| 新用户引导优化 | 10,000 | 每季度 |
| 高级搜索功能 | 5,000 | 每季度 |
| 移动端适配 | 50,000 | 每季度 |

### Impact（影响程度）

**定义**：对单个用户的影响有多大？

**评分标准（3分制）**：
- **3** = 巨大影响（解决核心痛点）
- **2** = 高影响（显著改善体验）
- **1** = 中等影响（有帮助但不关键）
- **0.5** = 低影响（锦上添花）
- **0.25** = 微小影响（几乎无感）

**示例**：
| 功能 | Impact评分 | 理由 |
|------|------------|------|
| 支付流程优化 | 3 | 直接影响转化率 |
| 页面加载速度 | 2 | 显著改善体验 |
| 主题颜色调整 | 0.5 | 美观但不影响核心功能 |

### Confidence（信心指数）

**定义**：你对上述估算有多大把握？

**评分标准（百分比）**：
- **100%** = 高信心（有数据支持、已验证）
- **80%** = 中等信心（有部分数据、类似经验）
- **50%** = 低信心（主要是假设、缺乏数据）
- **20%** = 很低信心（完全猜测）

**何时降低信心**：
- 缺乏历史数据
- 涉及新技术或未知领域
- 依赖外部因素
- 团队没有相关经验

### Effort（投入成本）

**定义**：需要多少工作量？

**测量单位**：人天（person-days）或人周（person-weeks）

**估算方法**：
1. 分解任务为小单元
2. 估算每个单元的工作量
3. 考虑设计、开发、测试、部署
4. 包含沟通和协调成本

**示例**：
| 功能 | 设计 | 开发 | 测试 | 总计 |
|------|------|------|------|------|
| 新用户引导 | 3天 | 5天 | 2天 | 10天 |
| 高级搜索 | 5天 | 15天 | 5天 | 25天 |
| 移动端适配 | 2天 | 10天 | 3天 | 15天 |

## 实战案例

### 案例：SaaS产品功能优先级排序

**背景**：某项目管理SaaS产品，需要决定下一个季度开发哪些功能。

**候选功能**：
1. 实时协作编辑
2. 移动端App
3. API开放平台
4. 高级报表功能
5. 自动化工作流

**RICE评分表**：

| 功能 | Reach | Impact | Confidence | Effort | RICE Score | 优先级 |
|------|-------|--------|------------|--------|------------|--------|
| 实时协作编辑 | 8,000 | 3 | 80% | 30天 | 640 | 2 |
| 移动端App | 20,000 | 2 | 70% | 60天 | 467 | 3 |
| API开放平台 | 2,000 | 2 | 60% | 45天 | 53 | 5 |
| 高级报表功能 | 5,000 | 1 | 90% | 15天 | 300 | 4 |
| 自动化工作流 | 15,000 | 3 | 80% | 25天 | 1,440 | 1 |

**计算示例（自动化工作流）**：
```
RICE = (15,000 × 3 × 0.8) / 25 = 36,000 / 25 = 1,440
```

**决策**：
- **P0（必做）**：自动化工作流
- **P1（高优）**：实时协作编辑
- **P2（中优）**：移动端App
- **P3（低优）**：高级报表功能
- **P4（暂缓）**：API开放平台

## 常见陷阱与避免方法

### 1. Reach虚高
**问题**：高估影响用户数
**解决**：使用实际数据，区分"可能使用"和"会使用"

### 2. Impact主观
**问题**：Impact评分因人而异
**解决**：
- 建立统一评分标准
- 多人独立评分后取平均
- 用数据验证（如A/B测试）

### 3. Confidence忽视
**问题**：所有项目都用高信心
**解决**：
- 强制要求为信心评分找依据
- 信心<80%时，必须说明原因
- 考虑做小规模验证提升信心

### 4. Effort低估
**问题**：常见的是低估工作量
**解决**：
- 包含沟通、测试、部署时间
- 考虑技术债务和依赖
- 使用历史数据校准
- 预留buffer（建议20-30%）

## 进阶用法

### 1. 加权RICE
根据战略目标调整权重：
```
Weighted RICE = (Reach × w1) + (Impact × w2) + (Confidence × w3) - (Effort × w4)
```

**示例**：如果公司目标是增长，可以提高Reach的权重。

### 2. 分层RICE
按用户分层计算：
- 新用户 vs 老用户
- 免费用户 vs 付费用户
- 不同行业/地区用户

### 3. 时间维度RICE
考虑时间价值：
```
Time-Adjusted RICE = RICE Score × Time Sensitivity Factor
```

**时间敏感度**：
- 高（竞品即将发布）：1.5
- 中（季度内需要）：1.0
- 低（可以等待）：0.7

### 评估清单

在评分前，确认以下问题：

**Reach评估**：
- [ ] 有明确的用户数据支持吗？
- [ ] 时间范围是否明确？
- [ ] 是否考虑了用户增长？

**Impact评估**：
- [ ] 是否有用户调研支持？
- [ ] 是否参考了类似功能的效果？
- [ ] 评分标准是否统一？

**Confidence评估**：
- [ ] 信心评分的依据是什么？
- [ ] 有哪些不确定因素？
- [ ] 是否需要先做验证？

**Effort评估**：
- [ ] 是否包含所有环节（设计、开发、测试、部署）？
- [ ] 是否考虑了技术债务？
- [ ] 是否预留了buffer？

## 与其他框架结合

### RICE + OKR
将RICE高分项目与OKR对齐：
```
公司OKR → 产品OKR → RICE优先级排序 → 季度规划
```

### RICE + 用户故事地图
在用户故事地图中使用RICE：
1. 绘制用户故事地图
2. 为每个故事计算RICE
3. 按RICE分数排序
4. 规划MVP和后续迭代

### RICE + 影响力矩阵
将RICE结果可视化：

```
High Impact
    │
    │  ★ 自动化工作流
    │  ★ 实时协作
    │
    │  ● 移动端App
    │
    │  ○ 高级报表
    │
    │  ○ API平台
    │
Low Impact ──────────────────── High Reach
```

## 总结

RICE是一个强大的优先级工具，但记住：

1. **RICE是起点，不是终点**：帮助结构化思考，但最终决策需要判断
2. **定期重新评估**：随着数据积累，更新评分
3. **团队共识**：让团队参与评分过程，增加认同感
4. **结合其他工具**：RICE + 决策矩阵 + 用户调研 = 更全面的决策
