# AI 决策案例

> 当 AI 成为决策参与者——而非单纯工具——决策过程、责任边界和质量标准都发生了根本变化。

---

## 案例 1：AI 辅助产品路线图决策

### 背景

某 SaaS 公司的产品团队需要决定 Q3 路线图。产品经理收集了 47 条用户反馈、3 份竞品分析报告和过去 6 个月的功能使用数据。传统做法是 PM 凭直觉 + RICE 评分手动排序，但 47 条需求两两对比的复杂度让手动分析难以兼顾全局。

团队决定引入 AI（LLM）辅助决策，但面临核心问题：**AI 在这个过程中应该扮演什么角色？参谋？分析师？还是决策者？**

### 决策过程

#### 第 1 步：用 AI 自主度光谱定位

```
完全人工 ←──────────────────────────────→ 完全自动
  │                                              │
  ├─ AI仅检索 ──┬─ AI建议 ──┬─ AI裁决 ──┬─ AI执行 │
  │  人工决策    │  人工审批  │  人工监督  │  人工监控│
  └─────────────┴───────────┴───────────┴─────────┘
       L0           L1           L2           L3
```

团队评估：

| 评估维度 | L0 | L1 | L2 | L3 |
|----------|:--:|:--:|:--:|:--:|
| 决策可逆性（路线图可调整） | ★ | ★ | ★ | ☆ |
| 数据充分度（47条反馈+使用数据） | ☆ | ★ | ★ | ★ |
| 影响范围（全产品方向） | ★ | ★ | ★ | ☆ |
| 事后验证能力（上线后可AB测试） | ★ | ★ | ★ | ★ |
| 伦理风险 | ☆ | ☆ | ☆ | ☆ |

**结论：选择 L1（AI 建议，人工审批）**——路线图影响面大，数据虽充分但需要商业判断，可逆但调整成本高。

#### 第 2 步：设计 AI 辅助决策流程

```
                    ┌──────────────┐
                    │   输入层     │
                    │ 47条用户反馈 │
                    │ 3份竞品分析  │
                    │ 6月使用数据  │
                    └──────┬───────┘
                           ↓
                    ┌──────────────┐
                    │   AI 处理层  │
                    │ 聚类分析     │ → 将47条需求归为8个主题
                    │ RICE预评分   │ → 基于数据给出初始分
                    │ 依赖关系图   │ → 识别功能间前后依赖
                    │ 风险标记     │ → 标记数据不足的需求
                    └──────┬───────┘
                           ↓
                    ┌──────────────┐
                    │   人工判断层 │
                    │ 战略对齐校验 │ → AI不知道公司战略
                    │ 客户承诺检查 │ → AI不知道已签合同
                    │ 团队能力评估 │ → AI不知道团队能做什么
                    │ 最终排序决定 │
                    └──────┬───────┘
                           ↓
                    ┌──────────────┐
                    │   反馈层     │
                    │ 人工标注偏差 │ → AI是否遗漏了关键需求
                    │ 记录决策依据 │ → 为什么接受/否决AI建议
                    │ 更新记忆     │ → 反馈到MEMORY.md
                    └──────────────┘
```

#### 第 3 步：执行与校验

AI 输出的 RICE 预评分与 PM 手动评分对比：

| 需求主题 | AI RICE 分 | PM RICE 分 | 偏差 | 原因 |
|----------|:----------:|:----------:|:----:|------|
| 批量操作 | 82 | 78 | +4 | 数据驱动，AI更准确 |
| 移动端适配 | 75 | 90 | -15 | PM知道大客户要求 |
| API 开放 | 68 | 65 | +3 | 基本一致 |
| 数据导出 | 61 | 45 | +16 | AI高估了用户覆盖面 |
| 暗色模式 | 55 | 30 | +25 | AI无法判断品牌优先级 |

**关键发现：AI 在纯数据维度（覆盖面、使用频率）更客观，但在战略维度（客户承诺、品牌优先级）系统性偏差。**

### 最终决策

| 阶段 | 时间 | 目标 | 关键任务 |
|------|------|------|----------|
| 第1周 | W1-W2 | AI辅助分析 | 聚类+预评分+依赖图 |
| 第2周 | W3-W4 | 人工校准 | 战略对齐+偏差修正 |
| 第3周 | W5-W6 | 决策确认 | 路线图评审+ADR记录 |

**AI 辅助决策质量指标：**

| 指标 | 目标值 |
|------|--------|
| AI与PM评分偏差 >20%的需求数 | <15% |
| 人工修正AI建议的比例 | 20-40% |
| 决策耗时（vs 纯人工） | 减少50% |
| 后续返工需求比例 | <10% |

## 案例 2：AI 自主度升级决策——何时从 L1 升到 L2

### 背景

某电商平台的内容审核系统，当前采用 L1 模式（AI 标记，人工审核）。日均审核量 50 万条，人工团队 30 人，审核延迟 4 小时。业务方要求将延迟降至 30 分钟内。

核心决策：**是否将 AI 自主度从 L1 升级到 L2（AI 裁决，人工监督）？**

### 决策过程

#### 第 1 步：用 AI 决策准备度评估

| 评估维度 | 当前状态 | L2 门槛 | 达标？ |
|----------|----------|---------|:------:|
| 准确率 | 92% | ≥95% | ❌ |
| 召回率 | 88% | ≥97% | ❌ |
| 漏判危害 | 可能放行违规内容 | 漏判率 <0.1% | ❌ |
| 误判危害 | 正常内容被删 | 误判率 <5% | ✅ |
| 可解释性 | 黑箱模型 | 可输出判罚依据 | ❌ |
| 申诉机制 | 有人工申诉通道 | 申诉自动分流 | ✅ |
| 回退能力 | 可一键切回L1 | 切回时间 <5分钟 | ✅ |

**3 项未达标，不满足 L2 升级条件。**

#### 第 2 步：设计分阶段升级路径

```
当前 L1 ──→ L1+ ──→ L2- ──→ L2
  │          │        │       │
  │  AI标记   │ AI自动  │ AI裁决 │ AI裁决
  │  人工审核 │ 通过低风险│ 高风险  │ 人工抽检
  │          │ 人工审高 │ 人工审  │
  │          │ 风险     │        │
  └──────────┴─────────┴────────┘
   准确率92%  准确率94%  准确率96%  准确率97%+
   延迟4h     延迟1h    延迟15min 延迟5min
```

#### 第 3 步：制定升级触发条件

| 升级 | 触发条件 | 验证方法 | 回退条件 |
|------|----------|----------|----------|
| L1→L1+ | 低风险类准确率≥95%，持续2周 | 每日抽检100条 | 低风险误判率>3% |
| L1+→L2- | 全类准确率≥96%，可解释性达标 | 独立第三方评测 | 任何漏判率>0.5% |
| L2-→L2 | 全类准确率≥97%，运行1个月无事故 | 全量审计+抽检 | 连续3天准确率<96% |

### 最终决策

暂不升级到 L2，先执行 L1→L1+ 过渡。

| 阶段 | 时间 | 目标 | 关键任务 |
|------|------|------|----------|
| 准备期 | M1-M2 | 模型优化 | 训练新模型，目标准确率94% |
| L1+试运行 | M3 | 低风险自动通过 | 日抽检+周评审 |
| L1+稳定期 | M4-M5 | 积累数据 | 收集L1+运行数据 |
| L2-评估 | M6 | 评估升级条件 | 第三方评测 |

**关键指标：**

| 指标 | L1 现状 | L1+ 目标 | L2- 目标 | L2 目标 |
|------|---------|----------|----------|---------|
| 审核延迟 | 4h | 1h | 15min | 5min |
| 人工工作量 | 50万/日 | 15万/日 | 5万/日 | 2万/日 |
| 准确率 | 92% | 94% | 96% | 97%+ |
| 漏判率 | 0.8% | 0.3% | 0.1% | <0.05% |

## 案例 3：AI 决策可解释性权衡

### 背景

某金融科技公司的信用审批系统使用 AI 模型自动审批贷款。模型 AUC 0.91，但监管要求审批决策必须可解释。团队需要在**模型性能**和**可解释性**之间做出权衡。

三种方案：

- **方案A**：黑箱模型 + 事后解释（SHAP/LIME）
- **方案B**：可解释模型（逻辑回归/决策树）+ 特征工程
- **方案C**：混合模型（黑箱决策 + 可解释模型验证）

### 决策过程

#### 第 1 步：用 AI 决策权衡分析框架

```
              可解释性
                 ↑
          方案B  │   方案C
           ●    │     ●
                 │
                 │
                 │
                 │
          方案A  │
           ●    │
                 └──────────→ 模型性能
```

#### 第 2 步：多维度评估

| 维度 | 方案A（黑箱+事后解释） | 方案B（可解释模型） | 方案C（混合） |
|------|----------------------|-------------------|--------------|
| AUC | 0.91 | 0.83 | 0.89 |
| 可解释性 | 弱（事后近似） | 强（直接可读） | 中（双层验证） |
| 监管合规 | 有风险 | 合规 | 合规 |
| 开发成本 | 低（已有模型） | 中（重新特征工程） | 高（两套模型） |
| 运维成本 | 低 | 低 | 中 |
| 申诉处理 | 困难（解释不可靠） | 简单 | 中等 |
| 公平性审计 | 困难 | 简单 | 可行 |

#### 第 3 步：风险评估

| 风险 | 方案A | 方案B | 方案C |
|------|-------|-------|-------|
| 监管否决 | 高 | 低 | 低 |
| 误拒优质客户 | 低 | 高 | 中 |
| 歧视性审批 | 高（难检测） | 低（易检测） | 低（可检测） |
| 运维复杂度 | 低 | 低 | 中 |

### 最终决策

**选择方案C（混合模型）**，分阶段实施：

| 阶段 | 时间 | 目标 | 关键任务 |
|------|------|------|----------|
| 第1期 | M1-M3 | 双模型并行 | 黑箱决策+可解释模型验证 |
| 第2期 | M4-M6 | 差异分析 | 两模型不一致案例深度分析 |
| 第3期 | M7-M9 | 渐进替代 | 高置信区间用可解释模型 |

**核心原则：金融场景中，可解释性不是"nice to have"，而是合规底线。性能损失可以通过更好的特征工程和混合架构来弥补。**

## 案例 4：AI 决策中的责任归属设计

### 背景

某互联网公司的推荐算法团队需要决定：当 AI 推荐导致用户投诉或负面影响时，责任如何分配？

这不仅是法律问题，更是产品设计和组织设计问题。

### 决策过程

#### 第 1 步：用 AI 责任链路分析

```
用户投诉
  ↓
问题归因
  ├── 数据问题 → 数据团队负责
  ├── 模型问题 → 算法团队负责
  ├── 产品设计问题 → 产品团队负责
  ├── 运营配置问题 → 运营团队负责
  └── 不可归因 → 系统性风险，需升级决策
```

#### 第 2 步：设计责任分配矩阵（RACI-AI）

| 决策类型 | AI | 产品经理 | 算法工程师 | 法务/合规 |
|----------|:--:|:--------:|:----------:|:---------:|
| 推荐内容选择 | A | R | C | I |
| 推荐策略调整 | C | A | R | I |
| 模型版本上线 | I | A | R | C |
| 用户投诉处理 | C | R | C | C |
| 合规风险判定 | I | C | C | R/A |

> R = Responsible（执行） A = Accountable（负责） C = Consulted（咨询） I = Informed（知会）

#### 第 3 步：设计 AI 决策审计日志

```
决策审计记录
├── 决策ID
├── 时间戳
├── 决策类型
├── AI 输入
│   ├── 模型版本
│   ├── 输入特征
│   └── 特征来源
├── AI 输出
│   ├── 推荐结果
│   ├── 置信度
│   └── 备选方案
├── 人工介入
│   ├── 是否有人工审核
│   ├── 审核人
│   └── 修改记录
├── 上下文
│   ├── 业务规则版本
│   ├── A/B 实验组
│   └── 特殊事件标记
└── 后果追踪
    ├── 用户反馈
    ├── 业务指标变化
    └── 后续调整
```

### 最终决策

| 原则 | 描述 |
|------|------|
| **人负最终责任** | AI 可以建议，但人对结果负责 |
| **可追溯性** | 每个AI决策必须有完整审计日志 |
| **可解释性** | 受影响方有权知道"为什么" |
| **可申诉性** | 用户可以对AI决策提出申诉，申诉必须由人工处理 |
| **持续改进** | 投诉和申诉数据必须反馈到模型训练 |

## AI 决策特点总结

1. **自主度光谱**：AI 决策不是"有或没有"，而是 L0-L3 的连续光谱，关键是选对位置
2. **准备度评估**：升级自主度前必须量化评估技术、流程、组织的准备度
3. **可解释性权衡**：在性能和可解释性之间找到行业合适的平衡点
4. **责任归属**：AI 决策必须有明确的责任链，人负最终责任

## 决策工具应用

| 场景 | 推荐工具 | 原因 |
|------|----------|------|
| AI 自主度选择 | AI决策自主度光谱 | 量化评估适合的参与程度 |
| AI辅助质量校验 | 偏差对比分析 | AI评分vs人工评分，识别系统性偏差 |
| 可解释性权衡 | 权衡分析模板 | 性能vs合规的结构化比较 |
| 责任归属设计 | RACI-AI矩阵 | 明确AI和人在每类决策中的角色 |
| 升级触发条件 | 条件决策框架 | 量化阈值+回退机制 |
