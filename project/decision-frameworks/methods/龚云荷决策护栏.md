# 龚云荷决策护栏清单

> 用途：AI 每次辅助决策时逐条过一遍，任一硬护栏触发必须在决策简报「护栏检查」里显式提示。这是把个人偏好和 Q3 约束变成可执行规则，让辅助决策稳定不跑偏。
> 来源：`USER.md` 决策偏好 + `CURRENT.md` 明确不做 + `MEMORY.md` 关键决策原则。
> 维护：约束变化时更新（季度切换、团队人力变化、主线调整）。

---

## 一、硬护栏（触发即红灯，必须显式提示）

| # | 护栏 | 判断标准 | 触发后动作 |
|---|------|----------|------------|
| G1 | 不挤占 Q3 P0 | 是否分散打透 `nl-sql` 这条主线的精力？ | 红灯：说明会挤占什么，除非能证明服务主线 |
| G2 | 不碰「明确不做」 | 是否命中 `CURRENT.md`/`WORK-PLAN.md` 的明确不做清单？ | 红灯：提示已列入不做，需用户显式推翻 |
| G3 | 资源现实性 | 2 人团队 + 预留 30% 支撑日常，这事有人有时间做吗？ | 红灯：指出要挤占谁的时间 |
| G4 | 数据支撑 | 关键判断有没有数据？ | 无数据的结论一律标「推断」，并给验证动作 |
| G5 | 不可逆则慢 | 决策可逆吗？回撤成本高吗？（满足任一即不可逆：影响Q3主线 / 需要投入超过3人天 / 对外承诺后无法撤回） | 不可逆/高成本：降速，强制走完预 mortem，必须写 `DECISIONS.md` |

## 二、软护栏（倾向性原则，作为评分权重）

| # | 原则 | 含义 |
|---|------|------|
| S1 | 单点打透优先 | 能深化 `nl-sql`/核心看板的，优先级天然更高 |
| S2 | 高频标准化优先 | 越是每天重复、可范式化的场景，AI 化价值越高 |
| S3 | 保核心价值区 | 异动归因 / OKR / 跨团队协调 / 人员管理是个人核心，别轻易外包给流程 |
| S4 | 先结论后细节 | 输出永远结论先行 |
| S5 | 边界与异常必写 | 方案要说明边界条件、异常流程、风险取舍 |

## 三、决策分层（决定投入多少分析）

| 层级 | 特征 | 走多重的流程 |
|------|------|--------------|
| 运营级（天 / 可逆） | 错了代价小 | 快速给结论，不必套框架 |
| 战术级（周-月 / 资源分配） | 影响本周-本月排期 | 走完整生产线 + 选项对比 |
| 战略级（季 / 方向 / 不可逆） | 影响 Q3 主线 | 生产线 + 预 mortem + 第一性原理，必写 `DECISIONS.md` |

## 四、当前约束快照（2026 Q3）

> 这一节会随季度变化，季度切换时刷新。

- **主线**：打透 `nl-sql`，覆盖 80% 日常取数，准确率 90%+
- **团队**：2 人（1 社招 + 1 实习），30% 精力保日常业务与核心看板
- **明确不做**：大规模 MCP/CLI 建设、方案设计类 Skill、异动归因/指标预测等高级 Skill、探索项目（TokenWisdom/个人网站）挤占 P0、重构现有目录
- **季度更新检查清单**：季度切换时，对照 `CURRENT.md`、`WORK-PLAN.md`、`MEMORY.md` 逐项刷新主线、团队配置、明确不做清单，确保护栏不过期
- **触发更新**：季度切换、人力变化、主线调整时刷新本节
