# 同期群分析（Cohort Analysis）

## 什么是同期群分析
同期群（Cohort）指**同一时期具有相同特征的用户群体**，最常用的是按"新增时间"分组，也可以按"首次行为时间""渠道"等特征分组。
Cohort分析本质是"按时间分组的留存/行为分析"，核心是把不同阶段进来的用户分开看，避免老用户占比高掩盖新用户问题。

## 和留存分析的区别
- 留存分析是Cohort分析的最常用场景
- Cohort分析可以看任何指标，不只是留存：留存率、付费率、ARPU、使用频次等都可以按Cohort看

## 适用场景
1. ✅ 版本迭代效果评估：新版本新用户是不是比老版本好？
2. ✅ 新用户质量趋势：每月/每周新用户质量是变好还是变差？
3. ✅ 用户生命周期行为变化：用户进来第一个月/第二个月/第三个月行为有什么规律？
4. ✅ 活动效果长期影响：活动拉来的新用户长期留存和自然量比怎么样？
5. ✅ 产品健康度监控：用户生命周期价值（LTV）是不是在提升？
6. ✅ 付费转化分析：用户进来多久开始付费？付费金额随时间怎么变化？

不适用场景：
❌ 用户量极小的早期产品（每批就几十人，统计不显著）
❌ 没有明确"新增时点"的单次消费场景（用RFM或趋势分析）
❌ 只关心当下大盘、不关心用户生命周期的研究（直接看趋势即可）

## 核心指标

| 指标 | 定义 | 计算方式 |
|------|------|----------|
| Cohort规模 | 该批次用户总数 | 新增当日去重用户数 |
| 留存率 | 第N天仍活跃用户占比 | 第N天活跃用户数 / Cohort规模 |
| 付费率 | 第N月发生付费的用户占比 | 第N月付费用户数 / Cohort规模 |
| ARPU | 第N月人均贡献收入 | 第N月总收入 / Cohort规模 |
| 累计ARPU | 截至第N月人均累计收入 | 截至第N月累计收入 / Cohort规模 |
| 流失率 | 第N月不再活跃的用户占比 | 1 - 第N月留存率 |

## 标准Cohort表格
以付费Cohort为例：

| 用户新增月 | 用户数 | 首月付费率 | 第2月付费率 | 第3月付费率 | 第4月付费率 | 首月ARPU | 第2月ARPU | 累计ARPU |
|------------|--------|------------|------------|------------|------------|----------|----------|----------|
| 1月 | 10,000 | 5% | 8% | 6% | 4% | 12元 | 18元 | 45元 |
| 2月 | 12,000 | 6% | 9% | 7% | - | 15元 | 22元 | 42元 |
| 3月 | 15,000 | 7% | 10% | - | - | 18元 | 25元 | 32元 |
| 4月 | 18,000 | 8% | - | - | - | 22元 | - | 22元 |

**看两个方向**：
- **横向（行）**：同一批用户在生命周期不同阶段的行为变化，看用户成熟过程
- **纵向（列）**：不同批次用户在相同生命周期阶段的指标对比，看产品/运营变化带来的提升

## 标准分析流程

### 第1步：确定Cohort分组维度
1. **时间维度**：按新增日/周/月分组，高频产品用日/周，中低频用月
2. **特征维度**：按渠道、版本、首行为、画像分组
3. **分组粒度选择**：每个Cohort至少要有500+用户，否则指标波动太大没意义

**示例：浏览器信息流按新增月Cohort**
```
1月Cohort = 1月份首次打开信息流页的用户
2月Cohort = 2月份首次打开信息流页的用户
```

### 第2步：定义指标和观测周期
1. **核心指标**：留存率？付费率？ARPU？使用频次？
2. **观测周期**：看7天/30天/90天？根据产品频次定
3. **统计口径**：去重方式、活跃定义（打开APP？消费内容？完成核心行为？）

### 第3步：构建Cohort矩阵表
输出标准矩阵表（见上方付费Cohort示例），用颜色深浅做热力图，便于一眼看出高低。

### 第4步：横向看生命周期规律
- 用户进来后第几天/第几个月指标达到稳态？
- 有没有" honeymoon期"（首月特别活跃，之后断崖下跌）？
- 有没有"延迟激活"（首月不活跃，第2-3月才起来）？

### 第5步：纵向看批次质量趋势
- 新Cohort同阶段指标比老Cohort高还是低？
- 哪个Cohort出现拐点？对应时间点发生了什么（版本/活动/渠道调整）？
- 趋势是持续改善还是单点波动？

### 第6步：异常Cohort下钻
对异常Cohort按渠道、版本、画像等维度下钻，找到异常的根源。
```sql
-- Presto/Kyuubi：信息流新用户次留Cohort，按渠道拆分
WITH new_users AS (
  SELECT device_id, channel, first_active_date
  FROM dws_user_retention
  WHERE first_active_date BETWEEN date'2026-04-01' AND date'2026-04-30'
)
SELECT
  n.channel,
  count(DISTINCT n.device_id) AS cohort_size,
  count(DISTINCT CASE WHEN r.active_date = date_add('day',1,n.first_active_date)
                      THEN n.device_id END) AS d1_retained
FROM new_users n
LEFT JOIN dwd_user_event r
  ON r.device_id = n.device_id AND r.event_name = 'feed_impression'
GROUP BY n.channel
ORDER BY cohort_size DESC;
```

### 第7步：输出结论和行动建议
- 批次质量是改善还是恶化？对应到具体产品/运营动作
- 哪类Cohort价值最高？加大投入
- 用户成熟周期是多少？据此定召回策略、新手引导时长

## 高级玩法

### 1. 多指标Cohort交叉
- 同一批用户同时看留存 + 付费 + 频次，交叉分析
- 例：留存高但付费低的Cohort，说明用户喜欢但不掏钱，定价或付费引导有问题

### 2. 首行为Cohort
- 按用户第一次使用的产品功能分组，对比长期价值
- 例：首行为"搜索"的用户 vs 首行为"信息流推荐"的用户，3个月后留存和消费深度差异

### 3. CDP/LTV预测建模
- 用前7天/前30天行为特征，预测该Cohort的90天/365天LTV
- 用于渠道ROI评估、预算分配

### 4. 虚拟Cohort对比
- 活动Cohort vs 自然量Cohort，控制画像一致后对比长期价值
- 评估活动是否真的带来了高质量用户，还是只是把未来用户提前拉来

## 常见坑和避坑指南

| 坑 | 后果 | 避坑 |
|----|------|------|
| Cohort样本太小 | 指标波动剧烈，看不出趋势 | 单批至少500+用户，小产品用周Cohort代替月 |
| 只看横向不看纵向 | 看到老Cohort留存高，误以为产品好，其实是新用户质量差被掩盖 | 纵向对比同阶段指标才是关键 |
| 把Cohort当趋势看 | 把不同批次数据串成一条线，混淆了批次差异和时间趋势 | Cohort矩阵和趋势图分开看 |
| 活跃定义不一致 | 首月"活跃"和第N月"活跃"口径不同，留存率失真 | 全周期统一活跃定义 |
| 不控制变量就对比 | 活动Cohort和自然Cohort画像本就不同，直接对比得错结论 | 用PSM或画像匹配控制变量 |
| 忽略时间窗口对齐 | 1月Cohort看第3月、4月Cohort也看第3月，但4月还没到第3月 | 严格按生命周期阶段对齐，未达成的格子留空 |

## 实战案例：浏览器信息流新用户质量评估

**背景**：浏览器信息流Q1做了推荐算法升级，大盘DAU涨了15%，但产品经理怀疑新用户质量在下降（拉来了但留不住）。

**分析过程**：
1. 按"首次打开信息流月"分Cohort，看次留、7留、30留三个关键节点
2. 纵向对比：1月Cohort 7留42%，2月40%，3月35%，4月31%，新用户留存持续下滑
3. 横向看：每个Cohort都是7留到30留跌幅最大，说明新用户在第一周后流失集中
4. 渠道下钻：发现3-4月新增里"信息流广告"渠道占比从20%升到45%，该渠道7留只有22%，远低于自然量45%
5. 行为对比：广告渠道用户首刷平均只看3条内容就退出，自然量用户看12条
6. 结论：算法升级带来更多曝光，但广告渠道拉新质量差，拉低了整体新用户留存

**结论**：大盘DAU上涨是老用户活跃提升 + 低质量新用户堆积的结果。建议广告渠道降低出价、优化投放素材人群，同时新用户首刷做内容冷启动优化，把首刷内容深度从3条提升到8条以上。
