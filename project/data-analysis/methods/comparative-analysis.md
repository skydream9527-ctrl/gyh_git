# 对比分析（Comparative Analysis）

## 什么是对比分析
对比分析是最基础、最常用、也是最有效的数据分析方法，没有之一。通过将两个或多个对象/时间段/群体的指标放在一起比较，发现差异、定位问题、评估效果。

**没有对比就没有结论**：
- "DAU100万" 是好是坏？不知道，和上周比、和去年比、和目标比才知道
- "转化率5%" 是高是低？和行业比、和改版前比、和其他渠道比才知道

## 适用场景
几乎所有数据分析场景都会用到对比，核心场景包括：
1. ✅ 指标异动诊断：今天DAU为什么跌了10%？
2. ✅ 改版/功能效果评估：新版本上线后转化有没有提升？
3. ✅ AB实验结果解读：实验组比对照组好多少？
4. ✅ 不同群体差异：新用户vs老用户、iOSvs Android、渠道Avs渠道B
5. ✅ 目标达成评估：实际完成和KPI比差多少？
6. ✅ 行业竞对分析：我们和竞品比处于什么水平？

## 核心对比维度（5种对比角度）

### 1. 同比/环比对比：时间维度
- **环比**：和上一个周期比，看短期趋势
  - 日环比：今天 vs 昨天
  - 周环比：本周 vs 上周
  - 月环比：本月 vs 上月
  - → 反映最近变化、短期波动
- **同比**：和去年同期比，消除季节性
  - 日同比：今天 vs 去年同天
  - 周同比：本周 vs 去年同周
  - → 剔除季节性/周期性，看真实增长
- **和历史平均比**：和过去30天/90天平均值比，看是否异常

例："本周DAU环比跌5%，但同比涨20%，说明是正常季节性波动，不是产品问题。"

### 2. 横向对比：分组维度
不同人群/渠道/平台/版本对比：
- 用户属性：新vs老、不同年龄/性别/地域
- 来源渠道：自然流量vs付费渠道、抖音vs快手vs微信
- 端/平台：iOS vs Android、H5 vs APP、小程序
- 版本：新版本vs旧版本
- 品类/业务线：不同商品品类、不同业务线

例："整体转化率没跌，但Android端跌了20%，原来是Android新版本出了bug。"

### 3. 纵向对比：流程/前后链路
用户全链路各环节对比，漏斗每一步对比。
例："和上周比，商详到加购率没变化，加购到提交订单也没变化，但提交订单到成功率跌了15%，定位到支付环节出问题。"

### 4. 目标对比：和预期比
- 和KPI/目标比：完成率多少？差多少？
- 和预测值比：比预期好还是差？
- 和AB实验的预期提升比：达到最小可检测效应了吗？

例："本月收入1200万，目标1000万，超额完成20%，但其中新用户收入只完成了目标的80%，新业务不及预期。"

### 5. 外部对比：和行业/竞品比
- 行业基准/大盘：行业整体涨了10%，我们只涨了5%，说明跑输大盘
- 竞品数据：竞品转化率8%，我们只有5%，还有提升空间
- 注意：外部数据口径可能不一致，要先对齐口径再对比

## 标准对比分析流程（以指标异动诊断为例）
当你发现某个指标异常（涨/跌超出预期），按这个流程走：

### 第1步：确认数据准确性（非常重要！）
先排除数据问题，不要分析半天是数据错了：
1. 指标口径有没有变？统计逻辑有没有改？
2. 数据管道有没有延迟/故障？埋点有没有丢？
3. 是不是技术原因导致的？（比如服务挂了1小时，某版本崩溃）
4. 确认异常是真实存在的，不是数据错误。

> 铁律：数据不对，分析全废。拿到异常先验数。

### 第2步：确认异常幅度和范围
1. 跌/涨了多少？是在正常波动范围内吗？（看历史波动区间，比如平时±3%波动，今天跌10%就是异常）
2. 是只今天异常，还是已经连续几天了？
3. 是全量异常，还是局部？

### 第3步：多维度拆解对比（下钻）
按核心维度一层一层拆，直到定位到差异来源：
```
整体指标异常
   ↓ 拆平台
发现Android端异常
   ↓ 拆版本
发现Android v3.2版本异常
   ↓ 拆渠道
发现应用宝渠道v3.2异常
   ↓ 拆路径
发现应用宝渠道v3.2用户在支付环节失败率高
```
**常用拆解维度**：
- 平台：iOS/Android/小程序/H5
- 版本：各版本号
- 渠道：各流量来源
- 用户：新/老，不同分群
- 地域：省市
- 时间：小时级（看是几点开始异常的，对应什么动作）
- 业务线/品类：哪个业务/品类贡献了异常

常用拆解逻辑：
- 贡献度分解：整体跌幅X%，A部分贡献了多少个百分点，B部分贡献了多少，找到贡献最大的因子
  公式：`部分贡献度 = (部分本期值 - 部分基期值) / 整体基期值`
- 率的拆解：转化率跌了，拆分子母（分子跌还是分母涨？）

### 第4步：关联事件对比
定位到大致范围后，找对应时间点前后的相关事件：
1. 产品侧：那个时间点有没有发版？有没有上线新功能？有没有改策略？
2. 运营侧：有没有做活动？活动结束了？有没有push/短信推送？
3. 技术侧：有没有故障？有没有接口超时？有没有CDN问题？
4. 外部因素：节假日？周末？行业大事件？竞对动作？政策变化？

### 第5步：验证因果，排除混淆
不要看到相关性就下结论，验证：
1. 异常群体是不是真的受到了该事件影响？（比如发版了，是不是只有新版本用户异常？）
2. 时间上是不是吻合？（事件发生在前，异常出现在后）
3. 有没有其他混淆因素？（同一时间有没有其他事情发生？）
4. 如果是AB实验，是不是实验分组带来的？

### 第6步：给出结论和建议
结论要包含：
- 异常是什么：指标涨跌多少
- 原因是什么：定位到具体原因
- 影响多大：影响多少用户/收入/指标
- 建议怎么做：修复问题/后续改进/如何避免

## 对比分析四大原则

### 1. 可比性原则：对比的双方要口径一致
这是最容易犯的错误：
- ❌ 错误："我们DAU1000万，竞品才500万，我们更厉害" → 你算的是日启动，竞品算的是日登录，口径不一样
- ❌ 错误："本月转化率比上月涨了20%" → 上月包含春节假期，本月是正常月份，季节性不可比
- ✅ 正确：对比前先对齐指标定义、统计时间、范围、口径，确保苹果比苹果，橘子比橘子。

### 2. 单一变量原则：一次对比只变一个变量
如果对比的两个群体有多个差异，你不知道是哪个导致的结果差异。
- ❌ 错误："新版本转化率比旧版本高10%" → 新版本用户主要是老用户，旧版本包含大量新用户，是版本差异还是新老差异？
- ✅ 正确：控制变量，只保留你要研究的那个变量不同，其他都一样（这就是AB实验的核心逻辑）。

### 3. 相对数+绝对数一起看
只看比例会误导，结合绝对数：
- ❌ 错误："A渠道转化率比B渠道高100%，全力投A" → A渠道只有100个用户转化20个（20%），B渠道10000个用户转化1000个（10%），B渠道实际带来的用户是A的50倍
- ✅ 正确：任何转化率/增长率都要结合基数看，相对数看效率，绝对数看规模。

### 4. 统计显著性原则
小样本差异不要下结论
- ❌ 错误："今天新用户转化率12%比昨天10%涨了20%，新引导有效" → 今天新用户只有100个，差异统计不显著，就是正常波动
- ✅ 正确：小样本看置信区间，做显著性检验，确保差异不是随机波动。

## 高级对比方法

### 1. 双重差分（DID）
当不能做AB实验时，用对照组+前后对比，排除自然增长/时间因素：
```
实验组：改版前 → 改版后
对照组：没改版 → 同样时间
效果 = (实验组后 - 实验组前) - (对照组后 - 对照组前)
```
→ 剔除了时间趋势，得到纯净的改版效果。

### 2. 合成控制
找不到合适对照组时，用其他群体加权合成一个"虚拟对照组"来对比，适合政策/大版本上线评估。

### 3. 基准对比法（Benchmarking）
设定多个基准线对比：
- 及格线：行业平均水平
- 优秀线：行业Top20%水平
- 目标线：我们自己的目标值
- 历史最好：我们历史上做到过的最好水平
→ 不同基准对比，知道我们处于什么位置。

## 常见坑和避坑指南

| 坑 | 后果 | 避坑 |
|----|------|------|
| 口径不一致就对比 | 结论完全错误，误导决策 | 对比前先对齐口径：指标定义、统计范围、时间周期 |
| 只看相对值不看绝对值 | 高估小群体的重要性，决策错误 | 相对数+绝对数同时展示 |
| 忽略基数/样本量 | 把小样本随机波动当显著差异 | 计算样本量，做统计显著性检验 |
| 相关当因果 | A和B有差异就认为是A导致B | 控制变量、验证时间顺序、排除混淆变量、AB验证 |
| 只比结果不比过程 | 只知道A比B好，不知道为什么 | 除了结果指标，还要对比过程指标，定位差异来源 |
| 拿自己的优势比别人劣势 | 自嗨式对比，不能发现问题 | 全方位对比，既要比优势也要比劣势 |
| 忽略季节性/周期性 | 把正常波动当异常，或者把异常当正常 | 同比+环比结合，建立历史波动区间 |
| 过度拆解 | 拆到太小的维度，每个维度差异都很小，找不到重点 | 先看整体贡献度，优先拆贡献最大的维度，不要盲目拆所有维度 |

## 实战案例：DAU下跌10%诊断

**背景**：周一上班发现昨日（周日）DAU比上周日跌了10%，需要定位原因。

**分析过程**：
1. **验证数据**：核对埋点、数据管道都正常，指标口径没变，数据真实。
2. **看异常幅度**：上周日DAU100万，昨日90万，跌10万，历史周日波动±3%，确实异常。
3. **维度拆解**：
   - 拆平台：iOS跌12%，Android跌8%，两端都跌，不是单平台问题
   - 拆新老用户：老用户DAU跌5%，新用户DAU跌30% → 主要是新用户跌了
   - 拆渠道：自然量正常，应用宝渠道新用户跌了80%，贡献了9万的跌幅，是主要原因
   - 拆时间：从周六凌晨2点开始应用宝新用户突然跌为0
4. **关联事件**：查发布记录，周五晚上发了Android新版本，应用宝渠道的包因为合规问题被下架了，周六凌晨生效。
5. **结论**：应用宝渠道Android包被下架，导致新用户无法下载，DAU跌10%。
6. **建议**：紧急联系应用宝解决合规问题重新上架，评估损失，后续发版提前检查各渠道审核状态。
