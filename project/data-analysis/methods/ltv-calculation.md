# LTV/CLV 用户终身价值计算（Lifetime Value）

## 什么是LTV
Life Time Value，用户终身价值，指用户从注册到流失整个生命周期内为产品创造的总价值（收入/利润）。

核心用途：判断获客成本（CAC）是否合理，**LTV/CAC ≥ 3 是健康的商业模式基准**；评估用户质量、指导资源分配、衡量变现能力。

## 适用场景
1. ✅ 获客ROI评估：判断某渠道/活动的获客成本是否可接受
2. ✅ 渠道质量对比：不同渠道用户的LTV差异，优化投放分配
3. ✅ 用户价值分层：高LTV用户重点运营，低LTV用户控制成本
4. ✅ 商业模式验证：LTV/CAC是否达到3倍健康线，回本周期是否≤6个月
5. ✅ 变现策略评估：提价/新增变现后LTV的变化

不适用场景：
❌ 新产品数据不足（用户生命周期还没走完，LTV预测误差大）
❌ 一次性交易场景（没有持续留存，用单次ROI更直接）
❌ 用户生命周期极短的产品（如一次性工具类）

## 核心指标

| 指标 | 定义 | 计算方式 |
|------|------|----------|
| LTV | 用户终身价值 | Σ（第n月留存率 × 第n月ARPU），或简化为 ARPU × 生命周期 |
| CAC | 用户获客成本 | 总营销费用 / 新增用户数 |
| ARPU | 每用户平均收入 | 总收入 / 活跃用户数 |
| ARPPU | 每付费用户平均收入 | 总收入 / 付费用户数 |
| 回本周期 | LTV累计覆盖CAC的时长 | CAC / 月ARPU（近似）或累计LTV≥CAC的月份 |
| LTV/CAC比 | 商业健康度 | LTV / CAC，健康线≥3 |
| 毛利率 | 收入中扣除成本后的利润比例 | (收入-成本)/收入 |
| 留存曲线 | 各月留存率衰减情况 | 第n月活跃用户 / 首月新增用户 |

## 标准分析流程

### 第1步：明确价值口径
1. **价值类型**：收入LTV（GMV）还是利润LTV（扣成本）？建议两者都算
2. **时间范围**：用户全生命周期（理论值）还是有限窗口（如180天、365天实测值）
3. **用户范围**：全量用户还是某批次/某渠道用户
4. **货币单位**：元、美元，注意汇率一致性

**示例**：浏览器信息流用户LTV，口径=广告变现收入，窗口=注册后365天

### 第2步：计算ARPU
1. 统计单位时间内（通常按月）总收入
2. 按活跃用户数平均（ARPU）或按付费用户数平均（ARPPU）
3. 区分新增用户和存量用户的ARPU

**示例：信息流月ARPU（Presto/Kyuubi）**
```sql
SELECT
    dt_month,
    SUM(ad_revenue) AS total_revenue,
    COUNT(DISTINCT user_id) AS active_users,
    SUM(ad_revenue) / COUNT(DISTINCT user_id) AS arpu
FROM dws_user_value
WHERE dt_month BETWEEN '2025-01' AND '2025-12'
GROUP BY dt_month
ORDER BY dt_month;
```

### 第3步：拟合留存曲线
1. 取一批同期用户（如2025年1月新增），统计后续每月的留存率
2. 留存曲线通常呈指数衰减，拟合函数：`R(n) = a × n^(-b)` 或 `R(n) = a × e^(-b×n)`
3. 用实际数据校准参数，预测长期留存

**示例：同期用户月留存（Presto/Kyuubi）**
```sql
SELECT
    months_since_register,
    COUNT(DISTINCT user_id) AS retained_users,
    first_month_cohort AS cohort_size,
    COUNT(DISTINCT user_id) * 1.0 / first_month_cohort AS retention_rate
FROM (
    SELECT
        user_id,
        date_diff('month', register_dt, active_dt) AS months_since_register,
        MAX(first_month_cohort) OVER () AS first_month_cohort
    FROM (
        SELECT
            user_id,
            register_dt,
            active_dt,
            COUNT(DISTINCT user_id) OVER () AS first_month_cohort
        FROM dwd_user_event
        WHERE register_dt BETWEEN '2025-01-01' AND '2025-01-31'
    ) t
) t2
GROUP BY months_since_register, first_month_cohort
ORDER BY months_since_register;
```

### 第4步：计算LTV
1. **简化法**：`LTV = ARPU × 平均生命周期`（生命周期=1/月流失率）
2. **精准法**：`LTV = Σ（第n月留存率 × 第n月ARPU）`，按生命周期求和
3. **实测法**：直接统计一批用户在窗口期（如365天）内的累计收入贡献

**示例：用留存曲线和ARPU计算LTV**
```
月ARPU = 2.5元
留存曲线：R(1)=60%, R(2)=48%, R(3)=42%, ... R(12)=22%
LTV(12个月) = 2.5 × (1 + 0.60 + 0.48 + 0.42 + ... + 0.22)
            = 2.5 × 5.8 = 14.5元
```

### 第5步：对比CAC评估健康度
1. 计算CAC：某渠道总投放费用 / 该渠道带来的新增用户数
2. 计算 LTV/CAC 比值：
   - **≥3 健康**：可加大投放
   - **1-3 警戒**：优化获客或变现
   - **<1 危险**：每获一个用户都在亏钱
3. 计算回本周期：累计LTV≥CAC的月份，**健康线≤6个月**

**输出标准LTV评估表**：
| 渠道 | CAC | 12月LTV | LTV/CAC | 回本周期 | 健康度 |
|------|-----|---------|---------|----------|--------|
| 信息流投放A | 8元 | 14.5元 | 1.8 | 5个月 | 警戒 |
| 搜索投放B | 5元 | 18元 | 3.6 | 3个月 | 健康 |
| push召回 | 0.5元 | 8元 | 16 | 1个月 | 健康 |
| 应用商店自然量 | 0元 | 12元 | ∞ | 0 | 健康 |

### 第6步：细分对比找差异
1. 按渠道拆分LTV，优化投放分配
2. 按用户属性（地域、设备）拆分，识别高价值群体
3. 按行为特征（活跃度、内容偏好）拆分，指导精细化运营

### 第7步：监控和迭代
1. LTV是预测值，需要用实测数据持续校准
2. 季度复盘各渠道LTV/CAC，调整投放策略
3. 变现策略变化后重新计算LTV

## 高级玩法

### 1. 预测LTV（pLTV）
- 新用户注册7天/14天内行为特征，预测365天LTV
- 用机器学习模型（XGBoost/LightGBM）训练历史用户行为+LTV
- 用于早期识别高价值用户、实时优化投放出价

### 2. 分群LTV
- 不同用户群体的LTV差异巨大，平均值会掩盖真相
- 按RFM/生命周期/行为聚类分别计算LTV
- 高LTV群体的特征反推投放定向人群

### 3. LTV敏感性分析
- 变现提价10%、留存提升5%对LTV的影响
- 找到对LTV影响最大的杠杆点（通常是留存）
- 指导产品优化优先级

## 常见坑和避坑指南

| 坑 | 后果 | 避坑 |
|----|------|------|
| 用收入LTV不扣成本 | 高估用户价值，决策激进 | 关键决策用利润LTV（扣内容成本、服务器成本） |
| 留存曲线拟合不准 | LTV预测偏差大 | 用足够长的历史数据拟合，定期校准 |
| 忽略渠道质量差异 | 平均LTV掩盖低质渠道 | 必须按渠道分别计算LTV/CAC |
| 窗口太短 | LTV被低估，错过长期价值 | 至少用180天窗口，长决策产品用365天+ |
| 把ARPU当LTV | 混淆单期价值和终身价值 | LTV要累加生命周期内各期价值 |
| 不考虑时间衰减 | 未来的钱和现在的钱等价 | 长周期LTV建议贴现（DCF） |
| CAC只算投放费 | 低估获客成本 | CAC含投放+渠道+运营+技术分摊 |

## 实战案例：信息流用户LTV评估驱动投放优化

**背景**：浏览器信息流2025年Q3加大投放，整体CAC从6元涨到9元，需要评估是否健康，决定Q4投放策略。

**分析过程**：
1. 明确口径：收入LTV（广告变现），窗口365天，按渠道分别计算
2. 计算月ARPU：整体2.5元，但搜索渠道用户3.2元，信息流投放渠道2.1元
3. 拟合留存曲线：1月新增用户12个月留存22%，搜索渠道28%，投放渠道15%
4. 计算LTV：
   - 搜索渠道：3.2 × 6.8 = 21.8元，CAC=5元，LTV/CAC=4.4，回本3个月，健康
   - 信息流投放A：2.1 × 4.5 = 9.5元，CAC=9元，LTV/CAC=1.1，回本11个月，警戒
   - push召回：1.8 × 5.0 = 9元，CAC=0.5元，LTV/CAC=18，健康
5. 细分发现：投放A渠道的低LTV用户集中在低端机型+三四线城市
6. 敏感性分析：留存提升5%，LTV提升12%；ARPU提升10%，LTV提升10%——留存杠杆更大

**结论**：整体LTV/CAC=2.1，低于3倍健康线，主要被信息流投放A渠道拖累。Q4策略：搜索渠道加预算30%，信息流投放A减预算40%并定向高价值人群（一二线+中高端机型），预期整体LTV/CAC提升到3.5以上。
