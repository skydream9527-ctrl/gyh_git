# 维度下钻分析（Drill Down）

## 什么是维度下钻
当整体指标异常时，按不同维度一层一层拆分，从宏观到微观，逐步定位到问题根源，是指标异动定位最核心的方法。

核心逻辑：**整体异常一定是某个局部异常导致的，拆分维度找到贡献最大的局部。**

## 适用场景
1. ✅ 指标异动归因：DAU/留存/转化率突然跌，定位到具体维度
2. ✅ 大盘波动诊断：整体看平稳，但拆开后发现局部恶化被增长掩盖
3. ✅ A/B测试异常解读：实验组整体不显著，下钻后发现某子群显著正向/负向
4. ✅ 活动效果拆解：活动整体达标，但哪个群体贡献最大？哪个群体没带动？
5. ✅ 质量问题定位：崩溃率/加载时长恶化，拆到具体端/版本/机型

不适用场景：
❌ 指标本身就在正常波动范围内（不要每次涨跌都下钻）
❌ 没有明确业务目标的"探索性拆分"（拆着拆着迷失方向）
❌ 数据量太小（拆到子层样本不足，差异不显著）

## 核心指标

| 指标 | 定义 | 计算方式 |
|------|------|----------|
| 整体变化量 | 整体指标本期与基期的差值 | 本期值 - 基期值 |
| 维度贡献度 | 某维度值对整体变化的贡献比例 | (该部分本期值 - 该部分基期值) / 整体变化量 |
| 相对变化率 | 某维度值自身的涨跌幅度 | (本期值 - 基期值) / 基期值 |
| 占比变化 | 某维度值在整体中占比的变化 | 本期占比 - 基期占比 |
| 异常显著性 | 该维度值变化是否超出正常波动 | z-score = (本期值 - 均值) / 标准差 |

## 核心拆解维度
| 维度类型 | 常用维度 |
|----------|----------|
| 用户维度 | 新/老用户、用户等级/分层、注册时间、地域、年龄性别、渠道来源 |
| 产品维度 | 端（iOS/Android/H5）、APP版本、平台、页面/功能模块 |
| 业务维度 | 内容品类、内容来源、业务线、创作者/账号 |
| 时间维度 | 小时级、天级、星期几、时段 |
| 行为维度 | 用户行为路径、是否完成核心行为、活跃度分层 |

## 标准分析流程

### 第1步：确认整体异常
1. **确认真实性**：先排除数据问题（埋点丢失、ETL延迟、口径变更）
2. **量化幅度**：跌了多少？相对历史均值偏离几个标准差？
3. **确认时间范围**：从什么时候开始跌？持续多久？是单点还是持续？

### 第2步：选择拆解维度优先级
按"业务相关性 + 历史经验"排序，不要盲目拆所有维度：
- 先拆**用户维度**（新老/渠道）→ 是不是某类用户问题
- 再拆**产品维度**（端/版本）→ 是不是某端/某版本问题
- 再拆**业务维度**（品类/页面）→ 是不是某块业务问题
- 最后拆**时间维度**（小时/星期）→ 是不是某时段集中

### 第3步：第一层下钻，计算贡献度
按优先级最高的维度拆分，计算每个维度值的贡献度，找到贡献最大的。
```sql
-- Presto/Kyuubi：信息流DAU下跌，按渠道拆分贡献度
SELECT
  channel,
  sum(CASE WHEN dt = date'2026-07-01' THEN dau END) AS base_dau,
  sum(CASE WHEN dt = date'2026-07-03' THEN dau END) AS cur_dau,
  (sum(CASE WHEN dt = date'2026-07-03' THEN dau END)
   - sum(CASE WHEN dt = date'2026-07-01' THEN dau END)) AS delta
FROM dws_channel_dau_d
WHERE dt IN (date'2026-07-01', date'2026-07-03')
GROUP BY channel
ORDER BY delta ASC;
```

### 第4步：对贡献最大的维度值，继续下钻
不要停在一层，继续往下拆：
- 渠道贡献最大 → 拆该渠道的端/版本/地域
- 版本贡献最大 → 拆该版本的页面/功能模块
- 直到定位到"能对应到具体行动"的粒度

### 第5步：多维度交叉验证
单一维度可能误导，需要交叉验证：
- "Android渠道跌" → 是Android所有版本都跌，还是某个版本集中？
- "新用户跌" → 是所有渠道新用户跌，还是某个渠道新用户跌？

### 第6步：关联事件匹配
定位到具体维度后，回溯该维度值对应的时间点和业务动作：
- 该版本上线了什么改动？
- 该渠道最近调整了什么投放？
- 该品类内容最近有什么策略变化？

### 第7步：输出结论和建议
- 异常根因（哪几个维度的哪个值贡献最大）
- 关联事件（对应什么产品/运营动作）
- 影响范围（绝对规模、持续时长）
- 修复建议（针对根因的具体行动）

## 贡献度计算
拆分后先算每个部分的贡献度，优先看贡献最大的：
```
指标变化贡献度 = (该部分本期值 - 该部分基期值) / (整体本期值 - 整体基期值)
```
例：整体DAU跌10万，其中Android渠道跌9万，贡献度90%，优先定位Android问题。

注意：贡献度可正可负。整体跌10万，某维度值涨2万（贡献度-20%），说明该部分在"对冲"下跌，反而要保护；某维度值跌12万（贡献度120%），是主要元凶。

## 高级玩法

### 1. 多维度归因模型
- 用Shapley值或对数回归量化各维度的独立贡献，避免维度间共线性干扰
- 适用于多个维度同时变化、难以单独归因的复杂场景

### 2. 自动化下钻
- 按"贡献度阈值 + 显著性阈值"自动逐层下钻，输出归因路径
- 适合做监控告警的自动归因，减少人工排查时间

### 3. 对比下钻（基线对比）
- 不只和昨天比，还可以和"同期均值""预测值""对照组"比
- 例：和过去30天同星期均值比，剔除星期效应

### 4. 反事实推演
- "如果某维度值保持基期水平，整体会怎样？"
- 用于评估修复后预期的恢复幅度

## 常见坑和避坑指南

| 坑 | 后果 | 避坑 |
|----|------|------|
| 拆分维度不MECE | 维度间有重叠，贡献度加起来>100%或<100%，归因不准 | 维度值要互斥穷尽，同一层用一个维度拆 |
| 只看一层就下结论 | "Android跌"就说是Android问题，其实是某版本bug | 至少拆2-3层，定位到可行动的粒度 |
| 只看贡献度不看显著性 | 某维度贡献度大但本身波动就大，可能是噪声 | 结合历史波动算z-score，确认是真异常 |
| 忽略维度间共线性 | "新用户跌"和"广告渠道跌"其实是同一批人，双重归因 | 交叉验证，用条件贡献度而非边际贡献度 |
| 拆到样本太小的粒度 | 拆到具体机型+版本+地域，每格就几十人，差异不显著 | 拆到样本量足够（建议每格≥1000）就停 |
| 把季节性当异常 | "周末DAU跌"其实是正常周末效应 | 和同星期历史均值比，而非和昨天比 |
| 停在数据层不关联业务 | 拆完了不知道为什么，给不出行动 | 每层下钻都要回溯对应业务动作 |

## 原则
- MECE原则：拆分维度要相互独立、完全穷尽，不要漏也不要重叠
- 先拆贡献大的，不要盲目拆所有维度
- 拆到能定位到具体原因、能对应到具体行动为止

## 实战案例：浏览器信息流消费时长下跌归因

**背景**：浏览器信息流用户人均消费时长从28分钟跌到24分钟，跌幅14%，需要定位原因。

**分析过程**：
1. 确认异常：跌幅14%，超过历史±5%波动区间，确认为真异常
2. 第一层按端拆分：Android贡献度85%，iOS贡献度15%，定位Android
3. 第二层按版本拆分：Android v9.8.2贡献度92%，其他版本基本持平，定位v9.8.2
4. 第三层按内容类型拆分：v9.8.2版本下"短视频"消费时长跌60%，"图文"持平
5. 交叉验证：v9.8.2版本只有Android，iOS没发；短视频时长跌但短视频曝光量没跌，说明是消费深度下降
6. 关联事件：v9.8.2版本上线了"短视频自动播放下一集"改为"手动滑动"，导致单条短视频消费深度下降
7. 进一步看：短视频完播率从65%升到78%，但人均观看条数从12条降到5条，总时长下降

**结论**：根因是v9.8.2版本短视频交互改动，虽然单条完播率提升，但人均观看条数大幅下降，总时长净跌。建议：保留自动播放但增加"已看过的不再自动播放"策略，既保证完播率又维持消费深度，预计时长恢复到27分钟。
