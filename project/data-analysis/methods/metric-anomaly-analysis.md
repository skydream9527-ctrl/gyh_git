# 指标异动分析（Metric Anomaly Analysis）

## 什么是指标异动
核心指标突然出现超出正常波动范围的涨跌，需要快速定位原因。

异动分析的核心目标：**在最短时间内判断"是真异常还是假异常"，如果是真异常，定位到具体原因并给出修复建议。**

## 适用场景
1. ✅ 核心指标监控告警：DAU/留存/转化率/收入突然偏离正常区间
2. ✅ 大盘异动应急响应：业务指标突降，需要1小时内给初步结论
3. ✅ 版本/活动上线后回归：上线后指标是否符合预期，有没有负向
4. ✅ 数据质量异常排查：指标突变得离谱，怀疑是数据问题
5. ✅ 外部事件影响评估：节假日/热点/政策对指标的影响

不适用场景：
❌ 指标在正常波动范围内（每次涨跌都分析会浪费资源）
❌ 长期缓慢变化（不是异动，是趋势，用趋势分析）
❌ 没有基线的新指标（无法判断什么是"正常"）

## 核心指标

| 指标 | 定义 | 计算方式 |
|------|------|----------|
| 偏离度 | 当前值与历史均值的偏离程度 | (当前值 - 均值) / 标准差（z-score） |
| 绝对变化量 | 当前值与基期值的差值 | 当前值 - 基期值 |
| 相对变化率 | 当前值相对基期的涨跌幅 | (当前值 - 基期值) / 基期值 |
| 影响规模 | 异动影响的用户数/业务量 | 异动波及的去重用户数 |
| 持续时长 | 异动从开始到恢复的时间 | 异动起止时间差 |
| 贡献度 | 某因素对异动的贡献比例 | 该因素变化量 / 整体变化量 |

## 标准分析流程（5步法）

### 第1步：确认真实性
先排除数据问题，避免"分析半天结果是埋点丢了"：
1. **埋点检查**：关键事件有没有正常上报？数据量有没有断崖下跌？
2. **管道检查**：ETL有没有延迟/失败？数据延迟到没到？
3. **口径检查**：指标定义/计算逻辑最近有没有变更？
4. **对照检查**：相关指标是否同步变化？（如DAU跌但启动数没跌，可能是口径问题）

```sql
-- Presto/Kyuubi：检查埋点上报量是否正常
SELECT
  dt,
  count(*) AS event_count,
  count(DISTINCT device_id) AS uv
FROM dwd_user_event
WHERE dt BETWEEN date'2026-06-25' AND date'2026-07-03'
  AND event_name = 'feed_impression'
GROUP BY dt
ORDER BY dt;
```

### 第2步：确认幅度和范围
1. **幅度**：涨跌多少？相对历史均值偏离几个标准差？
2. **范围**：是全量还是局部？哪些端/版本/渠道/地域受影响？
3. **时间**：从什么时候开始？持续多久？是单点还是持续？是瞬时还是渐进？

### 第3步：维度拆解下钻
按平台、版本、渠道、地域、用户群、时间等维度拆分，找到贡献最大的异常点（详见 `drill-down.md`）：
1. 计算每个维度值的贡献度
2. 优先看贡献度最大且显著性最高的
3. 逐层下钻，定位到可行动的粒度

### 第4步：关联事件匹配
异常时间点前后有没有这些事件：
| 事件类型 | 典型例子 |
|----------|----------|
| 版本发布 | 新版本灰度/全量，引入bug或功能改动 |
| 活动上线/下线 | 活动开始/结束，push推送，运营配置变更 |
| 技术故障 | 服务宕机、接口超时、崩溃、CDN问题、性能下降 |
| 外部因素 | 节假日、季节性、周末效应、政策变化、社会热点、竞对动作 |
| 数据变更 | 埋点改版、口径调整、ETL逻辑变更 |

### 第5步：验证因果，输出结论
1. **时间一致性**：事件发生时间和异动时间是否吻合？
2. **影响范围一致性**：事件影响的范围和异动的范围是否一致？
3. **反事实验证**：如果没有这个事件，指标会怎样？（对照组/历史同期对比）
4. **量化影响**：该事件解释了多少异动？还有多少未解释？
5. **输出结论**：根因 + 影响范围 + 修复建议 + 预期恢复时间

## 常见异动原因分类
| 类别 | 典型原因 |
|------|----------|
| 数据问题 | 埋点丢失、ETL延迟、口径变更、统计错误 |
| 技术问题 | 服务故障、接口超时、崩溃、CDN问题、性能下降 |
| 产品问题 | 新版本bug、功能改动带来负向影响、策略调整 |
| 运营问题 | 活动上线/结束、push推送、渠道投放变化、运营活动到期 |
| 用户问题 | 渠道质量变化、刷量/作弊、用户自然波动 |
| 外部因素 | 节假日、季节性、周末效应、政策变化、社会热点、竞对动作 |

## 异动判断标准
基于历史数据计算正常波动区间（±3σ原则），超出区间即判定为异常：
- 小指标波动大：±20%波动可能是正常
- 核心大盘指标稳定：±3%就可能是异常
- 不要每个涨跌都分析，建立告警阈值，只分析超出置信区间的异常

常用判断方法：
| 方法 | 适用 | 判断标准 |
|------|------|----------|
| 固定阈值 | 稳定指标 | 偏离±X%即告警 |
| 同比/环比 | 有季节性的指标 | 偏离同期±X%即告警 |
| 移动平均 | 趋势型指标 | 偏离MA7/MA14±X%即告警 |
| z-score | 有历史分布的指标 | \|z\| > 3即告警 |
| 同期均值 | 有星期效应的指标 | 偏离过去4周同星期均值±X%即告警 |

## 高级玩法

### 1. 自动化异动检测
- 用 Prophet/STL 做时间序列分解，剔除趋势和季节性，对残差做异常检测
- 用 Isolation Forest / LOF 做多维异常检测
- 建立告警规则引擎，自动触发归因下钻

### 2. 异动归因自动化
- 告警触发后，自动按维度下钻，输出各维度贡献度排名
- 结合事件库（版本/活动/故障记录）自动匹配可能原因
- 输出归因报告，人工只做最终确认

### 3. 多指标关联分析
- 一个指标异动，看相关指标是否同步异动，判断是"真业务变化"还是"数据问题"
- 例：DAU跌但启动数没跌 → 可能是活跃定义问题；DAU和启动数同步跌 → 真业务下降

### 4. 影响面推演
- 估算异动的业务影响（损失的收入/用户数）
- 推演如果不修复会怎样（持续恶化/自愈）
- 据此决定响应优先级

## 常见坑和避坑指南

| 坑 | 后果 | 避坑 |
|----|------|------|
| 不先排数据问题 | 分析半天结果是埋点丢失 | 第1步永远是数据真实性检查 |
| 用昨天做基线 | 周末/节假日效应导致误判 | 用同星期历史均值或移动平均做基线 |
| 只看整体不下钻 | 整体平稳但局部恶化被增长掩盖 | 必须按核心维度拆分看 |
| 把相关性当因果 | "异动时正好有活动"就说是活动导致 | 验证时间一致性 + 范围一致性 + 反事实验证 |
| 一个原因解释全部 | 实际多因素叠加，归因不完整 | 量化每个因素的贡献，剩余未解释部分要说明 |
| 响应太慢 | 等分析完异动已经自愈或扩大 | 先快速给初步结论（1小时内），再深入验证 |
| 不建立告警阈值 | 每次涨跌都人工判断要不要查 | 建立分级告警，只查超阈值的 |

## 实战案例：浏览器搜索转化率突降诊断

**背景**：浏览器搜索结果页"搜索→点击内容"转化率从32%跌到24%，跌幅25%，触发P1告警。

**分析过程**：
1. 确认真实性：检查搜索曝光、点击埋点上报量正常，ETL无延迟，口径未变，确认为真异常
2. 确认范围：跌幅从7月2日上午10点开始，持续到7月3日仍未恢复；全端同步下跌
3. 维度下钻：
   - 按端拆：Android贡献度68%，iOS贡献度32%，两端都跌，Android更严重
   - 按搜索类型拆：导航类搜索（搜网址）转化率持平，内容类搜索（搜资讯/视频）转化率跌40%
   - 按结果类型拆：视频类结果点击率跌55%，图文类跌15%
4. 关联事件匹配：7月2日上午搜索团队上线了"视频结果卡片改版"，视频缩略图从大图改为小图列表
5. 反事实验证：未升级的iOS旧版本视频结果点击率仍为38%，升级后版本降到17%，差异显著
6. 量化影响：视频结果改版解释了82%的转化率下跌，剩余18%为内容质量小幅波动

**结论**：根因是7月2日视频结果卡片改版，缩略图变小导致点击率下降。影响范围：全端内容类搜索，预计每日影响点击量约120万次。建议：回滚视频卡片样式或A/B测试新样式（大图+小图混合），预计转化率可恢复到30%以上。
