# 归因分析（Attribution Analysis）

## 什么是归因分析
将转化/价值按照一定规则分配给用户路径上的各个触点/渠道，评估每个触点/渠道的贡献，用于渠道投放优化、营销资源分配、触点效果评估。

核心价值：回答"这个转化是谁带来的"，避免只看末次触点导致资源错配，用数据驱动投放预算分配。

## 适用场景
1. ✅ 渠道投放优化：各渠道的归因贡献对比，指导预算分配
2. ✅ 营销ROI评估：某次活动/素材的真实转化贡献
3. ✅ 触点价值评估：信息流曝光、push、搜索、推荐各贡献多少转化
4. ✅ 用户路径优化：哪些触点是"种草"环节，哪些是"临门一脚"
5. ✅ 跨渠道协同：多渠道联动的协同效应量化

不适用场景：
❌ 用户只有单一触点（直接看转化率即可）
❌ 转化路径数据缺失（无法还原完整触点序列）
❌ 强实时竞价场景（归因有延迟，用实时规则更好）

## 核心指标

| 指标 | 定义 | 计算方式 |
|------|------|----------|
| 归因贡献值 | 某触点/渠道分到的转化价值 | 按归因模型分配的转化数/收入 |
| 归因贡献率 | 某触点贡献占总转化的比例 | 触点归因值 / 总转化值 |
| 渠道ROI | 渠道归因收入 / 渠道投入 | 归因收入 / 投放费用 |
| 触点路径长度 | 用户转化前经过的触点数 | 路径中触点个数 |
| 转化周期 | 从首次触达到转化的时长 | 转化时间 - 首次触达时间 |
| 渠道协助率 | 渠道出现在路径中但非末次的比例 | 协助路径数 / 含该渠道路径数 |

## 标准分析流程

### 第1步：明确归因目标和转化事件
1. **转化事件定义**：什么是"转化"？激活？付费？次日留存？
2. **价值口径**：按转化次数还是收入归因？建议两者都算
3. **归因窗口**：转化前多长时间内的触点算有效？（常见7天/30天）
4. **触点定义**：哪些行为算触点？曝光、点击、访问、互动？

**示例**：浏览器信息流push归因，转化事件=用户打开信息流并消费内容，窗口=触达后24小时

### 第2步：还原用户触点路径
1. 采集用户在归因窗口内的所有触点事件
2. 按时间顺序排列成路径
3. 标注每个触点的渠道、内容、时间

**示例：用户触点路径还原（Presto/Kyuubi）**
```sql
SELECT
    user_id,
    array_agg(
        named_struct(
            'touch_dt', touch_dt,
            'channel', channel,
            'touch_type', touch_type,
            'content_id', content_id
        )
    ) AS touch_path
FROM (
    SELECT
        user_id,
        touch_dt,
        channel,
        touch_type,
        content_id
    FROM dwd_user_event
    WHERE dt BETWEEN date_sub('${convert_dt}', 7) AND '${convert_dt}'
      AND user_id IN (
          SELECT user_id
          FROM dwd_user_event
          WHERE dt = '${convert_dt}'
            AND event_type = 'feed_consume'
      )
    ORDER BY user_id, touch_dt
) t
GROUP BY user_id;
```

### 第3步：选择归因模型
不同模型适用不同场景，需要根据业务特点选择：

| 归因模型 | 分配规则 | 适用场景 | 优缺点 |
|----------|----------|----------|--------|
| 末次触点归因 | 100%给最后一个触点 | 短决策、即时转化 | 简单但忽略种草环节 |
| 首次触点归因 | 100%给第一个触点 | 拉新评估、品牌曝光 | 适合拉新但忽略临门一脚 |
| 线性归因 | 平均分给所有触点 | 触点少、各触点同等重要 | 公平但没有重点 |
| 时间衰减归因 | 越靠近转化功劳越大 | 长决策周期、多触点 | 符合直觉但需调衰减参数 |
| U型归因 | 首次40%+末次40%+中间20% | 平衡拉新和转化 | 兼顾两端但中间被低估 |
| 数据驱动归因 | 算法计算真实贡献（Shapley/马尔可夫） | 数据量大、触点复杂 | 最准确但需足够数据 |

### 第4步：计算归因贡献
1. 按选定模型分配每个转化事件的价值
2. 汇总各触点/渠道的归因贡献
3. 计算归因贡献率和ROI

**示例：U型归因权重分配逻辑（Presto/Kyuubi）**
```sql
-- 路径长度=2：首末各50%；>2：首40%+末40%+中间20%平分
SELECT
    user_id,
    touch_path[1].channel AS first_channel,
    touch_path[cnt].channel AS last_channel,
    CASE WHEN cnt = 2 THEN 0.5 ELSE 0.4 END AS first_weight,
    CASE WHEN cnt = 2 THEN 0.5 ELSE 0.4 END AS last_weight,
    CASE WHEN cnt > 2 THEN 0.2 / (cnt - 2) ELSE 0 END AS mid_weight
FROM (
    SELECT user_id, touch_path, cardinality(touch_path) AS cnt
    FROM user_touch_paths
    WHERE cardinality(touch_path) >= 2
) t;
```

### 第5步：对比模型结果
1. 同一份数据用不同归因模型计算，对比差异
2. 重点关注渠道排名变化：末次归因下排第1的渠道，在U型或数据驱动下是否还是第1
3. 选择最符合业务逻辑的模型作为主模型，其他作为参考

**多模型对比示例**：
| 渠道 | 末次归因 | 首次归因 | 线性归因 | U型归因 | 数据驱动 |
|------|----------|----------|----------|---------|----------|
| 信息流投放 | 35% | 18% | 25% | 28% | 24% |
| 搜索 | 28% | 22% | 25% | 26% | 27% |
| push | 22% | 12% | 18% | 16% | 19% |
| 自然量 | 15% | 48% | 32% | 30% | 30% |

### 第6步：指导资源分配
1. 按归因贡献率×渠道ROI排序，识别高价值渠道
2. 协助率高的渠道（种草型）即使末次归因低也要保留
3. 末次归因高但协助率低的渠道可能是"截胡"型，需评估真实贡献

### 第7步：持续监控和优化
1. 归因模型定期校准（季度）
2. 新渠道上线后单独追踪归因贡献
3. AB测试验证归因结论的因果性

## 高级玩法

### 1. 数据驱动归因（DDA）
- 用Shapley值：把每个渠道当作"玩家"，计算其边际贡献
- 用马尔可夫链：移除某渠道后转化率下降多少，即为该渠道贡献
- 需要足够路径数据（建议单渠道路径>1000），否则结果不稳定

### 2. 多触点协同分析
- 看渠道组合的协同效应：A+B一起触达的转化率是否高于A+B单独触达之和
- 识别"黄金组合"：哪些渠道组合的转化率最高
- 指导跨渠道联投策略

### 3. 归因+LTV结合
- 不仅看转化归因，还看各渠道带来用户的LTV
- 高LTV渠道即使获客成本高也值得投入
- 短期归因+长期LTV综合评估渠道质量

## 常见坑和避坑指南

| 坑 | 后果 | 避坑 |
|----|------|------|
| 只用末次归因 | 种草渠道被低估，预算全给临门一脚渠道 | 至少用U型或数据驱动做对照 |
| 归因窗口太短 | 漏掉早期触达，归因失真 | 根据转化周期定窗口，常见7-30天 |
| 触点定义不清 | 曝光和点击混算，贡献虚高 | 区分曝光/点击/互动，分别归因 |
| 忽略渠道协助 | 只看末次贡献，砍掉种草渠道后转化暴跌 | 看协助率，保留种草型渠道 |
| 数据量不足用DDA | 数据驱动归因结果不稳定 | 单渠道路径>1000才用DDA，否则用规则模型 |
| 归因结果不验证 | 把相关性当因果，预算调整后效果反降 | AB测试验证归因结论的因果性 |
| 跨设备路径断裂 | 同一用户多设备，路径被切断 | 用统一用户ID（账号）打通跨设备 |

## 实战案例：信息流push归因驱动触达策略优化

**背景**：浏览器信息流每日发3条push，但push的末次归因贡献只有8%，运营质疑push价值被低估，预算面临削减。需要重新评估push的真实归因贡献。

**分析过程**：
1. 明确归因目标：转化事件=用户打开信息流并消费内容，窗口=触达后24小时
2. 还原触点路径：采集用户24小时内所有触点（push触达、信息流曝光、搜索、推荐点击）
3. 多模型对比：
   - 末次归因：push仅8%（很多用户最后是被推荐触发）
   - 首次归因：push 35%（push是用户回访的首个触点）
   - U型归因：push 22%（首尾兼顾）
   - 数据驱动（Shapley）：push 26%（移除push后转化率下降26%）
4. 协助率分析：push协助率高达62%（push触达后用户被其他触点完成转化）
5. 渠道协同：push+推荐组合的转化率比单独推荐高40%
6. AB验证：停发push一组，转化率下降18%，确认push的因果贡献

**结论**：push的真实归因贡献是22-26%，远高于末次归因的8%。push是核心"种草"触点，协助率62%，停发会导致整体转化下降18%。策略：保留push预算，优化push触达时机（早晚高峰），push内容与推荐内容协同（push推热点，推荐深化消费），整体转化率提升12%。
