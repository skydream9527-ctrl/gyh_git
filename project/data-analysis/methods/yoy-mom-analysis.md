# 同比环比分析（YoY/MoM）

## 什么是同比环比
时间维度对比的基础方法，用于判断趋势、剔除季节性。

| 指标 | 定义 | 计算公式 | 用途 |
|------|------|----------|------|
| 环比（MoM/WoW/DoD） | 和上一个相邻周期比 | (本期值 - 上期值)/上期值 | 看短期变化、近期趋势 |
| 同比（YoY） | 和去年同一周期比 | (本期值 - 去年同期值)/去年同期值 | 剔除季节性/周期性，看真实增长 |

## 适用场景
1. ✅ 趋势判断：业务是在增长还是衰退？
2. ✅ 季节性剔除：今年Q4比Q3好，是季节性还是真增长？用同比验证
3. ✅ 目标达成评估：实际增速对比年度目标增速
4. ✅ 活动/改版效果评估：对比同环比变化，剔除季节性干扰
5. ✅ 周期性波动识别：哪些波动是周期性的，哪些是异常
6. ✅ 预算/预测基准：基于同环比做下一年/下一季度预测

不适用场景：
❌ 没有历史数据的全新业务（无法做同比，至少要1年数据）
❌ 业务模式发生重大变化的对比期（同比口径不可比）
❌ 强外部冲击期（如疫情爆发期，同期数据失真）

## 核心指标

| 指标 | 定义 | 计算方式 |
|------|------|----------|
| 同比增长率 | 和去年同期的相对变化 | (本期值 - 去年同期值) / 去年同期值 |
| 环比增长率 | 和上一周期的相对变化 | (本期值 - 上期值) / 上期值 |
| 同比增量 | 和去年同期的绝对差值 | 本期值 - 去年同期值 |
| 环比增量 | 和上一周期的绝对差值 | 本期值 - 上期值 |
| 复合增长率（CAGR） | N年间的年均增长率 | (期末值/期初值)^(1/N) - 1 |
| 定基指数 | 以某期为基期=100的相对值 | 本期值 / 基期值 × 100 |

## 怎么用
- **环比涨同比跌**：短期比上月好，但比去年差，说明季节性因素，长期趋势向下
- **环比跌同比涨**：短期回调但长期趋势向上，可能是正常波动
- **同环比都涨**：趋势向好
- **同环比都跌**：趋势恶化，需要重视

## 标准分析流程

### 第1步：选择对比周期
根据业务频次选择合适的同环比周期：
| 业务频次 | 环比周期 | 同比周期 |
|----------|----------|----------|
| 高频（日活/日收） | WoW（周环比） | 去年同周 |
| 中频（月活/月收） | MoM（月环比） | 去年同月 |
| 低频（季度/年度） | QoQ（季环比） | 去年同季 |
| 实时监控 | DoD（日环比） | 去年同日 |

### 第2步：确认数据可比性
1. **口径一致**：对比期的指标定义、统计口径、计算逻辑是否一致？
2. **业务可比**：对比期业务模式是否相似？（如期间有大改版/合并/拆分，需调整）
3. **日历对齐**：同比要剔除闰年、节假日错位（春节、中秋）影响
   - 月同比：今年1月vs去年1月，但春节可能在不同月
   - 周同比：用"去年同期同一周"避免星期错位

### 第3步：计算同环比指标
```sql
-- Presto/Kyuubi：信息流DAU月环比+同比
SELECT
  this_month.dt AS month,
  this_month.dau AS cur_dau,
  last_month.dau AS prev_dau,
  last_year.dau AS yoy_dau,
  round((this_month.dau - last_month.dau) * 100.0 / last_month.dau, 2) AS mom_pct,
  round((this_month.dau - last_year.dau) * 100.0 / last_year.dau, 2) AS yoy_pct
FROM (SELECT substr(dt,1,7) AS month, avg(dau) AS dau FROM dws_app_dau_d
       WHERE dt BETWEEN date'2026-06-01' AND date'2026-06-30' GROUP BY 1) this_month
LEFT JOIN (SELECT substr(dt,1,7) AS month, avg(dau) AS dau FROM dws_app_dau_d
       WHERE dt BETWEEN date'2026-05-01' AND date'2026-05-31' GROUP BY 1) last_month
  ON 1=1
LEFT JOIN (SELECT substr(dt,1,7) AS month, avg(dau) AS dau FROM dws_app_dau_d
       WHERE dt BETWEEN date'2025-06-01' AND date'2025-06-30' GROUP BY 1) last_year
  ON 1=1;
```

### 第4步：剔除季节性影响
1. **看同比为主**：同比剔除了年内季节性，反映真实增长
2. **季节性指数法**：计算每月相对年均的季节性指数，用实际值/指数做季节调整
3. **春节/节假日调整**：春节移动假日，按"春节前后N天"对齐而非按月对齐

### 第5步：同环比组合判断趋势
| 同比 | 环比 | 判断 |
|------|------|------|
| 涨 | 涨 | 趋势向好，加速增长 |
| 涨 | 跌 | 长期向好，短期回调（多为季节性） |
| 跌 | 涨 | 长期承压，短期有改善 |
| 跌 | 跌 | 趋势恶化，需重视 |

### 第6步：细分维度看结构
整体同环比可能掩盖结构差异，需要拆分：
- 按业务线/内容品类拆：是不是某块业务拉动/拖累？
- 按新老用户拆：新用户增长 vs 老用户留存，哪个在变化？
- 按渠道拆：哪个渠道贡献了增长/下跌？

### 第7步：输出结论和预测
- 趋势判断：增长/平稳/衰退
- 驱动因素：哪些业务/群体拉动，哪些拖累
- 预测：基于同环比趋势 + 季节性，预测下一周期

## 常见季节性场景
- 电商：Q4（双11/双12/年货节）是旺季，Q1是淡季
- 旅游/出行：节假日、暑期是旺季，工作日/寒假是淡季
- 工具产品：工作日活跃高，周末低
- 内容/娱乐产品：周末/节假日活跃高，工作日低

> 看趋势不能只看环比，必须结合同比，否则会把季节性当增长或下跌。

## 高级玩法

### 1. 滚动同环比（Rolling YoY/MoM）
- 不看单月，看"过去3个月滚动均值"的同比，平滑短期波动
- 适合波动大的指标，避免单月噪声误判

### 2. 定基指数化
- 以某期为基期=100，后续各期换算成指数
- 多个指标用同一基期对比，消除量纲差异
- 适合多业务线/多指标综合对比增长速度

### 3. 季节性分解（STL/Prophet）
- 用时间序列分解把指标拆成：趋势 + 季节性 + 残差
- 趋势看长期方向，残差看异常，比单纯同环比更准

### 4. CAGR与预测
- 用CAGR衡量长期复合增长速度
- 趋势外推 + 季节性叠加做下一年预测
- 注意：预测要给区间，不要给单点

## 常见坑和避坑指南

| 坑 | 后果 | 避坑 |
|----|------|------|
| 只看环比不看同比 | 把季节性上涨当增长，决策失误 | 同环比必须一起看 |
| 春节等移动假日不对齐 | 1月vs去年1月，但春节在不同月，同比失真 | 按"春节前后N天"对齐，或剔除春节周 |
| 闰年/月份天数差异 | 2月28天vs29天，环比失真 | 用日均值的环比，不用月总值 |
| 口径变更不调整 | 对比期口径变了，同环比不可比 | 历史数据回刷或标注口径变更点 |
| 业务结构变化不调整 | 今年新增了业务线，同比虚高 | 用可比口径（同口径对比） |
| 基期异常导致误判 | 去年同期有疫情/故障，基数太低，今年同比虚高 | 看多年同比，或用复合增长率 |
| 短期波动当趋势 | 单月环比跌就喊衰退 | 看至少3个月趋势 + 同比 |

## 实战案例：浏览器内容中心消费量同环比分析

**背景**：浏览器内容中心（聚合信息流+搜索+推荐的内容消费大盘）2026年6月人均内容消费量环比5月涨8%，但产品经理担心是暑期季节性，需要判断真实增长情况。

**分析过程**：
1. 选周期：月环比（6月vs5月）+ 月同比（6月vs去年6月）
2. 数据可比性：口径一致，2025-2026无大改版，日历对齐（6月都是30天）
3. 计算同环比：
   - 6月人均消费量：34.2条
   - 5月：31.7条 → 环比+7.9%
   - 去年6月：32.8条 → 同比+4.3%
   - 去年5月：30.1条 → 去年6月环比去年5月+8.9%
4. 季节性判断：去年6月环比5月也涨8.9%，说明6月环比上涨是季节性规律（暑期前活跃提升），并非今年产品改善
5. 同比拆解：
   - 信息流：同比+2%（成熟业务，季节性主导）
   - 搜索：同比+12%（搜索内容库扩展拉动）
   - 推荐频道：同比+25%（新推荐策略上线，3月后效果显现）
6. 结论：6月环比上涨主要来自季节性（占比约70%），真实增长主要来自推荐频道（同比贡献60%）

**结论**：不要把季节性当成果。真实增长动力是推荐策略升级，建议加大推荐算法投入；信息流作为成熟业务季节性主导，暑期后会回落，提前准备暑期内容运营保活策略。
