# 数据分析报告写作指南

> 结论先行，业务语言，每个结论有数据支撑。报告是给决策者看的，不是给自己看的。

## 报告结构（结论先行）

```
1. 核心结论（3 句话说清楚）
   - 发生了什么？
   - 原因是什么？
   - 建议怎么做？
2. 关键数据支撑（3-5 个核心数据）
3. 详细分析过程（按逻辑拆解）
4. 行动建议清单（优先级、预期效果、负责人方向）
5. 附录（原始数据、详细表格、SQL）
```

## 好报告 vs 坏报告

| 坏报告 | 好报告 |
|--------|--------|
| 堆数据和图表，看完不知道要说什么 | 结论先行，开门见山 |
| "建议优化用户体验"这种空话 | "建议优化 Android v3.2 支付流程，预计提升支付成功率 5%" |
| 只说"是什么"，不说"为什么"和"怎么办" | 现象→原因→建议完整链路 |
| 用复杂术语炫技 | 业务语言，所有人都能看懂 |
| 只有结论没有证据 | 每个结论都有数据支撑 |
| 篇幅冗长，重点淹没 | 结构清晰，核心结论一屏可见 |

## 写作要点

1. **先写结论再写过程**：决策者第一屏就要看到"所以呢"
2. **量化所有建议**：预期效果、影响范围、优先级都要有数字
3. **区分现象与原因**："转化率跌了"是现象，"支付渠道故障导致"才是原因
4. **附录放细节**：正文只放决策必需的信息，数据明细、SQL、长表格进附录
5. **适配读者**：给老板看结论和建议，给同行看方法和细节

## 常见错误

- 只描述数据不解读业务含义
- 结论含糊（"有一定提升"→ 改为"提升 12.3%"）
- 建议不可执行（"加强运营"→ 改为"对新用户首日 push 召回，预计提升 D1 留存 3%"）
- 忽略护栏指标（只说收益不说副作用）
