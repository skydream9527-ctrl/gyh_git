# DECISIONS.md — 决策记录

> 作用：沉淀重要选择、取舍和依据，让 AI 下次辅助规划/决策时不从零开始。重要决策做完后同步回写 `MEMORY.md`。

## 决策原则

1. 先判断是否影响 P0；影响 P0 的决策优先级最高。
2. 资源不足时先收缩范围，不扩大战线。
3. 没有数据支撑的判断必须标注「推断」。
4. 重要决策必须记录：背景、选择、依据、风险、回看时间。
5. 决策后的执行动作要进入 `WORK-PLAN.md` 或 `CURRENT.md`。

## 2026-07-01 工作全景 AI 可替代性分级与工作区执行台升级

### 背景
- 首次基于 3 份上半年工作台账（197 条事项）+ 59 份交付文档逐一分析，系统盘点真实工作构成。
- 需判断哪些工作优先交给 AI、哪些必须自己做，为 `nl-sql` 战略与时间分配提供依据。

### 选择
- 工作归纳为 6 大类 19 子类，按 AI 可替代性分🟢高(5)/🟡中(9)/🔴低(6)三级。
- 优先 AI 化🟢高可替代的取数、人群包圈选、看板播报、需求台账、周报。
- 保留🔴低可替代的异动归因、OKR/Roadmap、跨团队协调、人员管理为个人核心价值区。
- 工作区从「知识库 + 记录」升级为「执行台」：沉淀数据资产层（SQL 模板库 / 指标口径字典 / 人群包登记）服务 `nl-sql`。

- 历史经验：团队仅 2 人 + 杨沈君 6.15 离职，AI 替代是刚需。

### 风险
- 风险1：数据资产层与既有 P0「数据资产知识库建设」需统一，避免重复造轮子。
- 风险2：可替代性占比为推断值，需后续用实际工时校准。

### 后续动作

## 2026-07-01 `nl-sql` 范式库首批实战 case 选择

### 背景
- Q3 一号位目标是打透 `nl-sql` 自然语言取数场景，但抽象优化 prompt 或平台能力容易脱离真实业务。
- 今日已完成浏览器流量地图现状摸底、信息流有效 DAU/时长 4-6 月分天取数，并沉淀到知识库。
- 飞书今日新增 P0 数据需求（抖音上滑售卖跑数、push 增量价值分析、首调过滤实验）说明高频真实需求仍会持续涌入。

### 选择
- 选择将**浏览器流量地图**与**信息流有效 DAU/时长**作为 `nl-sql` 范式库首批实战 case。
- 后续优先围绕真实业务需求积累 SQL 范式、口径说明、badcase 和 few-shot 样例。
- 暂不单独投入复杂 RAG 或大规模平台化能力，除非能直接提升这些高频场景的准确率和交付效率。

### 依据
- 数据依据：今日产出 10 组流量地图 SQL、91 天有效 DAU/时长数据、CSV/XLSX、HTML、飞书归档、知识库案例；20260401 校验与飞书口径完全一致。
- 业务依据：流量地图、push 增量、抖音上滑跑数均属于高频、可范式化、业务方关注度高的取数/分析场景。
- 历史经验：2026-06-30 已确认 `nl-sql` 核心瓶颈是业务查询范式库，而不是先做复杂 workflow/RAG。

### 风险
- 风险1：临时 P0 需求过多，可能挤占范式库系统化整理时间，需要把需求交付同步转化为范式沉淀。
- 风险2：流量地图旧口径仍需新口径补跑校准，未更新前相关策略建议需标注口径风险。

### 后续动作
- [ ] 7/2 完成流量地图新口径 UV/时长补跑，更新飞书文档、HTML 和知识库案例。
- [ ] 将抖音上滑售卖跑数、push 增量价值分析纳入下一批 `nl-sql` 范式样例。
- [ ] 按「业务问题 → 口径 → SQL → 校验值 → 输出格式 → badcase」结构整理范式库索引。

### 回看时间
- 2026-07-15：检查首批范式是否能复用到 3 个以上真实取数需求，简单查询准确率是否提升。

## 决策回填闭环

> 目的：让决策不止于记录，而是能复盘学习、越来越准。到回看时间，AI 主动对照实际结果回填，形成贝叶斯式修正。这是「辅助决策」越用越准的关键一环。

**回填触发**：到达下表「回看时间」，或每周总结时扫描到期项。

**回填动作**：

1. 对照当初的结论和依据，看实际结果如何。
2. 判定：✅ 成立 / ⚠️ 部分成立 / ❌ 推翻，写清偏差原因。
3. 提炼一条可复用经验，同步到 `MEMORY.md` 或对应剧本/护栏。
4. 决策成立度低的，检查是不是护栏漏了某一条，补进 `00-知识库/decision-frameworks/methods/龚云荷决策护栏.md`。

**回填格式**（回看后在对应决策的 ADR 末尾追加）：

```markdown
### 回看结果（YYYY-MM-DD）
- 判定：✅ 成立 / ⚠️ 部分成立 / ❌ 推翻
- 实际结果 vs 预期：
- 偏差原因：
- 沉淀经验：
```

## 2026-07-07 建立「需求→资产」回流 + 决策回填闭环自动化

### 背景
- WORKFLOWS.md backlog P0 项：「从 DECISIONS.md 自动生成待回看提醒」一直未建设
- 数据资产层建设需要从真实需求交付中自动提取可复用资产，当前靠人工沉淀容易遗漏
- 决策系统越用越准的关键是回填闭环，但当前完全靠手动触发

### 选择
- 新增 `decision_review.py`：每日 09:00 cron，解析 DECISIONS.md 待回看表格，搜集证据生成回填简报，飞书提醒
- 新增 `asset_recall.py`：手动触发，扫描 03-数据分析/ 等目录，按信号关键词分类提取 SQL/指标/人群包/方法论，生成回填建议报告
- 复用现有模式：飞书推送（daily_metrics_report.py）、信号抽取（daily_work_review.py）、配置驱动

### 依据
- 数据依据：当前 DECISIONS.md 有 6 条待回看决策，其中 1 条已过期（工作区 Agent 化入口文件，7/6 到期）
- 业务依据：资产回流首次扫描发现 135 条建议（SQL 103、指标口径 18、方法论 14），说明有大量可复用资产待沉淀
- 历史经验：daily_work_review.py 的信号抽取模式已验证有效，可直接复用

### 风险
- 风险1：asset_recall.py 的 SQL 提取可能产生噪音（单个 CTE 被拆成多块），需后续优化正则
- 风险2：飞书 webhook 需要配置才能推送，否则只能生成本地报告

### 后续动作
- [ ] 配置飞书 webhook 到 decision_review_config.json
- [ ] 添加 crontab：`0 9 * * * run_decision_review.sh`
- [ ] 优化 asset_recall.py 的 SQL 提取正则，减少噪音

- 2026-07-14：检查自动化机制是否正常运行，决策回填是否覆盖到期项，资产回流建议质量
