# AI 写 SQL 提示词模板（基础 + 进阶）

> 用途：让 AI 帮你写取数 SQL，直接复制修改即可用。搭配模块3/4 的"5 要素提示词 + 4 步校验"。
> 核心理念（呼应全课）：**AI 写作业，你批改作业**——提示词说清楚 + 生成后校验，准确率能到 90%。

---

## 一、提示词 5 要素（说清这 5 点，AI 就不容易乱编）
1. **【表】** 用哪张表（写全表名）
2. **【字段】** 关键字段的含义、类型、单位、枚举值
3. **【需求】** 要算什么指标、按什么维度、什么时间范围
4. **【过滤】** 限定什么人群/场景
5. **【注意】** 口径要求（分区、去重、单位、除法、NULL 处理等）

---

## 二、基础模板（单表 / 简单查询，直接复制改需求）
```
请帮我写 Hive SQL，使用以下表：

表名：iceberg_zjyprc_hadoop.browser.dwm_browser_event_aggregation_miui_di
字段说明：
- date: 日期，格式yyyyMMdd，分区字段
- did: 设备ID，用户唯一标识
- is_app_dau_2024: =1表示有效活跃用户
- is_new_2024: =1新用户，=0老用户
- app_open_cnt: APP打开次数
- app_dura: APP使用时长，单位毫秒
- app_launch_way: 启动方式，枚举：点击icon/第三方调起/点击push/subscribe_push/...

表名：iceberg_zjyprc_hadoop.browser.dwm_browser_event_aggregation_label_di
字段说明：
- date: 日期分区；did: 设备ID
- is_dau_feed_dapan_2024: =1信息流活跃用户
- is_top: 0=自然推荐，1=置顶内容
- expos_cnt/click_cnt: 曝光/点击次数
- news_vv_cnt/short_vv_cnt/mini_vv_cnt: 图文/短视频/小视频VV
- feed_dura: 信息流消费时长，单位毫秒

我的查询需求：
【写清楚：要查的指标、时间范围、过滤条件、分组维度】

要求：
1. 必须加 date 分区过滤，必须加 is_app_dau_2024=1（或 is_dau_feed_dapan_2024=1）过滤有效用户，必须过滤 COALESCE(did,'')!=''
2. 计算用户数用 COUNT(DISTINCT did)
3. 除法乘 1.0 避免整数除法，保留合适小数位
4. 时长是毫秒，转分钟除以 60000
5. 自然信息流查询加 is_top=0
6. 代码加必要注释
```

## 三、进阶模板（复杂 / 多表，拆步骤写，准确率提升 80%）
```
请帮我写 Hive SQL，需求如下：

【使用的表结构】：
1. 表名：...miui_di    粒度：did+date（每个用户每天一行）
   核心字段：date(分区,yyyyMMdd)、did、is_app_dau_2024、is_new_2024、
            app_launch_way(枚举:点击icon/第三方调起/点击push/subscribe_push/新全搜调起)、
            app_open_cnt、app_dura(毫秒)
2. 表名：...label_di   粒度：did+date
   核心字段：date、did、is_dau_feed_dapan_2024、is_top(0自然/1置顶)、
            expos_cnt、click_cnt、news/short/mini_vv_cnt、feed_dura(毫秒)

【实现步骤】：（把需求拆成 1/2/3/4 步，每步说明逻辑）
步骤1：xxx
步骤2：xxx
步骤3：xxx

【指标口径要求】：
- 用户数统一 COUNT(DISTINCT did)
- 比率乘 1.0、保留 4 位小数
- 时长毫秒 → 分钟除以 60000
- LEFT JOIN 后右表字段用 COALESCE(字段,0) 把 NULL 转 0
- 必加 date 分区、必加有效用户过滤、必过滤空 did
- 自然信息流加 is_top=0
- 多天关联用 date+did 两个 key

【输出要求】：
1. 用 WITH CTE 拆分步骤，每个临时表加注释
2. 核心计算逻辑加注释
3. 结果按日期/分组维度排序
```

## 四、常见场景提示词示例

### 场景1：PUSH 用户留存分析
```
查 2026-06-01 到 2026-06-30，每日点击 PUSH 启动的用户的次日/7日/14日留存率，对比同期全体 APP 用户留存。
留存定义：D0 活跃，Dn 仍为 APP 活跃用户(is_app_dau_2024=1)即 Dn 留存。
步骤1：取每日全体活跃并标记是否 PUSH 启动 → D0 标签表
步骤2：取更长区间的全量活跃 → 留存活跃表
步骤3：关联算 D1/D7/D14 是否仍活跃
步骤4：按 D0 日期 + 用户组聚合算留存率
```

### 场景2：指标波动排查
```
排查 2026-06-28 信息流 CTR 下降：按新老用户、按启动方式、按信息流频道分别算 CTR，
对比 2026-06-27，看哪个维度下降最多。
```

## 五、SQL 报错 / 结果异常排查提示词
```
我写的 SQL 有问题，请帮我排查：
【SQL 代码】：[贴你的 SQL]
【问题描述】：（报错？慢？还是结果不对？偏大还是偏小？和预期差多少？）
  例：跑出 2026-06-28 的 DAU 是 8000 万，比平时 4000 万高一倍，哪里错了？
【期望】：告诉我问题在哪一行、为什么错、给出修正后的 SQL
```

## 六、AI 生成后必做的 4 步校验
| 步 | 查什么 |
|----|--------|
| ① 字段校验 | 对照《核心表速查手册》，AI 有没有编不存在的字段 |
| ② 分区校验 | WHERE 有没有加 date |
| ③ 逻辑校验 | 去重了吗、除法 *1.0 了吗、时长 /60000 了吗、NULL 处理了吗 |
| ④ 结果校验 | 数量级合理吗（DAU 百万级、CTR 百分之几），和看板/历史对得上吗 |

> 🎯 记住：AI 生成的 SQL，**简单单表校验后可用，复杂多表一定要先查 1 天验证再放大**。
