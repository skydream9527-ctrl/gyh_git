# 启动课（第0课时）：数据是怎么来的 —— 从一次点击到一张表

> 课时：1小时
> 目标：在学 SQL 之前先打地基。理解「表」到底长什么样、SQL 执行时真正在操作什么、数据是怎么从用户的一次点击变成一张可查询的表。学完这节课，后面找表、写 SQL、看数据都会有踏实的底层认知。

---

## 一、开场：我们天天说的「数据」，到底长什么样？

先问三个问题，看看你能不能答上来：

1. 你写 `SELECT ... FROM 某张表` 取数时，这张「表」到底是什么东西？
2. 表里的数据是从哪来的？是谁、在什么时候、怎么放进去的？
3. 为什么查询一定要加 `date` 分区？为什么同一个人会有很多行？

如果这些问题答不太清楚，SQL 学起来就会「知其然不知其所以然」——照着模板改能跑，但一遇到报错或结果异常就不知道从哪查。

> **这节启动课不写 SQL**，只做一件事：把地基打好。搞清楚三件事——① 表是什么 ②数据怎么来的 ③ 数据是怎么一行行记录的。有了地基，后面每一课的内容都会顺很多。

**这节课的主线：**
```
先认识「表」  →  搞懂我们用的是什么表  →  看数据怎么从点击变成表  →  读懂一张真实的表
```

## 二、先认识「表」：一切数据的载体

### 2.1 表就是一张二维表格（就像 Excel）

不管是 Oracle、MySQL 还是我们公司用的大数据表，本质都一样：**数据存在一张张二维表格里**。你完全可以用 Excel 来理解它。

看一个最简单的「用户表」：

| user_id | 昵称     | 城市   | 注册天数 | 是否会员 |
|---------|----------|--------|----------|----------|
| 1001    | 小米粥   | 北京   | 320      | 1        |
| 1002    | 阿强     | 上海   | 45       | 0        |
| 1003    | Lucy     | 广州   | 128      | 1        |

拆开看，三个最基本的概念：

| 概念 | 英文 | 是什么 | 在上表里 |
|------|------|--------|----------|
| **行** | row / record | 一条记录，代表一个具体对象 | 一行 = 一个用户 |
| **列** | column / field | 一个字段，代表对象的某个属性 | user_id、昵称、城市… |
| **单元格** | cell | 某行某列交叉处的一个具体值 | 第1行「城市」= 北京 |

> 💡 **一句话记住**：**一行是一条记录，一列是一个属性（字段），SQL 里 `SELECT` 后面跟的就是「列名」。**

### 2.2 每一列都有「数据类型」

Excel 里一个单元格你想填啥填啥，但数据库里**每一列都规定死了只能放某一种类型的数据**。为什么要管这个？因为**类型决定了这列能做什么运算**。

| 类型大类 | 常见写法（了解即可） | 举例 | 能做什么 |
|----------|---------------------|------|----------|
| 整数 | INT、BIGINT | 注册天数 320、点击次数 5 | 能求和 SUM、求平均、比大小 |
| 小数 | DECIMAL、DOUBLE | 时长 35.5、金额 9.9 | 能算率、算人均 |
| 字符串（文本） | VARCHAR、STRING | 昵称、城市、启动方式 | 不能求和，只能等值/模糊匹配 |
| 日期时间 | DATE、DATETIME、TIMESTAMP | 2026-06-28、注册时间 | 用日期函数算区间、算间隔 |
| 布尔/标记 | 常用 0/1 表示 | 是否会员=1 | 用来做筛选、打标 |

> ⚠️ **这解释了两个后面会遇到的现象**：
> - 「时长」「次数」是数字类型，所以能 `SUM` 求总量；「城市」「启动方式」是字符串，只能拿来筛选和分组，不能求和。
> - 有的表里 `date` 存成数字 `20260628`，有的存成字符串 `'2026-06-28'`，写 SQL 时要对应着来，这就是类型在起作用。

### 2.3 主键（Primary Key）：一行的「身份证」

**主键 = 能唯一认出某一行的字段，不能重复、不能为空。**

上面的用户表里，`user_id` 就是主键：每个用户一个唯一 ID，靠它就能精确找到某一行。昵称不行（可能重名），城市更不行（一堆人同城）。

主键有三个作用：

| 作用 | 说明 | 对应后面学的 |
|------|------|-------------|
| ① 唯一定位一行 | 拿 user_id=1002 就能精确找到阿强这行 | WHERE 精确查询 |
| ② 保证不重复 | 同一个 user_id 不会出现两次，避免重复计数 | 为什么算人数要去重 |
| ③ 让不同表能「对上」 | 用户表的 user_id 和订单表的 user_id 一对应，两张表就能拼起来 | 模块4 的 JOIN 关联 |

**联合主键：几个字段一起当身份证**

有时候一个字段不够用，需要**几个字段组合起来才能唯一确定一行**，这叫联合主键。

> 举个后面天天见的例子：我们的核心表是「**一个用户一天一行**」，那么单看 `did`（设备号）不够（同一个人有很多天的数据），单看 `date` 也不够（同一天有几百万人）。**必须 `did + date` 两个字段合起来**，才能唯一定位「某个人在某一天」的那一行。这个「`did + date` 决定一行」，就是这张表的**粒度**。

> 💡 记住「**粒度**」这个词，后面模块会反复出现。粒度 = 这张表里「一行代表什么」。

### 2.4 顺带认识：外键、索引（了解即可，不用深究）

| 概念 | 一句话理解 | 类比 | 对应后面 |
|------|-----------|------|----------|
| 外键 | 一张表里引用了另一张表的主键，作为两表关联的依据 | 订单表里存了下单人的 user_id | JOIN 靠什么字段关联 |
| 索引 | 给某列建一个「目录」，查找时不用翻全表，直接定位 | 字典的偏旁部首检字表 | 大数据表里用「分区」起类似作用 |

> 💡 **索引这个概念先记一下**——它是「加速查找」的手段。等下我们会看到，我们用的大数据表不靠索引，而是靠「**分区**」来加速，这就是为什么模块2/3 反复强调「查询必须加 `date` 过滤」。

## 三、Oracle、MySQL 和我们用的大数据表，是什么关系？

你可能听过 Oracle、MySQL，它们和我们查数用的表有啥区别？先说结论：**「表、行、列、主键、SQL」这些概念是完全通用的，学会一个，其余触类旁通。** 区别在于「用来干什么」。

### 3.1 常见的数据库都是「关系型数据库」

Oracle、MySQL、PostgreSQL、SQL Server…… 这些都叫**关系型数据库**，共同点：数据都存在一张张有行有列的表里，都用 SQL 来增删改查。

| 数据库 | 特点 | 谁在用 |
|--------|------|--------|
| **MySQL** | 免费开源、轻量、最流行 | 绝大多数互联网 APP 的业务系统 |
| **Oracle** | 商业收费、稳定、功能强 | 银行、电信等大型传统企业 |
| PostgreSQL | 开源、功能强大 | 越来越多互联网公司 |

> 💡 对产品同学来说：**Oracle 和 MySQL 的「表结构、主键」概念一模一样，SQL 语法有些小差异（比如日期函数、分页写法不同），日常不用纠结区别。** 记住「它们都是把数据放在表里、用 SQL 操作」就够了。

### 3.2 关键区别：业务数据库 vs 数据仓库

这是这节课**最重要的一个认知**。虽然都叫「数据库、都是表」，但有两类，用途完全不同：

- **业务数据库**（Oracle / MySQL 这类）：**支撑 APP 实时运行**。你注册、登录、下单、发消息，数据实时写进去、随时被改。
- **数据仓库**（我们查数用的，底层是 **Hive / Iceberg** 这类大数据表，通过 **Kyuubi** 用 SQL 查询）：**专门存历史数据用来做分析**。

对比一下就清楚了：

| 对比项 | 业务数据库（Oracle/MySQL） | 数据仓库（我们用的 Hive/Iceberg） |
|--------|---------------------------|----------------------------------|
| 干什么用 | 支撑 APP 跑起来（下单、登录、发消息） | 存历史数据做分析、看报表 |
| 谁在操作 | 程序自动实时读写 | 数据/产品/分析同学写 SQL 查 |
| 数据量级 | 千万 ~ 亿级 | **百亿 ~ 万亿级** |
| 主要操作 | 增、删、改、查（实时改） | **几乎只有「查」，基本不改** |
| 有没有主键 | 有，强制唯一 | **一般没有强制主键**，靠业务字段+分区保证粒度（如 did+date） |
| 怎么加速 | 靠索引 | **靠分区**（按 date 一天一个区） |
| 数据怎么进去 | 一条条实时写入/修改 | **一批批追加，T+1（今天的数据明天凌晨才有）** |

**一个类比记住它俩的区别：**

> 🏦 业务数据库像「**实时记账本**」——每笔交易马上记上、随时能改、要求分毫不差。
> 📷 数据仓库像「**每天拍照存档的相册**」——每天给全站数据拍一张「快照」存起来，存下来就不改了，主要拿来回看和分析。

> ⚠️ **这个区别直接解释了模块2/3 的几条「铁律」，现在你就能理解为什么了：**
> - **为什么查询必须加 `date` 分区？** 因为数仓靠分区加速，一天的数据放一个区，加了 date 才只扫那一天；不加就扫全部几百亿行，几小时跑不出来。
> - **为什么表的粒度是 did + date？** 因为它没有传统主键，用「设备+日期」这个联合来标识唯一一行。
> - **为什么是 T+1、今天看不到今天的数据？** 因为它是「每天拍一张快照」——要等一整天的行为都产生完，凌晨批量加工，第二天才出。

## 四、数据是怎么来的：从一次点击到一张表（数据的一生）

现在最关键的问题来了：**表里那些数据，到底是怎么进去的？** 完整链路是这样的：

```
用户操作        ①埋点采集        ②日志上报        ③数据加工          ④生产成表        你写 SQL
点一下 / 滑一下 → SDK 记录行为 → 打包上报到服务器 → 清洗 / 去重 / 聚合 → 分层存成一张张表 → 从数据地图找到表取数
（用户）         （客户端/服务端）  （原始日志）      （数据同学的调度任务）  （ODS→DWD→DWM→ADS）  （产品同学）
```

下面拆开讲三个关键环节：**采集 → 加工 → 生产成表**。

### 4.1 第一步：埋点采集 —— 数据的源头

**埋点 = 在 APP 里预先埋好的「探针」，用户一有动作，就记录下来。**

用户在浏览器信息流里点了一条新闻，背后发生了什么：

```
用户点击新闻 → 埋点 SDK 立刻捕获这个动作 → 打包成一条「行为记录」→ 上报到服务器
```

**一条埋点记录了什么？**（回答四个问题）

| 问题 | 记录内容 | 举例 |
|------|----------|------|
| 谁 | 设备号 did、用户 id、版本、渠道 | did = d_1001 |
| 什么时候 | 精确到毫秒的时间戳 | 2026-06-28 20:33:41 |
| 做了什么 | 行为类型 + 在哪个页面/位置 | 在信息流点击了第3条 |
| 对象是什么 | 内容 id、标题、类型、位置 | 内容id=888，时政类，第3位 |

**两种采集方式**（了解即可）：

| 类型 | 在哪采集 | 记什么 | 举例 |
|------|----------|--------|------|
| 客户端埋点 | APP 里的 SDK | 界面上的行为 | 曝光、点击、播放、停留时长 |
| 服务端埋点 | 后台服务器 | 后台发生的事 | PUSH 下发、内容审核结果、支付成功 |

> 💡 为什么要分两种？因为有些行为只有客户端知道（用户划到哪了、停留多久），有些只有服务端知道（PUSH 到底发出去没、有没有到达）。**做 PUSH 分析时，「下发量」来自服务端、「点击」来自客户端**，这就是原因。

### 4.2 第二步：日志的加工 —— 从「杂乱原始记录」到「干净明细」

埋点上报后，最先落地的是**原始日志（log）**。原始日志很「脏」，大概长这样（一条点击日志的示意）：

```json
{"did":"d_1001","ts":1782648821000,"ev":"click","pg":"feed","item":"888","ver":"15.2.0","ext":{"pos":3,"type":"news"}...}
```

原始日志的问题：字段名是缩写、格式不统一、有重复上报、有测试设备和异常数据、嵌套很深不好用。所以需要**加工**。数据同学写好自动调度任务（每天凌晨自动跑），做这几件事：

| 加工动作 | 做什么 | 为什么 |
|----------|--------|--------|
| 解析 | 把嵌套的 json 拆成一个个规整的字段 | 方便用 SQL 按列查 |
| 清洗 | 去掉测试设备、异常值、无效数据 | 保证数据干净可信 |
| 去重 | 同一个行为上报了多次，去掉重复 | 避免指标虚高 |
| 统一口径 | 时间统一格式、字段统一命名、枚举值统一 | 全公司口径一致 |

加工完，就得到一张**明细表**：**一行 = 一次行为**（谁在几点几分点了哪条内容）。这就是数仓里的 **DWD 明细层**。

### 4.3 第三步：表的生产 —— 从明细到汇总，一层层加工上来

问题来了：明细表**一天就有几十上百亿行**（全站每个人每次点击都是一行），直接查它又慢又贵。所以要**再往上聚合**。

这就是「数仓分层」表的由来，现在你能理解每一层是怎么来的了：

| 层级 | 全称 | 一行代表什么 | 怎么来的 | 产品常用度 |
|------|------|-------------|----------|-----------|
| **ODS** | 操作数据层 | 一条原始日志 | 埋点日志原样同步进来 | ❌ 太原始，不用 |
| **DWD** | 明细层 | **一次行为** | ODS 清洗、解析、去重而来 | ⚠️ 太细太大，复杂分析才用 |
| **DWM** | 中间汇总层 | **一个人一天**（did+date） | DWD 按「人+天」聚合而来 | ✅ **日常取数首选** |
| **ADS** | 应用层 | 一个指标一天（如大盘 DAU） | DWM 再高度汇总而来 | ✅ 看固定报表用 |

**聚合到底在干什么？看个例子就懂：**

某个用户 `d_1001` 在 6月28日 这天启动了 5 次 APP、一共用了 35 分钟。在**明细层**里，他的每一次启动都是一行（5 行）；聚合到 **DWM 层**后，就变成**一行**：

```
明细层（DWD）· d_1001 在 20260628 的启动行为 —— 5 行：
  d_1001 | 08:12:33 | 启动 | 点击icon
  d_1001 | 12:05:10 | 启动 | 点击push
  d_1001 | 18:20:55 | 启动 | 点击icon
  d_1001 | 20:33:41 | 启动 | 点击icon
  d_1001 | 22:47:02 | 启动 | 点击icon
                    ↓  按「人+天」聚合（数一数、加一加）
汇总层（DWM）· d_1001 在 20260628 —— 1 行：
  d_1001 | 20260628 | app_open_cnt=5 | app_dura=2100000(毫秒，=35分钟) | ...
```

> 💡 **看懂这个例子，你就理解了两件大事**：
> - 「聚合」就是把很多行明细，按某个粒度（人+天）**数一数、加一加，压成一行汇总**。模块3 要学的 `COUNT`（数一数）、`SUM`（加一加）干的就是这事。
> - 我们日常查的 DWM 表已经帮我们聚合好了，所以「一个人一天就是一行」，性能好、字段全，不用自己从几百亿行明细里去算。

> ⚠️ **T+1 的由来**：因为要等 6月28日 一整天的行为全部产生完，凌晨才能把这一天的明细聚合成汇总。所以 6月28日 的数据，得 6月29日 凌晨才跑出来——这就是「今天看不到今天数据」的原因。

## 五、看一张真实的表：数据到底是怎么记录的

理论讲完，我们看一张**真实会用到的表**长什么样。以模块2 的核心表 `dwm_browser_event_aggregation_miui_di`（APP 整体行为汇总表）为例（以下为示意数据）：

| did | date | app_ver | app_launch_way | app_open_cnt | app_dura | is_app_dau_2024 |
|-----|------|---------|----------------|--------------|----------|-----------------|
| d_1001 | 20260628 | 15.2.0 | 点击icon | 5 | 2100000 | 1 |
| d_1002 | 20260628 | 15.1.8 | 点击push | 2 | 660000 | 1 |
| d_1003 | 20260628 | 15.2.0 | 第三方调起 | 1 | 90000 | 1 |
| d_1001 | 20260629 | 15.2.0 | 点击icon | 3 | 900000 | 1 |

### 5.1 逐行读懂它

- **第1行**：设备 `d_1001` 在 `20260628` 这天，用的是 15.2.0 版本，从桌面 icon 启动，一共打开 5 次、用了 2100000 毫秒（÷60000 = **35 分钟**），是当日活跃用户。
- **第2行**：设备 `d_1002` 这天是从 **点击 PUSH** 进来的，用了 11 分钟。
- **第4行**：注意，又是 `d_1001`！但 `date` 是 `20260629`——**同一个人的第二天，是另外一行**。

### 5.2 三个关键认知（这节课的核心收获）

| 问题 | 答案 |
|------|------|
| 这张表的**粒度**是什么？ | **did + date**，即「一个人一天一行」 |
| **一行**代表什么？ | 某个设备在某一天的行为汇总 |
| 为什么同一个 `did` 有多行？ | 因为**不同日期各一行**，d_1001 有 28 号、29 号两行 |
| 为什么 `app_dura` 是个大得离谱的数？ | 因为单位是**毫秒**，转分钟要 ÷60000 |

> 💡 这也解释了模块3 的一条铁律：**算「有多少活跃用户」必须 `COUNT(DISTINCT did)` 去重**——因为一个人跨多天有多行，直接数行数会把同一个人算很多次。

### 5.3 回到 SQL：现在再看这条查询，你就知道它在干嘛了

后面模块3 你会写这样一条最基础的 SQL，现在我们**不学语法，只把它翻译成人话**：

```sql
SELECT date, COUNT(DISTINCT did) AS dau      -- 要什么：日期，和「去重后的人数」
FROM   dwm_browser_event_aggregation_miui_di  -- 从哪张表：APP行为汇总表
WHERE  date = 20260628                        -- 筛哪些行：只要 6月28日 这天的
  AND  is_app_dau_2024 = 1                     --          且是当日活跃用户
GROUP BY date;                                 -- 按什么归堆：按天汇总
```

对着上面那张表，这条 SQL 做的事就是：

> 从「一人一天一行」的表里，**先筛出** `date=20260628` 且 `is_app_dau_2024=1` 的所有行，再把这些行里**不同的 did 数一数（去重计数）**，得到「6月28日这天有多少个活跃用户」。

**SQL 的本质，就这么三件事**（在一张二维表上）：

| SQL 关键词 | 在做什么 | 大白话 |
|-----------|----------|--------|
| `WHERE` | 筛选行 | 我只要符合条件的那些行 |
| `GROUP BY` | 把行分组 | 按天/按人群把行归堆 |
| `COUNT / SUM / AVG` | 对每堆做计算 | 数一数、加一加、算平均 |

> 💡 看懂了吗？**SQL 不神秘，它就是在一张二维表上做「筛选 + 分组 + 计算」。** 这就是模块3 要正式开始学的东西——现在你已经知道它在操作什么了。

### 5.4 一个重要理念：以后 SQL 让 AI 写，你负责「检查」

你可能在想：「我 SQL 基础差，学得会吗？」——放心，这门课要给你的是一个更省力的定位：

> 🤖 **日常取数的 SQL，以后大多可以让 AI 帮你写**（模块3、4 会专门教怎么让 AI 写 SQL）。**但 AI 经常犯错，而且自己发现不了，得靠你来检查。**

AI 写 SQL 最容易犯这几类错，恰好都是这门课要教你识别的：

| AI 常犯的错 | 具体表现 | 靠什么抓出来 |
|-------------|----------|-------------|
| 幻觉字段 | 编一个表里根本不存在的列名 | 懂表结构、会查数据地图（模块2） |
| 错口径 | CTR 分母用错（PUSH 该用到达、不是发送） | 懂业务口径（模块2、3） |
| 漏分区 | 忘了加 `date` → 扫全表跑不出来 | 记得"数仓靠分区"（本课第三节） |
| 粒度错 | 算人数漏 `DISTINCT` → 结果翻倍 | 懂"一人一天一行"的粒度（本课） |
| 单位错 | 时长是毫秒没 `/60000` → 结果大 6 万倍 | 懂字段单位（本课表样例） |

> 🎯 **所以这门课不是教你当「埋头写 SQL 的工人」（那部分 AI 会越来越强），而是教你当「验收 SQL 的监理」——看得懂、查得出错、判断结果对不对。**
>
> **一句话：AI 写作业，你批改作业。** 而要能批改，你得比 AI 更懂业务、更懂表、更懂口径——这恰恰是启动课要打的地基，也是后面每一课的落点。

## 六、本节课小结

1. **表 = 行（记录）+ 列（字段）**，每列有固定的数据类型；**主键**唯一标识一行，几个字段合起来当主键就叫「粒度」。
2. Oracle、MySQL、我们用的大数据表，**「表/主键/SQL」概念完全通用**；关键要分清两类用途：**业务库像实时记账本，数仓像每天拍照存档的相册**。
3. 我们分析用的是**数据仓库**，所以：**查询必加 date 分区**（靠分区加速）、**粒度是 did+date**（没有传统主键）、**T+1 更新**（每天批量拍快照）。
4. **数据的一生**：用户操作 → 埋点采集 → 日志加工（清洗/去重/统一口径）→ 分层聚合成表（ODS→DWD→DWM→ADS）→ 数据地图可查。
5. 读懂一张表要能回答：**粒度是什么、一行代表什么、字段怎么算来的**。
6. **SQL 的本质 = 在二维表上做「筛选（WHERE）+ 分组（GROUP BY）+ 计算（COUNT/SUM）」**。

> 🎯 **一句话总括**：数据是「用户行为被埋点记录下来、经过加工聚合、存成一张张按天拍照的表」，而 SQL 就是从这些表里筛选、分组、算数。地基打好了，下面开始真正上手。
