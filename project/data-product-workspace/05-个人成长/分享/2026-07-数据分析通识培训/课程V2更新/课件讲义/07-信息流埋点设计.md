# 模块7：信息流埋点设计

> 课时：1.5小时 | 层级：⭐ 专业
> 目标：理解数据的源头——会设计信息流埋点（表从哪来、四核心事件、时长原理、分题材差异、埋点到指标闭环、验收避坑）。
> 👉 面向产品（尤其信息流方向）；销售/运营了解"数据从埋点来、口径在埋点就定了"即可。

---

## 一、开场：产品是埋点需求的源头

数据的一生第一步就是埋点（启动课讲过）。**埋点设计错了，后面再牛的分析都是错的**（GIGO，垃圾进垃圾出）。「一次曝光怎么算」「时长从哪到哪」这些是**业务定义**，得产品想清楚。

---

## 二、你在数据工坊查的表，数据从哪来？（现场高频疑问）

培训现场有同学连着问：**"表里的字段都是 onetrack 打点来的吗？""有些数据我没埋点，是不是也能从数据工坊拿到？""埋点平台不是只有 Winchat 吗？"** 这里统一讲清楚。

**数据链路（从源头到你查的字段）：**

```
埋点（客户端 onetrack + 服务端埋点）   ← 数据的源头，产品定义
      ↓ 上报
日志层（onetrack 原始日志，最底层）
      ↓ 清洗 / 加工
ODS 层 → DWM 聚合表（你在数据工坊常查的 label_di / miui_di）
      ↓
你 SELECT 出来的那些字段
```

**几个要点（正面回答现场疑问）：**
- **表里能查到的数据，一定来自埋点**（客户端 onetrack 或服务端埋点），不是凭空生成的。**没埋点，就没这个数**。
- 但**埋点上报的不一定都进表**——数仓只加工了其中一部分常用的。想要的字段没有，往往是"没埋"或"没加工进这张表"。
- 埋点平台**不只 Winchat**：还有**服务端埋点**。onetrack 是**日志层**的东西，不等于你查的业务表——它是这些表的上游源头。
- 所以"这个指标能不能取到"，本质是"产品有没有埋这个点、数仓有没有加工进表"。这也正是本模块的意义。

> 🎯 一句话：**数据工坊的表 ← 数仓加工 ← 日志层 ← 埋点（你设计的）**。你在模块2-4 查的每个字段，源头都在这一节。

---

## 三、信息流业务与四核心事件

**通用结构**：频道 → 信息流列表 → 内容卡片 → 详情页/播放器。埋点就埋在这四层。

**四个核心事件**（抓住这四个抓住 80%）：

| 事件 | 定义 | 关键 |
|------|------|------|
| **曝光 expose** | 卡片被看到 | 定**有效曝光**（露出≥50% 且停留≥500ms），避免快速划过也算 |
| **点击 click** | 用户想看 | CTR = 点击/曝光 |
| **进入 enter** | 成功进入详情/播放，开始消费 | 和点击拆开：点了不一定进得去（加载失败/秒退），记**进入时间戳** |
| **退出 exit** | 离开详情/播放 | 记**退出时间戳** + 消费进度 |

消费链路：`曝光 → 点击 → 进入 →（消费）→ 退出`，本身是个漏斗。

---

## 四、时长怎么来：进入 + 退出（重点）

**消费时长 = 退出时刻 − 进入时刻**（算出来的，不是直接采的）。

**常见坑**：
- 只进入没退出（崩溃/切后台）→ 时长丢失或异常 → 用**心跳埋点**（每 15 秒上报"还在看"）+ 超时截断兜底
- 切后台算不算 → 一般算**前台有效时长**
- 多次进出同一内容 → 一般累加

> 💡 表里的 `feed_dura`/`consum_dura`/`app_dura`，就是这样采集聚合出来的（单位**毫秒**，`/60000` 转分钟）。现场也澄清过：`feed_dura` 含列表时长，`app_dura` 是 APP 整体时长，口径不同、值也不同，别混用。

## 五、分内容题材具体化（图文/小说/短视频/短剧）

四种题材**共用一套事件骨架**，靠 `content_type` 区分，只在细节上具体化：

| 维度 | 图文 | 小说 | 短视频 | 短剧 |
|------|------|------|--------|------|
| 进入= | 打开文章 | 打开阅读器(章节) | 开始播放 | 播放某一集 |
| 时长 | 阅读时长 | 阅读时长(跨章) | 播放时长 | 观看时长(跨集) |
| 特有事件/参数 | 翻页、是否读完 | 章节切换、订阅、付费解锁 | 播放进度、完播、点赞 | 剧id、集数、连播、付费解锁 |
| 核心指标 | 阅读VV、完读率 | 章节阅读数、追更率 | VV、完播率 | 追剧率、集均完播、付费解锁率 |

## 六、从埋点到指标：闭环

| 指标 | 来自哪些埋点 | 对应字段 |
|------|-------------|----------|
| 曝光量 | 有效曝光计数 | `expos_cnt` |
| CTR | 点击/曝光 | `click_cnt` |
| VV | 有效进入/播放计数 | `news/short/mini_vv_cnt` |
| 消费时长 | Σ(退出−进入)+心跳 | `feed_dura` |
| 信息流活跃(DAU) | 曝光+点击+播放+进空间次数>0 且非置顶 | `is_dau_feed_dapan_2024` |
| 完播/完读率 | 播放/阅读进度 | 专门口径 |

> 🎯 **你设计的埋点，就是前面模块查的那些数的源头。** 前面学"用数据"，这节学"数据怎么来"——闭环了。

**真实口径示例 `is_dau_2024`**（现场从"产出信息"里查到的加工逻辑）：`曝光次数 + 点击次数 + 视频播放次数 + 进入空间次数 > 0 且 is_top = 0（非置顶）` 才记为当日信息流活跃。一个看似简单的"活跃"标记，背后其实是**好几个埋点事件的组合口径**——这也说明为什么"口径"必须回到埋点去对，遇到看板对不上就查产出信息 / 找研发。

---

## 七、埋点需求文档 + 验收

**埋点需求表**（产品交给研发/数据）：事件名 | 触发时机 | 参数 | 类型 | 是否必填 | 说明。
**协作链**：产品(定需求) → 数据(口径/命名/元数据) → 研发(实现上报) → 测试验收 → 上线。

**六个高频坑**：重复上报、漏报、曝光未去重、时长异常(只进不出)、口径不一致、关键参数缺失(如没埋 content_type)。
**验收清单**：事件触发吗 / 参数齐吗 / 值对吗 / 去重生效吗 / 时长合理吗 / 小流量试跑指标在合理区间吗。

---

## 本节课小结
1. 产品是埋点源头，设计错→分析全错（GIGO）
2. **你查的表 ← 数仓 ← 日志层(onetrack) ← 埋点（客户端+服务端）**；没埋点就没数据
3. 四核心事件：曝光/点击/进入/退出；点击与进入要拆开
4. 时长 = 退出−进入，注意心跳兜底
5. 四题材共用骨架、靠 content_type 区分
6. 埋点→指标闭环（is_dau 是多事件组合口径）；六大坑 + 上线前验收
