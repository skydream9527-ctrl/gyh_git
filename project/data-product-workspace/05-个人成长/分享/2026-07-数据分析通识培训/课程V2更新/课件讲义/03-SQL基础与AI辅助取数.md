## 一、开场：SQL 没那么难

SQL = 和数据库对话的语言。产品/运营学 SQL 只要三件事：
- **看得懂**：别人的 SQL 在查什么
- **写得出**：日常取数的简单查询
- **会校验**：AI 生成的 SQL 对不对

> 💡 SQL 没有循环、判断这些复杂逻辑，写错最多报错、不会改坏数据（你只有查询权限）。放心大胆试。

## 二、SQL 五段式结构（固定顺序）

```sql
SELECT    要什么字段 / 算什么指标
FROM      从哪张表查
WHERE     筛选什么条件的数据
GROUP BY  按什么维度分组   -- 要分天/分群才加
ORDER BY  按什么排序       -- 可选
```

就像点菜：SELECT（要什么菜）FROM（哪家店）WHERE（什么口味）GROUP BY（按菜系归类）。

---

## 三、SELECT + FROM：查什么、从哪查

```sql
SELECT did, app_ver, app_dura
FROM   iceberg_zjyprc_hadoop.browser.dwm_browser_event_aggregation_miui_di
WHERE  date = 20260628
LIMIT  10;
```
> ⚠️ 探查数据时记得加 `LIMIT 10`，别一次拉海量数据。

## 四、WHERE：筛选数据

| 运算符 | 含义 | 例 |
|--------|------|-----|
| `=` `!=` | 等于/不等于 | `date = 20260628` |
| `>` `<` `>=` | 比较 | `app_dura > 60000` |
| `BETWEEN` | 区间 | `date BETWEEN 20260601 AND 20260628` |
| `IN` | 在列表里 | `app_launch_way IN ('点击icon','点击push')` |
| `LIKE` | 模糊匹配 | `app_ver LIKE '20.%'` |
| `AND`/`OR` | 且/或 | `date=… AND is_app_dau_2024=1` |

> 🔴 **分区过滤是重中之重**：`WHERE` 第一行必须写 `date`，否则扫全表、几小时跑不出来。（为什么？看第八节。）
> 💡 `IN ('a','b')` 是"或"的关系；但列表里值太多会拖慢性能。`LIKE` 必须配 `%`（任意多字符）或 `_`（单字符）才是模糊匹配。

## 五、GROUP BY + 聚合函数

| 函数 | 作用 | 用于 |
|------|------|------|
| `COUNT(DISTINCT x)` | 去重计数 | 算**多少人**（DAU/UV） |
| `COUNT(x)` | 计数 | 算**多少次**（PV/曝光） |
| `SUM(x)` | 求和 | 总时长、总 VV |
| `AVG(x)` | 平均 | 人均（更推荐 SUM/人数） |
| `ROUND(x,n)` | 保留小数 | 率保留几位 |

```sql
-- 按启动方式统计人数
SELECT app_launch_way, COUNT(DISTINCT did) AS user_cnt
FROM   …miui_di
WHERE  date = 20260628 AND is_app_dau_2024 = 1
GROUP BY app_launch_way
ORDER BY user_cnt DESC;
```

## 六、CASE WHEN：给数据分类打标

现场高频用到：把连续值/多个枚举**归到几类**（如启动方式归 5 大类、时长分档、新老用户），常配 `GROUP BY` 一起用。

```sql
SELECT
  CASE
    WHEN app_launch_way IN ('点击icon','点击桌面书签') THEN '主启'
    WHEN app_launch_way IN ('点击push','subscribe_push') THEN 'Push启动'
    WHEN app_launch_way LIKE '%搜索%' THEN '搜索入口'
    ELSE '其他'                       -- 前面都没命中才走这里
  END AS launch_type,
  COUNT(DISTINCT did) AS user_cnt
FROM …miui_di
WHERE date = 20260628 AND is_app_dau_2024 = 1
GROUP BY
  CASE
    WHEN app_launch_way IN ('点击icon','点击桌面书签') THEN '主启'
    WHEN app_launch_way IN ('点击push','subscribe_push') THEN 'Push启动'
    WHEN app_launch_way LIKE '%搜索%' THEN '搜索入口'
    ELSE '其他'
  END;
```

**三条铁律（现场最容易错）：**
1. **从上往下匹配、命中即停**——所以区间/条件要按顺序写、别重叠（如先写 `>=90` 再写 `>=60`）。
2. **必须 `END` 结尾**，可选 `ELSE` 兜底。
3. 🔴 **别用中文标点**：全角括号 `（）`、逗号会直接报错，一律英文半角。

> 💡 `GROUP BY` 里要重复写整段 CASE WHEN（有些平台可用列位置或别名，不通用）。分类打标、把长表转宽表都靠它。

## 七、核心指标 SQL 实战

**DAU**
```sql
SELECT COUNT(DISTINCT did) AS dau
FROM …miui_di WHERE date = 20260628 AND is_app_dau_2024 = 1;
```
**信息流 CTR / 人均时长**
```sql
SELECT
  ROUND(SUM(click_cnt)*1.0/SUM(expos_cnt), 4)              AS ctr,
  ROUND(SUM(feed_dura)*1.0/COUNT(DISTINCT did)/60000, 2)  AS avg_min
FROM …label_di
WHERE date = 20260628 AND is_dau_feed_dapan_2024 = 1 AND is_top = 0;
```
> 三个高频细节：除法 `*1.0`、时长 `/60000` 转分钟、大盘 CTR 用 `SUM/SUM` 不用 `AVG`。
> 📌 口径示例：`is_dau_feed_dapan_2024`/`is_dau_2024` 这类"活跃"标记，口径是"曝光+点击+播放+进入空间次数>0 且非置顶"——具体口径以数据地图"产出信息"为准。

## 八、SQL 是怎么跑起来的？——执行原理入门（了解级）

> 这一节不用记，只要建立一个直觉：**你写的是"要什么"，引擎决定"怎么跑"。** 想深入调优、搞懂为什么有时候慢，看模块4。

### 8.1 你的 SQL 不是在一台机器上跑
我们的表动辄几百亿行、上 TB，一台电脑根本扛不住。所以大数据平台把数据切成很多块，**分给成百上千台机器"并行"处理，最后把结果汇总**——这就是**分布式计算**。你写一条 SQL，背后可能是上千台机器在同时干活。

### 8.2 我们用的是什么引擎？
你在数据工坊写好 SQL，通过 **Kyuubi** 提交，交给 **Spark** 引擎去执行（有些老任务用 **Hive**）。你只管写 SQL，**把它翻译成"分布式任务"、分发到多少台机器、怎么调度，都是引擎的事**。

### 8.3 一条 SQL 的"一生"（4 步）
```
① 解析检查 → ② 生成执行计划 → ③ 拆成任务，分发到多台机器并行算 → ④ 汇总结果返回给你
（看语法/表/字段）  （引擎决定怎么跑最省）   （真正干活，读数据、过滤、聚合）    （把各机器的结果合并）
```

### 8.4 为什么"加 date 分区"最省钱？—— 分区裁剪
数据在存储时**按天分成一个个文件夹**（叫"分区"）。你 `WHERE date = 20260628`，引擎就**只打开那一天的文件夹**，其他几百天的数据碰都不碰——这叫**分区裁剪**，是最有效的提速手段。
> 🔴 不加 date：引擎只能把**所有分区**全读一遍（全表扫描），几百亿行，自然几小时跑不完。这就是第四节那条铁律的原因。

### 8.5 三个"了解级"的提速常识
| 常识 | 为什么 |
|------|--------|
| **只查需要的列**（别习惯性 SELECT *） | 列越少，读得越少、传得越少 |
| **先过滤，再计算**（WHERE 尽量早、条件尽量全） | 早点把数据变少，后面每一步都轻松 |
| **探查加 LIMIT** | 先看几行验证逻辑，别动不动全量扫 |

> 🎯 一句话：**能少读数据就少读（分区裁剪 + 列裁剪）、能早过滤就早过滤。** 至于 JOIN、GROUP BY 为什么慢、什么是 shuffle、数据倾斜怎么办——模块4 专门讲。

## 九、AI 辅助写 SQL（重点）

**这是销售/运营取数的主力方式**：说清需求，让 AI 写，你来校验。

**高质量提示词 5 要素**（说清楚，AI 准确率 90%）：
1. 【表】用哪张表
2. 【字段】关键字段含义
3. 【需求】要算什么指标
4. 【过滤】什么条件
5. 【注意】分区、去重、单位等口径

**AI 生成后必做的 4 步校验**：
| 步 | 查什么 |
|----|--------|
| ① 字段校验 | 对照数据地图，AI 有没有编不存在的字段 |
| ② 分区校验 | 有没有加 date |
| ③ 逻辑校验 | 去重了吗、除法 *1.0 了吗、单位转了吗 |
| ④ 结果校验 | 数量级合理吗（DAU 百万级、CTR 百分之几） |

> 🎯 呼应启动课：**AI 写作业，你批改作业。** 详细提示词模板见公共资料《AI 写 SQL 提示词模板》。

## 十、新手最常犯的错
1. 忘加 `date` 分区 → 扫全表（没走分区裁剪）
2. 算人数漏 `DISTINCT` → 翻倍
3. 除法没 `*1.0` → 率算成 0
4. 时长没 `/60000` → 大 6 万倍
5. CASE WHEN 忘 `END`、用了中文括号 → 报错
6. AI 生成不校验直接用 → 瞎编字段/错口径
