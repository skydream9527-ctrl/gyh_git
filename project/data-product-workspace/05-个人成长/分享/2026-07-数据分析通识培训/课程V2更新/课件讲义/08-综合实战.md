## 二、贯穿案例（完整版）：PUSH CTR 突然下降怎么查？

**场景**：周一早上你发现，周日 PUSH 整体 CTR 从平时的 8% 降到 6%，掉了 2 个百分点。老板问：是不是上周四上线的新个性化推送策略有问题？要不要回滚？

### 第一步 · 明确问题（模块5：分析流程）
- **目标**：定位下降原因 → 判断是否新策略导致 → 给决策建议（回滚/优化/观察）。
- **核心指标**：PUSH CTR = 点击量 / **到达量**（注意口径，不是发送量）。
- **时间范围**：重点看上周四~周日，对比上周同期。
- **拆分维度**：时间、PUSH 类型、用户分群、漏斗环节。

### 第二步 · 数据校验（先排除"数据本身错了"）
异常先别急着找业务原因，先确认数据可信：
1. 数据是否更新延迟？换个方式重跑确认。
2. 埋点有没有丢？点击/到达埋点是否正常。
3. 口径有没有变？CTR 公式是否被改。
→ 确认数据没问题，CTR 确实降了 2pct。

### 第三步 · 对比分析，确认异常范围（模块5：四把尺子）
用模块3/4 的 SQL 分维度取数，逐层拆：

| 拆解维度 | 发现 |
|----------|------|
| **拆时间** | 周四上线时 8.3%（略涨）→ 周五 8% → 周六 7.2% → 周日 6%，**逐步下降**；凌晨 2-4 点特别低 |
| **拆 PUSH 类型**（好维度） | 时政类 10%→9%、娱乐类 8%→7.5%、**新"热点推送"类 CTR 仅 3% 且发送量大**，其他基本不变 |
| **拆用户群** | 老用户 8%→7.5%，**新用户 7%→3.5%（重灾区）** |
| **拆漏斗** | 下发→到达、到达→展示都正常，**问题在"展示→点击"（10%→7.5%）** |

→ **定位**：新上线的"热点推送"类，周日凌晨给新用户批量推了不匹配的时政内容，这批 CTR 仅 3%，拉低了整体；其他类型和老用户基本正常。

> 💡 这一步用到的取数（分类型、分新老、算漏斗）正是模块3/4 的 SQL；用 AI 生成分维度 SQL、你按 4 步校验，效率最高。

### 第四步 · 深入原因（模块5：DIKW 走到"为什么"）
- 看内容：热点类都是较严肃的深度时政，标题严肃。
- 看新用户特征：刚装 APP，之前多看娱乐生活类，对时政兴趣低。
- 看时间：周日凌晨 2 点发，用户睡觉，早上起来一堆推送直接划走。
- 看灰度：上线前只灰度测了老用户（CTR 是涨的），**没测新用户**，全量后加进新用户就崩了。

### 第五步 · 结论与建议（模块1：结论先行 + 自证式汇报）
**结论（先行）**：周日 CTR 降 2pct，80% 来自新"热点推送"全量后给新用户凌晨推了不匹配内容（CTR 仅 3%）；新策略对老用户正向（+0.3pct），对新用户严重负向——**不必全量回滚，但要立刻调整**。

**行动建议**：
1. 新用户注册前 7 天不推时政类热点，只推泛娱乐 → 预计 CTR 回到 7.5%+。
2. 凌晨 0-8 点不批量发推送。
3. 以后灰度必须覆盖新用户，小流量观察 3 天再全量（呼应模块6：样本要覆盖全人群）。
4. 建 PUSH 分人群内容池，不同用户匹配不同内容。

> 🎯 一个漂亮的分析闭环 = 定位准（拆维度）+ 说清因（为什么）+ 给对策（可落地），这正是模块1 说的"讲清好在哪、差在哪、下一步怎么办"。

## 三、取数实战：一个刁钻命题怎么拆（现场命题）

现场有同学出了一道题：**"查昨天，浏览器自建源、视频沉浸式场景的『人均评论发布次数』"**。这类"条件一多就懵"的命题，正是日常取数的真实形态。拆法：

**第一步 · 想清楚口径**：人均评论发布次数 = `评论总次数 / 发过评论的人数`（分母是"发过评论的人"，不是全体用户）。

**第二步 · 分步骤取（别一口气写完）**：
1. 先把**评论总量**取出来，验证数量级对不对；
2. 再加**场景过滤**：自建源、`item_tab IN ('短视频','小视频')`、沉浸式 `page`；
3. 最后算**人均**（分母用 `CASE WHEN comment_pv>0 THEN did` 去重计数）。

**可跑骨架**（字段以数据地图为准，仅示意结构）：
```sql
SELECT
  SUM(comment_pv) * 1.0
  / COUNT(DISTINCT CASE WHEN comment_pv > 0 THEN did END) AS avg_comment
FROM ...内容评论表
WHERE date = '${date-1}'          -- 动态取昨天
  AND source = '自建源'
  AND item_tab IN ('短视频','小视频')
  AND page = '沉浸式';
```

> 💡 现场真实体会：**"一看代码都懂，一手搓全忘"**。复杂命题别指望一次写对——拆步骤、先单日验证、让 AI 补全再人工校验，还可以去"查询历史"翻别人写过的类似代码复用。这就是模块3/4 反复强调的"AI 写作业、你批改"。

## 四、业务版（销售/运营选学）：给一次活动做完整复盘

不写复杂 SQL 也能走完整流程，把模块1 的自证三板斧用起来：

1. **明确问题**：这次拉新活动到底值不值得再做？
2. **找数取数**：要"活动期日均新增、分渠道转化率、各渠道成本"——不会写 SQL 就按模块2 的"提需求五要素"找数据同学，或让 AI 帮取（模块3）。
3. **对比定位**（模块1 对比）：活动期 vs 活动前一周（前后对比）、A/B/C 渠道之间（分群对比）。
4. **判断归因**：新增里 A 渠道占 60% 且转化最高；C 渠道成本高、ROI 仅 0.6（亏）。**记得排除大盘**：大盘同期也涨了多少，扣掉才是活动净增。
5. **结论建议**：结论先行——"活动净拉新 X，A 渠道最优、C 渠道亏损，建议下次砍 C 保 A"。

> 💡 业务版的核心不是 SQL，是**用对比 + 排除大盘 + 结论先行**，把一件事讲成有数据支撑的故事。

## 五、常见卡点速查（做实战时最容易卡的地方）

| 卡点 | 怎么办 | 关联 |
|------|--------|------|
| 我算的和看板对不上 | 先对口径（过滤条件/表层级/去重），再找数据同学核对官方口径 | 模块2、公共资料 |
| SQL 跑很久没出来 | 90% 是没加 date 分区；先查 1 天验证逻辑再放大 | 模块3/4 |
| AI 写的 SQL 敢不敢用 | 简单单表校验后可用；复杂多表必查字段/口径/1天试跑 | 模块3/4 |
| 找不到根因 | 按 MECE 拆：时间→人群→模块→链路，再匹配业务动作 | 模块5 |
| 实验数据不显著 | 先看样本量够不够、效果是否太小、是否某细分显著 | 模块6 |

## 六、后续学习路径

1. **1~2 周**：把各模块练习跑一遍，日常取数先自己写/AI 辅助，写完对照公共资料模板检查。
2. **课后作业**：手写 SQL 算周报里那些指标（DAU、人均时长等）——为后续"AI 写周报 skill"打底（现场约定的作业）。
3. **1~2 个月**：独立完成 2~3 个完整小分析（功能上线效果、指标波动排查），按本课框架走。
4. **长期**：复杂分析多和数据同学对思路；做 AB 前先一起 review 设计；读《精益数据分析》《AB 实验：科学归因与增长的艺术》等。想系统用 AI 取数分析，看**选修手册**（结合 6 个 skill）。

> 💡 数据分析是实践技能，多写多练多踩坑成长最快，不用追求一开始就完美。

## 本节课小结
1. 任何分析都按一条线走：**明确问题 → 找数取数 → 拆维度定位 → 判断原因 → 结论先行**。
2. 完整版案例（PUSH CTR 下降）串起了找数、SQL、四把尺子、漏斗、归因、汇报。
3. 取数实战（自建源视频评论）示范"刁钻命题怎么拆步骤 + AI 补 + 手搓验证"。
4. 业务版（活动复盘）不靠复杂 SQL，靠对比 + 排除大盘 + 结论先行。
5. 卡住时对照"常见卡点速查"，多数问题都有标准解法。
