### 3.1 数据来源与表结构（启动课 / 模块2）
> 例：数据从哪来、粒度是什么、分区/T+1、DWM vs ADS…

**Q：数据地图看着就是个"搜索结果页"，它到底能干嘛？**
- 背景：现场打开数据地图，有同学第一反应"这不就是个搜索吗"。
- 解答：它是**数据的"图书馆检索系统"**。拿到表名一搜，就能看：表中文名 / 负责人、字段含义与枚举、分区规则、样例数据、表血缘，还有**产出信息**（点任务名能追字段的加工逻辑）。找数、对口径的第一站都在这。
- 关联：模块2、公共资料《数据工坊与权限操作手册》。
- 状态：✅ 已解决

### 3.2 SQL 取数（模块3 / 4）
> 例：为什么查不出来、JOIN 膨胀、去重、AI 写 SQL 校验…

**Q1：`IN` 是什么？可以放很多个吗？会影响性能吗？**
- 背景：讲 WHERE 条件时被追问。
- 解答：`IN (...)` 是一串"**或**"的简写——括号里任意一个匹配上就算通过，可理解为一个字符串数组（带引号的是字符串）。**能放多个，但放几十上百个会拖慢性能**，不建议堆太多。
- 示例：`app_launch_way IN ('点击icon','点击push')`
- 关联：模块3（WHERE）。
- 状态：✅ 已解决

**Q2：算比率为什么要 `*1.0`？**
- 背景：看 CTR 公式时问。
- 解答：因为点击、曝光都是**整数**，SQL 里"整数 ÷ 整数"会按整数处理、把小数截断（结果可能变 0）。乘 `1.0` 把它转成小数，整条式子就按小数算了。引擎的规则是"按式子里精度最高的类型算"，只要有一个是小数，结果就是小数。
- 示例：`ROUND(SUM(click_cnt)*1.0/SUM(expos_cnt), 4)`
- 关联：模块3（聚合/CTR）。
- 状态：✅ 已解决

**Q3：`AS` 是干嘛的？**
- 解答：**重命名**（给字段/表达式起别名）。不写 `AS` 也能跑，但结果列名会是一长串表达式、可读性差还可能乱码，建议都加。
- 示例：`... AS ctr`
- 关联：模块3。
- 状态：✅ 已解决

**Q4：有 `LEFT JOIN`，那有 `RIGHT JOIN` 吗？两组有啥区别？**
- 解答：有 `RIGHT JOIN`，但**日常几乎都用 `LEFT JOIN`**（以左表为准）。原因和执行顺序有关：引擎（如 Spark）从左往右读表，`A LEFT JOIN B` 先读 A 再读 B；如果左表小，还能把它复制到各机器做广播。用 `RIGHT JOIN` 顺序反过来，不符合"从左到右"的阅读/执行习惯。统一用 LEFT 更清楚。
- 关联：模块4（JOIN）。
- 状态：✅ 已解决

**Q5：`date = '${date-1}'` 是什么意思？**
- 背景：看历史代码时问。
- 解答：`date` 取当天日期，`-1` 表示减一天（即昨天），外面的 `${...}`（dollar + 大括号）把它变成一个**动态参数**注入到代码里。调度/复用时用它代替写死日期，就能"永远取昨天"。
- 关联：模块3、公共资料《数据工坊与权限操作手册》。
- 状态：✅ 已解决

**Q6：`CASE WHEN` 怎么用？有什么坑？**
- 解答：分类打标/条件分支。**从上往下匹配、命中即停（互斥）**，必须以 `END` 结尾。三个坑：① 区间从上往下写别重叠（如成绩 90/70-90/<70，一个人只落一个区间）；② 一定 `END` 结尾；③ **别用中文括号/标点**（会报错）。
- 示例：`COUNT(DISTINCT CASE WHEN comment_pv>0 THEN did END)`（发过评论的人数）
- 关联：模块3、模块8（取数实战）。
- 状态：✅ 已解决

**Q7：先聚合再 JOIN，是用 `WITH` 实现吗？子查询是什么？**
- 解答：`WITH` 或子查询都行——核心是**在关联前先把每个表按 `DID`（或 `DID+日期`）聚合成"一人一行"，再 JOIN**，避免"一对多 × 一对多"把指标翻倍。`WITH (...)` 里那段、或括号里那段完整查询，就是子查询（虚拟临时表）。
- 关联：模块4（CTE / 先聚合再 JOIN / 数据重复坑）。
- 状态：✅ 已解决

### 3.5 埋点设计（模块7）
> 例：有效曝光怎么定、时长怎么算、分题材差异、埋点验收…

**Q1：数据工坊里的表，字段都是 onetrack 打点来的吗？没做埋点也能拿到吗？哪些能拿哪些不能？**
- 背景：现场连环追问（很有代表性）。
- 解答：
  - 表里的数据**一定来自埋点**（客户端 onetrack **或服务端埋点**），不是凭空生成——**没埋点就没这个数**。
  - 但埋点上报的**不一定都进表**：数仓只加工了常用的一部分。想要的字段没有，通常是"没埋"或"没加工进这张表"。
  - `onetrack` 是**日志层**，不等于你查的业务表；它是这些表的上游源头。链路：`埋点（客户端+服务端）→ 日志层(onetrack) → ODS → DWM 表 → 你查的字段`。
- 关联：模块7（表从哪来）、启动课（数仓分层）。
- 状态：✅ 已解决

**Q2：埋点平台不是只有 Winchat 吗？**
- 解答：不只。除了客户端埋点，还有**服务端埋点**。所以"能不能取到某个数"，取决于产品有没有埋、以及是客户端还是服务端埋。
- 关联：模块7。
- 状态：✅ 已解决

### 3.6 工具与权限（数据地图 / 数据工坊 / AI 工具）
> 例：权限怎么申请、执行报错、AI 工具怎么用…

**Q1：表权限怎么申请？谁审批？被驳回怎么办？**
- 解答：
  - **申请**：数据地图/数据工厂 → 空间权限申请 → 选空间 **桌面内容中心-产品** → 有效期选 **365 天** → 写清原因 → 提交。
  - **审批链**：**上级 → 表管理员 → 空间管理员**，多级；表保密级别越高涉及的人越多。
  - **被驳回**：现场就遇到——查看"产出信息"里的**加工逻辑代码是单独权限**（提示"查看代码请找研发，查询数据请单独申请"）。解决：**截图发群、找数据研发**确认口径 / 给权限。
- 关联：模块2、公共资料《数据工坊与权限操作手册》。
- 状态：✅ 已解决

**Q2：分号分隔的多段 SQL 一起点执行，为什么第二段是空的？**
- 解答：数据工坊里**分号分隔的多段会"并行提交"**。如果第二段依赖第一段建的临时表，并行跑时第一段还没出结果，第二段就取到空。**有依赖就一段段按顺序执行**（选中要跑的那段再点执行）。
- 关联：模块2、公共资料《数据工坊与权限操作手册》。
- 状态：✅ 已解决

**Q3：怎么复用别人写过的代码？**
- 解答：数据工坊「查询历史」支持**关键词检索**过往跑过的 SQL（如搜"热点卡片人群圈选"能找到别人写过的），相似需求先搜历史复用，比从零写快。注意历史代码里可能有执行失败的无效段。
- 关联：模块2。
- 状态：✅ 已解决

### 3.7 指标口径专区（高频易错，重点维护）
> 例：CTR 分母用到达还是发送、人均时长口径、VV 怎么算、"我和你算的对不上"…

**Q1：我算的内容中心 DAU 和看板对不上（3423 vs 3398），怎么回事？**
- 背景：现场手搓 DAU 时出现。
- 解答：多半是**过滤条件 / 口径没对齐**。常见原因：① 没加**有效用户过滤**（如 `is_dau_feed_dapan_2024=1`）；② 没按日期分组（跑成多天去重的一个总数）；③ 表层级/去重方式不同。**理论上官方口径应唯一**——先自查过滤和口径，再找数据研发核对看板口径。
- 关联：模块5（分析前先校验）、模块8（速查）、公共资料。
- 状态：✅ 已解决

**Q2：`feed_dura` 和 `app_dura` 有什么区别？毫秒怎么换算？**
- 解答：都是**毫秒**，`/60000` 换算成分钟（`/1000` 是秒，再 `/60` 是分钟）。`app_dura` 是 **APP 整体使用时长**；`feed_dura` 是**信息流消费时长（含列表时长）**，一般算信息流人均时长用 `feed_dura`。两者口径不同、值也不同，别混用。
- 关联：模块7（时长）、核心表速查手册。
- 状态：✅ 已解决

**Q3：`is_dau_2024`（信息流当日活跃）到底怎么算的？**
- 背景：现场从数据地图"产出信息"查加工逻辑（一度因无权限被驳回，后由讲师演示）。
- 解答：口径 = `曝光次数 + 点击次数 + 视频播放次数 + 进入空间次数 > 0 且 is_top = 0（非置顶）`。即**至少曝光成功且有互动**才算——不是拉一半屏幕就算。是个多埋点事件组合的口径。
- 关联：模块7（埋点到指标闭环）、核心表速查手册。
- 状态：✅ 已解决

### 3.8 其他
**Q：为什么 SQL 里写中文（括号/命名）会报错、标红？**
- 解答：全角中文括号 `（）` 和中文命名，部分引擎不支持，会语法报错。一律用**英文半角**。平台把可能有问题的片段**标红**（也包括写到一半不完整的语句），是正常提示。
- 关联：模块3（CASE WHEN 坑）、模块8。
- 状态：✅ 已解决
