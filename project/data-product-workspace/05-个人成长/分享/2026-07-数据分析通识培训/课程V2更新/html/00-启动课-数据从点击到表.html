01

SECTION 01

先认识"表"
一切数据的载体

表，就是一张二维表格（像 Excel）

user_id昵称城市注册天数是否会员
1001小米粥北京3201
1002阿强上海450
1003Lucy广州1281
行 row

一条记录 = 一个用户

列 column

一个字段 = 一个属性

单元格 cell

某行某列的一个具体值

一行是一条记录，一列是一个属性 —— SQL 里 SELECT 后面跟的，就是"列名"。

每一列都有"数据类型" 决定这列能做什么

类型常见写法举例能做什么
整数INT / BIGINT点击次数 5求和、比大小
小数DECIMAL / DOUBLE时长 35.5算率、算人均
字符串VARCHAR / STRING城市、启动方式只能匹配 / 分组
日期时间DATE / TIMESTAMP2026-06-28算区间、算间隔
标记常用 0 / 1是否会员 = 1筛选、打标
⚠️ 所以"时长 / 次数"能 SUM 求和，"城市 / 启动方式"不能；date 有时是数字 20260628、有时是字符串，写法要对应。

主键：一行的"身份证"

主键 = 能唯一认出某一行的字段，不能重复、不能为空。上表里就是 user_id。
🎯
① 唯一定位

user_id=1002 精确找到阿强这一行

🚫
② 保证不重复

同一个 id 不会出现两次 → 呼应"算人数要去重"

🔗
③ 让表能对上

两张表靠同一个 id 拼起来 → 第 3 课 JOIN

昵称不行（可能重名）、城市更不行（一堆人同城）—— 只有 user_id 能唯一标识。

联合主键 = 粒度 全课高频词

有时一个字段不够，要几个字段合起来才能唯一确定一行：

-- 我们的核心表：一个用户一天一行 单看 did → 不够（同一个人有很多天） 单看 date → 不够（同一天有几百万人） did + date 合起来 → 才能锁定"某个人在某一天"的那一行
💡 "did + date 决定一行"，就是这张表的"粒度"。粒度 = 一行代表什么，第 1~6 课会反复出现。
02

SECTION 02

Oracle、MySQL
和我们用的表

概念都通用，区别在"用来干什么"

常见数据库，都是"关系型数据库"

共同点：数据都存在一张张有行有列的表里，都用 SQL 增删改查。

MySQL

免费开源、最流行，绝大多数互联网 APP 在用

Oracle

商业收费、稳定，银行 / 电信等大企业在用

PostgreSQL…

还有 SQL Server 等，"表 / 主键"概念一致

💡 Oracle 和 MySQL 的"表、主键"概念一模一样，SQL 语法有些小差异。产品同学记住"都是把数据放表里、用 SQL 操作"就够了。

关键：业务数据库 vs 数据仓库 最重要的认知

对比业务数据库（Oracle/MySQL）数据仓库（我们用的 Hive/Iceberg）
干什么支撑 APP 实时运行存历史数据做分析
数据量千万 ~ 亿级百亿 ~ 万亿级
主要操作增删改查（实时改）几乎只有查，基本不改
主键强制唯一一般没有，靠 did+date 定粒度
怎么加速索引分区（按 date）
数据怎么进一条条实时写一批批追加，T+1
🏦 业务库像实时记账本
随时记、随时改
📷 数仓像每天拍照的相册
一天一张，存下不改

这就解释了第 1 课的三条"铁律"

查询必加 date

数仓靠分区加速，一天一个区。加了 date 才只扫那天；不加就扫几百亿行，几小时出不来。

粒度是 did+date

它没有传统主键，用"设备 + 日期"联合标识唯一一行。

T+1 更新

每天"拍一张快照" —— 等一整天行为产生完，凌晨批量加工，第二天才出。

🎯 现在你不是"死记"这些规则，而是理解了它们为什么存在。

下面拆开看三个关键环节：采集 → 加工 → 生产成表。

① 埋点采集 —— 数据的源头

用户点了一条新闻，埋点 SDK 立刻捕获，打包成一条"行为记录"上报。一条埋点回答四个问题：

问题记录内容举例
谁设备号 did、版本、渠道did = d_1001
什么时候精确到毫秒的时间戳2026-06-28 20:33:41
做了什么行为类型 + 页面 / 位置信息流点击第 3 条
对象是什么内容 id、类型、位置内容 888、时政、第 3 位
💡 客户端埋点记界面行为（曝光 / 点击 / 停留），服务端埋点记后台事件（PUSH 下发 / 审核）—— 所以 PUSH"下发量"来自服务端、"点击"来自客户端。

② 日志加工 —— 从"杂乱原始记录"到"干净明细"

// 原始日志：字段缩写、格式乱、有重复、有脏数据 {"did":"d_1001", "ts":1782648821000, "ev":"click", "pg":"feed", "item":"888", "ver":"15.2.0", "ext":{"pos":3, "type":"news"}...}
加工做四件事

解析拆字段 · 清洗去脏数据 · 去重 · 统一口径（时间 / 命名）

加工完得到

一张明细表：一行 = 一次行为。这就是第 1 课说的 DWD 明细层。

数据同学写好调度任务，每天凌晨自动跑这些加工。

③ 表的生产 —— 一层层聚合上来

明细表一天几十上百亿行，直接查又慢又贵，所以要再往上聚合。这就是"数仓分层"的由来：

层级一行代表怎么来的常用度
ODS一条原始日志埋点日志原样同步❌
DWD一次行为ODS 清洗、去重⚠️ 复杂才用
DWM一个人一天DWD 按"人 + 天"聚合✅ 日常首选
ADS一个指标一天DWM 再高度汇总✅ 看报表

下一页看看"聚合"到底把数据变成了什么样。

"聚合"在干什么：5 行明细 → 1 行汇总

明细层（DWD）· d_1001 在 6/28 启动 5 次：

d_1001 | 08:12:33 | 点击icon d_1001 | 12:05:10 | 点击push d_1001 | 18:20:55 | 点击icon d_1001 | 20:33:41 | 点击icon d_1001 | 22:47:02 | 点击icon

汇总层（DWM）· 按"人 + 天"压成 1 行：

d_1001 | 20260628 | app_open_cnt = 5 | app_dura = 2100000 -- 毫秒 = 35 分钟
💡 聚合 = 把很多行明细，按粒度数一数、加一加，压成一行。第 2 课的 COUNT / SUM 干的就是这事；T+1 也源于此——要等一天数据齐了才能聚合。

SECTION 04

看一张真实的表
数据怎么记录

真实表样例：…dwm_browser…miui_di

diddateapp_launch_wayapp_open_cntapp_durais_app_dau_2024
d_100120260628点击icon521000001
d_100220260628点击push26600001
d_100120260629点击icon39000001
粒度

did + date：一个人一天一行

为什么 d_1001 两行？

28、29 号各一行，不同日期分开

app_dura 好大？

单位毫秒，2100000÷60000 = 35 分钟

⚠️ 一个人跨多天有多行 → 算活跃用户数必须 COUNT(DISTINCT did) 去重。

* 示意数据，用于说明表结构，非真实数据。

回到 SQL：现在你知道它在干嘛了

SELECT date, COUNT(DISTINCT did) AS dau -- 要什么：日期 + 去重人数 FROM dwm_browser_event_aggregation_miui_di -- 从哪张表 WHERE date = 20260628 -- 筛哪些行 AND is_app_dau_2024 = 1 -- 且是活跃用户 GROUP BY date; -- 按天归堆

翻译成人话：从"一人一天一行"的表里，筛出 6/28 且活跃的行，把不同的 did 数一数。

WHERE

筛选行

GROUP BY

把行分组

COUNT / SUM

对每组算数

🎯 SQL 的本质 = 在一张二维表上做「筛选 + 分组 + 计算」。就这么朴素。

SECTION 05 · 送给你的定心丸

以后 SQL 让 AI 写
你负责 检查

AI 写作业，人检查作业

"我 SQL 不好，学得会吗？"

好消息：日常取数的 SQL，以后大多可以让 AI 帮你写（第 2、3 课会专门教怎么让 AI 写 SQL）。

但 AI 不是万能的，它经常犯这几类错，而且自己发现不了：

幻觉字段

编一个不存在的列名

错口径

CTR 分母用错

漏分区

忘了加 date

粒度错

算人数漏去重

单位错

毫秒没转分钟

⚠️ 这些错，得靠懂业务、懂表、懂口径的你来抓 —— 下一页现场检查一段 AI 写的 SQL。
