SQL 进阶：JOIN、CTE 与执行调优

多表拼查 · 执行原理（MapReduce / Shuffle）· 语法调优

⏱️ 1.5 小时🎯 JOIN/CTE 销售运营了解，原理调优产品重点

单表不够用的时候

有些问题一张表答不了，要跨表看：APP 启动 + 信息流消费、PUSH 点击 + 后续行为、用户标签 + 行为。

JOIN = 用共同的"钥匙"（如 date + did）把多张表拼起来。

最常用两种 JOIN（99% 场景）

类型逻辑比喻
INNER JOIN只保留两表都匹配上的数学语文都考了的学生
LEFT JOIN左表全留，右表没匹配填 NULL所有考数学的，语文缺考记 NULL
⚠️ LEFT JOIN 后右表 NULL，要用 COALESCE(x, 0) 补 0，否则 SUM/AVG 异常。

案例：PUSH 用户的信息流消费

SELECT COUNT(DISTINCT a.did) AS push_user, ROUND(SUM(COALESCE(b.feed_dura,0))/COUNT(DISTINCT a.did)/60000,2) AS avg_min FROM ( /* 先过滤 PUSH 启动用户 */ ) a LEFT JOIN ( /* 信息流消费 */ ) b ON a.date=b.date AND a.did=b.did;

避坑：① 必写 ON（漏写=笛卡尔积爆炸）② 先过滤再 JOIN ③ 小表放左边。

🧨 真实案例：100 万条危险网址 × 千万级用户行为，漏了 ON 就是笛卡尔积，直接把集群跑挂。

CTE（WITH）：像搭积木一样写 SQL

WITH 临时表1 AS ( SELECT … FROM … WHERE … ), 临时表2 AS ( SELECT … FROM 临时表1 JOIN … ) SELECT … FROM 临时表2;

复杂需求拆成多步，每步清晰不易错，也让 AI 生成拆分后的 SQL 更准。

🧩 现场案例：PUSH 价值量分析就是拆成"先圈用户 → 再关联行为 → 再算价值量"多步 WITH 完成的。

多表常见坑

  • 忘加 date → 扫全表
  • JOIN 漏 ON → 笛卡尔积爆炸
  • 多表直接 SUM → 数据重复、指标翻倍
  • AVG 算人均 → 排除 NULL 用户，偏高
  • 没 COALESCE → 结果变 NULL

四个坑的真实后果带数字

坑翻车现场
JOIN 漏 ON100万 × 80万 = 8000 亿行，集群跑挂
多表直接 SUM时长被复制 3 份 → 30 分钟算成 90 分钟
忘加 date读 400 个分区，耗时 ×400
AVG 人均100/5=20（跳过 NULL）· 正确 100/10=10

一个 ON、一个 date 的差别，结果可能差百万倍。

SQL 怎么执行：MapReduce 三阶段

Map · 分

切块并行：读数据、过滤、按 key 打标

Shuffle · 搬

按 key 跨机器重新分发，同 key 汇一起

Reduce · 合

对同 key 数据聚合/关联

类比人口普查：各村统计（Map）→ 汇总到省中心（Shuffle）→ 算省总数（Reduce）。

🎯 SELECT/WHERE 在 Map（快）；GROUP BY/JOIN/DISTINCT/ORDER BY 触发 Shuffle（慢，是瓶颈）。

案例：GROUP BY app_launch_way 走三阶段

# ① Map（3 台各读一批、按启动方式打标） 机器1: d1→icon d2→push 机器2: d3→icon d4→icon 机器3: d5→push d6→icon # ② Shuffle（按 启动方式=key 重新分发） icon → 机器A: d1,d3,d4,d6 push → 机器B: d2,d5 # ③ Reduce（各算 DISTINCT did） 机器A: icon = 4 人 机器B: push = 2 人

💡 GROUP BY 的字段就是 shuffle 的 key；同 key 必须先"搬"到一起才能算——这一步最慢。JOIN 同理，key 换成 did。

Shuffle：性能瓶颈的根源

把数据跨机器按 key 重新分发，涉及网络传输 + 磁盘读写 + 排序，是最贵的操作。

谁触发 shuffle

  • GROUP BY
  • JOIN
  • DISTINCT
  • ORDER BY（全局排序）
  • 窗口 PARTITION BY

怎么减少代价

  • 先过滤（数据先变少）
  • 只带需要的列
  • 先聚合再 JOIN
  • 能不排就不排（Top N 用 LIMIT）

💡 调优的本质：少 shuffle、shuffle 时少搬数据。

案例：先聚合再 JOIN，shuffle 降 6 倍

label_di 明细当天 5000 万行，要 JOIN 算 PUSH 用户人均时长。

写法进 shuffle 的数据量
❌ 直接拿 5000 万行明细去 JOIN5000 万行，慢 + 易倾斜
✅ 先按 did 聚合成 800 万行再 JOIN800 万行，降 ~6 倍
LEFT JOIN ( SELECT date,did,SUM(feed_dura) feed_dura FROM …label_di WHERE date=20260628 AND is_top=0 GROUP BY date,did ) b -- 先压小再关联

数据倾斜：99% 卡在最后 1%

某个 key 数据特别大，shuffle 后全汇到一台机器，它一个人干了别人几百倍的活 → 个别 task 卡死甚至 OOM。

🔍

最常见元凶

did 为空 / 默认值 / 测试 id，大量空值全挤一起

🛠️

怎么解

先过滤脏 key（解 80%）· 小表广播 · 加盐打散 · 提前聚合

🔴 卡在 99%？第一反应查有没有空值/默认值 key。

案例：一个空 did 拖垮整个任务

# 两表按 did JOIN，3000 万行 did 为空（脏数据） 空 did("") → 全部 3000 万行 汇到同一台机器！ 正常 did → 均匀分到上千台，每台几万行
🐢

正常 task 各 30 秒；空 did 那个 task 跑 40 分钟 / OOM，卡在 99%

✅

WHERE COALESCE(did,'')!='' 过滤空值 → 40 分钟变 40 秒

写 SQL 就能做的调优

手段收益
分区裁剪(WHERE 带 date)只扫需要分区，最有效
列裁剪(别 SELECT *)少读少传
谓词下推(过滤尽量早)尽早减少数据量
先聚合再 JOIN大幅减少 shuffle 数据
小表广播 broadcast join免 shuffle，提速明显
避免笛卡尔积、控 ORDER BY防爆炸、减 shuffle

💡 AQE（自适应查询执行）会自动优化 shuffle 分区、处理部分倾斜——但好习惯还得自己养。

Hive vs Spark：我们用哪个

维度Hive（MapReduce）Spark SQL
计算每步落磁盘内存 + DAG
速度慢、稳快，主流
适合超大批处理交互式、日常取数

Kyuubi 连的就是 Spark SQL，日常主要用它；语法 90% 通用。

EXPLAIN 看计划 & 慢查询排查

EXPLAIN SELECT … 重点看：有没有全表扫描、有没有大 Shuffle、JOIN 是否走 Broadcast。

先看对策
加 date 分区了吗漏了=全表，先补
JOIN 漏 ON 吗补 ON、查关联键
Shuffle 太大吗先过滤、先聚合、只带需要列
数据倾斜吗查空值/默认值 key

🎯 先查 1 天验证逻辑，再放大时间范围。

AI 辅助复杂 SQL

  • 复杂需求拆成步骤喂给 AI（用什么表、输出什么字段），别一次丢大需求
  • AI 常见坑修复：幻觉字段、漏 date、没处理 NULL、口径错、人均用 AVG、大表直接关联（改 WITH 先聚合）
  • 让 AI 解释执行计划：贴 EXPLAIN，让它指出大 shuffle / 能否改 broadcast

🎯 越复杂越要"拆步骤 + 人校验"。
