⏱️ 1.5 小时🎯 产品（信息流方向）

产品是埋点需求的源头

数据的一生第一步就是埋点。埋点设计错了，后面再牛的分析都是错的（GIGO，垃圾进垃圾出）。

"一次曝光怎么算""时长从哪到哪"这些是业务定义，得产品想清楚。

你在数据工坊查的表，从哪来？

埋点（客户端 onetrack + 服务端） # 源头，产品定义 ↓ 上报 日志层（onetrack 原始日志，最底层） ↓ 清洗 / 加工 ODS → DWM 聚合表 # 你查的 label_di / miui_di ↓ 你 SELECT 的那些字段

🔴 没埋点就没数据。表里能查到的一定来自埋点（客户端或服务端）；但埋点上报的不一定都加工进表。

💡 "这个数能不能取到" = "产品有没有埋这个点、数仓有没有加工进表"。onetrack 是日志层，不等于你查的业务表。

四个核心事件（抓住 80%）

事件关键
曝光 expose定有效曝光（露出≥50% 且停留≥500ms）
点击 clickCTR = 点击/曝光
进入 enter和点击拆开：点了不一定进得去；记进入时间戳
退出 exit记退出时间戳 + 消费进度

消费链路：曝光 → 点击 → 进入 →（消费）→ 退出，本身是个漏斗。

时长 = 退出 − 进入

消费时长 = 退出时刻 − 进入时刻（算出来的，不是直接采的）。
  • 只进入没退出（崩溃/切后台）→ 用心跳埋点（每 15 秒上报"还在看"）+ 超时截断兜底
  • 切后台一般算前台有效时长
  • 多次进出同一内容一般累加

表里的 feed_dura/app_dura，就是这样采集聚合出来的（单位毫秒，/60000 转分钟）。

四题材共用骨架，靠 content_type 区分

图文/小说/短视频/短剧共用一套事件，只在细节上具体化。

六大坑 + 上线前验收

六个高频坑

  • 重复上报 / 漏报
  • 曝光未去重
  • 时长异常（只进不出）
  • 口径不一致 / 缺关键参数

验收清单

  • 事件触发吗
  • 参数齐吗、值对吗
  • 去重生效吗
  • 小流量指标在合理区间吗
