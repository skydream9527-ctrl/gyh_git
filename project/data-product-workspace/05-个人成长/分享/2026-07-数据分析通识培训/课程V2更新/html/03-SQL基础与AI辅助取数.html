SQL 没那么难

  • 看得懂：别人的 SQL 在查什么
  • 写得出：日常取数的简单查询
  • 会校验：AI 生成的 SQL 对不对
💡 SQL 没有循环判断，写错最多报错、改不坏数据（你只有查询权限）。放心大胆试。

SQL 五段式结构（固定顺序）

SELECT 要什么字段 / 算什么指标 FROM 从哪张表查 WHERE 筛选什么条件 GROUP BY 按什么维度分组 -- 要分天/分群才加 ORDER BY 按什么排序 -- 可选

就像点菜：要什么菜（SELECT）· 哪家店（FROM）· 什么口味（WHERE）· 按菜系归类（GROUP BY）。

WHERE：筛选数据

运算符例
= !=date = 20260628
BETWEENdate BETWEEN 20260601 AND 20260628
IN（"或"）app_launch_way IN ('点击icon','点击push')
LIKE（模糊）app_ver LIKE '20.%'（% 通配任意字符）
动态日期date = '${date-1}'（调度/复用时取昨天）

🔴 分区过滤是重中之重：WHERE 第一行必须写 date，否则扫全表、几小时跑不出来。

💡 IN 是一串"或"的简写，但枚举太多会拖慢性能；LIKE 必须配 % 才能模糊匹配。

GROUP BY + 聚合函数

函数用于
COUNT(DISTINCT x)算多少人（DAU/UV）
COUNT(x)算多少次（PV/曝光）
SUM(x)总时长、总 VV
ROUND(x,n)率保留几位小数

算人数一定 COUNT(DISTINCT did)——漏了 DISTINCT 结果翻倍。

CASE WHEN：分类打标

SELECT CASE WHEN app_launch_way IN('点击icon') THEN '主启' WHEN app_launch_way IN('点击push') THEN 'Push启动' ELSE '其他' END AS launch_type, COUNT(DISTINCT did) AS user_cnt FROM …miui_di WHERE date=20260628 GROUP BY /* 同段 CASE */;

🔴 三铁律：① 从上往下匹配、命中即停 ② 必须 END 结尾 ③ 别用中文括号/标点（会报错）。

核心指标：信息流 CTR / 人均时长

SELECT ROUND(SUM(click_cnt)*1.0/SUM(expos_cnt), 4) AS ctr, ROUND(SUM(feed_dura)*1.0/COUNT(DISTINCT did)/60000, 2) AS avg_min FROM …label_di WHERE date = 20260628 AND is_dau_feed_dapan_2024 = 1 AND is_top = 0;

三个高频细节：除法 *1.0 · 时长 /60000 · 大盘 CTR 用 SUM/SUM 不用 AVG。

SQL 是怎么跑起来的？了解级

  • 不在一台机器上跑：几百亿行 → 拆给上千台机器并行算再汇总（分布式）
  • Kyuubi → Spark 引擎：你写"要什么"，引擎决定"怎么跑"
  • 加 date = 分区裁剪：只读那一天的文件夹，其他碰都不碰——最省
3 个提速常识：只查需要的列 · 先过滤再算 · 探查加 LIMIT。 （原理深入见模块4）

AI 辅助写 SQL 业务岗主力方式

提示词 5 要素

  • 【表】用哪张表
  • 【字段】关键字段含义
  • 【需求】算什么指标
  • 【过滤】什么条件
  • 【注意】分区/去重/单位

生成后 4 步校验

  • ① 字段：有没有编字段
  • ② 分区：加 date 了吗
  • ③ 逻辑：去重/*1.0/单位
  • ④ 结果：数量级合理吗

🎯 呼应启动课：AI 写作业，你批改作业。模板见公共资料《AI 写 SQL 提示词模板》。

新手最常犯的 5 个错

  • 忘加 date 分区 → 扫全表
  • 算人数漏 DISTINCT → 翻倍
  • 除法没 *1.0 → 率算成 0
  • 时长没 /60000 → 大 6 万倍
  • AI 生成不校验直接用 → 瞎编字段/错口径
