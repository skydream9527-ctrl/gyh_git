数据从哪来

用户操作

💡 数据不是凭空来的：它是"用户行为 → 被埋点记录 → 经加工聚合 → 存成表"的结果。你查的每个数，背后都有这条链路。

四个核心事件

信息流的曝光 / 点击 / 进入 / 退出，抓住这四个就抓住了大半。

时长是"算"出来的

消费时长 = 退出时刻 − 进入时刻，不是直接采的。

📌 所以：曝光量、点击量、消费时长这些数，源头都是这些埋点事件的累加。

从日志到表：为什么有好几层

层级一行代表产品用吗
ODS 原始层一条原始日志❌ 太原始
DWD 明细层一次行为⚠️ 太大，复杂才用
DWM 汇总层一个人一天（did+date）✅ 日常首选
ADS 应用层一个指标一天✅ 看报表
💡 日常取数优先用 DWM 层：它已经按"人 + 天"聚合好了，一个人一天就是一行，性能好、字段全。

PART 02 · 重点

指标口径
与度量

新人最容易忽略、最该重视的一块 —— 数对不上，十有八九在这

为什么"口径"这么重要？

同一个「CTR」，口径不同，结果能差一倍还多。你说 3%、他说 5%，其实数据一模一样 —— 只是分母用的不一样。
📛
口径 = 一个指标的"精确定义"

算哪些人、除以什么、怎么过滤、什么单位。

🤝
口径不对齐 = 各说各话

"数对不上"90% 不是数据错了，是两个人口径不同。

下一页，我们用同一份数据，看看"口径"能把结果玩出多少花样 —— 点一点试试。

核心指标口径速查 先记这几个

指标口径 / 公式关键注意
DAUCOUNT(DISTINCT did)必须去重
信息流 CTRSUM(click_cnt)/SUM(expos_cnt)用 SUM/SUM 不用 AVG；自然流量 is_top=0
PUSH CTR点击 / 到达分母是"到达"不是"发送"！
总 VVnews+short+mini_vv_cnt三类相加
人均时长(分)SUM(feed_dura)/COUNT(DISTINCT did)/60000毫秒转分钟
渗透率用了的人 *1.0 / 总人数分子记得 *1.0

🎯 记指标，别只记名字，要记"分子分母是什么、怎么过滤"—— 那才是口径。

度量的六个坑 新人必踩

时长是毫秒
转分钟要 /60000，不转就大 6 万倍
算人数漏 DISTINCT
一人多行 → DAU 翻好几倍
除法漏 *1.0
整数除法 → 率算成 0
漏 date 分区
扫全表，几小时跑不出来
漏有效用户过滤
没加 is_app_dau_2024=1 → 混进测试/异常
自然流量漏 is_top=0
把置顶算进去 → CTR 虚高

⚠️ 这六个，是"数不对"的高频来源。写完 SQL 先对着过一遍。

"为什么我和你算的对不上？"

查同一个指标前，先和对方对齐这 6 项 —— 对齐了，数自然就一致：

  • 用的是同一张表吗？（ADS vs DWM）
  • 都去重了吗？（COUNT DISTINCT）
  • 过滤条件一样吗？（有效用户 / is_top）
  • 时间范围一样吗？（哪几天）
  • 单位一样吗？（毫秒 / 分钟）
  • 分子分母定义一样吗？（CTR 口径）
💡 90% 的"对不上"，都能在这 6 项里找到原因。先对口径，再对数。

定义一个指标，说清"三件套"

📌
① 口径定义

这个指标"是什么"、算哪些人 / 哪些行为

🧮
② 计算公式

分子、分母分别是什么，怎么聚合

⚠️
③ 注意事项

单位、过滤、去重、边界情况

// 示例：人均信息流时长（分钟） ① 口径：当日信息流活跃用户，人均消费的信息流时长 ② 公式：SUM(feed_dura) / COUNT(DISTINCT did) / 60000 ③ 注意：feed_dura 是毫秒；只算 is_dau_feed_dapan_2024=1；is_top=0
