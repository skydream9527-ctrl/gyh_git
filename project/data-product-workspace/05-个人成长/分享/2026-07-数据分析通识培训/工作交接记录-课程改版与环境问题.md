## ⚠️ 一、环境问题（最重要，请优先排查）

### 现象
本会话中，**文件的创建 / 移动操作经常不落盘或稍后"回退"**，表现为：
- `fs_write` 返回"创建成功"，但过一会读盘文件不存在（发生过：串讲 deck 第一次、skill 索引、第8课课件+作业、课程改版方案.md 一度消失）。
- `bash mkdir -p` 建 4 个目录，只成功 2 个；重试一次才补齐。
- `bash mv` 批量移动，部分或全部不生效（源文件还在原位、目标没出现）。
- 之前"搬进 V1/V2"的一批文件，后来**回退散落回了根目录**。

### 已验证的规律
1. **"重试 + 每步读盘验证" 能最终达成**：同一个操作重试一两次、每次用 Python 读盘确认，通常能成功。
2. **被用户中断（interrupt）时的写入尤其容易丢**——多个丢失的文件都是 `fs_write` 后紧接被打断的。
3. **终端对中文输出会乱码 / 重复 / 错位**：验证时必须用 **纯 ASCII 输出**（`True/False`、索引数字），不要 `ls` 看中文、也不要 print 中文路径。

### 建议排查方向（推断，需你确认）
- 目录是否挂在**云同步盘 / iCloud / 网盘**（同步冲突导致回退）？
- 是否在**容器 / 沙盒**里，存在快照回滚或 overlayfs 分层导致读写不一致？
- 磁盘空间 / 权限 / inode 是否异常？
- 建议：修复后，把课程目录放到一个**本地、无同步、稳定**的路径再继续。

## 六、续做操作规范（应对环境不稳）
1. **能用 `fs_write` 直接写目标位置就别用 `mv` 搬**（fs_write 比 mv 相对可靠；内容 AI 都能重新生成）。
2. **每写/移一个文件，立刻用 Python 读盘验证**：`os.path.isfile(...)`，输出用 **ASCII**（True/False、索引）。
3. **验证不通过就重试**，通常重试 1~2 次能成。
4. **严格一次一个文件 / 一个操作**，做完验证再下一步。
5. deck 的 HTML 引擎：复用现有任一 deck 的 `<style>` 和底部 `<script>`（翻页/fragment/进度条/打印），只改中间 `<section class="slide">` 内容。
6. HTML 校验用 `grep -c '<section'` vs `'</section>'` 配对（注意中文文件名在 bash 里要么用 Python 操作、要么忽略乱码看数字）。
