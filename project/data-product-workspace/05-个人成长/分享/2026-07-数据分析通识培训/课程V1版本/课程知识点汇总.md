## 二、全局速查卡（最高频，先记这些）

### 内容生态核心两张表

| 表名（库 `iceberg_zjyprc_hadoop.browser.`） | 粒度 | 用途 |
|------|------|------|
| `dwm_browser_event_aggregation_miui_di` | did + date（每人每天一行） | APP 启动、版本、启动方式、整体使用时长、基础行为 |
| `dwm_browser_event_aggregation_label_di` | did + date（每人每天一行） | 信息流曝光 / 点击 / 播放、消费时长、内容标签、用户标签 |

> **关联方式**：两表通过 `date + did` 做 JOIN（表是 did+date 粒度，同一用户不同日期是不同记录）。

### 高频字段

| 字段 | 含义 | 备注 |
|------|------|------|
| `did` | 设备 ID（代表一个用户） | 算人数用 `COUNT(DISTINCT did)` |
| `date` | 日期分区 | 格式 `yyyyMMdd`，**查询必加** |
| `is_app_dau_2024` | =1 为当日 APP 活跃用户 | miui_di 表过滤有效用户 |
| `is_dau_feed_dapan_2024` | =1 为信息流活跃用户 | label_di 表过滤有效用户 |
| `app_launch_way` | 启动方式 | icon / push / 搜索 / 第三方… |
| `app_dura` | APP 使用时长 | 单位**毫秒**，/60000 转分钟 |
| `is_top` | 是否置顶内容 | 自然信息流过滤 `is_top = 0` |
| `expos_cnt` / `click_cnt` | 曝光 / 点击次数 | CTR = click/expos |
| `feed_dura` | 信息流消费时长 | 单位**毫秒** |
| `news_vv_cnt` / `short_vv_cnt` / `mini_vv_cnt` | 资讯 / 短视频 / 小视频 VV | 三者相加为总 VV |

### 核心口径公式

| 指标 | 公式 | 注意 |
|------|------|------|
| DAU | `COUNT(DISTINCT did)` | 必须去重 |
| CTR（信息流） | `SUM(click_cnt) * 1.0 / SUM(expos_cnt)` | 用 SUM/SUM，不用 AVG |
| CTR（PUSH） | `点击 / 到达`（不是点击/发送！） | 分母是到达量 |
| 人均信息流时长（分钟） | `SUM(feed_dura) * 1.0 / COUNT(DISTINCT did) / 60000` | 毫秒 → 分钟 |
| 总 VV | `SUM(news_vv_cnt + short_vv_cnt + mini_vv_cnt)` | 三类相加 |
| 渗透率 | `SUM(CASE WHEN 指标>0 THEN 1 ELSE 0 END) * 1.0 / COUNT(DISTINCT did)` | 分子记得 *1.0 |

### 六条铁律（写 SQL 前默念）
1. `WHERE` 第一行写 **date 分区过滤**，否则扫全表跑几小时
2. 算「多少人」必须 **`COUNT(DISTINCT did)`**
3. 所有除法**分子乘 `1.0`**（避免整数除法结果为 0）
4. 时长是毫秒，转分钟 **`/60000`**、转秒 `/1000`
5. 分母可能为 0 时加 **`NULLIF(分母, 0)`**
6. 关联 / 汇总后，NULL 用 **`COALESCE(字段, 0)`** 补 0

## 第 1 课 · 数据地图与埋点入门

> **一句话主旨**：先学会「找到数据、看懂字段」—— 理解数据价值、埋点原理、数仓分层，掌握数据地图和核心表。

### 1. 为什么产品要懂数据
- **用户行为代表真实意图**：「不要看他说了什么，要看他做了什么」。调研 / 访谈有表述偏差，点击 / 停留 / 消费行为不会说谎。
- **行为数据噪声更小**：「汽车出现前，用户不会说想要汽车，只会说想要一辆更快的马车」—— 能发现用户没说出口的偏好。
- **迭代闭环的核心**：需求 → 上线 → 效果验证 → 下一轮迭代，数据是唯一的效果评判标准。

### 2. 数据埋点是什么
- **定义**：埋点是用户行为的记录。链路：`用户操作 → 触发埋点(SDK) → 客户端&服务端交互 → 数据采集上报 → 落表存储`。
- **一条埋点回答四个问题**：谁（did / 设备 / 版本 / 渠道）、何时（时间戳 / 日期）、做了什么（曝光 / 点击 / 播放 / 停留 + 对象）、结果如何（时长 / 是否看完 / 后续行为）。
- **常见类型与对应指标**：

  | 类型 | 产品关心指标 |
  |------|--------------|
  | 页面 | PV、UV、页面停留时长 |
  | 点击 | CTR、点击分布、路径转化 |
  | 曝光 | 曝光量、有效曝光 |
  | 播放 | 播放时长、完播率、跳出率 |
  | 自定义 | PUSH 到达、下载完成、支付成功 |

### 3. 埋点设计：事件模型（来自飞书培训材料）
- **事件三要素**：操作行为 · 参数/属性名 · 参数/属性值。SQL 里的字段，本质就是这些参数。
- **事件模型 5 要素**（OneTrack 埋点元数据）：
  - **When** 时间戳（SDK 自动采集）
  - **Who** 用户信息（自然维度参数）
  - **Where** 点位设计（在哪触发）
  - **What** 事件设计（发生了什么）
  - **How** 参与方式（事件维度参数）
- **结构层级**：`页面 page → 模块 module → 元素 item`（一个页面多个模块，一个模块多个元素）。
- **设计三步**：明确场景（起止点）→ 明确上报内容 → 明确上报参数（from_page / sessionID…）。
- **三大原则**：① 对行为抽象 ② 兼容 & 复用 ③ 自上而下设计。

### 4. 公司数据体系分层

| 层级 | 全称 | 说明 | 产品常用度 |
|------|------|------|-----------|
| ODS | 操作数据层 | 原始数据，未清洗 | ❌ 基本不用 |
| DWD | 明细数据层 | 清洗后的明细，单条行为 | ⚠️ 复杂分析，数据量大 |
| **DWM** | 中间汇总层 | 按「用户+日期」聚合的轻度汇总 | ✅ **日常取数首选** |
| ADS | 应用数据层 | 高度汇总结果表，用于报表 | ✅ 固定指标，灵活性差 |

> **取数原则**：优先用 DWM 层聚合表，性能是 DWD 的 10~100 倍，不用自己写复杂去重聚合。

### 5. 数据地图使用
- **能查什么**：表信息 / 负责人、字段含义 / 类型 / 枚举值、分区规则 / 更新频率、表血缘、样例数据。
- **核心 5 步**：搜索表 → 看分区（多数按 `date`）→ 看字段样例 → 查血缘（口径不满足向上游找）。
- **找表技巧**：优先找带 `_di`（日分区）的聚合表；浏览器业务表在 `iceberg_zjyprc_hadoop.browser.` 库下。

### 6. 数据权限申请
数据地图右上角入口 → 搜表申请查询权限 → 填理由 → 表负责人审批（半天内）→ Kyuubi 查询。**建议一次性申请常用表权限**。

### 本课要点
> 用行为数据决策；埋点 = 行为记录（事件+参数）；优先 DWM 层；数据地图找表；核心两表 date+did 关联；查询必加分区 + 有效用户过滤。

## 第 2 课 · SQL 基础语法（上）

> **一句话主旨**：SQL 不是编程，是固定几个关键词拼起来。目标：看得懂、写得出、会校验。

### 1. 心理建设
产品学 SQL 只要三件事：**看得懂**别人的 SQL、**写得出**简单取数、**会校验**结果。SQL 没有循环 / 判断等复杂逻辑，写错最多报错，不会改坏数据（只有查询权限）。

### 2. 五段式结构（固定顺序）
```sql
SELECT    你要什么字段 / 算什么指标
FROM      从哪张表查
WHERE     筛选什么条件的数据
GROUP BY  按什么维度分组   -- 要分天 / 分群才加
ORDER BY  按什么排序       -- 可选
```
> **铁律**：SELECT 里非聚合的字段（如 `date`），必须出现在 GROUP BY 里，否则报错。

### 3. WHERE 常用写法

| 写法 | 意思 | 例子 |
|------|------|------|
| `=` / `!=` | 等于 / 不等于 | `date = 20260628` |
| `>` / `<` | 大于 / 小于 | `feed_dura > 0` |
| `BETWEEN … AND` | 区间（含两端） | `date BETWEEN 20260620 AND 20260628` |
| `IN (…)` | 在这几个值里 | `app_launch_way IN ('点击icon','点击push')` |
| `LIKE '%词%'` | 包含关键词 | `app_launch_way LIKE '%搜索%'` |
| `AND` / `OR` | 都满足 / 满足其一 | `date=… AND is_app_dau_2024=1` |

> 字符串用单引号 `'点击icon'`；数字直接写 `20260628`。

### 4. 5 个聚合函数 + 1 个辅助

| 函数 | 作用 | 用于 |
|------|------|------|
| `COUNT(DISTINCT x)` | 去重计数 | 算**多少人**：DAU、UV |
| `COUNT(x)` | 不去重计数 | 算**多少次**：PV、曝光、点击 |
| `SUM(x)` | 求和 | 总量：总时长、总 VV |
| `AVG(x)` | 平均 | 人均（算人均更推荐 SUM/人数） |
| `MAX / MIN` | 最大 / 最小 | 找峰值 |
| `ROUND(x, n)` | 保留小数 | `ROUND(0.08234, 4)` → 0.0823 |

### 5. 核心指标 SQL 示例

**单日 DAU**
```sql
SELECT COUNT(DISTINCT did) AS dau
FROM   iceberg_zjyprc_hadoop.browser.dwm_browser_event_aggregation_miui_di
WHERE  date = 20260628
  AND  is_app_dau_2024 = 1
  AND  COALESCE(did, '') != '';
```

**信息流核心指标（CTR / 人均时长 / VV）**
```sql
SELECT date,
  COUNT(DISTINCT did) AS feed_dau,
  ROUND(SUM(click_cnt) * 1.0 / SUM(expos_cnt), 4)                    AS ctr,
  ROUND(SUM(feed_dura) * 1.0 / COUNT(DISTINCT did) / 60000, 2)       AS avg_min,
  SUM(news_vv_cnt + short_vv_cnt + mini_vv_cnt)                      AS total_vv
FROM   iceberg_zjyprc_hadoop.browser.dwm_browser_event_aggregation_label_di
WHERE  date = 20260628 AND is_dau_feed_dapan_2024 = 1
  AND  COALESCE(did,'') != '' AND is_top = 0
GROUP BY date;
```
> 三个高频细节：`*1.0` 避免整数除法 · 时长 `/60000` 转分钟 · 大盘 CTR 用 `SUM/SUM` 不用 `AVG`。

**分群统计用 `CASE WHEN`（相当于 Excel 的 IF）**
```sql
COUNT(DISTINCT CASE WHEN app_launch_way IN ('点击push','subscribe_push')
                    THEN did END) AS push_dau
```
> 满足条件的才返回 did，不满足返回空 → `COUNT(DISTINCT)` 只数满足条件的人。PUSH 指标、新老用户对比都是这个套路。

### 6. AI 辅助写 SQL
- **工具**：公司内部 NL-SQL 工具（首选）/ 通用大模型（Claude、GPT）/ IDE 内置助手。
- **高质量提示词模板**：说清楚【表】【字段说明】【查询需求】【过滤条件】【排序】—— AI 准确率 90% 取决于需求描述清不清楚。
- **必做 4 步校验**：① 字段校验（对照数据地图，别用瞎编字段）② 分区校验（有没有 date）③ 逻辑校验（DISTINCT / *1.0 / 单位 / 公式）④ 结果校验（DAU 几百万级、CTR 1%~20%、人均几分钟）。
- **最佳实践**：先查 1 天验证逻辑，再扩大时间范围。

### 7. 新手最常犯的 7 个错误
1. 忘加 `date` 分区 → 扫全表
2. 算人数漏 `DISTINCT` → DAU 翻倍
3. 除法没乘 `1.0` → 率算成 0
4. 时长单位没转 → 大 6 万倍
5. 漏有效用户过滤 → 含测试 / 异常
6. 分母为 0 没处理 → 报错（加 `NULLIF`）
7. AI 生成不校验直接用 → 瞎编字段 / 错口径

## 第 3 课 · SQL 基础语法（下）

> **一句话主旨**：解决更复杂的取数 —— JOIN 多表关联、CTE 拆积木、避坑与性能、AI 进阶。

### 1. JOIN 多表关联

**为什么需要**：单表不够用，要跨表看（APP 启动 + 信息流消费、PUSH 点击 + 后续行为、用户标签 + 行为）。JOIN = 用共同的钥匙（如 `date + did`）把多张表拼起来。

**最常用两种（99% 场景）**：

| 类型 | 逻辑 | 场景 | 比喻 |
|------|------|------|------|
| INNER JOIN | 只保留两表都匹配上的 | 既启动 APP 又看信息流的用户 | 同时考了数学和语文的学生 |
| LEFT JOIN | 左表全保留，右表没匹配填 NULL | 所有启动 APP 的用户有没有看信息流 | 所有考数学的学生，语文缺考记 NULL |

> LEFT JOIN 后右表没匹配上是 **NULL**，要用 `COALESCE(feed_dura, 0)` 补 0，否则 SUM / AVG 异常。

**案例：PUSH 启动用户的信息流消费（先过滤再关联）**
```sql
SELECT COUNT(DISTINCT a.did) AS push_user,
  ROUND(SUM(COALESCE(b.feed_dura,0)) / COUNT(DISTINCT a.did) / 60000, 2) AS avg_min
FROM ( SELECT date,did FROM …miui_di
       WHERE date=20260628 AND is_app_dau_2024=1
         AND app_launch_way IN('点击push','subscribe_push') ) a
LEFT JOIN ( SELECT date,did,feed_dura FROM …label_di
       WHERE date=20260628 AND is_dau_feed_dapan_2024=1 AND is_top=0 ) b
  ON a.date=b.date AND a.did=b.did;
```

**JOIN 避坑**：① 一定写 `ON`（漏写 = 笛卡尔积，百万×百万搞挂集群）② 关联字段类型一致（必要时 CAST）③ 注意多对多导致数据膨胀 ④ 小表放左边。

### 2. CTE（WITH 语句）：像搭积木一样写 SQL
```sql
WITH
临时表1 AS ( SELECT … FROM … WHERE … ),
临时表2 AS ( SELECT … FROM 临时表1 JOIN … )
SELECT … FROM 临时表2;
```
> 复杂需求拆成多步，每步逻辑清晰、不易错，而且让 AI 生成拆分后的 SQL 准确率更高。典型：PUSH 用户 vs 全体用户对比，拆成「全体 / PUSH / 信息流 / 关联打标 / 聚合对比」5 步。

### 3. 常见 SQL 错误

| 坑点 | 后果 | 避免 |
|------|------|------|
| 忘加分区 date | 扫全表跑几小时 | 先写 WHERE date |
| JOIN 漏 ON | 笛卡尔积爆炸 | 写完 JOIN 立刻写 ON |
| 多表关联直接 SUM | 数据重复、指标翻倍 | 先验证关联后记录数 |
| 直接 AVG 算人均 | 排除 NULL 用户，偏高 | `SUM(x)/COUNT(DISTINCT did)` |
| 没处理 NULL | 结果变 NULL | `COALESCE(x, 0)` |
| 整数除法没 *1.0 | 率算成 0 | 分子乘 1.0 |
| 时长单位没转 | 大 6 万倍 | 毫秒 /60000 |

### 4. 性能优化 5 原则
1. **越早过滤越好**：子查询里先过滤日期 / 有效用户 / 字段再 JOIN
2. **只查需要的字段**：别 `SELECT *`
3. **避免大表关联大表**：能先聚合再关联就别用明细
4. **数据量大分步跑**：查 3 个月以上拆成按月
5. **优先用 DWM 聚合表**

> 跑不出来：先查 1 天验证逻辑 → 看是否漏分区 / 笛卡尔积 → 再找数据同学，别一直重试占资源。

### 5. AI 辅助复杂 SQL
- **复杂需求拆成步骤喂给 AI**（用什么表、输出什么字段），别一次丢大需求。
- **AI 常见坑修复**：幻觉字段（对照数据地图）、关联漏 date、没处理 NULL、口径错（明确告诉公式）、大表直接关联（拆 WITH）、人均用了 AVG（改 SUM/COUNT DISTINCT）。

## 第 4 课 · 常用数据分析方法

> **一句话主旨**：从「报数」升级到「给结论」。两个重点 —— **DIKW 数据驱动思维** 与 **排查数据问题的四把尺子**。

### 1. DIKW 模型（重点）

数据要经过四层向上加工，才能变成决策依据：

| 层级 | 含义 | 举例（PUSH CTR） | 口诀 |
|------|------|------------------|------|
| **D** 数据 Data | 未加工的原始数字 | 昨天 PUSH 点击 100 万次 | 事实 |
| **I** 信息 Information | 加工后有意义的指标 | CTR 8%，比前天降 2pct | 是什么 |
| **K** 知识 Knowledge | 对比后发现的规律 / 问题 | 主要是新用户群体下降 | 为什么 |
| **W** 智慧 Wisdom | 基于知识的行动方案 | 回滚新用户推送策略 | 怎么办 |

**天气类比**：温度 30°C / 湿度 80%（数据）→ 高温高湿（信息）→ 容易中暑（知识）→ 团建改室内（智慧）。

> **关键认知**：产品做分析的目标不是停在「报数据」，而是走到 **K（为什么）和 W（怎么办）**：发现问题、定位原因、给出建议。

**数据驱动三原则**：① 用数据说话，不凭感觉 ② 相关性 ≠ 因果性 ③ 数据是验证假设的工具（先假设后验证，不是先结论后找数据）。

### 2. 完整分析流程（5 步）
> **明确目标 → 拆解指标维度 → 取数验证 → 定位原因 → 输出建议**

- **步骤 1 明确目标（最重要）**：解决什么问题？谁看？做什么决策？什么算好 / 坏？
  - ❌ 反例：「帮我看看 PUSH 数据怎么样」（不知道看什么、看多久、比什么）
  - ✅ 正例：「周一 CTR 为什么比上周一降 2pct，是不是新策略问题，要不要回滚」
- **步骤 2 拆解指标维度**：指标（看什么数）+ 维度（怎么拆：时间 / 新老 / 渠道 / 类型 / 版本），可用公式一层层拆。

### 3. 排查数据问题的四把尺子（重点，来自飞书培训材料）

排查（如「浏览器日活下降怎么排查」）的关键，是选对拆解的尺子：

- **① 好维度 = 强区分度**：在该维度上，指标变化有很强的区分度，一拆就能看出「是谁出了问题」。
  - 强区分（好）：按 PUSH 类型拆，热点类 3% vs 其他 7.8%，一眼定位。
  - 弱区分（差）：按性别拆，男 6.1% vs 女 5.9%，几乎没差别，帮不上忙。
  - 一言以蔽之：**找到可能的影响因素，并在维度上做区分**。
- **② 好参照 = 基准线**：没有对比就没有结论。参照有时间参照（上周同日 / 上月 / 去年同期）、对照组参照、大盘 / 目标参照。最强的一种是 **DID 双重差分**（详见第 5 课）。
- **③ 好视角 = 对的样本 + 对的周期**：
  - 样本 = 最小作用颗粒度：目标决定样本颗粒度（优化播放流畅度，样本应是「播放会话 / 次」而非笼统「用户」）。
  - 周期由目标决定：点击类指标几天，留存类 1~2 周。（为什么常以「天」统计？—— 抹平一天内波动、对齐用户作息。）
- **④ 好起止点 = 链路分析**：核心原理 —— **如果数据变化和某功能有关，它只会影响该功能「往后」的环节**。从改动点往后逐环节看转化，就能判断是不是该功能造成的；改动点之前的变化多半无关。

### 4. 四种常用分析方法（覆盖 90% 需求）

| 方法 | 做什么 | 一句话 |
|------|--------|--------|
| 对比分析 | 同比 / 环比 / 对照 / 细分 | 找差异、判好坏 |
| 趋势分析 | 时间序列看周期、异常点 | 看规律、找异动 |
| 分群分析 | 按属性 / 行为拆分 | 定位问题群体 |
| 漏斗分析 | 多步链路转化 | 找流失瓶颈 |

- **对比分析原则**：口径一致 · 单变量对比 · 既看绝对值也看相对值。
- **分群分析 · 辛普森悖论**：整体 CTR 降了，但分人群看每个群体都在涨 —— 因为高 CTR 人群占比下降拉低了整体平均。**大盘异动时先拆维度**，别只看平均。
- **漏斗分析 · PUSH 全链路**：`下发 → 到达 → 展示 → 点击 → 落地页加载 → 消费/留存`。
  - **漏斗六步法**：明确目标(SMART) → 设计路径(按行为逻辑) → 采集数据 → 搭漏斗算转化率 → 深度分析瓶颈 → 优化验证迭代(A/B)。
  - CTR 下降别只盯点击，拆漏斗看是「到达少了」还是「展示了没人点」。

### 5. 真实案例：PUSH CTR 下降 2pct 排查

| 步骤 | 发现 |
|------|------|
| 拆时间 | 周四上线 8.3% → 周日 6%，逐步下降；凌晨 2-4 点特别低 |
| 拆 PUSH 类型 | 新热点推送 CTR 仅 3%，贡献 80% 下降 |
| 拆用户 | 新用户从 7% 降到 3.5%，是重灾区 |
| 拆漏斗 | 到达 / 展示正常，问题在「展示→点击」 |

> **结论**：新策略对老用户正向、对新用户严重负向 —— 不必全量回滚，但要立刻调整（新用户前 7 天不推时政、避开凌晨、灰度含新用户）。

### 6. 汇报：结论先行
顺序：**一句话结论 → 核心数据 → 原因拆解 → 可落地建议 → 附录**。

### 7. 常见分析误区
唯数据论（异常先查埋点 / 口径）· 忽略样本量 · 相关当因果 · 过度拆解（拆到每组几百人没意义）· 只看相对不看绝对。

## 第 5 课 · AB 实验原理与判断

> **一句话主旨**：用实验「验证」策略到底行不行。重点 —— **DID 双重差分法**。

### 1. 为什么做 AB
不做 AB 最容易犯三个错：**幸存者偏差**（只看用了说好的，没看流失的）、**归因错误**（涨了就归功新功能，其实可能是自然趋势 / 活动）、**主观判断**。
> AB 核心价值：通过随机分流排除其他因素干扰，准确测量策略的真实效果。

### 2. 基本逻辑
把用户**随机**分两组，同时间同环境运行，唯一区别是待测策略：
- **对照组 A**：保持原策略，作为基准
- **实验组 B**：上线新策略
最后对比指标差异 = 策略的真实效果。

### 3. 完整流程（6 步）
> **实验设计 → 流量分配(随机+AA) → 实验上线(先灰度) → 数据收集 → 效果评估 → 结论决策**

**实验设计最关键**：明确目标、按用户分流、定三类指标、算好样本量和周期。

### 4. 统计基础（通俗版）

| 概念 | 通俗解释 | 判断标准 |
|------|----------|----------|
| 随机波动 vs 真实效果 | 抛 10 次硬币不一定 5 正 5 反 | 判断差异是真效果还是巧合 |
| p 值 | 若策略没用，观察到当前差异的概率 | **p < 0.05 认为显著** |
| 置信区间 | 真实效果的可能范围，如 [+1%,+5%] | **区间不含 0 才显著** |
| 样本量 / MDE | 太小易误判，太大影响广 | 比例类开 3~7 天，留存 1~2 周 |

### 5. 三类指标（缺一不可）

| 类型 | PUSH | 信息流 |
|------|------|--------|
| 核心指标 | CTR、带来的启动 / 时长 | 人均时长、CTR、人均 VV、有效 DAU |
| 辅助指标 | 到达率、展示率、点击量 | 人均曝光 / 点击、完读 / 完播率 |
| 护栏（反向） | 通知关闭率、卸载率、7 日留存 | 跳出率、APP 整体时长、卸载率 |

> **别只看北极星指标**：给 PUSH 加声音震动 CTR 涨了，但用户嫌烦关通知 / 卸载 —— 护栏指标必须看。

### 6. 7 大陷阱（90% 的错误）
1. 样本量不足 / 时间太短（有新奇效应）
2. SRM 样本比例失衡
3. 多重比较（天天看、看多指标易假阳性）
4. 辛普森悖论
5. 实验互相干扰（流量重叠）
6. 只看相对不看绝对
7. 没有 AA 测试

### 7. DID 双重差分法（重点，来自飞书培训材料）

**解决什么**：当没法做纯随机实验时，同时剔除两个干扰 ——
- 只比「组间」不行：忽略两组**固有差异**（发达地区本就增长快）
- 只比「前后」也不行：忽略**大盘时间趋势**（整体上行，不改也涨）

**核心公式**：
> **政策效应 =（政策后政策组 − 政策前政策组）−（政策后对照组 − 政策前对照组）**

**四格理解**：

| 分组 / 时间 | 干预前 | 干预后 | 组内差分 |
|-------------|--------|--------|----------|
| 政策组 | Y₁₀ | Y₁₁ | Δ₁ = Y₁₁ − Y₁₀ |
| 对照组 | Y₀₀ | Y₀₁ | Δ₀ = Y₀₁ − Y₀₀ |
| — | 固有差异 | 政策后差异 | **DID = Δ₁ − Δ₀** |

> Δ₁ 含「真实效应 + 时间趋势」，Δ₀ 只含「时间趋势」，一减，时间趋势被抵消，剩下真实效应。（配套 deck 有可拖动的 DID 计算器演示。）

**三大前提假设**：① **平行趋势**（最关键，无干预时两组趋势平行）② SUTVA（个体独立、非双边市场）③ 无遗漏变量偏差。

**与其他方法对比**：DID（控时间趋势+固有差异，数据要求低）· PSM（匹配相似个体）· SCM（合成对照组）· RD（断点附近近似随机）。

**双边市场的实验难题**：双边市场（电商 / 外卖 / 网约车 / 社交，供给方+需求方相互依赖）普通 AB 会失效。例：闲鱼给实验组买家发券促成交，会**挤占对照组买家「原本能买到」的机会** → 实验组涨、对照组被拖累，高估效果。解法：按市场 / 地域 / 时间分流，或用 DID / 合成控制等准实验方法。

### 8. 结果可信度 · 8 条 Checklist
1. 开了多久？样本量够吗？覆盖周末？新奇效应过了吗？
2. 分流均匀吗？SRM / AA 校验过吗？两组基线一致吗？
3. 核心指标提升多少？p 值？置信区间含 0 吗？
4. 护栏 / 反向指标有没有负向？
5. 分维度看效果一致吗？有没有辛普森悖论？
6. 有没有其他同时开的实验干扰？
7. 绝对值提升多少？业务价值大吗？
8. 结论和建议符合数据吗？

## 第 6 课 · 综合实战与答疑

> **一句话主旨**：用一个完整案例把前 5 课串成能力。

### 1. 综合实战案例：PUSH CTR 突然下降

| 步骤 | 要点 |
|------|------|
| STEP 1 明确问题 | 定位下降原因 / 判断是否新策略导致 / 给回滚 or 优化 or 观察建议 |
| STEP 2 数据校验 | 先排除数据自身问题（更新延迟 / 埋点丢失 / 口径变化） |
| STEP 3 对比分析拆维度 | 时间逐步降、热点推送 3%（贡献 80%）、新用户 7%→3.5%、问题在「展示→点击」 |
| STEP 4 深入原因 | 时政内容不匹配新用户 + 凌晨发送 + 灰度只测老用户 |
| STEP 5 结论建议 | 不全量回滚但立刻调整；预计恢复到 7.5%+ |

> **知识点串联**：取数拆维度(第2·3课) + 数据校验(第1·4课) + 好维度定病灶 / 对比分群漏斗(第4课) + 灰度与 AB 判断(第5课) + 结论先行(第4课)。

### 2. 常见问题 Q&A
- **Q1 SQL 和看板对不上**：排查口径 → 表（ADS vs DWM）→ 去重 → 更新时间。
- **Q2 SQL 跑很久没出来**：先查是不是漏 date 分区（90% 是这个）→ 先查 1 天 → 别 SELECT * → 先过滤再 JOIN → 大数据量拆月跑。
- **Q3 AI 生成的 SQL 敢用吗**：简单单表校验后可用；复杂多表必须查字段真实性 + 过滤是否加全 + 先查 1 天看数值合理 + 对比历史。
- **Q4 AB 一周不显著**：算样本量能测多大 MDE、看效果是否真的很小、分维度看某群体是否显著、不要为显著硬凑 p 值。
- **Q5 怎么找根因（MECE 原则）**：拆时间 → 拆人群 → 拆业务模块 → 拆链路 → 匹配业务动作（上线 / 活动 / bug / 运营动作）。

### 3. 后续学习路径
- **第一阶段（1~2 周）· 基础上手**：把练习题跑一遍，日常取数自己写 + AI 辅助。
- **第二阶段（1~2 月）· 独立分析**：独立完成 2~3 个完整小分析。
- **第三阶段（长期）· 长期提升**：复杂分析和数据同学对思路；做 AB 前一起 review 设计。

> 最后一句：**数据分析是实践技能，多写多练多踩坑，成长最快。**

## 附录 A · 常见坑速查表

| 坑 | 后果 | 正确做法 |
|----|------|----------|
| 忘加 date 分区 | 扫全表跑几小时 | WHERE 第一行写 date |
| 算人数漏 DISTINCT | DAU 翻好几倍 | `COUNT(DISTINCT did)` |
| 除法没乘 1.0 | 率算成 0 | 分子乘 `*1.0` |
| 时长单位没转 | 结果大 6 万倍 | 毫秒 `/60000` 转分钟 |
| 分母为 0 | SQL 报错 | `NULLIF(分母, 0)` |
| 没处理 NULL | 结果变 NULL | `COALESCE(x, 0)` |
| JOIN 漏 ON | 笛卡尔积爆炸 | 写完 JOIN 立刻写 ON |
| 多表关联直接 SUM | 指标翻倍 | 先验证关联后记录数 |
| 只看大盘平均 | 辛普森悖论误判 | 大盘异动先拆维度 |
| 只看北极星指标 | 忽略负向 | 必看护栏 / 反向指标 |
| AB 提前看数据 | 多重比较假阳性 | 定好周期和核心指标再看 |
| AI 生成不校验 | 瞎编字段 / 错口径 | 4 步校验后再跑 |
