01 为什么需要多表关联（JOIN）？

上节课学的SQL都是从单张表查询，但实际业务中我们经常需要：

  • 把APP启动数据和信息流消费数据关联起来看（miui_di + label_di）
  • 把PUSH点击用户和后续的留存/消费行为关联起来
  • 把用户属性标签和行为数据关联起来做分群分析

这时候就需要用JOIN把多张表通过共同的"key"（比如date+did）拼接在一起查。

02 JOIN关联查询核心知识

JOIN的基本语法

SELECT a.字段, b.字段
FROM 表1 a
JOIN 表2 b
  ON a.关联key = b.关联key
WHERE 过滤条件
💡 技巧：表名后面加a/b是给表起别名，简化书写，避免字段重名冲突。

最常用的两种JOIN：INNER JOIN vs LEFT JOIN

这两个是产品同学99%场景会用到的，理解区别非常重要：

INNER JOIN（内连接）
逻辑：只保留两张表都能匹配上的数据，匹配不上的直接丢掉
适用场景：查"既启动了APP，又看了信息流"的用户
通俗比喻：查"同时参加了数学考试和语文考试"的学生，缺考一门的都不算
LEFT JOIN（左连接）
逻辑：以左边表为基准，左边表数据全部保留，右边表匹配不上的字段填NULL
适用场景：查"所有启动了APP的用户，他们有没有看信息流（没看的显示0）"
通俗比喻：查"所有参加数学考试的学生，他们的语文成绩是多少（缺考语文的显示NULL）"

案例：关联两张核心表，查PUSH启动用户的信息流消费

需求：查2026-06-28通过PUSH启动APP的用户，他们的人均信息流时长是多少？

-- 思路：
-- 1. 从miui_di表中筛选出通过PUSH启动的用户（左表a）
-- 2. LEFT JOIN关联label_di表（右表b），取他们的信息流消费数据
-- 3. 聚合计算指标
SELECT
    COUNT(DISTINCT a.did) AS push_user_cnt,
    ROUND(SUM(COALESCE(b.feed_dura, 0)) / COUNT(DISTINCT a.did) / 60000, 2) AS avg_feed_duration_min,
    ROUND(SUM(COALESCE(b.news_vv_cnt + b.short_vv_cnt + b.mini_vv_cnt, 0)) * 1.0 / COUNT(DISTINCT a.did), 2) AS avg_vv_per_user
FROM (
    -- 子查询先筛出PUSH启动用户
    SELECT date, did
    FROM iceberg_zjyprc_hadoop.browser.dwm_browser_event_aggregation_miui_di
    WHERE date = 20260628
      AND is_app_dau_2024 = 1
      AND COALESCE(did, '') != ''
      AND app_launch_way IN ('点击push', 'subscribe_push')
) a
LEFT JOIN (
    -- 子查询先筛出有效信息流数据
    SELECT date, did, feed_dura, news_vv_cnt, short_vv_cnt, mini_vv_cnt
    FROM iceberg_zjyprc_hadoop.browser.dwm_browser_event_aggregation_label_di
    WHERE date = 20260628
      AND is_dau_feed_dapan_2024 = 1
      AND COALESCE(did, '') != ''
      AND is_top = 0
) b
ON a.date = b.date AND a.did = b.did

关键细节：

  • 关联条件必须同时带上date和did，因为表是按did+date粒度的，同一个用户不同日期是不同记录
  • 用COALESCE(字段, 0)把匹配不到的NULL值转换成0，避免SUM/AVG时结果异常（NULL和任何数运算都是NULL）
  • 先在子查询里过滤数据再关联，性能比直接关联全表好很多！

JOIN避坑指南

  1. 一定要写关联条件ON：漏写ON会产生笛卡尔积，100万用户×100万用户=1万亿条数据，直接把集群搞挂
  2. 关联字段类型要一致：a.did是varchar字符串，b.did如果是bigint数字类型，关联会失败或结果不对，可以用CAST转类型
  3. 注意多对多关联：如果两张表不是1对1关系，关联后数据会重复膨胀，导致指标计算偏大
  4. 小表放前面：LEFT JOIN把数据量小的表放左边，性能更好

03 子查询与CTE（WITH语句）：把复杂SQL拆成简单块

复杂需求一步写完会很乱，容易出错，推荐用WITH语句把逻辑拆成多个临时表，像搭积木一样写SQL，可读性极强。

CTE（WITH语句）语法

WITH 
临时表1 AS (
    SELECT ... FROM ... WHERE ...
),
临时表2 AS (
    SELECT ... FROM 临时表1 JOIN ... 
)
-- 最后从最终的临时表查询
SELECT ... FROM 临时表2

案例：对比PUSH启动用户 vs 全体用户的指标差异

需求：2026-06-28，对比PUSH启动用户和全体用户的人均APP时长、人均信息流时长、信息流VV渗透率

WITH 
-- 步骤1：全体APP用户基础指标
all_user AS (
    SELECT did, app_dura
    FROM iceberg_zjyprc_hadoop.browser.dwm_browser_event_aggregation_miui_di
    WHERE date = 20260628
      AND is_app_dau_2024 = 1
      AND COALESCE(did, '') != ''
),
-- 步骤2：PUSH启动用户
push_user AS (
    SELECT DISTINCT did
    FROM iceberg_zjyprc_hadoop.browser.dwm_browser_event_aggregation_miui_di
    WHERE date = 20260628
      AND is_app_dau_2024 = 1
      AND COALESCE(did, '') != ''
      AND app_launch_way IN ('点击push', 'subscribe_push')
),
-- 步骤3：信息流消费数据
feed_data AS (
    SELECT did, feed_dura,
        news_vv_cnt + short_vv_cnt + mini_vv_cnt AS vv
    FROM iceberg_zjyprc_hadoop.browser.dwm_browser_event_aggregation_label_di
    WHERE date = 20260628
      AND is_dau_feed_dapan_2024 = 1
      AND COALESCE(did, '') != ''
      AND is_top = 0
),
-- 步骤4：关联拼接用户标签
user_label AS (
    SELECT
        a.did,
        a.app_dura,
        CASE WHEN b.did IS NOT NULL THEN 1 ELSE 0 END AS is_push_user,
        COALESCE(c.feed_dura, 0) AS feed_dura,
        COALESCE(c.vv, 0) AS vv
    FROM all_user a
    LEFT JOIN push_user b ON a.did = b.did
    LEFT JOIN feed_data c ON a.did = c.did
)
-- 步骤5：最终聚合对比
SELECT
    CASE WHEN is_push_user = 1 THEN 'PUSH启动用户' ELSE '非PUSH启动用户' END AS user_group,
    COUNT(DISTINCT did) AS user_cnt,
    ROUND(AVG(app_dura) / 60000, 2) AS avg_app_duration_min,
    ROUND(AVG(feed_dura) / 60000, 2) AS avg_feed_duration_min,
    ROUND(SUM(CASE WHEN vv > 0 THEN 1 ELSE 0 END) * 1.0 / COUNT(DISTINCT did), 4) AS vv_penetration
FROM user_label
GROUP BY CASE WHEN is_push_user = 1 THEN 'PUSH启动用户' ELSE '非PUSH启动用户' END
💡 优势：用WITH拆成5个步骤，每一步逻辑都很清晰，不容易出错，而且AI生成这种拆分后的SQL准确率会高很多。

04 常见SQL错误避坑（都是血泪教训）

坑点后果怎么避免
忘记加分区过滤date扫全表，查几小时出不来，影响集群写SQL先写WHERE date条件
JOIN漏写ON条件笛卡尔积，数据量爆炸写完JOIN立刻写ON关联条件
多表关联后直接SUM/COUNT数据重复，指标翻倍/翻好几倍先验证关联后的记录数是不是对的
直接用AVG算人均会把NULL的用户排除，导致人均偏高用SUM(指标)/COUNT(DISTINCT did)更稳妥
没处理NULL值计算结果是NULL，看起来像0但实际不是用COALESCE(字段, 0)把NULL转成0
整数除法没乘1.0CTR/渗透率算出来是0除法都乘1.0转成小数再算
时长单位没转结果大6万倍毫秒转分钟除以60000，转小时除以3600000

05 SQL性能优化入门：让你的查询跑的更快

产品同学不用懂太深的优化，记住这几个原则，查询速度能快10倍：

  1. 越早过滤越好：先在子查询里把日期、有效用户、需要的字段过滤好，再做JOIN，不要先关联全表再过滤
  2. 只查需要的字段：SELECT *会把所有字段都查出来，浪费资源，需要什么字段写什么字段
  3. 避免大表关联大表：如果能先聚合再关联，就不要用明细数据关联
  4. 数据量太大时分步跑：如果查3个月以上的数据，可以拆成按月查，或者先汇总成临时表再关联
  5. 优先用DWM层聚合表：不要自己从DWD明细表算，DWM表已经提前聚合好了，快很多

查询跑不出来怎么办？

  • 先查小范围日期（比如先查1天）验证逻辑对不对，再放大时间范围
  • 看是不是没加分区过滤
  • 看是不是产生了笛卡尔积
  • 找数据同学帮忙优化，不要一直重试浪费集群资源

06 AI辅助复杂SQL进阶 🤖

复杂需求怎么提给AI？

不要一次性丢一个大需求给AI，要拆解成步骤告诉AI，每一步要做什么，用什么表，输出什么字段，准确率会大幅提升：

复杂需求提示词示例： 
请帮我写Hive SQL，按以下步骤实现：
步骤1：从miui_di表筛选2026-06-28的有效用户（is_app_dau_2024=1，did非空），标记是否是PUSH启动（app_launch_way in ('点击push','subscribe_push')），取did、app_dura字段，别名t1
步骤2：从label_di表筛选2026-06-28的信息流有效消费用户（is_dau_feed_dapan_2024=1，is_top=0，did非空），取did、feed_dura、(news_vv_cnt+short_vv_cnt+mini_vv_cnt) as vv，别名t2
步骤3：t1 LEFT JOIN t2 ON t1.did = t2.did，关联后COALESCE把NULL转成0
步骤4：按是否PUSH启动分组，统计用户数、人均APP时长（分钟，保留2位）、人均信息流时长（分钟，保留2位）、VV渗透率（vv>0的用户占比，保留4位）
要求：用WITH语句拆分步骤，加注释，注意过滤条件和单位转换

AI写复杂SQL的常见坑怎么修？

AI常见错误修复方法
幻觉不存在的字段对照数据地图把字段名替换成正确的
关联条件漏了date补上date关联条件（如果是多天查询的话）
没处理NULL值加上COALESCE把NULL转成0
指标口径错了（比如CTR公式）把正确的公式明确告诉AI
性能差，大表直接关联让AI先过滤聚合再关联，拆成多个WITH子句
计算人均用了AVG而不是SUM/COUNT改成SUM(指标)/COUNT(DISTINCT did)的写法

慢查询/报错怎么让AI帮忙排查？

把完整SQL+报错信息/问题描述发给AI：

我写的SQL跑了半小时没出来，帮我看看哪里可以优化：
[贴SQL]

AI基本能快速定位到忘记加过滤条件、笛卡尔积、重复关联这类常见问题。

07 本节课小结

✅ 要点2
复杂SQL用WITH（CTE）拆成多个临时表，像搭积木一样写，逻辑清晰不容易错。
✅ 要点3

✅ 要点4
性能优化原则：早过滤、少取字段、优先用聚合表、分步跑。
✅ 要点5
复杂需求拆成步骤告诉AI，生成后重点校验关联逻辑、口径、NULL处理，不要直接复制就用。
