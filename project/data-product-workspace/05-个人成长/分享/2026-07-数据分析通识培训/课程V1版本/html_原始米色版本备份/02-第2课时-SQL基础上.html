一基础概念扫盲（先搞懂这些词是什么意思）

在学SQL之前，先把最常听到的几个概念用大白话讲清楚，完全不用记，有个印象就行：

名词大白话解释你需要知道的
数据库 存所有数据的"大Excel文件"，比Excel能存多得多的数据（能存几百亿行），查询速度更快 你不用管它存在哪里，只要知道从哪张表查就行
表（Table） 类似Excel里的一个sheet页，按行列存数据 我们日常用的都是数据同学加工好的汇总表（DWM/ADS层），不是原始日志，不用自己处理脏数据
字段（列） 类似Excel里的列，每个字段存一类信息：日期、用户ID、点击次数、时长... 字段名都是英文缩写，比如did是设备ID（代表一个用户），date是日期，cnt结尾的一般是次数，dura结尾的一般是时长
记录（行） 类似Excel里的一行，一条记录就是一个用户一天/一个PUSH的汇总数据 我们用的用户行为汇总表，基本都是「每个用户每天一行」，不会有重复行
SQL 和数据库说话的语言，你告诉它你要什么数，它帮你算出来返回给你 语法非常像英语，读出来就知道是什么意思，不用记复杂逻辑
查询 就是你给数据库提需求，让它算数据给你 你写一段SQL，点运行，等几十秒到几分钟出结果，结果可以直接导出Excel
分区 数据库按日期把数据分成一小块一小块，就像把Excel按天拆成不同文件 最重要！查数据一定要加date日期过滤，不然会查所有历史数据，跑几小时都出不来，还可能影响别人用
聚合 把多行数据汇总成一个数，比如把1000万个用户的时长加起来算总时长 我们算指标几乎都要用到聚合，不用你自己手动加，SQL自动算
去重 算人数的时候，同一个用户只算一次 算DAU/UV这种"多少人"的指标，必须去重，不然会重复计数
语法 就是写SQL的固定格式，就像中文要说"我吃饭"而不是"饭吃我"，顺序错了数据库看不懂 语法就那固定5个关键词，顺序固定，不用死记硬背
注释 SQL里写的说明文字，数据库不会执行，写给人看的 两个减号--后面的内容就是注释，我们例子里有很多注释帮你理解
运行/执行 把你写的SQL提交给数据库，让它开始算数据 点界面上的"运行"按钮就行，跑的过程中可以取消
结果集 SQL运行完返回给你的表格数据，和Excel长得一样 可以直接复制、导出Excel，用来做分析或做PPT

💡 给0基础同学的几个心理建设

  1. SQL真的不难：它不是编程语言，没有循环、判断这些复杂逻辑，就是固定的几个关键词拼起来，告诉数据库你要什么
  2. 不用背语法：忘了就查，或者直接让AI帮你写，我们的目标是"能看懂、会改简单条件、会校验结果"
  3. 写错了没关系：SQL写错了最多就是报错，不会把数据改坏，也不会把数据库搞挂（我们都是查数据，没有修改权限），大胆试
  4. 先抄再改：我们给的所有例子都是日常用的，你先把例子里的日期改成你要查的，跑通了再慢慢改其他条件

二SQL是什么？长什么样？

SQL（Structured Query Language）是结构化查询语言，它的语法非常像英语，你按顺序读就能明白意思。

举个最简单的例子，你一看就懂：

-- 查询2026年6月28日的DAU（日活跃用户数）
SELECT COUNT(DISTINCT did) AS dau
FROM browser.dwm_browser_event_aggregation_miui_di
WHERE date = 20260628
  AND is_app_dau_2024 = 1
翻译成大白话就是：
从dwm_browser_event_aggregation_miui_di这张表里，
筛选出日期是20260628、且是有效活跃用户的数据，
对设备ID去重后数一下有多少个，
给这个结果起个名字叫dau。

是不是很简单？SQL的核心就是这固定的几部分，按顺序写就行。

三基础查询固定结构：SELECT - FROM - WHERE

所有简单查询都遵循这个固定顺序，不要搞混顺序：

SELECT 【你要什么字段/算什么指标】
FROM 【从哪张表查】
WHERE 【筛选什么条件的数据】
GROUP BY 【按什么维度分组】（如果要分天/分群算就加）
ORDER BY 【按什么排序】（可选）

就这5个关键词，记住顺序：SELECT → FROM → WHERE → GROUP BY → ORDER BY，90%的日常查询都够用了。

我们一个一个讲，从最简单的开始，每个都配真实业务例子。

3.1 最简单的单值查询：查单日DAU

需求：查2026-06-28当天，浏览器APP的DAU（日活跃用户数）是多少？

SELECT COUNT(DISTINCT did) AS dau
FROM iceberg_zjyprc_hadoop.browser.dwm_browser_event_aggregation_miui_di
WHERE date = 20260628
  AND is_app_dau_2024 = 1
  AND COALESCE(did, '') != ''
逐行解释（面向0基础）：
• COUNT(DISTINCT did) AS dau：
  • COUNT：数个数
  • DISTINCT did：对did（设备ID，代表一个用户）去重，同一个用户只算一次
  • AS dau：给算出来的结果起个别名叫dau，结果里列名就显示dau
  • 👉 这一行的意思：数一下有多少个不重复的用户ID，这就是DAU
• FROM iceberg_zjyprc_hadoop.browser.dwm_browser_event_aggregation_miui_di：
  • 👉 从用户日汇总行为表（miui_di表）里查数据
• WHERE date = 20260628：
  • 👉 非常重要！只查2026年6月28日这一天的数据，这就是分区过滤，不加会跑死
• AND is_app_dau_2024 = 1：
  • 👉 只统计判定为有效活跃的用户，过滤掉测试账号、异常数据
• AND COALESCE(did, '') != ''：
  • 👉 过滤掉设备ID为空的异常数据

运行结果就是一个数：

dau
8562341

就是说当天DAU是856万左右。

📝 补充：WHERE条件常用写法（过滤数据就靠它）

WHERE就是筛选条件，和Excel里的筛选一模一样，常用的就这几种：

写法大白话意思例子
字段 = 值等于某个值date = 20260628 只看6月28日；is_app_dau_2024 = 1 只看有效活跃用户
字段 != 值不等于某个值is_top != 1 过滤掉置顶内容
字段 > 值 / 字段 < 值大于/小于feed_dura > 0 只看有时长的用户
字段 BETWEEN 值1 AND 值2在两个值之间（包含两端）date BETWEEN 20260620 AND 20260628 看20号到28号
字段 IN (值1, 值2, 值3)在这几个值里面app_launch_way IN ('点击icon', '点击push') 只看这两种启动方式
字段 LIKE '%关键词%'包含某个关键词app_launch_way LIKE '%搜索%' 启动方式里带"搜索"的都算
字段 IS NULL / 字段 IS NOT NULL是空值/不是空值did IS NOT NULL 过滤掉没有设备ID的异常数据
条件1 AND 条件2两个条件都满足date = 20260628 AND is_app_dau_2024 = 1
条件1 OR 条件2满足其中一个条件就行app_launch_way = '点击push' OR app_launch_way = 'subscribe_push'
💡 提示：字符串（文字内容）要用单引号包起来，比如'点击icon'；数字不用引号，直接写就行，比如20260628、1。

3.2 查趋势：按日期分组，查每天的DAU

需求：查2026-06-20到2026-06-28这9天，每天的DAU是多少，看趋势。

SELECT 
    date,
    COUNT(DISTINCT did) AS dau
FROM iceberg_zjyprc_hadoop.browser.dwm_browser_event_aggregation_miui_di
WHERE date BETWEEN 20260620 AND 20260628  -- 日期范围，包含20号和28号
  AND is_app_dau_2024 = 1
  AND COALESCE(did, '') != ''
GROUP BY date  -- 按日期分组，每天算一个DAU
ORDER BY date  -- 按日期从小到大排序，看趋势更清楚
新增的语法解释：
• date BETWEEN 20260620 AND 20260628：日期在这个区间内，等价于date >= 20260620 AND date <= 20260628
• GROUP BY date：按日期分组，相同日期的数据放在一起算，每天出一行结果
• ORDER BY date：按日期升序排列（从小到大），如果要倒序就写ORDER BY date DESC
⚠️ 重要规则：SELECT里出现的、不是聚合函数包起来的字段（这里是date），必须出现在GROUP BY后面！不然会报错。

202606218654321
202606228712345

四5个必学聚合函数：算所有指标都靠它们

聚合函数就是把多行数据汇总计算成一个/多个结果，日常用的就5个，记下来：

函数作用什么时候用注意事项
COUNT(DISTINCT 字段)去重后计数算多少人：DAU、UV、点击用户数、留存用户数✅ 算人数必须加DISTINCT！不然会重复计数
COUNT(字段)不去重计数算多少次：PV、曝光次数、点击次数、发送量不用加DISTINCT，一次点击就算一次
SUM(字段)求和算总量：总时长、总VV、总发送量、总点击量把每个用户的数值加起来
AVG(字段)平均值算人均XX：人均时长、人均点击次数、人均VV注意：AVG是对行求平均，算人均最好用SUM/人数
MAX(字段) / MIN(字段)最大值 / 最小值找峰值、最小值：单日最高DAU、最低CTR用得比较少

还有一个非常常用的辅助函数：

函数作用
ROUND(数字, 小数位数)四舍五入保留指定小数位，比如ROUND(0.08234, 4)得到0.0823也就是8.23%

五核心指标SQL示例大全（覆盖日常90%需求）

所有例子都用真实表，直接改日期/条件就能用。

📊 指标1：分渠道DAU

需求：查2026-06-28当天，按启动方式拆分的DAU：主启、PUSH启动、第三方调起、搜索启动、其他

SELECT 
    date,
    COUNT(DISTINCT did) AS total_dau,  -- 总DAU
    -- 主启：点击icon/桌面书签
    COUNT(DISTINCT CASE WHEN app_launch_way IN ('点击icon', '点击桌面书签') THEN did END) AS main_launch_dau,
    -- PUSH启动：点击push/订阅push
    COUNT(DISTINCT CASE WHEN app_launch_way IN ('点击push', 'subscribe_push') THEN did END) AS push_launch_dau,
    -- 第三方调起
    COUNT(DISTINCT CASE WHEN app_launch_way = '第三方调起' THEN did END) AS third_party_dau,
    -- 搜索启动
    COUNT(DISTINCT CASE WHEN app_launch_way LIKE '%搜索%' THEN did END) AS search_launch_dau
FROM iceberg_zjyprc_hadoop.browser.dwm_browser_event_aggregation_miui_di
WHERE date = 20260628
  AND is_app_dau_2024 = 1
  AND COALESCE(did, '') != ''
GROUP BY date
这里用到了新语法：CASE WHEN，相当于Excel里的IF函数： 
CASE WHEN 条件 THEN 满足条件返回的值
     WHEN 条件2 THEN 满足条件2返回的值
     ELSE 都不满足返回的值（可选）
END
在这个例子里，满足启动方式条件的才返回did，不满足的返回空，COUNT(DISTINCT)就不会统计它，实现分群计数。
📊 指标2：信息流基础指标（曝光、点击、CTR、人均时长、总VV）

需求：查2026-06-28当天，信息流的核心指标：DAU、总曝光、总点击、CTR、人均信息流时长（分钟）、总VV。

用的是label_di信息流汇总表：

SELECT 
    date,
    COUNT(DISTINCT did) AS feed_dau,                    -- 信息流DAU：看了信息流的用户数
    SUM(expos_cnt) AS total_expose,                    -- 总曝光次数：多少次内容被展示
    SUM(click_cnt) AS total_click,                     -- 总点击次数：多少次内容被点击
    -- CTR（点击率）= 总点击 / 总曝光，*1.0是为了避免整数除法，保留4位小数
    ROUND(SUM(click_cnt) * 1.0 / SUM(expos_cnt), 4) AS ctr,
    -- 人均信息流时长：总时长（毫秒）/ 用户数 / 60000 转成分钟，保留2位小数
    ROUND(SUM(feed_dura) * 1.0 / COUNT(DISTINCT did) / 60000, 2) AS avg_feed_duration_min,
    -- 总VV = 资讯VV + 短视频VV + 小视频VV
    SUM(news_vv_cnt + short_vv_cnt + mini_vv_cnt) AS total_vv,
    -- 人均VV
    ROUND(SUM(news_vv_cnt + short_vv_cnt + mini_vv_cnt) * 1.0 / COUNT(DISTINCT did), 2) AS avg_vv_per_user
FROM iceberg_zjyprc_hadoop.browser.dwm_browser_event_aggregation_label_di
WHERE date = 20260628
  AND is_dau_feed_dapan_2024 = 1  -- 有效信息流活跃用户
  AND COALESCE(did, '') != ''
  AND is_top = 0  -- 过滤置顶内容，只看自然推荐的内容
GROUP BY date
关键细节解释（0基础必看）：
1. 为什么要乘*1.0？
  • 数据库里两个整数相除，默认会取整，比如1/2结果是0而不是0.5
  • 乘一个1.0转成小数，除法就能得到正确的小数结果了
  • 所有算率、算比例、算人均的时候，都记得乘1.0
2. 时长单位：
  • 表里存的feed_dura字段单位是毫秒，转成秒除以1000，转成分钟除以60000
  • 不要直接看毫秒数值，会大6万倍
3. 为什么用SUM(click_cnt)/SUM(expos_cnt)而不是AVG(ctr)？
  • 因为AVG是对每个用户的CTR求平均，会受每个用户曝光量多少的影响，结果不对
  • 正确的大盘CTR一定是「总点击/总曝光」，先求和再相除
📊 指标3：分内容类型的VV和时长

需求：查2026-06-28当天，按内容类型拆分的VV和时长：资讯、短视频、小视频各自的VV、占比、人均消费时长。

SELECT 
    date,
    -- 总VV和总时长
    SUM(news_vv_cnt + short_vv_cnt + mini_vv_cnt) AS total_vv,
    SUM(feed_dura) AS total_duration_ms,
    -- 资讯
    SUM(news_vv_cnt) AS news_vv,
    ROUND(SUM(news_vv_cnt) * 1.0 / SUM(news_vv_cnt + short_vv_cnt + mini_vv_cnt), 4) AS news_vv_ratio,
    ROUND(SUM(news_dura) * 1.0 / COUNT(DISTINCT CASE WHEN news_vv_cnt > 0 THEN did END) / 60000, 2) AS avg_news_duration_min,
    -- 短视频
    SUM(short_vv_cnt) AS short_vv,
    ROUND(SUM(short_vv_cnt) * 1.0 / SUM(news_vv_cnt + short_vv_cnt + mini_vv_cnt), 4) AS short_vv_ratio,
    ROUND(SUM(short_dura) * 1.0 / COUNT(DISTINCT CASE WHEN short_vv_cnt > 0 THEN did END) / 60000, 2) AS avg_short_duration_min,
    -- 小视频
    SUM(mini_vv_cnt) AS mini_vv,
    ROUND(SUM(mini_vv_cnt) * 1.0 / SUM(news_vv_cnt + short_vv_cnt + mini_vv_cnt), 4) AS mini_vv_ratio,
    ROUND(SUM(mini_dura) * 1.0 / COUNT(DISTINCT CASE WHEN mini_vv_cnt > 0 THEN did END) / 60000, 2) AS avg_mini_duration_min
FROM iceberg_zjyprc_hadoop.browser.dwm_browser_event_aggregation_label_di
WHERE date = 20260628
  AND is_dau_feed_dapan_2024 = 1
  AND COALESCE(did, '') != ''
  AND is_top = 0
GROUP BY date
📊 指标4：PUSH核心指标（发送、到达、点击、CTR）

需求：查2026-06-20到2026-06-28每天的PUSH核心指标：发送量、到达量、点击量、到达率、CTR（点击/到达）、PUSH启动DAU。

如果PUSH有单独的表就用PUSH表，这里用miui_di表先算：

SELECT 
    date,
    -- PUSH发送/到达/点击量（如果用PUSH专用表更准确）
    SUM(push_send_cnt) AS total_send,    -- 总发送量
    SUM(push_arrive_cnt) AS total_arrive, -- 总到达量
    SUM(push_click_cnt) AS total_click,  -- 总点击量
    -- 到达率 = 到达/发送
    ROUND(SUM(push_arrive_cnt) * 1.0 / NULLIF(SUM(push_send_cnt), 0), 4) AS arrive_rate,
    -- CTR = 点击/到达（注意不是点击/发送！）
    ROUND(SUM(push_click_cnt) * 1.0 / NULLIF(SUM(push_arrive_cnt), 0), 4) AS ctr,
    -- PUSH启动DAU
    COUNT(DISTINCT CASE WHEN app_launch_way IN ('点击push', 'subscribe_push') THEN did END) AS push_dau
FROM iceberg_zjyprc_hadoop.browser.dwm_browser_event_aggregation_miui_di
WHERE date BETWEEN 20260620 AND 20260628
  AND is_app_dau_2024 = 1
  AND COALESCE(did, '') != ''
GROUP BY date
ORDER BY date
新语法解释：NULLIF(字段, 0)
• 防止分母是0导致报错，比如某天没有发送PUSH，到达量是0，直接除会报错
• NULLIF的意思是：如果字段等于0，就返回NULL，除法结果就是NULL，不会报错
• 所有除法都建议加NULLIF保护
📊 指标5：分新老用户的指标拆分

需求：查2026-06-28当天，新用户（安装7天内）和老用户的DAU、人均时长、CTR对比。

SELECT 
    date,
    -- 用户分群
    CASE WHEN datediff(from_unixtime(date, 'yyyyMMdd'), install_date) < 7 THEN '新用户(7天内)'
         ELSE '老用户'
    END AS user_type,
    -- 指标
    COUNT(DISTINCT did) AS dau,
    ROUND(SUM(feed_dura) * 1.0 / COUNT(DISTINCT did) / 60000, 2) AS avg_duration_min,
    ROUND(SUM(click_cnt) * 1.0 / NULLIF(SUM(expos_cnt), 0), 4) AS ctr
FROM iceberg_zjyprc_hadoop.browser.dwm_browser_event_aggregation_label_di
WHERE date = 20260628
  AND is_dau_feed_dapan_2024 = 1
  AND COALESCE(did, '') != ''
  AND is_top = 0
GROUP BY date,
    CASE WHEN datediff(from_unixtime(date, 'yyyyMMdd'), install_date) < 7 THEN '新用户(7天内)'
         ELSE '老用户'
    END

常用小技巧：查字段枚举值

如果你不知道某个字段都有哪些取值（比如启动方式有哪几种、内容类型怎么分），用DISTINCT查：

-- 查app_launch_way字段都有哪些取值
SELECT DISTINCT app_launch_way
FROM iceberg_zjyprc_hadoop.browser.dwm_browser_event_aggregation_miui_di
WHERE date = 20260628
  AND app_launch_way IS NOT NULL

运行结果就会列出所有不同的启动方式名称，你就知道怎么写CASE WHEN了。

六AI辅助写SQL：不用记语法也能写

现在完全不需要死记硬背语法，AI可以帮你生成90%的SQL代码，你只要会提需求、会校验结果就可以了。

6.1 推荐工具

  1. 公司内部NL-SQL工具（首选）：直接用中文说需求，自动生成可直接运行的SQL，还能直接查结果
  2. 通用大模型（Claude/豆包/GPT）：适合复杂需求，但需要你提供表结构和字段说明
  3. IDE内置AI助手：写SQL的时候自动补全，报错了直接把错误贴给它帮你改

6.2 高质量提示词模板（照着填空就行）

AI生成的SQL准不准，90%取决于你需求说的清不清楚，按照这个模板写：

请帮我写Hive SQL，要求如下：

【使用的表】：iceberg_zjyprc_hadoop.browser.dwm_browser_event_aggregation_label_di（信息流用户日汇总表）
【主要字段说明】：
- date：日期，格式yyyyMMdd，分区字段
- did：设备ID
- expos_cnt：信息流曝光次数
- click_cnt：信息流点击次数
- feed_dura：信息流消费时长，单位毫秒
- news_vv_cnt/short_vv_cnt/mini_vv_cnt：资讯/短视频/小视频VV
- is_dau_feed_dapan_2024：是否有效信息流活跃用户，1=是
- is_top：是否置顶内容，0=自然内容，1=置顶

【查询需求】：
查询2026年6月20日到6月28日，每天的以下指标：
1. 信息流DAU
2. 总曝光、总点击、CTR=点击/曝光，保留4位小数
3. 人均信息流时长（分钟），保留2位小数
4. 总VV（三类VV相加）、人均VV
过滤条件：只看自然内容（is_top=0），有效活跃用户，did不为空
按日期升序排列
❌ 反面例子（AI会瞎编字段、错口径）：
"帮我查一下信息流的CTR和时长"

6.3 必须做的4步校验（AI会幻觉，一定要检查！）

AI生成的SQL一定要检查这4点，再运行：

  1. 字段校验：用到的字段是不是这张表里真的有的？对照数据地图看，不要有AI瞎编的字段
  2. 分区校验：有没有加date过滤？不加的话会跑几小时出不来，这是最常见错误
  3. 逻辑校验：
    • 算"多少人"是不是用了COUNT(DISTINCT did)？有没有漏DISTINCT？
    • 除法是不是乘了1.0？有没有加NULLIF防止分母为0？
    • 单位对不对？毫秒转分钟有没有除以60000？
    • 指标公式对不对？CTR是点击/曝光还是点击/到达？有没有搞反分子分母？
  4. 结果校验：跑出来的数值合不合理？
    • DAU是不是在几百万的量级？如果跑出来只有几万肯定错了
    • CTR是不是在1%~20%之间？如果出来80%肯定错了
    • 人均时长是不是在几分钟到几十分钟？出来几万分钟肯定单位错了
    • 和你已知的历史数据/看板数据对一对，差很多肯定哪里错了
💡 最佳实践：先查1天的数据验证逻辑对不对，结果没问题再扩大时间范围。

七新手最常犯的7个错误（避坑！）

我把大家刚学SQL最容易踩的坑列出来，写完对照检查：

❌ 错误 01
忘记加date分区过滤
后果：查询扫全表，跑几小时出不来，甚至影响集群
✅ 修复：WHERE第一行就写date条件，养成习惯
❌ 错误 02
算人数没加DISTINCT
后果：COUNT(did)算出来是记录数，DAU比真实大好几倍
✅ 修复：只要算"XX用户数"，必须COUNT(DISTINCT did)
❌ 错误 03
除法没乘1.0
后果：两个整数相除取整，比如CTR算出来是0
✅ 修复：所有除法都在分子乘*1.0
❌ 错误 04
单位忘记转换
后果：时长直接用毫秒数值，出来的结果大6万倍
✅ 修复：时长/毫秒字段，转分钟除以60000，转秒除以1000
❌ 错误 05
漏掉有效用户过滤
后果：没加is_app_dau_2024=1，包含测试账号/异常数据，指标不准
✅ 修复：查用户行为表都加上对应is_dau过滤
❌ 错误 06
分母为0没处理
后果：某天分母是0，SQL直接报错跑不出来
✅ 修复：所有除法分母都加NULLIF(分母, 0)
❌ 错误 07
直接用AI生成的SQL不校验
后果：AI瞎编字段/错口径，结果完全错
✅ 修复：按上面4步校验完再跑

八本节课小结

  1. 不用怕SQL：语法很简单，固定5个关键词顺序：SELECT → FROM → WHERE → GROUP BY → ORDER BY
  2. 核心概念：表=sheet，字段=列，记录=行，分区=按日期分块，算人数必须去重
  3. 5个聚合函数：COUNT(DISTINCT)算人数，SUM求和，AVG平均，ROUND保留小数
  4. 核心指标：DAU、分渠道DAU、分内容类型VV/时长、PUSH指标、分新老用户，都给了现成SQL，改日期就能用
  5. AI提效：用模板写提示词让AI生成，但是一定要做4步校验再运行
  6. 避坑：加date分区、算人数加DISTINCT、除法乘1.0、注意单位转换

  2. 查昨天的信息流总曝光、总点击、CTR、人均时长（分钟）、总VV
  3. 查昨天的资讯/短视频/小视频各自的VV和VV占比
  4. 尝试用AI帮你写上面的SQL，然后对照我们讲的要点做校验
