为什么产品
要懂数据？

先讨论一个问题 没有标准答案

在浏览器信息流里，一条时政类图文 A 和一条娱乐类短视频 B，
该怎么评估谁更好、更受用户喜欢、更值得被推给更多人？
🙋
凭感觉？

"我觉得 A 更有价值" —— 每个人口味不同，说服不了别人。

💬
问用户？

用户嘴上说的，和实际点的，往往是两回事。

📊
看数据

用点击、停留、完播等真实行为说话 —— 今天的主角。

为什么要看数据？

"不要看他说了什么，
要看他做了什么。"用户行为代表真实意图，比语言更可信

"汽车出现前，用户不会说想要汽车，
只会说想要一辆更快的马车。"行为数据噪声更小，能发现用户没说出口的偏好

📈 更关键的是：需求 → 上线 → 效果验证 → 下一轮迭代，数据是这个闭环里唯一的"评判标准"。

埋点记录什么？有哪些类型？

一条埋点，回答四个问题：

  • 谁 —— did / uid、设备、版本、渠道
  • 什么时候 —— 时间戳、日期分区
  • 做了什么 —— 曝光 / 点击 / 播放 / 停留 + 对象
  • 结果如何 —— 停留时长、是否看完、后续行为
埋点类型产品关心的指标
页面PV、UV、页面停留时长
点击CTR、点击分布、路径转化
曝光曝光量、有效曝光
播放播放时长、完播率、跳出率
自定义PUSH 到达、下载完成、支付成功

类型再多，本质都一样：把一次行为，拆成"事件 + 参数"记录下来。

埋点怎么设计：事件模型 5 要素

基于事件模型，埋点在 OneTrack 里被拆成"元数据"。设计一个埋点，就是想清楚这 5 件事：

🕐
When

时间戳
SDK 自动采集

👤
Who

用户信息
自然维度参数

📍
Where

点位设计
在哪触发

⚡
What

事件设计
发生了什么

🔧
How

参与方式
事件维度参数

事件三要素：操作行为 · 参数 / 属性名 · 参数 / 属性值 —— 后面 SQL 里的字段，就是这些参数。
来源：内容生态产运班 · 数据埋点该怎么设计

埋点的结构层级与设计原则

自上而下的结构层级：

页面 page
→
模块 module
→
元素 item

一个页面由多个模块构成，一个模块又包含多个元素 —— 埋点顺着这个层级铺开。

设计三步 + 三大原则：

  • 明确场景（起止点）
  • 明确上报内容（时长 / 点击 / 曝光…）
  • 明确上报参数（from_page / sessionID…）
① 对行为抽象 ② 兼容 & 复用 ③ 自上而下设计

SECTION 03

公司数据体系分层

同一份数据，为什么有好几张表？

ODS → DWD → DWM → ADS 从原始到汇总

层级全称说明产品常用度
ODS操作数据层原始数据，直接从埋点 / 日志同步，未清洗❌ 基本不用
DWD明细数据层清洗后的明细，单条行为记录⚠️ 复杂分析，数据量大
DWM中间汇总层按"用户 + 日期"聚合的轻度汇总表✅ 日常取数首选
ADS应用数据层高度汇总结果表，用于报表 / 看板✅ 固定指标，灵活性差
💡 产品取数原则：优先用 DWM 层聚合表 —— 性能是 DWD 的 10~100 倍，不用自己写复杂的去重聚合。

SECTION 04

数据地图使用方法

数据的"字典"，找表查字段的入口

数据地图能查什么 · 怎么用

它是公司的数据资产目录，可以查：

  • 表的基本信息、负责人
  • 字段含义、类型、枚举值
  • 分区规则、更新频率
  • 表血缘（从哪来、被谁用）
  • 样例数据预览

核心操作 5 步：

搜索表
按业务线 / 功能 / 表名关键词搜
看分区
绝大多数按 date 分区（yyyyMMdd）
看字段 + 样例
理解维度 / 指标字段，预览数据
查血缘
口径不满足就向上游找更明细的表

🔎 小技巧：优先找名字带 _di（日分区）的聚合表；浏览器业务表都在 iceberg_zjyprc_hadoop.browser. 库下。

05

SECTION 05 · 重点

内容生态核心常用表

两张表，覆盖 80% 的日常取数需求

浏览器 & 信息流：核心两表

…aggregation_miui_di

粒度：did + date（每个用户每天一行）
用途：APP 启动、版本、启动方式、整体使用时长、基础行为

…aggregation_label_di

粒度：did + date（每个用户每天一行）
用途：信息流曝光 / 点击 / 播放、消费时长、内容标签、用户标签

-- 两表通过 date + did 关联 FROM …miui_di a JOIN …label_di b ON a.date = b.date AND a.did = b.did

完整库表名：iceberg_zjyprc_hadoop.browser.dwm_browser_event_aggregation_miui_di / _label_di

核心字段速查

miui_di · APP 整体行为

did 设备 ID，算 DAU 用 COUNT(DISTINCT did)
date 日期分区 yyyyMMdd，查询必加！
is_app_dau_2024 =1 为当日 APP 活跃用户
app_launch_way 启动方式：icon / push / 搜索…
app_dura APP 时长（毫秒，/60000 转分钟）

label_di · 信息流消费

is_dau_feed_dapan_2024 =1 为信息流活跃用户
is_top 是否置顶，自然流量过滤 is_top=0
expos_cnt / click_cnt 曝光 / 点击次数
news_vv_cnt / short_vv_cnt 图文 / 短视频 VV
feed_dura 信息流消费时长（毫秒）

🧠 记不住没关系 —— 这些下节课写 SQL 会反复用到，用着用着就熟了。

常用过滤条件 抄下来就能用

-- 1. 分区过滤：必须加！否则扫全表，慢到跑不出来 WHERE date BETWEEN 20260601 AND 20260630 -- 2. 过滤有效活跃用户 AND is_app_dau_2024 = 1 -- 3. 过滤空 did AND COALESCE(did, '') != '' -- 4. 自然信息流过滤置顶内容 AND is_top = 0
⚠️ 头号坑：忘记加分区过滤 date → 查询扫全表，跑几小时出不来，甚至把集群搞挂被数据同学"关心"。
