02

SECTION 02

SQL 是什么？
长什么样？

它的语法，读出来就懂

-- 查询 2026 年 6 月 28 日的 DAU（日活跃用户数） SELECT COUNT(DISTINCT did) AS dau FROM …dwm_browser_event_aggregation_miui_di WHERE date = 20260628 AND is_app_dau_2024 = 1
翻译成大白话：从用户日汇总表里，筛选出日期 = 6/28 且有效活跃的数据，对设备 ID 去重数一下有多少个，结果起名叫 dau。

是不是很简单？SQL 的核心就是这固定几部分，按顺序写就行。

03

SECTION 03

基础查询：五段式结构

SELECT → FROM → WHERE → GROUP BY → ORDER BY

记住这个固定顺序

SELECT 【你要什么字段 / 算什么指标】
FROM 【从哪张表查】
WHERE 【筛选什么条件的数据】
GROUP BY 【按什么维度分组】 ← 要分天 / 分群才加
ORDER BY 【按什么排序】 ← 可选
就这 5 个关键词，顺序固定，90% 的日常查询都够用了。

逐行拆解：查单日 DAU

SELECT COUNT(DISTINCT did) AS dau FROM …dwm_browser_event_aggregation_miui_di WHERE date = 20260628 AND is_app_dau_2024 = 1 AND COALESCE(did, '') != ''
  • COUNT(DISTINCT did) —— 对设备 ID 去重后数个数，这就是 DAU
  • WHERE date = 20260628 —— 分区过滤，不加会跑死
  • is_app_dau_2024 = 1 —— 只统计有效活跃用户，滤掉测试 / 异常
  • COALESCE(did,'') != '' —— 过滤空设备 ID

WHERE 常用写法 和 Excel 筛选一样

写法意思例子
= / !=等于 / 不等于date = 20260628
> / <大于 / 小于feed_dura > 0
BETWEEN … AND区间（含两端）date BETWEEN 20260620 AND 20260628
IN (…)在这几个值里app_launch_way IN ('点击icon','点击push')
LIKE '%词%'包含关键词app_launch_way LIKE '%搜索%'
AND / OR都满足 / 满足其一date=… AND is_app_dau_2024=1

💡 字符串用单引号 '点击icon'；数字不用引号，直接写 20260628。

查趋势：GROUP BY 按天分组

-- 查 6/20 ~ 6/28 每天的 DAU，看趋势 SELECT date, COUNT(DISTINCT did) AS dau FROM …miui_di WHERE date BETWEEN 20260620 AND 20260628 AND is_app_dau_2024 = 1 GROUP BY date -- 每天算一个 DAU ORDER BY date -- 按日期升序，趋势更清楚
⚠️ 铁律：SELECT 里非聚合的字段（这里是 date），必须出现在 GROUP BY 里，否则报错。
04

SECTION 04

5 个聚合函数
算所有指标都靠它

日常就这 5 个 + 1 个辅助

函数作用什么时候用
COUNT(DISTINCT x)去重计数算多少人：DAU、UV、点击用户数
COUNT(x)不去重计数算多少次：PV、曝光、点击、发送量
SUM(x)求和算总量：总时长、总 VV、总发送
AVG(x)平均人均（算人均更推荐 SUM/人数）
MAX / MIN最大 / 最小找峰值、最小值
ROUND(x, n)保留小数ROUND(0.08234, 4) → 0.0823
🔑 记住一条：算"多少人"必须 COUNT(DISTINCT did)，漏了 DISTINCT，DAU 会翻好几倍。
05

SECTION 05 · 重点

核心指标 SQL 大全

改日期 / 条件就能用，覆盖日常 90% 需求

信息流核心指标：CTR · 人均时长 · VV

SELECT date, COUNT(DISTINCT did) AS feed_dau, -- CTR = 点击/曝光，*1.0 避免整数除法 ROUND(SUM(click_cnt) * 1.0 / SUM(expos_cnt), 4) AS ctr, -- 人均时长：总时长(ms) / 人数 / 60000 → 分钟 ROUND(SUM(feed_dura) * 1.0 / COUNT(DISTINCT did) / 60000, 2) AS avg_min, SUM(news_vv_cnt + short_vv_cnt + mini_vv_cnt) AS total_vv FROM …label_di WHERE date = 20260628 AND is_dau_feed_dapan_2024 = 1 AND COALESCE(did,'') != '' AND is_top = 0 GROUP BY date

三个高频细节：*1.0 避免整数除法 · 时长 /60000 转分钟 · 大盘 CTR 用 SUM/SUM 不用 AVG。

Demo · 口径对不对，差在两个细节 拨开关看

原始数据（固定）

总曝光　12,000,000
总点击　960,000
总时长　5.09e12 ms
用户数　8,560,000
CTR 分子乘 1.0（避免整数除法）
时长 ÷ 60000（毫秒转分钟）

* 关掉任一开关，看结果怎么"崩"。

算出来的指标

CTR = 点击 / 曝光0.08
人均信息流时长9.91 分钟
✅ 两个开关都开着 —— 口径正确，结果合理。

💡 对应新手错误 03、04：整数除法让 CTR 变 0；忘记 /60000 让"人均时长"变成天文数字。

来源：内容生态产运班 · 结合业务口径

分群统计的利器：CASE WHEN

-- 按启动方式拆分 DAU（CASE WHEN 相当于 Excel 的 IF） SELECT date, COUNT(DISTINCT did) AS total_dau, COUNT(DISTINCT CASE WHEN app_launch_way IN ('点击push','subscribe_push') THEN did END) AS push_dau FROM …miui_di WHERE date = 20260628 AND is_app_dau_2024 = 1 GROUP BY date
满足条件的才返回 did、不满足返回空，COUNT(DISTINCT) 就只数满足条件的人 —— 实现"一次查询、多个分群"。PUSH 指标、新老用户对比，都是同一套路。

SECTION 06

AI 辅助写 SQL
不用记语法也能写

高质量提示词模板 照着填空

请帮我写 Hive SQL： 【表】…label_di（信息流用户日汇总表） 【字段】date 日期分区 yyyyMMdd；did 设备ID； expos_cnt 曝光；click_cnt 点击；feed_dura 时长(毫秒)； is_dau_feed_dapan_2024=1 有效活跃；is_top 置顶(0=自然) 【需求】查 6/20~6/28 每天：DAU、CTR=点击/曝光(4位)、 人均时长(分钟,2位)、总VV(三类相加) 【过滤】is_top=0、有效活跃、did非空；按日期升序
❌ 反面："帮我查一下信息流的 CTR 和时长" —— AI 会瞎编字段、错口径。

AI 会幻觉：必做 4 步校验

字段校验
用到的字段这张表真有吗？对照数据地图，别用 AI 瞎编的
分区校验
有没有加 date 过滤？这是最常见错误
逻辑校验
算人数有 DISTINCT？除法乘 1.0？单位转了？CTR 公式对吗？
结果校验
数值合理吗？DAU 几百万级、CTR 1%~20%、人均几分钟
💡 最佳实践：先查 1 天验证逻辑，再扩大时间范围。

新手最常犯的 7 个错误

🕐
忘加 date 分区

扫全表跑几小时 → WHERE 第一行写 date

👥
算人数漏 DISTINCT

DAU 翻好几倍 → COUNT(DISTINCT did)

➗
除法没乘 1.0

整数取整 CTR = 0 → 分子乘 *1.0

⏱️
单位没转换

毫秒当分钟大 6 万倍 → /60000

🧹
漏有效用户过滤

含测试/异常 → 加 is_dau 过滤

🚫
分母为 0 没处理

直接报错 → NULLIF(分母, 0)

🤖
AI 生成不校验直接用

瞎编字段 / 错口径 → 按上面 4 步校验完再跑
