SECTION 01

JOIN：把多张表
拼起来查

为什么需要 JOIN？

单表不够用，实际业务经常要"跨表看"：

  • 把 APP 启动数据和信息流消费数据关联起来看（miui_di + label_di）
  • 把 PUSH 点击用户和后续的留存 / 消费行为关联起来
  • 把用户属性标签和行为数据关联起来做分群分析
JOIN = 用共同的"钥匙"（比如 date + did）把多张表拼接在一起查。

最常用两种：INNER vs LEFT 99% 场景

INNER JOIN 内连接

只保留两张表都能匹配上的数据，匹配不上的丢掉。
→ 查"既启动 APP 又看了信息流"的用户。
比喻：同时参加了数学和语文考试的学生。

LEFT JOIN 左连接

以左表为基准全保留，右表匹配不上的填 NULL。
→ 查"所有启动 APP 的用户，有没有看信息流（没看显示 0）"。
比喻：所有考数学的学生，语文成绩缺考记 NULL。

Demo · 同样两张表，两种 JOIN 结果 切换看

两张小表（did 为钥匙）

表A · 启动APP表B · 看信息流(feed_dura)
u1u2 → 30
u2u3 → 12
u3u4 → 8

* 以表A为左表，ON a.did = b.did。

关联结果

💡 LEFT JOIN 后，右表没匹配上的是 NULL —— 记得用 COALESCE(feed_dura, 0) 转成 0，否则 SUM/AVG 结果异常。

来源：内容生态业务线取数场景

案例：PUSH 启动用户的信息流消费

-- 先在子查询里各自过滤，再关联（性能更好） SELECT COUNT(DISTINCT a.did) AS push_user, ROUND(SUM(COALESCE(b.feed_dura,0)) / COUNT(DISTINCT a.did) / 60000, 2) AS avg_min FROM ( SELECT date,did FROM …miui_di WHERE date=20260628 AND is_app_dau_2024=1 AND app_launch_way IN('点击push','subscribe_push') ) a LEFT JOIN ( SELECT date,did,feed_dura FROM …label_di WHERE date=20260628 AND is_dau_feed_dapan_2024=1 AND is_top=0 ) b ON a.date=b.date AND a.did=b.did
  • 关联条件带上 date + did —— 表是 did+date 粒度
  • COALESCE(…,0) 把 NULL 转 0，避免 SUM/AVG 异常
  • 先在子查询过滤再关联，比关联全表快很多

JOIN 避坑指南

💥
一定写 ON 条件

漏写 = 笛卡尔积，百万×百万，直接搞挂集群

🔤
关联字段类型一致

字符串 did 关联数字 did 会失败，必要时 CAST

📈
注意多对多

不是 1 对 1 会数据膨胀，指标算大

📦
小表放前面

LEFT JOIN 把小表放左边，性能更好

🚨 头号事故：JOIN 漏写 ON → 笛卡尔积，1 万亿条数据。写完 JOIN 立刻写 ON。

SECTION 02

CTE：把复杂 SQL
拆成简单块

WITH 语句：像搭积木一样写 SQL

WITH 临时表1 AS ( SELECT … FROM … WHERE … ), 临时表2 AS ( SELECT … FROM 临时表1 JOIN … ) -- 最后从最终的临时表查询 SELECT … FROM 临时表2
好处 1 · 清晰

复杂需求拆成多步，每步逻辑一目了然，不容易出错。

好处 2 · AI 友好

让 AI 生成拆分后的 SQL，准确率会高很多。

案例：PUSH 用户 vs 全体，5 步拆解

WITH all_user AS ( -- ① 全体 APP 用户 + app_dura … ), push_user AS ( -- ② PUSH 启动用户 … ), feed_data AS ( -- ③ 信息流消费数据 … ), user_label AS ( -- ④ 关联拼接，CASE WHEN 打标 + COALESCE 补 0 SELECT a.did, a.app_dura, CASE WHEN b.did IS NOT NULL THEN 1 ELSE 0 END AS is_push, COALESCE(c.feed_dura,0) AS feed_dura FROM all_user a LEFT JOIN push_user b ON a.did=b.did LEFT JOIN feed_data c ON a.did=c.did ) SELECT is_push, COUNT(DISTINCT did), -- ⑤ 最终分组聚合对比 ROUND(AVG(app_dura)/60000,2) FROM user_label GROUP BY is_push

💡 5 步各司其职，逻辑清晰、便于校验 —— 复杂查询都建议这么拆。

常见错误 & 性能优化

常见 SQL 错误（血泪教训）

坑点后果怎么避免
忘加分区 date扫全表跑几小时先写 WHERE date
JOIN 漏 ON笛卡尔积爆炸写完 JOIN 立刻写 ON
多表关联直接 SUM数据重复、指标翻倍先验证关联后记录数
直接 AVG 算人均排除 NULL 用户，偏高SUM(x)/COUNT(DISTINCT did)
没处理 NULL结果变 NULLCOALESCE(x, 0)
整数除法没 *1.0率算出来是 0分子乘 1.0
时长单位没转结果大 6 万倍毫秒 /60000 转分钟

性能优化 5 原则 快 10 倍

越早过滤越好
子查询里先过滤日期 / 有效用户 / 字段，再 JOIN
只查需要的字段
别 SELECT *，要什么写什么
避免大表关联大表
能先聚合再关联，就别用明细关联
数据量大分步跑
查 3 个月以上拆成按月，或先汇总临时表
优先用 DWM 聚合表
别从 DWD 明细自己算，DWM 已聚合好，快很多
🐢 跑不出来先查 1 天验证逻辑 → 看是否漏分区 / 笛卡尔积 → 再找数据同学，别一直重试占资源。

SECTION 04

AI 辅助写复杂 SQL

复杂需求，拆成步骤喂给 AI

请帮我写 Hive SQL，按步骤实现： 步骤1 从 miui_di 筛 6/28 有效用户，标记是否 PUSH 启动，取 did、app_dura（别名 t1） 步骤2 从 label_di 筛 6/28 信息流有效消费，取 did、feed_dura、vv（别名 t2） 步骤3 t1 LEFT JOIN t2 ON did，COALESCE 补 0 步骤4 按是否 PUSH 分组，统计人数 / 人均时长 / VV 渗透率 要求 用 WITH 拆分、加注释、注意过滤与单位转换
🔑 不要一次丢大需求 —— 拆成"用什么表、输出什么字段"的步骤，AI 准确率大幅提升。

AI 写复杂 SQL 的坑，怎么修

AI 常见错误修复方法
幻觉不存在的字段对照数据地图替换成正确字段名
关联漏了 date补上 date 关联条件
没处理 NULL加 COALESCE(x, 0)
指标口径错（CTR 公式）把正确公式明确告诉 AI
大表直接关联，慢让它先过滤聚合再关联，拆 WITH
人均用了 AVG改成 SUM(x)/COUNT(DISTINCT did)

🐢 慢查询 / 报错：把完整 SQL + 报错发给 AI，基本能定位漏过滤、笛卡尔积、重复关联。
