## 综合实战案例：PUSH CTR突然下降怎么办？
我们用一个完整的真实场景，把前5节课学的内容串起来，走完从问题发现到结论输出的全流程。

### 第一步：明确问题
**场景**：你周一早上看数据，发现周日PUSH整体CTR从平时的8%降到了6%，下降了2个百分点，老板让你查一下原因，看看是不是上周四上线的新个性化推送策略有问题，要不要回滚。

**明确分析目标**：
- 定位CTR下降的原因
- 判断是不是新策略导致的
- 给出决策建议：回滚/优化/继续观察

**时间范围**：重点看上周四到周日，对比上周同期（上上周四到周日）
**核心指标**：PUSH CTR = 点击量 / 到达量
**拆分维度**：时间、PUSH类型、用户分群、漏斗环节、版本/渠道

---

### 第二步：数据校验（先确认数据没问题）
首先排除数据本身的问题：
1. 检查数据是不是更新延迟？用不同的查询方式重新跑一遍，确认数据准确
2. 检查埋点上报有没有问题？点击/到达埋点有没有丢失？
3. 检查口径有没有变化？CTR公式是不是被改了？
→ 确认数据没问题，CTR确实是降了2个点

---

### 第三步：对比分析，确认异常范围
先做对比，搞清楚是整体降了还是局部降了：
1. **时间对比**：
   - 和上周日比：上周日CTR是8.1%，本周日是6%，确实下降
   - 看每日趋势：周四新策略刚上线时CTR是8.3%（略涨），周五8%，周六7.2%，周日6%，是逐步下降的，不是突然掉下来
   - 看小时级：周日全时段都低，不是某个小时异常，凌晨2-4点特别低
2. **分PUSH类型对比**：
   - 时政新闻类：10% → 9%（降1pct）
   - 娱乐内容类：8% → 7.5%（降0.5pct）
   - 热点推送类：新上线的类型，周日发送量很大，CTR只有3%
   - 其他类型：基本不变
   → 热点推送类贡献了80%的下降！
3. **分用户群对比**：
   - 老用户：8% → 7.5%（降0.5pct）
   - 新用户（7天内新增）：7% → 3.5%（降3.5pct）
   → 新用户下降特别明显，是重灾区
4. **漏斗拆解**：
   - 下发→到达：80% → 80%（没变）
   - 到达→展示：75% → 74%（基本没变）
   - 展示→点击：10% → 7.5%（这里掉了2.5pct，问题出在点击环节）
→ 问题定位：**新上线的热点推送类PUSH，周日凌晨给新用户发了一大批，这批PUSH点击率特别低（只有3%），拉低了整体CTR**，其他类型和老用户基本正常。

---

### 第四步：深入分析原因
为什么热点类PUSH对新用户点击率这么低？
1. 看这批PUSH的内容：都是比较深度的时政社会新闻，标题比较严肃
2. 看新用户特征：刚安装APP的新用户，之前看的都是娱乐、生活类内容，对时政新闻兴趣低
3. 看发送时间：周日凌晨2点发送，大部分用户在睡觉，早上起来看到一堆推送，直接划走了
4. 对比小流量测试数据：上线前小流量灰度只测了老用户，没有测新用户，灰度时CTR是提升的，全量后加了新用户就崩了

---

### 第五步：结论与建议
**结论输出（结论先行）**：
1. 周日PUSH CTR下降2个百分点，80%的原因是新上线的热点推送全量后，给新用户在凌晨批量推送了不匹配的时政内容，这批PUSH点击率仅3%，拉低了整体指标
2. 新策略本身对老用户是正向的（CTR略涨0.3pct），但对新用户有严重负向，不是完全不能用，是需要优化
3. 不需要全量回滚，但需要立刻调整

**行动建议**：
1. 立刻调整热点推送策略：新用户注册前7天不发时政类热点推送，只发泛娱乐内容，预计CTR能恢复到7.5%以上
2. 优化推送时间：凌晨0点到早8点不批量发推送，避免用户早上起来被一堆推送打扰
3. 后续新策略灰度测试必须包含新用户群体，不能只测老用户，小流量观察3天没问题再全量
4. 建立PUSH分人群内容池，不同用户匹配不同类型内容

---

### 附：完整过程用到的知识
- SQL：分群聚合、多表关联，用AI辅助生成SQL，校验结果

- 实验/灰度知识：小流量测试要覆盖全人群，不能只测部分用户，灰度时间要足够

## 常见问题答疑Q&A
整理一些日常大家经常问的问题：

### Q1：我自己写SQL跑出来的数据和看板不一样，怎么办？
A：按这个顺序排查：
1. 先看口径：你用的过滤条件和看板是不是一样？有没有加is_app_dau_2024=1？有没有过滤is_top=0？时间范围是不是一致？
2. 看表：看板是不是用的ADS层汇总表，你用的是DWM层？口径可能不一样，找数据同学确认看板口径
3. 看去重：是不是COUNT(DISTINCT did)？有没有重复计算？
4. 看更新时间：看板数据是不是延迟更新了？

### Q2：SQL跑了很久没出来怎么办？
A：
1. 先检查是不是没加date分区过滤！90%的慢查询都是这个原因
2. 先查1天的数据验证逻辑对不对，再扩大时间范围
3. 不要SELECT *，只查需要的字段
4. 先在子查询里过滤好数据再JOIN，不要关联全表
5. 查3个月以上大数据量，建议拆成按月跑，或者找数据同学优化

### Q3：AI生成的SQL敢直接用吗？
A：简单的单表查询校验一下可以用，复杂的多表关联一定要：
1. 先检查字段是不是都存在，有没有AI幻觉的字段
2. 检查过滤条件是不是加全了
3. 先查1天数据看结果数值在合理范围吗？（比如DAU是不是几百万级，CTR是不是在合理区间）
4. 和已知的历史数据/看板数据对比，对得上再用

### Q4：AB实验开了一周数据不显著怎么办？
A：
1. 先算一下现在的样本量能检测出多大的MDE？如果能检测出的最小效果都比你预期的大，说明样本量不够，需要扩大流量或者延长时间
2. 看效果是不是真的很小，如果提升不到0.5%，业务价值也不大，可以考虑停掉
3. 分维度看，是不是某个细分人群显著，整体不显著？如果是的话可以考虑只对那个群体上线
4. 不要为了显著而显著，硬凑p值<0.05是自欺欺人

### Q5：做分析时怎么找到问题的根因？
A：用MECE原则拆分维度，互相独立、完全穷尽：
1. 先拆时间：什么时候开始出问题？是突发还是逐步？
2. 再拆人群：哪些用户出问题？新/老/高/低活/版本/渠道？
3. 再拆业务模块：哪个功能/哪个内容/哪个入口出问题？
4. 再拆链路：漏斗哪个环节掉了？
5. 最后匹配对应时间点的业务动作：上线/活动/bug/运营动作，基本就能定位到

课程结束后，大家可以按这个路径继续提升：
1. **第一阶段（1~2周）**：把课件里的练习题都自己跑一遍，日常取数先尝试自己写SQL，用AI辅助，写完对照模板检查，遇到问题随时问
2. **第二阶段（1~2个月）**：独立完成2~3个完整的小分析，比如某个功能上线后的效果分析、某个指标波动排查，按照课程里的框架走

   - 复杂分析可以多和数据同学对思路，学习拆解问题的方法
