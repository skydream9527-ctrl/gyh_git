# 第7课时：信息流埋点设计（上）—— 核心事件模型

> 课时：1.5小时
> 目标：从「用数据」进阶到「设计数据」。理解信息流业务形态，掌握四个核心埋点事件（曝光 / 点击 / 进入 / 退出），搞懂「消费时长」到底是怎么算出来的，学会给事件设计参数。
> 前置：启动课（数据的一生）、第1课（埋点与事件模型 5 要素）。

---

## 一、开场：为什么产品也要懂「埋点设计」？

前 6 课我们学的是**用数据**——取数、分析、看实验。但有个更根本的问题：**这些数据是从哪来的？**

回顾启动课那条链路：`用户操作 → 埋点采集 → 日志加工 → 生产成表 → 你取数`。**埋点是整条链路的源头。**

- **源头错了，后面全错**：埋点漏了、口径定错了、事件设计缺失，那么后面再厉害的 SQL 和分析模型，算出来的都是错的数——业内叫 **GIGO（Garbage In, Garbage Out，垃圾进垃圾出）**。
- **产品是埋点需求的源头**：「一次曝光怎么算」「消费时长从哪到哪」「要不要区分题材」——这些是**业务定义**，得产品想清楚，不是研发拍脑袋、也不是数据同学能替你定的。
- **会设计埋点 = 你能拿到「对的、全的、口径一致」的数据**，这是做好一切分析的地基。

> 🎯 这两节课，我们就聚焦「数据的一生」的第一步：**信息流的埋点，到底该怎么设计。**

## 二、先看懂：信息流业务长什么样

### 2.1 什么是信息流（feed）
信息流 = 把内容（图文 / 视频 / 小说…）以**列表 / 瀑布流**的形式，按推荐算法**源源不断**地呈现给用户，用户往下滑就能一直看。

### 2.2 浏览器信息流 vs 资讯类 APP
两者形态相似，但场景不同，埋点设计侧重点也不同：

| 对比 | 浏览器信息流 | 资讯类 APP |
|------|-------------|-----------|
| 入口 | 工具型 APP 首页下方的 feed、快捷入口 | APP 本身就是一个大 feed |
| 用户心智 | **顺带看**（用完工具顺手刷两条） | **主动消费**（专门来看内容） |
| 内容形态 | 图文 / 短视频 / 小视频**混排** | 图文 / 视频 / 专题 / 直播 |
| 关注重点 | 从工具场景**导流**到内容、停留时长 | 消费深度、留存、题材偏好 |

> 💡 不管哪种，**信息流的通用结构是一样的**，埋点就埋在这个结构的各个"点位"上。

### 2.3 信息流的通用结构（埋点埋在哪）
```
频道 channel（推荐 / 视频 / 小说 / 本地…）
   └── 信息流列表 feed list（一屏屏往下滑）
          └── 内容卡片 card / item（一张张卡片）
                 └── 详情页 / 播放器 detail / player（点进去消费的地方）
```
这四层，正好对应第1课讲的埋点结构层级「**页面 page → 模块 module → 元素 item**」。**埋点设计，就是想清楚在这四层的哪些地方、记录用户的哪些动作。**

### 2.4 一次典型的用户消费旅程
```
打开 APP → 进入某频道 → 往下滑列表 → 卡片进入屏幕(曝光) → 点了一张卡(点击)
        → 进入详情页/播放器(进入) → 阅读/观看(消费) → 返回(退出) → 回列表继续滑…
```
> 🎯 **埋点，就是把这条旅程里的关键动作一个个记录下来。** 抓住其中最核心的 4 个动作，就抓住了信息流埋点的 80%。

## 三、核心事件模型：曝光、点击、进入、退出（本课重点）

> 一个事件 = 一个被记录下来的用户动作。信息流最核心的四个事件如下。

### 3.1 曝光 expose —— 内容「被看到」
- **定义**：一个内容卡片出现在用户屏幕上、被用户看到。
- **关键：什么才算「看到」？** 不能卡片一露头就算，否则飞快划过也算曝光、数据虚高。要定**有效曝光标准**：
  - 卡片进入屏幕可视区域的**面积 ≥ 50%**（露出一半以上）
  - **停留 ≥ 500 毫秒**（不是一闪而过）
  - 两个条件都满足，才上报一次曝光。
- **对应指标**：曝光量 `expos_cnt`（**CTR 的分母**）。

### 3.2 点击 click —— 用户「想看」
- **定义**：用户点击卡片，表达消费意愿。
- **在曝光的基础上，额外记**：点击时间、点击的区域（标题 / 封面图 / 作者 / 按钮）。
- **对应指标**：点击量 `click_cnt`（**CTR 的分子**）。`CTR = 点击 / 曝光`。

### 3.3 进入 enter —— 真正「开始消费」
- **定义**：点击后**成功进入**详情页 / 播放器，内容开始呈现。
- **为什么点击和进入要分开记？** 因为**点了不一定进得去**：加载失败、网络慢、秒退。点击→进入之间存在流失，分开记才能发现"点了进不去"的问题。
- **关键：记录进入时间戳**——这是后面算消费时长的**起点**。

### 3.4 退出 exit —— 「结束消费」（含时长，重点中的重点）
- **定义**：用户离开详情页 / 播放器（返回、切走、关闭 APP）。
- **关键：记录退出时间戳**——这是算消费时长的**终点**，同时记退出方式、消费进度（读到哪 / 看到第几秒）。
- **⚠️ 最重要的一句话：时长不是「采」出来的，是「算」出来的**——见第四节。

### 3.5 串起来：一次完整的消费链路
```
曝光 expose → 点击 click → 进入 enter →（消费）→ 退出 exit
   ▼            ▼           ▼                    ▼
 expos_cnt   click_cnt   进入量/秒退率      消费时长 = 退出 - 进入
```
这条链路本身就是一个**漏斗**（呼应第4课漏斗分析）：

| 环节 | 示意量级 | 能发现什么问题 |
|------|----------|---------------|
| 曝光 | 1000 | —— |
| 点击 | 80（CTR 8%） | 卡片吸引力够不够 |
| 进入 | 75（点击→进入 94%） | 有没有"点了进不去"（加载 / 秒退） |
| 有效消费 | 68 | 进去了有没有真的看 |

> 💡 只有把「点击」和「进入」拆开，你才能定位问题出在**卡片不吸引人**（曝光→点击低），还是**页面加载太差**（点击→进入低）。

## 四、时长怎么算：进入 + 退出的「配对」（本课重点）

### 4.1 时长的本质：一道减法
**消费时长 = 退出时刻 − 进入时刻。**

> 例：用户 20:00:00 进入一篇文章，20:03:20 退出 → 消费时长 = 200 秒。

为什么这么设计？因为**没办法直接"采集一个时长"**——用户还在看的时候，时长一直在变。只能记下「进入」和「退出」两个**时刻**，两者相减就是时长。

### 4.2 时长最容易踩的坑（⚠️ 面试和实战高频）
| 坑 | 场景 | 后果 | 怎么兜底 |
|----|------|------|----------|
| 只进入、没退出 | APP 崩溃、被系统杀进程、直接关机 | 时长丢失，或算出"看了 8 小时"的异常值 | **心跳埋点** + 超时截断 |
| 切到后台算不算 | 用户切微信去了，10 分钟后回来 | 前台时长 vs 总时长口径不清 | 明确定义：一般算**前台有效时长** |
| 同一内容多次进出 | 看一半退出，又点回来看 | 时长该累加还是取最长？ | 定义清楚：一般**累加**为该内容总时长 |
| 时间戳时区 / 精度 | 客户端和服务端时间不一致 | 时长为负数或异常 | 统一时间源、异常值过滤 |

### 4.3 心跳埋点 heartbeat（长时消费的兜底机制）
对于**看视频、读小说**这类长时消费，光靠"进入 / 退出"两个点不够稳（万一没退出就丢了）。所以增加**心跳埋点**：消费过程中**每隔一段时间（如 15 秒）上报一次「我还在看」**。

- 好处：就算最后没正常退出，也能用"最后一次心跳时间"兜底估算时长。
- 视频完播率、有效播放时长，也常基于心跳来算。

> 💡 **回到取数**：速查手册里的 `feed_dura`（信息流消费时长）、`consum_dura`（总消费时长）、`app_dura`（APP 时长），本质都是这样"进入/退出 + 心跳"采集后、聚合出来的。**你现在懂它们从哪来了。**

## 五、事件的参数设计

有了事件，还要给每个事件设计**参数**（属性）——参数就是后面 SQL 里能查的字段。

### 5.1 公共参数（几乎每个事件都要带）
沿用第1课的事件模型 5 要素，落到信息流：

| 5 要素 | 参数（示意） | 含义 |
|--------|-------------|------|
| **Who** 谁 | `did` / `uid`、`app_ver`、`channel_from` | 设备 / 用户、版本、渠道 |
| **When** 何时 | `event_time`（毫秒时间戳） | 事件发生时间 |
| **Where** 在哪 | `page`、`feed_channel`、`position`、`module` | 页面 / 频道 / 第几位 / 模块 |
| **What** 什么内容 | `content_id`、`content_type`、`author_id` | 内容 ID、**题材类型**、作者 |
| **How** 什么场景 | `from_page`、`session_id`、`request_id` | 来源页、会话、推荐请求 ID |

### 5.2 一次点击事件的报文（示意）
```json
{
  "event": "click",
  "did": "d_1001",
  "event_time": 1782648821000,
  "feed_channel": "推荐",
  "page": "feed",
  "position": 3,
  "content_id": "88132",
  "content_type": "short_video",
  "from_page": "home",
  "session_id": "s_99f0"
}
```

### 5.3 哪些参数、哪些事件要带（设计要点）
| 参数 | 曝光 | 点击 | 进入 | 退出 | 说明 |
|------|:---:|:---:|:---:|:---:|------|
| `content_id` / `content_type` | ✅ | ✅ | ✅ | ✅ | 全程带，才能按内容 / 题材归因 |
| `position` | ✅ | ✅ | | | 分析"第几位点击率最高" |
| `event_time` | ✅ | ✅ | ✅ | ✅ | 进入 / 退出的时间用来算时长 |
| `session_id` | ✅ | ✅ | ✅ | ✅ | 串起一次会话内的完整链路 |
| 消费进度 / 时长相关 | | | ✅ | ✅ | 退出时记看到哪、看了多久 |

> 🎯 **设计原则（承接第1课）**：① 对行为抽象（不同题材复用同一套事件）② 兼容 & 复用（参数命名统一）③ 自上而下（先想清楚要分析什么，再决定埋什么）。**先想清楚"要回答什么业务问题"，再倒推"该埋哪些事件和参数"。**

## 六、本节课小结
1. **产品是埋点需求的源头**，埋点设计错 → 后面分析全错（GIGO）。
2. 信息流通用结构：**频道 → 列表 → 卡片 → 详情 / 播放器**，埋点埋在这四层。
3. **四个核心事件**：曝光（被看到）、点击（想看）、进入（开始消费）、退出（结束消费）。
4. **点击与进入要拆开**，才能区分"卡片不吸引人"还是"页面加载差"。
5. **时长 = 退出 − 进入**，是算出来的；注意"只进入没退出"的坑，用**心跳埋点**兜底。
6. **参数设计**对应 Who/When/Where/What/How；`content_type`（题材）全程带，是下节课分题材分析的基础。
