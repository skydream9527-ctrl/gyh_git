# 数据产品 26年Q2复盘

> 复盘时间：2026-06-27
> 文档状态：草稿待确认

<callout emoji="💡">
【核心认知】**聚焦是资源受限下的第一原则**——在人力极度紧张的现实下，必须放弃全面铺开的思路，集中所有资源在单点上打透突破，用一个成功的点带动全局正向趋势，而不是分散兵力在多个战场同时作战。

【小结】Q2在团队资源配置与规划严重不匹配（规划8人实际仅2人到岗，缺口75%）的极端困难情况下，AI Native方向依然取得了关键突破：ICE Data Workbench v3落地、核心SQL类Skill上线、版本分析/实验分析Agent场景跑通，验证了AI在数据工作流中的价值和可行性，为Q3平台化建设打下了基础。数据基建方面，埋点重构基本停滞，但看板体系持续完善，日常业务需求未受影响。

【亮点】
- **ICE Data Workbench v3落地**：AI数据工作流工作台完成开发，成为团队AI能力的统一承载平台，是从零散Skill走向平台化的关键一步
- **核心Skill工具箱初成**：nl-sql、nl-mapping-table-sql、auto-analysis等自然语言生成SQL的核心Skill上线可用，支撑了最基础也是最高频的取数场景
- **Agent场景跑通验证**：借助产品同学支持，浏览器灰度版本分析Agent、实验分析Agent已可用，日报Agent在跑通阶段，验证了从单点Skill到完整工作流Agent的可行性
- **看板体系持续完善**：经营分析、实验看板、竞品监控日报等看板体系稳定运行，支撑了日常业务数据需求，看板稳定性、数据准确性保持99%+
- **认知升级**：
  1. **聚焦原则**：资源有限时必须收缩战线，单点打透再逐步扩展，贪多求全只会一事无成
  2. **场景切入**：AI落地必须从高频、标准化的真实业务场景切入，解决团队每天都在做的重复工作，避免务虚
  3. **平台化路径**：零散Skill无法形成规模化效应，必须通过统一工作台+标准化MCP协议承载能力，形成生态
  4. **基建前提**：数据资产（埋点、指标、知识库）的标准化和质量是AI效果的基础前提，基建不稳则上层应用效果无法保证

【不足之处】
- **资源规划严重失误**：OKR制定时对人员到岗风险预估不足，按8人配置规划工作，实际仅2人到岗，导致大部分KR无法按计划推进
- **埋点重构基本停滞**：Q2规划的信息流元素曝光点击二期、滑动埋点、浏览器启动退出埋点等均未推进，AI方向占用了几乎全部可用资源
- **KR整体完成度低**：Q2末提升3倍人效的核心O未达成，周度数据需求仍需人工处理，AI覆盖范围还比较有限
- **知识沉淀不足**：AI开发经验、Skill/Agent开发手册、踩坑记录等知识未系统化沉淀，团队无法快速复用
- **业务价值产出不够**：短剧/外投等业务方向受资源限制，未深入产出可量化的深度分析价值
</callout>

## 核心问题与反思

1. **资源规划严重脱离实际，未遵循聚焦原则**：OKR规划时按8人配置（1社招+2校招+2实习生+3外包），实际仅2人到岗（1社招+1实习生），资源缺口75%，这是Q2进展不达预期的根本原因。更重要的是，在资源明显不足的情况下，初期依然按原规划全面铺开多条战线，没有及时收缩聚焦，导致精力分散，单点突破力度不够。**教训：** 资源不足时必须立即收缩，集中全部力量打透一个点，再考虑扩展。

2. **AI Native方向验证成功，落地路径从零散Skill转向平台化**：Q2验证了AI在数据工作流中的价值，但也证明零散Skill建设效率低、难成体系，必须走**平台化+MCP生态+CLI工具**路线，通过统一ICE工作台承载能力，通过标准化MCP协议连接工具，通过CLI让能力触达每个工作环节，才能规模化提效。

3. **AI必须从真实高频业务场景切入，避免务虚**：版本分析、实验分析、日报等高频标准化场景能快速跑通验证，而过于宏大的目标（比如3倍人效、0积压）在资源不足时反而容易落空。**教训：** 优先解决团队每天都在做的重复工作，用一个场景的实际提效赢得认可，再逐步扩大覆盖范围。

4. **数据基建与AI建设需要平衡**：Q2几乎将全部资源投入AI方向，导致埋点重构停滞。长期来看，数据资产标准化是AI效果的基础，但在资源极度紧张时，需要找到平衡点——先做支撑AI所需的最小基建，再逐步补全。
