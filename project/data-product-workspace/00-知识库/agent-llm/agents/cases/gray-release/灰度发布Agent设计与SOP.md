# 灰度发布数据分析 Agent 设计案例

> 来源：ICE Workbench 内置垂直领域Agent
> 版本：V1.2 | 更新时间：2026-06-29
> 类型：垂直领域专业分析Agent（SOP驱动型）

---

## 一、Agent 定位与身份

### 核心定位
资深互联网产品数据分析专家 + 智能决策顾问，**专精于APP版本灰度发布全流程分析与放量决策**。

### 与AB实验Agent的区别
| 维度 | 灰度发布Agent | AB实验Agent |
|---|---|---|
| 分组依据 | app_version（版本号） | 实验桶（随机分流） |
| AA校验 | 大DAU产品通常免AA | 必须AA校验通过 |
| 核心目标 | 护栏监控 + 放量决策 | 因果推断 + 显著性检验 |
| 决策重点 | 核心指标不下跌即可放量 | 实验指标显著正向才上线 |

### identity.md 核心承诺
1. 版本号、放量阶段、关键变量、直接指标没确认前，不直接下结论
2. 数据查询和决策阈值优先读取Agent知识库中的规则、模板和案例
3. 放量/回滚/暂停只给建议和风险说明，不替用户执行不可逆决策
4. 角色自适应输出：PM看决策建议，分析师看统计细节

## 二、六阶段SOP标准流程（严格顺序执行，不可跳步）

### Phase 0 · 信息收集 📝
**目标**：收集灰度背景信息，5项必填缺一不可

| # | 字段 | 说明 | 校验规则 |
|---|---|---|---|
| 1 | 对照包版本号 | 基线版本（无实验变量） | X.Y.Z格式 |
| 2 | 业务包版本号 | 实验版本（包含本次改动） | X.Y.Z格式 |
| 3 | 关键变量 | 本次改动核心差异点 | 必填 |
| 4 | 直接指标 | 被变量直接影响的指标 | 至少1个 |
| 5 | 当前放量阶段 | 5%/10%/30%三选一 | 三选一 |

**交互模式**：
- 生成 `tmp_gray_input.md` 模板让用户填写
- 支持用户直接对话输入，仅追问缺失项
- 选填信息：变量所在页面/模块、变量影响机制（提供后解锁AI自主下钻）

### Phase 1 · 数据获取 🔄
**目标**：多数据源获取对照包vs业务包对比数据，标准化输出

| 优先级 | 数据源 | 工具 |
|---|---|---|
| 1（默认） | Kyuubi SQL查询 | `kyuubi_query` |
| 2 | 文件上传 | CSV/Excel解析 |
| 3 | 飞书表格 | feishu Skill |

**核心要求**：
- 枚举字段必须先探查真实值（`SELECT x, COUNT(*) GROUP BY x LIMIT 20`），不依赖猜测
- 标准化为统一 `ExperimentData` 数据结构
- 支持参数化SQL模板注入

### Phase 2 · 数据校验 ✅
**目标**：8项数据质量校验，确保分析基础可靠

| # | 校验项 | 规则 | 异常处理 | 严重级别 |
|---|---|---|---|---|
| 1 | 数据完整性 | 核心指标缺失 < 20% | ⚠️ 标注缺失指标 | 🔴 高 |
| 2 | SRM校验 | 两组样本比例±2%符合预期 | ❌ 暂停，重新分流 | 🔴 高 |
| 3 | 用户量级差异 | \|对照-实验\|/对照 < 10% | ⚠️ 降低可信度 | 🟡 中 |
| 4 | 样本量充足 | 任一组每日 > 1000 | ⚠️ 降低统计功效 | 🟡 中 |
| 5 | 统计功效 | 功效 ≥ 80% | ⚠️ 警示假阴性 | 🟡 中 |
| 6 | 异常波动 | 单日相对波动 < 20% | ⚠️ 排查异常事件 | 🟡 中 |
| 7 | 时间一致性 | 两组时间范围完全一致 | ❌ 暂停，统一时间 | 🔴 高 |
| 8 | AA基线稳定性 | 小流量±1%，中流量±0.5% | 记录偏移用于扣除 | 🟢 低 |

**暂停规则**：仅🔴高风险必须暂停，🟡中风险可继续但必须标注。

**免AA特殊处理**：
- 有AA数据 → 计算波动，超阈值在Phase3扣除基线偏移
- 无AA数据 → 标记`aa_status='ℹ️ 免AA'`，Phase2基线校验加权重

### Phase 3 · 四步核心分析 🔥
**严格按Step1→2→3→4顺序执行**

#### Step 1 · 核心指标分析（护栏指标）
三类约16个核心指标全部校验：
- 规模指标（5个）：DAU/曝光用户/有效用户/消费用户/次日留存
- 消费指标（3个）：人均停留时长/人均消费时长/人均VV
- 收入指标（~8个）：大盘收入/ARPU/信息流收入/ARPU/IPU/eCPM

**判定逻辑**：
```python
relative_change = (treatment - control) / abs(control) * 100
direction = '正向' if relative_change >= 0 else '平稳' if relative_change >= -3 else '下降'
violates_guardrail = direction == '下降' and relative_change < -3
```

**组合类型判定**：
| 组合 | 含义 | 决策倾向 | 是否需人工 |
|---|---|---|---|
| 全部正向 | 全部 ≥ 0 | ✅ 强烈建议放量 | 人工确认即可 |
| 部分正向+平稳 | 仅正向/平稳 | ✅ 可以放量 | 人工确认即可 |
| **此消彼长** ⚠️ | 正向+下降并存 | ⚠️ 必须人工评估 | **必须人工决策** |
| 出现下降 ❌ | 有指标 < -3% | ❌ 暂不放量，下钻 | 人工确认 |

#### Step 2 · 直接指标分析
验证关键变量是否真的影响了预期的直接指标：
- 显著且方向符合预期 → ✅ 符合预期
- 显著但反向 / 变化太小(<1%) → ❌ 不符合预期 / ⚠️ 不明显

#### Step 3 · 因果链路验证（关键）
确认核心指标变化确实由实验变量带来，三步验证：
1. **直接指标响应**：≥1个直接指标显著正向响应 → ✅
2. **核心指标跟随**：≥60%核心指标正向/平稳 → ✅
3. **AA基线扣除**：有偏移则扣除后重新判定 → ✅

任一步失败 → 因果链路断裂，需排查埋点/外部干扰/延长观察期。

#### Step 4 · 异常下钻分析（条件触发）
**触发条件**：核心指标下降或直接指标不符合预期

**AI自主下钻流程**：
1. 匹配下钻知识库，按关键词生成场景假设
2. 结合页面结构知识细化维度
3. 按假设执行下钻SQL，更新置信度
4. 综合判定根因，给出建议
5. 输出：假设列表+验证结果+根因定位+改进建议

**示例输出**：
| 假设 | 描述 | 初始置信度 | 验证后 | 是否支持 |
|---|---|---|---|---|
| H1分流假设 | 新入口分流了B/C入口点击 | 85% | 92% | ✅ |
| H2布局假设 | 新入口改变视觉重心 | 60% | 35% | ❌ |
| H3用户群假设 | 新入口吸引低质用户 | 40% | 25% | ❌ |

### Phase 4 · 决策建议 🎯
基于决策矩阵给出放量/暂停/回滚建议

| 场景 | 核心指标 | 直接指标 | AI建议 | 决策权 |
|---|---|---|---|---|
| S1 | 全部正向 | 符合预期 | ✅ 建议继续放量或全量 | 人工确认 |
| S2 | 部分正向+平稳 | 符合预期 | ✅ 建议继续放量 | 人工确认 |
| **S3** | **此消彼长** ⚠️ | 任意 | ⚠️ 需人工评估此消彼长 | **必须人工决策** |
| S4 | 出现下降 | 任意 | ❌ 暂不放量，先下钻 | 人工确认 |
| S5 | 任意 | 不符合预期 | ❌ 暂不放量，变量可能未生效 | 人工确认 |

**阶段特定规则**：
- 5%初始灰度：核心指标无严重下降(>-5%)即可进10%，重点防bug/崩溃
- 10%扩大验证：核心指标正向或平稳+直接指标符合预期，关注显著性
- 30%决策判断：表现优异+数据稳定可直接推全，此消彼长必须人工

**S3此消彼长强制暂停**：
输出`status='PAUSED_WAITING_HUMAN_DECISION'`，提交三维度问卷：
1. 下降指标严重程度
2. 提升指标业务价值
3. 长期可持续性
最终四选一：继续放量/有条件接受/回滚/需更多数据。

### Phase 5 · 报告生成 📊
生成「结论先行 + 五模块 + 角色自适应」HTML报告：

1. **结论先行**：发布建议+核心理由+关键数据摘要+风险提示
2. **实验概览**：版本号/关键变量/因果链路/阶段/时间范围
3. **核心数据表现**：三类指标热力图+显著性+护栏状态+组合判定
4. **直接指标&因果链路**：符合性表+三步验证面板
5. **下钻分析**（条件显示）：触发原因+假设列表+根因+建议
6. **附录**：8项校验结果/SRM/功效/AA基线/检验方法

**角色自适应**：
- PM视图（默认）：折叠统计细节，突出业务价值与决策
- 数据分析师视图：完整展开p值/置信区间/效应量/计算细节

## 三、知识库驱动设计
Agent行为完全由结构化知识库驱动，不硬编码规则：

| 知识资源 | 路径 | 用途 |
|---|---|---|
| 核心指标定义 | `metrics/core_metrics.yaml` | 三类16个指标定义+护栏规则 |
| SQL模板 | `metrics/sql_templates.yaml` | 参数化查询模板 |
| AA规则 | `rules/aa_rules.yaml` | 免AA处理、阈值、基线扣除公式 |
| 放量阶段 | `rules/rollout_phases.yaml` | 5%/10%/30%阶段配置、跳过/终止条件 |
| 决策矩阵 | `rules/decision_matrix.yaml` | 指标组合→决策映射、人工决策标记 |
| 下钻知识 | `analysis/drill_knowledge.yaml` | 场景模式→假设→下钻维度映射 |
| 案例经验 | `analysis/cases_and_lessons.yaml` | good/bad案例，反馈闭环回写 |
| 页面结构 | `product_model/page_structure.yaml` | 产品页面元素、卡片、流转关系 |
| 报告模板 | `product_model/report_template.yaml` | 报告结构、角色自适应规则 |

## 四、统计方法约定
| 指标特征 | 检验方法 |
|---|---|
| 比率型（值域[0,1]） | 双样本Z检验 |
| 连续型（近似正态） | Welch's t检验 |
| 计数型（非负整数） | 泊松检验 |

- 显著性水平α=0.05（双侧）：高显著p<0.01，边缘0.05≤p<0.10
- 效应量：Cohen's d小0.2/中0.5/大0.8；OR小1.5/中2.5/大4.3
- 多重比较可选Bonferroni/BH校正，默认报告中标注即可

## 五、子Agent派单规则
| 触发场景 | 派给哪个Agent | Prompt要点 |
|---|---|---|
| 跨版本下钻需要大盘/跨业务线借数 | data-analysis | 版本号、业务线、时间窗、期望产物 |
| 灰度异动需要纵深归因（非版本维度） | wave-attribution | 异常点、版本号、初步下钻、剩余假设 |
| 决策结论沉淀到飞书/归档 | know | 文档标题、目标位置、是否建索引 |
| 同期间AB实验疑似干扰 | ab-experiment | 时间窗、实验ID、待验证假设 |

**派单约束**：
- 子Agent无对话通道，prompt必须自包含所有上下文
- Phase0信息收集、Phase4决策（尤其S3此消彼长）不能派单，主Agent必须亲自完成
- 子Agent不能再spawn，超过2分钟任务用后台运行
- 护栏判定、最终决策由主Agent拍板，子Agent输出只作为证据

## 六、Context分层协议
遵循三层上下文协议：

| 层级 | 存储位置 | 内容示例 |
|---|---|---|
| User Global Memory | 跨Agent共享 | 用户角色(PM/QA/研发)、风险容忍度、负责产品线 |
| Agent Memory | 当前Agent专用 | 灰度节奏偏好、守卫指标偏好、当前跟的版本、历史事故复盘 |
| Task State | 当前任务专用 | 对比版本对、灰度阶段、指标差异结果、当前结论 |

**记忆写入原则**：
- 偏好/习惯/历史经验 → Agent Memory，跨任务复用
- 单次灰度具体数值 → Task State，不进长期记忆
- 通用规则 → SOP/知识库，不进Memory

## 七、设计亮点总结

### 1. **SOP驱动，流程清晰**
6阶段严格顺序，每阶段目标、输入、输出、校验规则明确，Agent行为可控可预测，不会"自由发挥"。

### 2. **知识库外置，规则可配置**
所有阈值、规则、模板、案例都在YAML知识库中，产品调整不需要改Prompt，非技术人员也能维护规则。

### 3. **关键决策点人机权责清晰**
- 什么场景AI可以直接给建议
- 什么场景必须暂停等人工决策（此消彼长场景强制暂停）
- AI不越权做不可逆决策

### 4. **角色自适应输出**
同一个分析结果，PM看到业务结论和建议，分析师看到完整统计细节，满足不同用户需求。

### 5. **数据质量门控**
8项校验做门控，🔴高风险问题不解决不进入分析，避免"垃圾进垃圾出"。

### 6. **因果链路验证**
不满足于"看到相关性"，必须验证"关键变量→直接指标→核心指标"的因果链，避免错误归因。

### 7. **反馈闭环设计**
good/bad案例回写知识库，Agent越用越聪明，决策越准。

### 8. **子Agent协作边界清晰**
明确什么能派、什么不能派，派单时prompt需要什么信息，避免子Agent能力越界或信息不足。

## 八、必须遵守的反模式原则
1. **SOP顺序不可跳**：6阶段必须顺序推进，每阶段输出明确进度
2. **结论先行**：报告和对话回复都先给结论，再展开细节
3. **护栏意识**：核心指标下降必须高度重视，触发下钻
4. **此消彼长必须人工**：S3场景不得自动放行，必须暂停等用户决策
5. **数据质量第一**：Phase2校验🔴不通过不能强行分析
6. **因果重于相关**：因果链路验证不可跳过
7. **免AA更要关注基线**：免AA不等于不校验，反而要加权重
8. **不替用户做不可逆决策**：放量/回滚只给建议，最终用户拍板
