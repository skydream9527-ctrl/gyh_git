## 一、为什么很多产品做出来没人用（一段扎心的论述）

产品经理最容易陷入的三个误区：

> "这个功能很酷/技术很先进，我们做吧。"
> "竞品都做了，我们也得跟上。"
> "老板/业务方要的，赶紧做了上线。"

这些都是**真实存在的工作动力**，但作为产品方法论是**灾难性的**——你在用"输入（做了多少功能）"衡量产出，而不是用"输出（解决了什么用户问题、创造了什么业务价值）"。

更扎心的真相是：**80%的功能上线后没人用，80%的时间花在了没价值的需求上**。很多产品忙了一年，回头看真正对业务有贡献的功能就两三个。

产品方法论要解决的就是：**在资源永远有限、需求永远做不完、用户永远说不清楚自己要什么的情况下，怎么持续做对的事情，把资源投入到真正创造用户价值和业务价值的地方。**

> 当你有想法不确定对不对时，优先做AB实验验证。详见 [../ab-testing/](../ab-testing/)。当实验做不了需要判断因果时，用 [../causal-inference/](../causal-inference/) 的方法。当需要做优先级和路线决策时，用 [../decision-frameworks/](../decision-frameworks/) 的工具。

## 二、产品工作的三层模型

把产品工作简化成三层，绝大多数人停留在第二层：

```
┌───────────────────────────────────────────────────────────┐
│  3. 价值层（Why）       为什么做？创造什么价值？            │ ← 用户价值、业务价值、商业模式
│  2. 方案层（What）      做什么？做成什么样？                │ ← 需求、PRD、功能、流程
│  1. 执行层（How）       怎么做？什么时候上线？              │ ← 项目管理、进度、协调、上线
└───────────────────────────────────────────────────────────┘
            影响力↑                           可替代性↓
```

**重点**：
- 第1层执行是基础，要靠谱，但价值有限，很多人/工具都能做
- 第2层方案是核心，把需求转化为产品方案，是产品经理的基本功
- 第3层价值判断是壁垒，决定"做不做"比"怎么做"重要100倍，这是优秀和平庸产品经理的分水岭

**记一条铁律**：用执行层的勤奋掩盖价值层的懒惰，是产品经理最容易犯的错误。忙了一年做了50个功能，不如做对1个真正有价值的功能。

## 三、数据产品的核心能力框架

数据产品经理和通用产品经理有共性，也有特殊性，核心能力是这四个：

```
┌──────────────────────────────────────────────────────────┐
│                    数据产品核心能力                        │
├──────────────┬──────────────────┬────────────────────────┤
│  业务理解     │   数据能力        │   产品能力              │
│  (Business)  │   (Data)         │   (Product)            │
├──────────────┼──────────────────┼────────────────────────┤
│ 业务目标拆解  │ 指标体系设计      │ 用户需求洞察            │
│ 问题识别判断  │ 数据敏感度        │ 产品方案设计            │
│ 成本收益意识  │ 分析方法掌握      │ 优先级判断              │
│ 行业认知      │ AB实验/因果思维   │ 项目推动能力            │
└──────────────┴──────────────────┴────────────────────────┘
                          ↓
                    AI协作能力
                    （新时代放大器）
```

**AI时代新要求**：不是要你会写代码训模型，而是要你懂AI能做什么不能做什么，知道怎么用AI提效，知道怎么设计AI Native的产品形态。详见 [../agent-llm/](../agent-llm/)。

## 四、产品工作全流程SOP

从0到1做一个产品/功能的标准流程：

```
1. 问题定义
   ├─ 用户遇到了什么问题？（不是"我要做XX功能"）
   ├─ 这个问题影响多少用户？频率多高？痛点多强？
   ├─ 解决了能带来什么业务价值？（指标能提升多少？）
   └─ 这个问题值得解决吗？（投入产出比怎么样？）

2. 指标设计
   ├─ 北极星指标是什么？（衡量核心价值）
   ├─ 护栏指标是什么？（不能变差的东西）
   └─ 观测指标是什么？（过程指标，辅助判断）

3. 方案设计
   ├─ 用户路径是什么？核心流程怎么走？
   ├─ 有哪些备选方案？优缺点分别是什么？
   ├─ 埋点怎么设计？上线后怎么验证效果？
   └─ PRD写清楚：需求背景、目标、方案、指标、埋点、排期

4. 开发上线
   ├─ 需求评审，和研发/设计/测试对齐
   ├─ 灰度发布，先小流量验证
   └─ 数据监控，实时看指标有没有异常

5. 效果评估
   ├─ AB实验结果显著吗？核心指标涨了吗？
   ├─ 护栏指标有没有变差？有没有副作用？
   ├─ 用户反馈怎么样？符合预期吗？
   └─ 全量上线还是回滚？还是继续优化？

6. 复盘沉淀
   ├─ 哪些做对了？哪些做错了？为什么？
   ├─ 有什么经验可以复用？
   └─ 下一轮迭代怎么改进？
```

## 五、最常见的五个产品陷阱（先记住，后细学）

1. **功能堆砌（Feature Creep）**
   - 现象：不断加功能，产品越来越复杂，核心体验越来越差
   - 后果：用户找不到核心功能，维护成本飙升，哪个都没做好
   - 解法：做减法，问自己"如果去掉这个功能，用户会骂吗？不会就去掉"；新功能上线看使用率，低于5%就考虑下线

2. **自嗨型产品**
   - 现象：我觉得用户需要，我觉得这个体验好，用户应该喜欢
   - 后果：做出来自己觉得很完美，用户根本不买账
   - 解法：走出去见用户，看用户真实怎么用；用数据说话，别凭感觉；小步快跑，快速验证

3. **竞品跟随**
   - 现象：竞品做了XX，我们也得做，不然就落后了
   - 后果：永远跟在别人后面，没有自己的核心竞争力，做出来也比别人晚
   - 解法：想清楚竞品为什么做这个？他们的用户和我们一样吗？我们有必要做吗？不做会怎么样？做了能带来什么差异化价值？

4. **过早优化（Premature Optimization）**
   - 现象：还没验证需求成立，就开始做性能优化、做复杂架构、做完美体验
   - 后果：需求本身不成立，所有优化都白费，浪费时间资源
   - 解法：先用最简方案验证需求成立，有用户用、有价值，再谈优化

5. **指标绑架**
   - 现象：为了涨指标不择手段，什么手段能让数字好看就做什么
   - 后果：指标涨了，用户价值没涨，甚至伤害用户长期价值（比如为了DAU疯狂发骚扰推送）
   - 解法：指标是结果不是目标，永远盯着用户价值，区分"好的增长"和"坏的增长"

## 六、AI时代产品经理变了什么，没变什么

**没变的（核心能力）**：
- 用户需求洞察能力——用户不会因为有了AI就不说谎了
- 价值判断能力——做什么比怎么做更重要，AI也回答不了战略问题
- 沟通协作能力——推动事情落地需要人
- 产品 sense——对体验、对产品的直觉和品味

**变了的（效率杠杆）**：
| 工作 | 以前怎么做 | 现在怎么做 |
|---|---|---|
| 竞品分析 | 自己找资料、截图、整理 | AI帮你找资料、整理对比框架、生成初稿 |
| PRD撰写 | 自己从0写到尾 | AI帮你写初稿、生成验收标准、检查逻辑漏洞 |
| 数据分析 | 自己跑SQL、做报表、写分析 | AI帮你写SQL、做初步分析、找异常点 |
| 用户反馈整理 | 自己一条条看反馈、归类 | AI帮你自动归类、提炼痛点、统计频次 |
| 原型设计 | 自己画Axure/Figma | AI帮你生成初版原型，你调整细节 |
| 知识沉淀 | 自己整理文档、建知识库 | AI帮你自动沉淀、关联知识、智能检索 |

**新机会**：AI Native产品——不是在老产品上加个AI聊天框，而是从0开始基于AI能力设计新的产品形态，这是最大的机会。

## 七、推荐学习路径

### 入门（1-2周上手）
1. 本OVERVIEW建立产品心智
2. [templates/](./templates/) 里的六个模板先学会用：PRD、竞品分析、周报、会议纪要、用户反馈、用户故事
3. 搞懂产品全流程SOP，知道每个环节该做什么
4. 练习：拿一个你觉得不好用的产品，按"问题-方案-指标"拆解一遍

### 进阶（1-2个月）
5. 学会指标体系设计：OSM模型、UJM、北极星指标、护栏指标
6. 学会需求优先级判断：用RICE评分、KANO模型（见decision-frameworks域）
7. 掌握AB实验方法（见ab-testing域），能用数据验证产品想法
8. 练手：完整跟一个需求从定义到上线复盘的全流程

### 高阶（按需）
9. 商业模式、产品战略、生态设计
10. AI Native产品设计（见agent-llm域），探索新的产品形态
11. 组织能力、团队管理、跨部门协作
12. 行业深度认知，成为领域专家

## 九、心法

如果只能记三句话：

1. **不要给我功能，给我问题和解决方案**——用户要的不是打孔机，是墙上的洞；先想清楚洞在哪，再设计打孔机。
2. **少做就是多做**——资源永远不够，砍掉80%的需求，把20%真正重要的事情做到极致。
3. **上线只是开始，不是结束**——上线后看数据、听反馈、快速迭代，产品是长出来的不是设计出来的。

下次接需求做产品前，Checklist：
1. 用户遇到的真实问题是什么？（不是"要做XX功能"）
2. 这个问题影响多少人？痛点有多强？值得解决吗？
3. 成功指标是什么？怎么验证有没有解决问题？
4. 有哪些备选方案？为什么选这个？
5. 最坏情况是什么？如果失败了怎么办？

回答不出来，先别急着排期开发。
