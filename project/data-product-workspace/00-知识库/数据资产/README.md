# 数据资产层（服务 nl-sql）

> 定位：把高频、可复用的「取数 SQL 模板、指标口径、人群包」沉淀为结构化资产，直接喂给 `nl-sql` 自然语言取数，让 AI「越用越准」。这是把工作区从「知识库 + 记录」升级为「执行台」的核心一层，也是 Q3 P0「数据资产知识库建设」的落地载体。

## 为什么单独建这一层

2026 H1 的 3 份工作台账（197 条事项）+ 59 份交付文档分析显示：取数、人群包圈选、看板填报占实际工作量过半，且 SQL 骨架高度重复、模板化程度极高。这些工作的痛点不是"不会写 SQL"，而是：

- SQL 散落在各需求文档附录，交付靠私聊/截图，无法复用
- 同类需求（短剧 push 包、浮层售卖、版本圈选）反复写相似 SQL，只改参数
- 口径反复口头确认，没有权威字典
- 人群包交付后无登记，无法追溯和复用

数据资产层就是解决这四个痛点，把「一次性交付」变成「可复用资产」。

## 怎么用（喂给 nl-sql）

1. **沉淀**：每次交付真实取数/圈人需求后，把可复用的 SQL 抽象成模板，回填到对应子域。
2. **校准**：模板中标注 `${参数}` 和「待校准」项，用真实 badcase 逐步补全实际表名、字段、口径。
3. **供给**：模板库 + 口径字典作为 `nl-sql` 的 few-shot 样例和检索知识，提升准确率。
4. **登记**：人群包交付后在登记表追加一行，形成可追溯资产。

## 维护规则

- 每个模板必须标注：来源、数据源表、参数、交付形态、验证状态（✅已验证 / ⚠️待校准）
- 引用数据必须标注来源和时间；未经真实查询验证的一律标「⚠️待校准」，不冒充已验证
- 新增/修改先找对子域，不在本层堆临时脚本（临时脚本仍放 `03-数据分析/`）
