# Skills 约束条件添加规范

> 基于对 `/Users/mi/.micode/skills/` 目录下全部 25 个 Skills 的系统性分析，提炼出的约束条件添加规范。

---

## 一、约束条件的分类体系

Skill 中的约束条件按作用范围分为五个层级：

| 层级 | 作用范围 | 示例 |
|------|---------|------|
| L1 入口约束 | 决定 Skill 是否被触发 | description 中的触发词和适用场景 |
| L2 前置约束 | 执行前必须满足的条件 | 环境变量、认证状态、依赖检查 |
| L3 流程约束 | 执行过程中必须遵守的规则 | 强制读取参考文件、指标范围校验 |
| L4 输出约束 | 产出结果必须满足的质量标准 | SQL 自检清单、数据完整性检查 |
| L5 边界约束 | 异常和边界情况的处理 | 超时降级、空数据打断、错误自动修复 |

## 二、L1 入口约束

### 2.1 触发词约束

在 frontmatter 的 `description` 中明确定义触发条件：

```yaml
description: |
  当用户提到"版本灰度分析"、"版本数据对比"、"灰度指标分析"、
  "版本AB分析"、"灰度报告"时使用本技能。
```

**规范要求**：
- 触发词必须具体，避免过于宽泛（如"数据分析"应限定为特定业务场景）
- 同时提供"必须使用"和"也要使用"两种触发强度
- 明确排他性：当多个 Skill 可能匹配时，说明优先级

### 2.2 不适用场景约束

明确说明什么情况下**不应**使用此 Skill：

```yaml
description: |
  NOT for Manim Community Edition (which uses `manim` imports and `manim` CLI).
```

```markdown
## 不适用场景
- 需要实时数据推送的场景（本 Skill 仅支持批量查询）
- 非浏览器/信息流业务的数据分析
- 需要用户级明细数据导出的场景
```

## 四、L3 流程约束

### 4.1 强制读取参考文件约束

**这是最常见的流程约束**，在数据分析类 Skill 中尤为重要：

```markdown
⚠️ **严禁跳过**：在编写任何 SQL 之前，**必须**读取 `reference/` 文件夹中对应业务子目录下的文件。
禁止凭记忆推测表名、字段名或指标口径，所有信息**必须**来自上述真实文件。
```

**约束模式**：

| 约束强度 | 标记 | 含义 |
|---------|------|------|
| 强制执行 | ⚠️ / ⛔ | 不可跳过，违反即终止 |
| 推荐执行 | 💡 | 建议执行，特殊情况下可跳过 |
| 条件执行 | 🔀 | 满足特定条件时执行 |

### 4.2 范围校验约束

**指标/事件范围约束**是防止 Skill 越界的关键机制：

```markdown
> [!WARNING]
> **指标/事件范围约束（强制执行）**
> 
> 在读取索引文件后，**必须严格检查**用户请求的指标或事件是否存在于索引文件中：
> - ✅ **存在**：继续执行后续步骤
> - ❌ **不存在**：立即停止，执行以下操作：
>   1. 告知用户：**"当前指标/事件范围不支持生成此 SQL 代码。"**
>   2. 提供可选方案：列出当前业务线下支持的相似指标/事件
>   3. 引导联系：**"如需添加此指标/事件，请联系开发者补充定义。"**
>   4. **终止 Skill 执行**：不再继续后续任何步骤
```

**标准回复格式**：

```markdown
⚠️ 指标不支持

您请求的指标「{用户输入的指标名}」不在当前支持的指标范围内。

📌 当前业务线支持的相似指标：
- {指标1}
- {指标2}
- {指标3}

如需添加此指标，请联系开发者补充表结构和字段定义。

[Skill 执行已终止]
```

### 4.3 多轮交互约束

需要收集多个信息的 Skill 必须定义交互顺序：

```markdown
## 需求引导流程（必须严格按顺序执行，不可跳过）

### Step 1：询问业务线
### Step 2：询问指标类型
### Step 3：询问时间周期
### Step 4：需求确认后生成 SQL

三步信息齐备后，向用户做一次简要确认，确认后立即进入执行阶段，不再追问其他问题。
```

### 4.4 路径约束

涉及文件系统操作的 Skill 必须明确路径规范：

```markdown
⚠️ **严禁使用错误路径**：该 Skill 的目录名是 `data-sql`，绝对不能使用 `/skills/sql/` 或 `/skills/sql/scripts/`。
唯一正确的路径前缀是 `/Users/mi/.micode/skills/data-sql/`，不得自行猜测或简写目录名。
```

## 五、L4 输出约束

### 5.1 SQL 自检清单约束

生成 SQL 的 Skill 必须包含完整的自检清单：

```markdown
**SQL 自检清单（必须全部通过，否则修正后重检）：**

| # | 检查项 | 要求 |
|---|--------|------|
| ① | 日期格式 | 使用 `'YYYYMMDD'` 格式，禁止连字符 |
| ② | 字段合法性 | 所有字段必须在参考文件中出现 |
| ③ | 表名合法性 | 包含完整三段式前缀 |
| ④ | 别名规范 | 列别名使用纯英文 |
| ⑤ | 分区过滤 | WHERE 必须包含 `date` 过滤条件 |
| ⑥ | 指标口径 | 聚合逻辑与参考文件一致 |
| ⑦ | 语法完整性 | 包含 SELECT、FROM、WHERE 等必要子句 |
| ⑧ | 字段歧义 | 多表 JOIN 时同名字段必须加表别名前缀 |
```

**自检闭环约束**：

```markdown
> **自检-修正-再检闭环（强制）**：
> 1. SQL 生成完毕后，**立刻执行一次完整的 8 项自检**
> 2. 如果发现任意一项不通过，**必须当场修正 SQL**
> 3. 修正完成后，**必须重新从第①项开始再次执行完整自检**
> 4. **只有自检 100% 通过后**，才允许执行查询
```

### 5.2 数据完整性约束

```markdown
**数据完整性检查**：
- 确认目录下存在预期的所有文件
- 如果缺少某些文件，向用户反馈缺失的文件列表
- 阅读每份文件的表头和前几行数据，确认数据格式正确
```

### 5.3 空数据强制打断约束

```markdown
⛔ **空数据强制打断（最高优先级）**：在写入 CSV 之前，**必须首先**确认查询结果行数 > 0。
- 若结果为空（0 行），**立即停止本次任务所有后续步骤**
- 禁止继续写 CSV、禁止进行分析、禁止生成飞书文档
- 向用户输出原因和调整建议
```

### 5.4 禁止捏造数据约束

```markdown
**绝对禁止大模型捏造数据**：必须通过 Python 代码调用脚本，把取回的完整、真实的原始查询结果
直接原封不动地写入文件。作为 AI，绝对不能尝试通过自行拼凑或回忆部分数据去伪造结果。
```

### 5.5 产出文件完整性约束

```markdown
在写入飞书前，必须确认输出目录下以下文件全部存在且非空：
1. `[前缀]_query.md` — 取数 SQL 归档
2. `[前缀]_data.csv` — 真实查询数据
3. `[前缀]_analysis.md` — 分析结论汇总
4. `[前缀]_chart.png` — 数据可视化图表（图片文件大小必须 > 0 字节）

⚠️ 若 `_chart.png` 不存在或大小为 0，**必须返回重新绘制图表**，不得跳过。
```

## 六、L5 边界约束

### 6.1 错误自动修复约束

定义常见错误的识别和修复规则：

```markdown
| 错误特征 | 原因 | 修复方式 |
|---------|------|--------|
| 含 `Unexpected character` | 中文别名导致解析失败 | 替换中文别名为英文 |
| 含 `Column ... not found` | 字段名不合法 | 回到参考文件重新核查 |
| 含 `is ambiguous` | 多表 JOIN 同名字段未加前缀 | 加表别名前缀 |
| 含 `Timeout` / `OOM` | 查询超时或内存超限 | 按降级策略处理 |
```

### 6.2 超时降级约束

```markdown
**查询超时 / 内存超限自动降级**：

| 重试轮次 | 策略 | 向用户说明 |
|--------|------|---------|
| 第 1 次 | 将 date 过滤范围收窄为最近 7 天 | 「查询超时，已自动缩小至最近 7 天重试...」 |
| 第 2 次 | 若仍失败，再收窄为最近 1 天 | 「最近 7 天仍失败，已进一步缩小至最近 1 天...」 |
| 第 3 次 | 若仍失败，反馈用户并给出优化建议 | 说明原因并建议优化方向 |
```

### 6.3 时间范围约束

```markdown
⚠️ **时间周期建议**：如果分析时间范围超过 7 天，建议压缩到 7 天以内，
避免稀释版本差异效果。
```

### 6.4 白名单修改约束

```markdown
**任何修改到搜索白名单的操作 MUST 经用户明确确认后才能执行。**
LLM 严禁自主运行任何白名单修改命令。
```

### 6.5 破坏性操作约束

```markdown
⚠️ **overwrite 是破坏性操作**：会清空整篇文档，图片和白板将永久丢失。
优先使用 `append`/`replace` 进行局部修改。
非 TTY 环境下使用 overwrite 必须加 `--force`，否则报错。
```

## 七、约束条件的标记规范

### 7.1 视觉标记

| 标记 | 含义 | 使用场景 |
|------|------|---------|
| ⚠️ | 警告/强制要求 | 不可跳过的步骤、重要注意事项 |
| ⛔ | 禁止 | 绝对不允许的操作 |
| ✅ | 通过/正确 | 校验通过的状态 |
| ❌ | 失败/错误 | 校验失败的状态 |
| 💡 | 建议 | 推荐但非强制的做法 |
| 🔀 | 条件分支 | 满足条件时执行 |

### 7.2 文字标记

| 标记 | 含义 |
|------|------|
| **必须** / **强制** | 不可违反的约束 |
| **严禁** / **禁止** | 绝对不允许的操作 |
| **推荐** / **建议** | 最佳实践但非强制 |
| **可选** | 可根据情况决定 |

### 7.3 约束块的格式

使用 GitHub Alert 语法或引用块突出约束：

```markdown
> [!WARNING]
> **指标范围约束（强制执行）**
> 详细内容...

> [!NOTE]
> 建议操作，非强制。

> [!CAUTION]
> 破坏性操作，需谨慎。
```
