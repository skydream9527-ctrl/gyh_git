# Skills 验收需要的注意点

> 基于对 `/Users/mi/.micode/skills/` 目录下全部 25 个 Skills 的系统性分析，提炼出的验收检查要点。

---

## 一、验收总览

Skill 验收应从以下六个维度进行系统性检查：

| 维度 | 核心问题 | 权重 |
|------|---------|------|
| 需求理解 | 是否正确理解并覆盖了用户需求 | ⭐⭐⭐ |
| 目录规范 | 目录结构和命名是否符合标准 | ⭐⭐ |
| SKILL.md 完整性 | 主文件是否包含所有必需内容 | ⭐⭐⭐ |
| 约束充分性 | 约束条件是否覆盖关键风险点 | ⭐⭐⭐ |
| 流程可执行性 | 执行流程是否可以端到端跑通 | ⭐⭐⭐ |
| 交付就绪度 | 是否具备交付给用户使用的条件 | ⭐⭐ |

---

## 二、需求理解验收

### 2.1 需求覆盖度检查

| # | 检查项 | 通过标准 | 常见问题 |
|---|--------|---------|---------|
| 1 | 核心功能是否明确 | SKILL.md 中有清晰的功能描述 | 功能描述模糊，如"帮助用户处理数据" |
| 2 | 适用场景是否完整 | description 中列出了所有触发场景 | 遗漏常见触发词，导致 Skill 不被激活 |
| 3 | 不适用场景是否定义 | 明确说明了边界和排除项 | 未定义边界，导致 Skill 被错误触发 |
| 4 | 输入输出是否明确 | 每个步骤的输入输出都有说明 | 输入模糊，Agent 不知道需要收集什么信息 |
| 5 | 用户交互是否设计 | 多轮交互的顺序和模板已定义 | 交互顺序混乱，缺少确认环节 |

### 2.2 需求准确性检查

- **触发词是否精准**：避免过于宽泛（如"数据分析"）或过于狭窄（如仅一个特定词）
- **角色定位是否匹配**：Skill 的角色描述是否与其功能一致
- **业务术语是否准确**：涉及业务领域时，术语使用是否与业务方一致

## 三、目录规范验收

### 3.1 目录结构检查

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 1 | 主文件名是否为 `SKILL.md` | 大写，位于 Skill 根目录 |
| 2 | 是否包含 `scripts/` 目录 | 即使为空也必须存在 |
| 3 | 是否包含 `references/` 目录 | 即使为空也必须存在 |
| 4 | 是否包含 `assets/` 目录 | 如有模板/图片资源则必须存在 |
| 5 | 是否存在不应有的文件 | 无 README.md、CHANGELOG.md 等辅助文档 |
| 6 | 目录命名是否规范 | 使用小写字母和连字符（如 `auto-analysis`） |

### 3.2 文件组织检查

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 1 | scripts 中的脚本是否有明确职责 | 每个脚本的功能可从文件名推断 |
| 2 | references 是否按需组织 | 大型 Skill 按业务线/领域分子目录 |
| 3 | 是否存在重复内容 | 同一信息不应同时出现在 SKILL.md 和 references 中 |
| 4 | 敏感信息是否排除 | 无硬编码的 API Key、Token、密码 |

## 四、SKILL.md 完整性验收

### 4.1 Frontmatter 检查

| # | 检查项 | 通过标准 | 常见问题 |
|---|--------|---------|---------|
| 1 | 是否包含 `name` 字段 | ✅ 必需 | 缺失 name |
| 2 | 是否包含 `description` 字段 | ✅ 必需 | 缺失 description |
| 3 | `name` 是否与目录名一致 | ✅ 应一致 | name 与目录名不匹配 |
| 4 | `description` 是否包含触发场景 | ✅ 必需 | description 过于简略，缺少触发条件 |
| 5 | `description` 是否包含功能说明 | ✅ 必需 | 只有触发词没有功能描述 |
| 6 | frontmatter 中是否有多余字段 | 仅保留 name 和 description | 包含 version、author 等非标准字段（除非有特殊需要） |

### 4.2 Body 内容检查

| # | 检查项 | 通过标准 | 常见问题 |
|---|--------|---------|---------|
| 1 | 是否包含功能描述 | 清晰说明 Skill 做什么 | 功能描述缺失或模糊 |
| 2 | 是否包含执行流程 | 步骤编号有序，逻辑清晰 | 流程缺失或步骤无编号 |
| 3 | 执行流程是否完整 | 从输入到输出全链路覆盖 | 缺少关键步骤（如数据校验、结果输出） |
| 4 | 是否包含使用示例 | 至少 2-3 个典型示例 | 无示例或示例过于简单 |
| 5 | 是否标注了强制步骤 | 用 ⚠️ 或"必须"标记 | 所有步骤权重相同，无法区分主次 |
| 6 | 是否定义了交互模板 | 涉及用户交互时有完整对话模板 | 交互内容模糊，Agent 不知道如何提问 |
| 7 | 是否说明了参考文件的读取时机 | 明确标注何时读取哪个 reference | references 存在但 SKILL.md 中未引用 |
| 8 | Body 长度是否合理 | 控制在 500 行以内 | 过长导致上下文膨胀 |

### 4.3 渐进式披露检查

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 1 | SKILL.md 是否只保留核心流程 | 详细规范已移入 references |
| 2 | references 是否在 SKILL.md 中被引用 | 每个 reference 文件都有明确的读取时机说明 |
| 3 | 引用层级是否为一层深度 | 避免 reference 文件再引用其他 reference |
| 4 | 长 reference 文件是否有目录 | 超过 100 行的文件顶部有 Table of Contents |

## 五、约束充分性验收

### 5.1 前置约束检查

| # | 检查项 | 通过标准 | 风险等级 |
|---|--------|---------|---------|
| 1 | 是否包含环境/认证检查 | 首次使用前有检查步骤 | 🔴 高 |
| 2 | 环境缺失时是否有引导 | 缺失时向用户说明并提供解决方案 | 🔴 高 |
| 3 | 依赖是否明确列出 | Python 依赖、系统依赖、其他 Skill 依赖 | 🟡 中 |
| 4 | 是否有版本兼容性说明 | CLI 版本、API 版本的最低要求 | 🟡 中 |

### 5.2 流程约束检查

| # | 检查项 | 通过标准 | 风险等级 |
|---|--------|---------|---------|
| 1 | 是否强制读取参考文件 | 标注"严禁跳过"、"必须读取" | 🔴 高 |
| 2 | 是否包含范围校验 | 指标/事件不在支持范围时有终止机制 | 🔴 高 |
| 3 | 范围校验是否有标准回复格式 | 不支持时输出结构化的提示信息 | 🟡 中 |
| 4 | 多轮交互是否有序 | 明确步骤顺序，标注"不可跳过" | 🟡 中 |
| 5 | 路径引用是否精确 | 使用绝对路径，禁止猜测或简写 | 🔴 高 |

### 5.3 输出约束检查

| # | 检查项 | 通过标准 | 风险等级 |
|---|--------|---------|---------|
| 1 | 是否包含 SQL/代码自检清单 | 逐项校验，必须全部通过 | 🔴 高 |
| 2 | 自检是否有闭环机制 | 不通过时修正后重新自检 | 🔴 高 |
| 3 | 是否禁止数据捏造 | 明确标注"绝对禁止" | 🔴 高 |
| 4 | 是否包含空数据打断机制 | 结果为空时终止后续步骤 | 🔴 高 |
| 5 | 产出文件是否有完整性检查 | 列出必需文件清单，逐一确认存在且非空 | 🟡 中 |
| 6 | 报告输出是否有格式规范 | 指标名称中文化、按维度分模块 | 🟡 中 |

### 5.4 边界约束检查

| # | 检查项 | 通过标准 | 风险等级 |
|---|--------|---------|---------|
| 1 | 是否定义了错误自动修复规则 | 常见错误有识别特征和修复方式 | 🔴 高 |
| 2 | 是否有超时降级策略 | 多轮降级，每轮向用户说明 | 🟡 中 |
| 3 | 破坏性操作是否有警告 | 标注 ⚠️ 并提供安全替代方案 | 🔴 高 |
| 4 | 白名单/配置修改是否需确认 | 禁止 LLM 自主修改，必须用户确认 | 🔴 高 |
| 5 | 时间范围是否有建议约束 | 超过阈值时提醒用户 | 🟢 低 |

## 六、流程可执行性验收

### 6.1 端到端流程检查

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 1 | 流程是否可以完整执行 | 从用户输入到最终输出，每一步都有明确操作 |
| 2 | 步骤之间是否有衔接 | 上一步的输出是下一步的输入 |
| 3 | 分支逻辑是否完整 | 每个条件分支都有对应的处理路径 |
| 4 | 是否有遗漏的中间步骤 | 如数据保存、文件归档等容易被忽略的操作 |
| 5 | 脚本调用是否正确 | 命令格式、参数、路径都经过验证 |

### 6.2 脚本可用性检查

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 1 | 脚本是否可以正常运行 | 实际执行一次，确认无报错 |
| 2 | 脚本参数是否与 SKILL.md 一致 | 文档中的调用示例与脚本实际参数匹配 |
| 3 | 脚本输出格式是否与后续步骤兼容 | 输出可以被下一步正确解析 |
| 4 | 依赖是否可安装 | requirements.txt 中的包可以正常安装 |
| 5 | 错误处理是否完善 | 脚本异常时有明确的错误信息 |

### 6.3 参考文件可用性检查

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 1 | SKILL.md 中引用的 reference 文件是否都存在 | 每个引用路径对应一个真实文件 |
| 2 | reference 文件内容是否与 SKILL.md 描述一致 | 文件内容覆盖了 SKILL.md 声称的信息 |
| 3 | reference 文件是否最新 | 表名、字段名、API 端点等信息未过时 |
| 4 | 索引文件是否完整 | 索引文件覆盖了所有支持的指标/事件 |

## 七、交付就绪度验收

### 7.1 用户体验检查

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 1 | 首次使用是否顺畅 | 环境检查 → 认证 → 执行，流程清晰 |
| 2 | 错误提示是否友好 | 用户能理解错误原因和修复方式 |
| 3 | 交互是否最少化 | 只收集必要信息，不过度追问 |
| 4 | 输出是否可直接使用 | 报告、文件等产出无需二次加工 |

### 7.2 文档质量检查

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 1 | SKILL.md 是否有语法错误 | Markdown 格式正确，无断链 |
| 2 | 代码示例是否可运行 | 所有代码块都可以直接复制执行 |
| 3 | 表格是否对齐 | 表格列数一致，内容完整 |
| 4 | 是否有拼写/格式错误 | 无明显拼写错误、格式混乱 |

### 7.3 安全性检查

| # | 检查项 | 通过标准 |
|---|--------|---------|

## 八、验收评分标准

### 8.1 评分维度

| 维度 | 满分 | 及格线 | 说明 |
|------|------|--------|------|
| 需求理解 | 20 | 14 | 核心功能、场景、边界是否清晰 |
| 目录规范 | 10 | 7 | 结构、命名、文件组织是否标准 |
| SKILL.md 完整性 | 25 | 18 | Frontmatter、Body、渐进式披露 |
| 约束充分性 | 25 | 18 | 前置/流程/输出/边界约束覆盖度 |
| 流程可执行性 | 15 | 11 | 端到端可跑通、脚本可用 |
| 交付就绪度 | 5 | 4 | 用户体验、文档质量、安全性 |

### 8.2 交付判定

| 总分 | 判定 | 建议 |
|------|------|------|
| 85-100 | ✅ 可交付 | 直接交付，后续根据使用反馈迭代 |
| 70-84 | ⚠️ 有条件交付 | 修复高风险项后可交付 |
| 60-69 | ❌ 需返工 | 存在较多问题，需系统性修复 |
| <60 | ❌ 不可交付 | 需重新设计 |

### 8.3 高风险项（一票否决）

以下任一项不通过，直接判定为不可交付：

| # | 高风险项 | 原因 |
|---|---------|------|
| 1 | 缺少 SKILL.md 或 frontmatter 不完整 | Skill 无法被触发 |
| 2 | description 缺少触发场景 | Skill 无法被正确激活 |
| 3 | 执行流程无法端到端跑通 | Skill 核心功能不可用 |
| 4 | 存在硬编码的密钥/Token | 安全风险 |
| 5 | 缺少数据捏造禁止约束 | 可能输出虚假分析结果 |
| 6 | 缺少空数据打断机制 | 可能在无数据时继续生成虚假报告 |

## 九、验收流程

### 9.1 验收步骤

```
第一步：目录结构扫描
  → 检查目录命名、文件组织、敏感信息

第二步：SKILL.md 静态审查
  → 检查 Frontmatter、Body 完整性、约束充分性

第三步：参考文件一致性校验
  → 检查引用路径、内容覆盖、索引完整性

第四步：脚本功能验证
  → 实际执行脚本，验证输入输出

第五步：端到端流程测试
  → 模拟用户场景，完整走通执行流程

第六步：边界场景测试
  → 测试异常输入、空数据、超时等边界情况

第七步：评分与交付判定
  → 按评分标准打分，给出交付建议
```
