阶段一（选表）和阶段二（确认查询条件）已在 SKILL.md 主流程中完成，从阶段三开始。

**严格按顺序，每个关卡必须等用户确认后才继续。**

## 阶段三：匹配分析方法（阶段二之后、写 SQL 前必须执行）

根据用户需求，从分析方法库中自动匹配。可组合多种方法。

> 匹配前必须先 `Read references/methods-basic.md`；需要高阶方法时追加 `Read references/methods-advanced.md`

1. **自动匹配**：扫描用户需求关键词，与每个方法的「场景关键词」比对，分为三档：
   - **推荐**：与需求直接相关
   - **可选**：有补充价值
   - **不推荐**：不适用于当前需求
2. **输出推荐清单**（仅展示推荐档和可选档，不推荐档折叠）：
   - **推荐档**：每项含编号 + 方法名 + 一句话适用理由（结合用户具体需求说明）
   - **可选档**：每项含编号 + 方法名 + 能额外获得的洞察
   - **不推荐档**：折叠为一行汇总，如「其余 N 种方法不适用（如需查看详情请在 Other 中输入"展开"）」
3. **关卡 ②（AskUserQuestion，单选）**：
   - 选项 1：**用推荐的**（仅使用推荐档方法）
   - 选项 2：**推荐 + 可选**（使用推荐和可选两档方法）
   - Other：用户自由输入编号组合（如 "1,3,5,12"）可选任意方法含不推荐项，或输入"展开"查看不推荐档详情后再决定
4. 用户确认后，按选定的分析方法指导后续 SQL 编写和结果输出

## 阶段四：方案规划

1. **读取所有已确认数据表的字段信息**（按优先级依次尝试）：
   - **优先**：通过 sql 技能执行 `DESCRIBE table_name` 或 `SHOW COLUMNS FROM table_name` 获取完整字段列表（含类型和注释）
   - **补充**：如果阶段一已从飞书知识库获取了字段说明文档，与 SQL 结果交叉验证
   - **嵌套字段解析（最高优先级）**：如果用户需要的字段在顶层列中未找到，**必须用 `SELECT data, properties FROM table LIMIT 5` 抽样查看 JSON/Map 类型字段内部结构**。很多业务字段嵌套在 `data`、`properties` 等复合字段内部。找不到字段时，先解析这些嵌套结构再判断字段是否存在
   - **展示策略**：不展示全量字段列表，只展示与分析需求相关的字段（按章节分组），由用户确认是否遗漏
2. **关卡 ③（AskUserQuestion）**：确认字段理解 → 筛选弃用字段 → 校验表关联
3. 生成分析框架 MD 文档（**遵循写作规范中的金字塔要求**）：总目标（一句话）→ **按用户需求划分分析章节（业务主题，MECE）**，每个章节规划：章节主题、预期回答的核心问题、涉及的维度和指标、**该章节使用哪些选定的分析方法（标注编号）**、对应的数据获取思路（Hive 模式写 SQL，本地数据模式写 pandas 操作）和预期图表。所有用户选定的方法必须被分配到至少一个章节中，不得遗漏
4. **关卡 ④**：用户确认方案

## 阶段五：数据查询与深度分析

5. 按阶段四确认的章节规划执行数据获取（Hive 模式：通过 sql 技能执行查询；本地数据模式：通过 pandas 处理 DataFrame），**每个章节内按规划的分析方法逐一产出数据和洞察**，图表类型从方法库「可视化」列选取
6. **数据完整性校验**：每次数据获取完毕后立即检查——空结果 → 暂停该章节并用 AskUserQuestion 与用户确认条件是否需要调整；异常值（如某天数据量骤降 90%+）→ 标记异常并说明可能原因，询问用户是否排除或保留
7. **AI 自主深度分析**（边界规则）：
   - **可以**：对已有数据做维度交叉、异常捕捉、隐藏关联挖掘、风险预警；在已选方法范围内自行发起补充查询（Hive 模式：SQL；本地数据模式：pandas）以获取更细粒度数据
   - **不可以**：引入用户未选择的分析方法（如用户未选 RFM，不得自行做 RFM 分层）；变更阶段四已确认的章节结构
   - **补充查询规则**：每次自主补充查询时必须向用户说明原因（如"为定位渠道 A 的异常下降，补充查询了子渠道维度的明细数据"），不得静默执行

## 阶段六：分析确认

8. **纯文字输出完整分析**（含用户需求分析 + AI 自主发现）——不生成图表，不写飞书。按章节组织，**每章注明使用了哪些分析方法**，确保用户选定的所有方法都有对应产出
9. **关卡 ⑤**：用户确认分析内容
   - **模式 B**：确认后流程结束，输出纯文字报告即为最终交付物。**模式 B 到此结束，不进入阶段七。**
   - **模式 C**：确认后进入阶段七生成正式交付物

## 阶段七：交付（仅模式 C）

### Step 1：生成 matplotlib 图表

> 首次生成图表前必须 `Read references/chart-guide.md`

1. 本地创建 `dar-charts/` 目录
2. 通过 Bash 执行 Python 脚本，使用 matplotlib 生成所有图表 PNG 到 `dar-charts/` 目录
   - 文件命名：`chart_1.png`, `chart_2.png`...（按章节和观点顺序编号）
   - 参数：`dpi=150`, `bbox_inches='tight'`
   - 中文字体：`rcParams["font.sans-serif"] = ["SimHei", "Microsoft YaHei", "PingFang SC", "Noto Sans CJK SC"]`
3. 生成后校验：`ls -lh dar-charts/` 逐项检查每个 PNG 存在且文件大小 > 0 字节

### Step 2：写入飞书文档

> 首次写飞书前必须 `Read references/report-format.md`

1. **AskUserQuestion**：写入知识库 or 个人空间（不得默认选择）
2. 通过 feishu 技能创建文档
3. **逐章循环写入**（严禁一次性生成整篇后补图）：

```
for 每个章节 (第一章, 第二章, ...):
  a. feishu append → 章节标题 + callout 结论
  b. for 每个观点 (N.1, N.2, ...):
       feishu append → 观点子标题 + 文字分析
       feishu docx upload-image → 该观点对应的图表 PNG
       feishu append → 数据表格（如有）
  c. feishu append → 行动建议
  d. 自检：确认本章所有图片已在章节内部，非末尾
```

### Step 3：交付自检

- `ls -lh dar-charts/` 列出所有生成的文件，逐项核对：
  - [ ] 每个 `chart_N.png` 都存在（数量应与分析观点数一致）
  - [ ] 所有 PNG 文件大小 > 0（空文件说明生成失败）
- 飞书文档检查：
  - [ ] 每张图表 PNG 已通过 feishu 技能正确插入到对应章节
  - [ ] 核心结论前置于报告开头（不在末尾）
  - [ ] 每章结论→观点→图表→表格→行动完整
- **缺失文件处理**：列出缺失清单，重试生成一次；仍失败则在交付清单中标注哪些文件未生成，告知用户

### Step 4：可选关卡 ⑥ — ECharts HTML 交互式报告

matplotlib 图表 + 飞书文档交付完成后：

**关卡 ⑥（AskUserQuestion）**：「飞书文档已交付完成。是否需要额外生成 ECharts HTML 交互式报告？」

- 选项 1：**不需要**（流程结束，输出交付清单）
- 选项 2：**需要 HTML 报告**
  - → `Read references/html-report.md`
  - → 按 ECharts 规范生成 HTML 报告
  - → 截图生成报告全图 PNG
  - → 输出完整交付清单（matplotlib PNG + 飞书文档 + HTML 报告 + 报告全图）
