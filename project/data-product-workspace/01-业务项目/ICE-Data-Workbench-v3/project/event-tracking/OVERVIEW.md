# 内容产品埋点 — 开篇导读

> 这是 [event-tracking/](.) 目录的总入口。读完应能回答：埋点设计为什么是产品 / 数据 / 工程的核心话题、内容产品埋点和电商 / SaaS 有什么不同、应该按什么顺序学、最容易踩的坑在哪。

---

## 一、为什么埋点是被严重低估的话题

业内常见现象：

- 产品上线半年，跑 AB 实验时发现"关键指标埋点没埋"
- 埋点 SDK 升级，**所有数据点都偏了 5%**，没人发现
- 推荐团队抱怨"完播率信号脏"，因为 5 个端的"完播"定义都不一样
- 数据看板显示 DAU 涨了——结果是埋点 bug 导致一个事件触发两次
- 增长投放团队和产品团队用**两套不同的"激活"定义**，互相打架

**埋点失败不是"数据问题"，是产品决策的根基崩了**。任何上层的 BI、AB、因果、推荐、增长都建立在埋点的真实性和一致性上。

行业里一个被反复验证的经验：

> **每花在埋点设计上的 1 小时，省下数据团队后期清洗的 10 小时**。

这是为什么头部公司都把 "事件分类法 / 埋点规范"做成专门的工程产物——但很多公司还停留在 "PM 写文档发给前端" 的原始阶段。

---

## 二、内容产品埋点的特殊性

电商 / SaaS / 工具产品的埋点已经有成熟范式（Segment Spec、Amplitude Taxonomy 都偏这两类）。**内容产品的埋点比它们复杂得多**，原因：

| 维度 | 电商 / SaaS | 内容产品 |
|---|---|---|
| 核心动作 | 注册、加购、下单、付费——离散、清晰 | 曝光、播放、滑动、互动——连续、模糊 |
| 单次会话事件量 | 10-50 | 100-1000+ |
| 关键指标 | 转化率、付费率 | 留存、时长、互动率、推荐反馈 |
| 推荐系统耦合 | 弱 | **极强**（每次曝光都是模型反馈信号） |
| 内容生命周期 | N/A | 生产 → 审核 → 分发 → 消费 → 二次传播 |
| 跨设备 / 跨端 | 一般 | **常见**（手机看、电脑看、电视看） |
| 实时性要求 | 一般 | **极高**（推荐 / 风控秒级反馈） |

**内容产品的埋点本质上是给"推荐系统"和"内容生态"做的反馈神经系统**——不只是给 BI 看数据。这一点决定了它的设计原则和电商 / SaaS 不一样。

## 三、心智模型：埋点 = 反馈神经系统

借用控制论的框架（详见 [../cybernetics/concepts/feedback-and-homeostasis.md](../cybernetics/concepts/feedback-and-homeostasis.md)）：

```
内容产品 = 一个巨大的反馈控制系统

   产品 / 内容（输出）
        │
        ▼
    用户行为（信号）
        │
        ▼ ← ← ← 埋点（传感器）
        │
        ▼
    BI / 推荐 / AB / 增长（控制器）
        │
        ▼
   产品 / 内容（修正）
```

**埋点的角色 = 这个反馈环的传感器**。

- 传感器精度差 → 控制器决策偏
- 传感器丢数据 → 控制器盲区
- 传感器定义飘移 → 控制器越调越错

按 [Ashby 必要多样性定律](../cybernetics/concepts/ashby-requisite-variety.md)：**控制器的多样性 ≥ 干扰的多样性**。映射到埋点：

> **埋点能区分的状态种类 ≥ 业务真实变化的种类，否则任何上层决策都会失稳。**

这是为什么"事件类型够多 + 参数够丰富"是产品长期价值的基础设施。

## 四、四根支柱

把整个埋点体系简化为四件事，剩下的都是细节：

```
┌──────────────────────────────────────────────────────────────┐
│                         埋点体系                              │
├────────────────┬───────────────┬─────────────┬──────────────┤
│ 1. 事件设计    │ 2. 参数设计    │ 3. 数据契约 │ 4. 治理      │
│ (Events)       │ (Parameters)  │ (Schema)    │ (Governance) │
├────────────────┼───────────────┼─────────────┼──────────────┤
│ 命名规范        │ 公共参数       │ Schema      │ 版本管理     │
│ 分类法          │ 事件特有参数   │ Registry    │ 数据质量     │
│ 内容核心事件    │ 必填 / 选填    │ 兼容性      │ SRM 监控     │
│ 推荐反馈信号    │ 类型规范       │ 验证        │ 弃用流程     │
└────────────────┴───────────────┴─────────────┴──────────────┘
```

学习时也按这四根柱子走：先想清楚"要追踪什么"（事件），再看"要附带什么信息"（参数），然后看"怎么形式化"（契约），最后看"怎么持续治理"（治理）。

## 五、最常见的七个陷阱

放在导读，是因为新做埋点的团队 80% 在这里栽：

### 1. 命名混乱
- 现象：`click_btn`、`btn_click`、`btnClick`、`clickButton` 各团队各写各的
- 后果：数据查询时一个意图要 `WHERE event_name IN (...)` 列十几个变体
- 解法：建立**强制命名规范** + Lint 工具

### 2. 没有事件分类法
- 现象：300 个事件平铺，找不到要的
- 后果：新人查埋点要问老员工
- 解法：`action_object` 模式 + 分类树（详见 [concepts/event-taxonomy.md](concepts/event-taxonomy.md)）

### 3. 公共参数和事件参数混乱
- 现象：同一个 `user_id` 在 30 个事件里写了 30 遍 schema
- 后果：升级时改 30 处
- 解法：**公共参数与事件参数严格分层**（详见 [concepts/parameter-design.md](concepts/parameter-design.md)）

### 4. 客户端 vs 服务端不分
- 现象：付费事件埋在客户端 → 用户改包绕过 → 数据失真
- 后果：核心指标永远不对
- 解法：**关键事件必须服务端埋点**

### 5. 没有 schema 校验
- 现象：前端误传 `user_id: null`，写库失败但没告警
- 后果：上线一周才发现某 SDK 整体掉线
- 解法：上报前 schema 校验 + 服务端落库前再校验 + 失败告警

### 6. 埋点改动无版本管理
- 现象："曝光"定义从 v1.5 起改了（新增"必须可见 50%"），但旧数据按旧定义
- 后果：跨版本数据不可比
- 解法：**事件版本 + 计算口径文档化**

### 7. 推荐系统反馈信号设计错
- 现象：把"快速滑走"埋成正反馈（"用户看完了"），导致模型训练向负方向跑
- 后果：推荐质量持续退化
- 解法：**显式区分正 / 负 / 中性反馈**（详见 [content-events/core-content-events.md](content-events/core-content-events.md)）

## 七、和本工作区其他模块的关联

- **AB 测试**（[../ab-testing/](../ab-testing/)）：实验完全依赖埋点。**SRM 检测**（[../ab-testing/pitfalls/peeking.md](../ab-testing/pitfalls/peeking.md)）的本质就是埋点数据质量监控。
- **因果推断**（[../causal-inference/](../causal-inference/)）：所有协变量、处理变量、结果变量都来自埋点。埋点不全 = 因果识别不完整。
- **Agent + LLM**（[../agent-llm/production/observability.md](../agent-llm/production/observability.md)）：LLM 应用的 trace 本质也是埋点；事件与参数设计的原则相通。
- **控制论**（[../cybernetics/](../cybernetics/)）：埋点是反馈控制系统的传感器层；Ashby 多样性定律、信息熵都直接适用。

---

## 八、一句话总结

**埋点不是"事后给数据补数"，是产品决策的反馈神经系统。事件与参数设计的质量决定上层一切（AB、推荐、增长、洞察）的天花板。把它从"脏活"提升为"工程资产"，是数据团队成熟度的标志。**

## 九、Checklist：你的埋点体系健康吗

```
□ 1. 有写下来的"命名规范文档"吗？大家用一致的词典吗？
□ 2. 事件按分类法组织（不是 300 个平铺）？
□ 3. 公共参数 vs 事件参数严格分层？
□ 4. 关键事件（付费 / 核心动作）是服务端埋点吗？
□ 5. 有 schema 校验和失败告警？
□ 6. 事件 / 参数有版本管理 + 弃用流程？
□ 7. 推荐反馈信号显式区分正 / 负 / 中性？
□ 8. 上线前有埋点 review + QA 流程？
□ 9. 有 SRM-like 监控（每天看埋点量是否在合理区间）？
□ 10. 跨端 / 跨版本的数据有"可比性"保证？
```

回答不出来其中任何一个，回去 [concepts/](concepts/) 补课。
