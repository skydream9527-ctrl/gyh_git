# MLOps 与平台工程

> 模型训出来只是开始——从"一个 checkpoint"到"线上稳定服务、持续迭代、成本可控"，中间是整个 MLOps 平台。一个没有 MLOps 的团队，每次上线都是手工搬砖、每次回滚都是事故、每次扩容都是赌博。本文讲清实验管理、模型仓库、调度平台（Ray/K8s）、监控、灰度发布、成本治理。

---

## 一、MLOps vs DevOps：为什么不能直接套用

| 维度 | DevOps（传统软件） | MLOps（模型） |
|---|---|---|
| 产物 | 代码 + 配置 | 代码 + 数据 + 模型权重 + 超参 + 环境 |
| 版本 | git 即可 | 模型/数据也要版本化（大文件） |
| 测试 | 单测/集成测试 | 离线评测 + 在线 A/B + 数据漂移检测 |
| 部署 | 镜像 + 滚动 | 镜像 + 权重 + GPU 调度 + 灰度 |
| 监控 | 延迟/错误率/资源 | + **预测质量、数据漂移、概念漂移** |
| 回滚 | 上一版本 | 上一版本 + 上一份权重 + 上一份数据 |

**核心差异**：模型的行为由"数据 + 代码 + 权重"共同决定，三者都要版本化和可追溯。这是 MLOps 比 DevOps 复杂的根本原因。

## 二、实验管理：训了 200 个实验，哪个最好？

### 1. 要解决的问题

训练过程中产生大量实验（不同超参/数据/代码版本），需要：

- 记录每次实验的全部上下文（代码 commit、超参、数据版本、环境）
- 跟踪指标曲线（loss、eval、自定义业务指标）
- 对比实验、复现实验、共享给团队

### 2. 主流工具

| 工具 | 定位 | 特点 |
|---|---|---|
| **MLflow** | 开源通用 | 实验追踪 + 模型仓库 + 部署，生态最广 |
| **Weights & Biases** | 商业 SaaS | 可视化强、协作好，学术界流行 |
| **TensorBoard** | 轻量 | TF/PyTorch 自带，本地调试够用 |
| **ClearML / Comet** | 商业 | 类似 W&B |

### 3. 实践要点

- **每次实验必记**：git commit、超参、数据 hash、环境（镜像/依赖）、随机种子
- **指标分级**：训练指标（loss）→ 评测指标（benchmark）→ 业务指标（线上 AB）
- **复现性测试**：随机抽 5% 实验重跑，验证指标可复现（不可复现 = 有隐藏状态没记）
- **实验命名规范**：`{模型}_{数据}_{关键超参}_{日期}`，别用 `exp_final_v2_real`

## 三、模型仓库与版本化

### 1. 要版本化的东西

| 产物 | 大小 | 版本化方式 |
|---|---|---|
| 代码 | 小 | git |
| 超参 / 配置 | 小 | git（yaml） |
| 数据 | 大 | DVC / 对象存储 + hash |
| 模型权重 | 大（GB-TB） | 模型仓库（MLflow / HF Hub / 对象存储） |
| 评测结果 | 中 | 实验追踪系统 |

### 2. 模型生命周期

```
训练 → 评测（离线） → 注册到模型仓库（Staging） → 灰度（Canary） → 生产（Production）
                                                              ↓
                                                         监控异常 → 回滚
```

**模型仓库的核心能力**：

- 版本管理：每个 checkpoint 一个版本，不可变
- 阶段标记：Staging / Production / Archived
- 血缘追溯：模型 ← 哪次实验 ← 哪个 commit + 哪份数据
- 权限控制：谁能推 Production

### 3. 工具

- **MLflow Model Registry**：开源，和 MLflow Tracking 一体
- **Hugging Face Hub**：模型分发，适合开源/内部共享
- **W&B Artifacts**：商业，血缘追踪强
- **对象存储 + 元数据库**：自建，大文件存 S3/OSS，元数据存 DB

## 四、调度平台：Ray 与 Kubernetes

### 1. 两类计算负载

| 负载 | 特点 | 典型 |
|---|---|---|
| 训练 | 长时、固定资源、GPU 密集 | 分布式训练任务 |
| 推理 | 长驻服务、弹性扩缩、GPU | 模型 serving |
| 批处理 | 短时、可并行、CPU/GPU | 数据处理、批量推理、评测 |

### 2. Kubernetes：通用容器编排

**适合**：推理服务（长驻）、批处理作业、平台底座

- 标准容器编排，生态成熟
- GPU 调度：通过 device plugin（NVIDIA GPU Operator）
- 弹性扩缩：HPA（CPU/内存）/ VPA / 自定义指标（QPS）
- 局限：GPU 共享、拓扑感知调度较弱，需额外组件（如 Volcano、Kueue）

**推理 serving on K8s**：

```
KServe / Seldon Core / vLLM + K8s Deployment
  - 模型权重用 PVC / 对象存储拉取
  - GPU 资源 request/limit
  - HPA 按 QPS 扩缩
  - 滚动更新 / 金丝雀
```

### 3. Ray：分布式计算统一框架

**适合**：训练、批处理、推理（Ray Serve）、强化学习

- 统一 API：任务（task）、Actor（长驻）、部署（Serve）
- 弹性：节点动态加入/退出
- Ray Serve：推理 serving，支持多模型路由、自动扩缩
- Ray Data / Ray Train：数据处理 / 分布式训练

**Ray vs K8s 取舍**：

| 维度 | Kubernetes | Ray |
|---|---|---|
| 定位 | 通用容器编排 | 分布式计算框架 |
| GPU 调度 | 设备级 | 任务级 + 共享 |
| 适合 | 长驻服务、混合负载 | 训练/批处理/推理统一 |
| 学习曲线 | 陡（生态复杂） | 中（Python 原生） |
| 典型组合 | 推理服务底座 | 训练 + 数据 + 推理一体化 |

**实战**：很多团队用 **K8s 做底座 + Ray 跑计算负载**（Ray on K8s），各取所长。

### 4. 训练调度

- **Slurm**：HPC 传统调度，GPU 集群事实标准，队列/抢占/拓扑感知强
- **Kubeflow / Volcano / Kueue**：K8s 上的批处理调度
- **Ray Train**：Ray 上的分布式训练

大模型训练集群多用 Slurm（拓扑感知、IB 配置成熟），推理/平台多用 K8s。

## 五、监控：模型上线后看什么

### 1. 三层监控

```
基础设施层：GPU 利用率、显存、温度、网络、磁盘
   ↓
服务层：QPS、延迟（TTFT/TPS）、错误率、队列长度
   ↓
模型质量层：预测分布、数据漂移、概念漂移、业务指标
```

前两层和传统服务监控一样（Prometheus + Grafana），**第三层是 MLOps 独有且最易被忽视的**。

### 2. 数据漂移与概念漂移

| 类型 | 含义 | 检测方法 |
|---|---|---|
| 数据漂移（Data Drift） | 输入分布变了（用户问的变了） | 输入特征分布对比（KS 检验、PSI） |
| 概念漂移（Concept Drift） | 输入→输出的关系变了（标注标准变了） | 输出分布、业务指标下降 |
| 上游数据问题 | 特征管道出错、缺失值 | schema 校验、完整性检查 |

**LLM 特有信号**：

- prompt 长度分布变化（用户开始问长文）
- 工具调用成功率下降（环境变了）
- 拒答率上升（触发了新的安全边界）
- 用户反馈（点赞/点踩）比例变化

### 3. 监控工具

- **基础设施/服务**：Prometheus + Grafana、Datadog、夜莺
- **模型可观测**：Arize / Fiddler / WhyLabs（商业）、Evidently（开源）
- **LLM 专用**：LangSmith、Langfuse、Helicone（追踪 prompt/链路/成本）

### 4. 告警原则

- 业务指标（转化率、满意度）下降 → P0
- 数据漂移显著（PSI > 0.2）→ P1
- 延迟 / 错误率超阈值 → P1
- GPU 利用率低 → P2（成本信号，非故障）

## 六、灰度发布与回滚

### 1. 为什么模型灰度比代码灰度更复杂

- 模型行为不可预测（同样的输入，新旧模型输出可能差很多）
- 质量指标滞后（要积累足够样本才能判断）
- 回滚要同时回代码 + 权重 + 配置

### 2. 灰度策略

| 策略 | 做法 | 适用 |
|---|---|---|
| 影子（Shadow） | 新模型只推理不返回，对比结果 | 无风险验证 |
| 金丝雀（Canary） | 1% → 5% → 25% → 100% 流量 | 标准做法 |
| A/B 测试 | 按用户分桶，统计对比 | 量化决策 |
| Multi-Armed Bandit | 自动调流量到表现好的 | 探索/利用平衡 |

### 3. 灰度时盯什么

- **业务指标**：核心转化、满意度、留存（最重要，但滞后）
- **代理指标**：点击率、完成率、人工评分采样
- **安全指标**：拒答率、违规率、投诉率
- **系统指标**：延迟、错误率、资源（别让新模型把服务压垮）

### 4. 回滚机制

- **一键回滚**：模型仓库保留 Production 上一版本，配置切换即可
- **权重预热**：回滚目标版本的权重提前加载到 GPU，避免冷启动
- **流量切换**：通过服务网格（Istio）或网关层切流量，秒级生效
- **回滚演练**：定期演练，确保回滚链路真的可用（很多团队回滚按钮第一次用就失败）

## 七、成本治理

### 1. LLM 推理成本拆解

```
总成本 = GPU 租用 / 折旧
       + 网络 / 存储
       + 人力（运维 + 算法）
       + 外部 API（如果混用闭源模型）
```

推理场景 GPU 占 70-90%，是治理重点。

### 2. 降本杠杆（按收益排序）

| 杠杆 | 收益 | 风险 | 说明 |
|---|---|---|---|
| 模型路由（大小模型分流） | 50-80% | 中 | 简单请求走小模型，难的走大模型 |
| 量化（INT4/FP8） | 50% | 低-中 | 见 [04-inference-optimization.md](04-inference-optimization.md) |
| Prefix Caching | 30-70% | 低 | 多轮/Agent 场景收益巨大 |
| 批处理优化 | 3-5× | 低 | Continuous Batching + PagedAttention |
| 弹性扩缩 | 20-40% | 低 | 按流量自动伸缩，闲时缩容 |
| 请求降级 | 视情况 | 中 | 高峰期降质量保可用 |
| 蒸馏小模型 | 5-10× | 高 | 质量损失需评测 |

### 3. 模型路由（最大杠杆）

```
用户请求 → 路由器判断难度
            ├─ 简单（80% 流量）→ 小模型（7B，便宜 10×）
            └─ 复杂（20% 流量）→ 大模型（70B，贵但准）
```

- 路由依据：prompt 长度/关键词/分类器/先让小模型试再决定
- 收益：80% 流量走小模型 → 整体成本降 70%+
- 风险：路由错误导致质量下降，需持续监控路由准确率

### 4. GPU 利用率治理

- **分时复用**：训练集群闲时跑批处理推理/评测
- **GPU 共享**：MIG（H100/A100 物理切分）/ 时间分片，小模型共享一张卡
- **spot/抢占式实例**：批处理任务用 spot，省钱 60%+
- **利用率告警**：GPU 利用率持续 < 40% → 该缩容或合并

### 5. 成本可观测

- 按 API / 用户 / 模型 / 场景拆分成本（token 级计费）
- 每日成本看板 + 异常波动告警
- ROI 视角：每个场景的"成本 / 业务收益"，砍掉负 ROI 的调用

## 八、平台工程化：从"能跑"到"好维护"

### 1. 平台分层

```
应用层：业务方调用（SDK / API 网关）
   ↓
能力层：模型路由、prompt 管理、评测、灰度
   ↓
运行时：推理引擎（vLLM/TRT-LLM）、训练框架
   ↓
调度层：K8s / Ray / Slurm
   ↓
资源层：GPU 集群、存储、网络
```

### 2. 关键平台能力

| 能力 | 价值 |
|---|---|
| 模型即服务（MaaS） | 业务方不碰 GPU，调 API 即可 |
| Prompt 管理 | prompt 版本化、AB、回滚 |
| 评测平台 | 自动化离线 + 在线评测 |
| 特征/数据平台 | 统一数据接入、版本化 |
| 实验平台 | 一键起训练、自动调参 |
| 成本平台 | 全链路成本归因 |

### 3. 内部开发者体验（IDP）

- 标准化模板：新模型上线用脚手架，不用从零搭
- 自助式：业务方自助配置路由/灰度，不用平台团队介入
- 文档 + Runbook：常见故障有标准处置流程

## 九、常见踩坑

1. **只监控不告警**：指标都采集了，但没设告警规则，出问题靠用户投诉。每个关键指标都要有阈值告警。
2. **数据漂移没人看**：模型质量悄悄下降，等业务指标崩了才发现。漂移检测要常态化。
3. **回滚没演练**：以为能回滚，真出事时权重没预热/配置对不上/链路断了。定期演练。
4. **成本不分摊**：所有团队共用集群，没人对成本负责。按场景/团队分摊成本。
5. **灰度看错指标**：只看延迟/错误率（系统指标），不看业务质量 → 新模型"系统正常但效果变差"上线了。
6. **GPU 利用率低不治理**：买一堆 H100，利用率 20%。要么缩容要么分时复用，别让卡空转。
7. **实验不可复现**：没记全上下文，三个月后想复现某个好结果，发现忘了随机种子/数据版本。
