# 第一性原理 (First Principles Thinking)

> "我们不能用制造问题时的同一水平思维来解决问题。" — 阿尔伯特·爱因斯坦

## 什么是第一性原理

第一性原理是一种**从最基本的事实和假设出发、逐层向上推导**的思维方式。它拒绝类比推理和经验惯性，要求回到事物的本质，从零开始构建认知。

**类比推理 vs. 第一性原理：**

| 思维方式 | 逻辑 | 特点 | 风险 |
|----------|------|------|------|
| 类比推理 | 别人怎么做 → 我也这么做 | 快速、省力 | 继承他人的假设和局限 |
| 第一性原理 | 基本事实 → 逻辑推导 → 结论 | 缓慢、费力 | 可能忽略实践验证 |

## 核心流程

```
┌───────────────────────────────────────────────────────┐
│                   第一性原理流程                         │
├───────────────────────────────────────────────────────┤
│                                                       │
│  ① 识别问题                                           │
│     "我们在解决什么？"                                  │
│         ↓                                             │
│  ② 分解假设                                           │
│     "哪些是事实？哪些只是假设？"                         │
│         ↓                                             │
│  ③ 质疑假设                                           │
│     "这个假设必须成立吗？为什么？"                       │
│         ↓                                             │
│  ④ 回归基本事实                                       │
│     "哪些是不可再分的基本真理？"                         │
│         ↓                                             │
│  ⑤ 从零重构                                          │
│     "仅基于基本事实，最优解是什么？"                     │
│         ↓                                             │
│  ⑥ 验证与迭代                                         │
│     "结论经得起现实检验吗？"                             │
│                                                       │
└───────────────────────────────────────────────────────┘
```

## 各阶段详解

### ① 识别问题

**目标：** 明确真正要解决的问题，而非表象

**方法：**
1. **5 Whys 追问法**：连续追问"为什么"5次，直达根因
2. **问题重构**：用3种不同的方式描述同一个问题
3. **约束识别**：列出所有已知的硬约束和软约束

**检查清单：**
- [ ] 问题描述不包含解决方案
- [ ] 区分了"问题"和"解决方案"
- [ ] 识别了问题的边界和范围
- [ ] 确认了谁拥有这个问题

### ② 分解假设

**目标：** 把隐含假设暴露出来

**方法：**
1. **假设清单**：列出所有前提条件，标记来源
2. **分类标注**：事实（已验证） / 假设（未验证） / 信念（不可验证）
3. **依赖映射**：画出假设之间的依赖关系

**假设分类框架：**

| 类型 | 定义 | 验证方式 | 示例 |
|------|------|----------|------|
| 事实 | 已被数据或实验证实 | 无需再验证 | "用户日活 500 万" |
| 假设 | 未验证但可验证 | A/B 测试、调研 | "用户更偏好深色模式" |
| 信念 | 不可证伪的价值判断 | 领导层对齐 | "用户体验优先于短期收入" |

### ③ 质疑假设

**目标：** 找出哪些假设可以被推翻

**方法：**
1. **逆向思维**：假设反面成立，推导后果
2. **极端案例**：在极端条件下假设是否还成立
3. **跨域类比**：其他行业如何处理类似假设

**质疑清单：**
- [ ] 每个假设都经过"如果反过来呢？"检验
- [ ] 识别了"所有人都这么认为"的从众假设
- [ ] 区分了"不可能"和"没人试过"
- [ ] 检查了假设的时效性（去年成立的今年还成立吗？）

### ④ 回归基本事实

**目标：** 找到不可再分的基本真理

**基本事实的标准：**
- 可被独立验证
- 不依赖其他假设
- 在当前条件下必然成立

**互联网行业常见的基本事实：**

| 领域 | 基本事实 | 不是基本事实的 |
|------|----------|----------------|
| 用户 | 用户的时间有限且零和 | "用户喜欢短视频" |
| 技术 | 摩尔定律持续降低计算成本 | "微服务一定比单体好" |
| 商业 | 收入 = 流量 × 转化率 × 客单价 | "免费模式一定能赢" |
| 组织 | 沟通成本随团队规模非线性增长 | "大团队一定比小团队快" |

### ⑤ 从零重构

**目标：** 仅基于基本事实，设计最优方案

**方法：**
1. **约束优化**：在基本约束下求最优解
2. **理想最终结果 (IFR)**：想象资源无限的理想方案，再逐步加约束
3. **功能分解**：将需求分解为最小功能单元，独立优化每个单元

**重构清单：**
- [ ] 方案中每个元素都有基本事实支撑
- [ ] 没有因为"别人都这么做"而添加的元素
- [ ] 方案的复杂度是必要的，而非习惯性的
- [ ] 考虑了至少一个违反常规的替代方案

### ⑥ 验证与迭代

**目标：** 确保推导出的结论在现实中可行

**验证方式：**
1. **最小可行实验**：用最低成本验证核心假设
2. **反例搜索**：主动寻找推翻结论的证据
3. **同行评审**：让不了解推导过程的人挑战结论

## 互联网行业应用场景

### 场景 1：技术选型

**类比推理：** "大厂都用 Go，我们也用 Go"

**第一性原理：**
1. 基本事实：我们的服务是 I/O 密集型，QPS 峰值 10K，团队 8 人擅长 Python
2. 推导：Python + 异步框架可满足 QPS，开发效率高，团队学习成本低
3. 结论：选 Python + FastAPI，而非跟风 Go

### 场景 2：产品定价

**类比推理：** "竞品月费 99 元，我们定 89 元"

**第一性原理：**
1. 基本事实：用户价值 = 每月节省 20 小时，时薪 100 元 = 2000 元价值
2. 推导：定价可锚定用户价值而非竞品，99 元只是价值的 5%
3. 结论：分阶梯定价（基础 199 / 专业 499 / 企业 999），锚定价值而非竞品

### 场景 3：组织架构

**类比推理：** "科技公司都该用扁平架构"

**第一性原理：**
1. 基本事实：沟通路径 = n(n-1)/2，每增加 1 人沟通成本非线性增长
2. 推导：团队规模超过 8 人时，需要中间协调层减少沟通路径
3. 结论：小团队（≤8人）扁平，大团队分 squad + chapter 混合架构

## AI 时代的特殊应用

### AI 辅助第一性原理思考

AI 擅长快速分解和质疑假设，但**不擅长判断基本事实的可靠性**。最佳模式：

```
人类定义问题边界 + 识别基本事实
        ↓
AI 辅助分解假设 + 生成质疑角度
        ↓
人类筛选有效质疑 + 确认基本事实
        ↓
AI 辅助从零重构 + 生成方案空间
        ↓
人类评估方案可行性 + 做出决策
```

### AI 做第一性原理决策的风险

| 风险 | 描述 | 缓解措施 |
|------|------|----------|
| 幻觉事实 | AI 编造看似合理但虚假的"基本事实" | 人类逐一验证每个事实 |
| 过度抽象 | AI 倾向于过度一般化，忽略具体约束 | 加入领域专家审核 |
| 忽略隐性知识 | AI 无法获取组织内的非正式知识 | 在 AGENTS.md/MEMORY.md 中显式记录 |
| 确认偏误放大 | AI 可能强化而非质疑用户的偏好 | 要求 AI 生成反对意见 |

## 与其他框架的结合

### 第一性原理 + OODA 循环

```
观察 (Observe)     ← 收集基本事实
    ↓
定向 (Orient)      ← 第一性原理分解假设
    ↓
决策 (Decide)      ← 从零重构方案
    ↓
行动 (Act)         ← 最小可行实验验证
```

### 第一性原理 + 决策矩阵

1. 用第一性原理确定评估维度（而非照搬通用维度）
2. 用决策矩阵量化各选项在基本事实维度上的得分
3. 结合两者：第一性原理保证维度正确，决策矩阵保证比较公平

### 第一性原理 + AI 上下文基础设施

- AGENTS.md 中的"关键规则"应来自第一性原理推导
- MEMORY.md 中的"关键决策"应记录推导链路
- USER.md 中的"工作环境"应只包含基本事实约束

## 常见误区

### 误区 1：把"回归本质"当成"简化问题"

- **问题**：忽略了问题的真正复杂性
- **后果**：方案看似优雅但不可行
- **正确做法**：第一性原理是找到正确的基本单元，而非忽略复杂单元

### 误区 2：拒绝一切类比

- **问题**：类比推理有快速传递经验的价值
- **后果**：重复造轮子，浪费时间
- **正确做法**：类比用于启发灵感，第一性原理用于验证结论

### 误区 3：过度推导

- **问题**：从基本事实推导过长链路，每步引入新假设
- **后果**：结论不可靠，比类比推理更差
- **正确做法**：每步推导不超过 3 层，每层验证

### 误区 4：忽视实践验证

- **问题**：沉迷于理论推导，不接受现实反馈
- **后果**：理论上完美，实践中失败
- **正确做法**：推导后必须做最小可行实验

## 实战案例：在线教育平台转型决策

### 背景

一家 K12 在线教育公司面临政策变化，需要转型。管理层提出了 3 个方向。

### 第一性原理推导

**Step 1：识别问题**
- 表象问题：政策限制了 K12 学科培训
- 真正问题：如何基于已有能力找到新增长点

**Step 2：分解假设**
- 假设1：我们是一家 K12 教育公司 → 未验证
- 假设2：转型意味着完全放弃现有业务 → 未验证
- 假设3：必须进入全新赛道 → 未验证

**Step 3：质疑假设**
- 质疑1：我们不是"K12 教育公司"，我们是"拥有直播技术、内容生产能力和 500 万家庭用户的公司"
- 质疑2：转型不等于放弃，可以迁移核心能力
- 质疑3：新赛道也可能是已有能力的新组合

**Step 4：回归基本事实**
- 事实1：拥有成熟的直播互动技术
- 事实2：拥有内容生产和审核流程
- 事实3：拥有 500 万家庭用户的数据和信任
- 事实4：素质教育和成人培训不受政策限制

**Step 5：从零重构**

| 方向 | 利用的事实 | 新增投入 | 市场规模 |
|------|-----------|----------|----------|
| A. 素质教育（编程/美术） | 事实1+2+3+4 | 中 | 中 |
| B. 成人职业培训 | 事实1+2+4 | 高 | 大 |
| C. 教育技术服务（SaaS） | 事实1+2 | 低 | 大但竞争激烈 |
| D. 家庭生活服务 | 事实3 | 高 | 未知 |

**Step 6：验证**
- 对 A 做最小可行产品测试，3 周内验证转化率
- 对 C 做 3 个客户访谈验证需求

### 最终决策

选择 A（素质教育）作为主方向，C（教育 SaaS）作为副方向。理由：基于第一性原理，A 最大化利用已有基本事实，新增投入最少，风险最低。

---

## 记住

1. **第一性原理不是拒绝经验，而是拒绝未经检验的假设**
2. **基本事实必须可验证——不可验证的不是事实，是信念**
3. **推导链路越短越可靠——每多一层推导就多一层风险**
4. **AI 是第一性原理的放大器，但也可能放大错误——人类必须验证基本事实**
5. **实践是最终裁判——再完美的推导也需要现实检验**
