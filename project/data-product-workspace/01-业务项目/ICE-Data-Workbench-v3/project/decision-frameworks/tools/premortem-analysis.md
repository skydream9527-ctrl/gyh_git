# 预-mortem分析 (Pre-mortem Analysis)

> "事前验尸"——假设决策已经失败，倒推失败原因，从而提前规避风险。

## 什么是预-mortem分析？

预-mortem分析（Pre-mortem Analysis）由心理学家 Gary Klein 提出，是一种**逆向思维**的决策风险识别工具。传统的项目复盘（Post-mortem）是在失败后分析原因，而预-mortem 则是在决策**执行之前**，假设项目已经失败，然后倒推可能的失败原因。

```
传统复盘 (Post-mortem)          预-mortem 分析
┌──────────────────┐          ┌──────────────────┐
│  项目执行         │          │  假设已经失败     │
│      ↓           │          │      ↓           │
│  失败发生         │          │  倒推失败原因     │
│      ↓           │          │      ↓           │
│  分析原因（太晚） │          │  提前规避（正好） │
└──────────────────┘          └──────────────────┘
```

**核心价值**：
- 克服**乐观偏误**（团队天然倾向高估成功率）
- 激发** dissenting voices**（反对意见在"假设失败"语境下更容易表达）
- 识别**盲点**（常规风险评估容易遗漏的失败路径）
- 生成**应对预案**（不只是识别风险，还要准备对策）

## 核心步骤

### 步骤 1：设定假设（5 分钟）

团队负责人宣布：

> "假设现在是 6 个月后，我们的决策/项目已经彻底失败了。不是部分成功，而是灾难性的失败。"

**要点**：
- 用"彻底失败"而非"遇到困难"——极端假设才能激活深层担忧
- 设定具体时间（3 个月/6 个月/1 年），与项目周期匹配
- 所有人必须接受这个假设，不能说"不可能"

### 步骤 2：独立生成失败原因（10 分钟）

每位参与者独立写下 **3-5 个可能导致失败的原因**。

**引导问题**：
- 什么外部事件可能导致失败？
- 我们的哪个假设可能是错的？
- 什么内部问题可能被我们忽略了？
- 如果失败了，回头看，最可能的原因是什么？
- 哪个利益相关方可能阻碍我们？

**独立生成的原因类型**：

| 类别 | 示例 |
|------|------|
| 技术风险 | 核心技术方案不可行、性能瓶颈 |
| 市场风险 | 用户需求变化、竞争对手抢先 |
| 组织风险 | 关键人员离职、跨部门协作失败 |
| 资源风险 | 预算超支、资源不足 |
| 执行风险 | 时间线过于激进、依赖方延期 |
| 合规风险 | 政策变化、法律风险 |

### 步骤 3：汇总与聚类（10 分钟）

收集所有失败原因，进行归类和去重。

**聚类方法**：
1. 每人在便签上写出原因
2. 贴到白板上，按相似性聚类
3. 合并同类项，标注出现频次
4. 标记"独特洞察"（只有一人提出但很有价值的原因）

**输出格式**：

| 失败原因 | 提出人数 | 类别 | 严重程度 |
|----------|----------|------|----------|
| 核心技术方案不可行 | 5/8 | 技术 | 🔴 高 |
| 竞品提前发布类似功能 | 3/8 | 市场 | 🟡 中 |
| 跨部门协作流程卡点 | 4/8 | 组织 | 🔴 高 |
| 关键工程师离职 | 2/8 | 资源 | 🟡 中 |
| 数据合规政策变化 | 1/8 | 合规 | 🟡 中 |

### 步骤 4：评估与排序（10 分钟）

对每个失败原因评估：
- **发生概率**：高/中/低
- **影响程度**：灾难性/严重/轻微
- **可预防性**：可控/部分可控/不可控
- **当前预警信号**：现在能观察到什么征兆？

**优先级矩阵**：

```
影响程度 ↑
灾难性   │  ② 监控+预案  │  ① 立即行动  │  ① 立即行动  │
严重     │  ③ 记录关注   │  ② 监控+预案  │  ② 监控+预案  │
轻微     │  ④ 暂不处理   │  ③ 记录关注   │  ③ 记录关注   │
         └──────────────┴──────────────┴──────────────┘
              低               中               高       → 发生概率
```

### 步骤 5：制定应对策略（15 分钟）

针对高优先级的失败原因，制定具体应对措施：

| 失败原因 | 预防措施 | 应急预案 | 预警指标 | 责任人 |
|----------|----------|----------|----------|--------|
| 技术方案不可行 | 第2周做PoC验证 | 备选方案B已预研 | PoC关键指标不达标 | 技术负责人 |
| 跨部门协作卡点 | 提前与协作方对齐 | 升级机制+并行推进 | 协作方响应超3天 | PM |
| 竞品抢先发布 | 缩短MVP周期 | 差异化功能快速跟进 | 竞品动态监测 | 产品负责人 |

**应对策略类型**：
- **预防**：降低发生概率（如 PoC 验证、提前沟通）
- **对冲**：减少发生后影响（如备选方案、并行推进）
- **接受**：无法预防也无法对冲，记录并监控
- **转移**：将风险转移给第三方（如保险、外包）

## 实战案例：电商平台推荐系统升级预-mortem

### 背景

某电商平台计划将推荐系统从规则引擎升级为深度学习模型，预计 3 个月完成。团队 8 人，已在决策矩阵中选出方案 A（自研模型）。

### 预-mortem 执行

**步骤 1**：假设 6 个月后，推荐系统升级项目彻底失败。

**步骤 2**：独立生成失败原因

| # | 失败原因 | 提出人 | 类别 |
|---|----------|--------|------|
| 1 | 模型训练数据质量差，导致推荐效果不如规则 | 算法工程师 | 技术 |
| 2 | 线上推理延迟超 200ms，用户体验下降 | 后端工程师 | 技术 |
| 3 | 推荐结果不可解释，运营无法干预 | 运营负责人 | 业务 |
| 4 | 老系统下线后新系统出 Bug，无法回滚 | 技术负责人 | 技术 |
| 5 | A/B 测试期间新模型 CTR 下降，决策层叫停 | PM | 组织 |
| 6 | 隐私法规变化，训练数据不合规 | 法务 | 合规 |
| 7 | 关键算法工程师在项目中期离职 | HR | 资源 |
| 8 | 与搜索团队协作接口未对齐 | 搜索团队 | 组织 |

**步骤 3-4**：评估排序

| 失败原因 | 概率 | 影响 | 优先级 |
|----------|------|------|--------|
| 训练数据质量差 | 高 | 灾难性 | ① 立即行动 |
| 推理延迟超标 | 中 | 灾难性 | ① 立即行动 |
| A/B测试CTR下降 | 中 | 严重 | ② 监控+预案 |
| 老系统无法回滚 | 低 | 灾难性 | ② 监控+预案 |
| 运营无法干预 | 高 | 严重 | ② 监控+预案 |
| 关键人员离职 | 低 | 严重 | ③ 记录关注 |

**步骤 5**：制定应对

| 失败原因 | 预防措施 | 应急预案 | 预警指标 |
|----------|----------|----------|----------|
| 数据质量差 | 第1周做数据质量审计 | 规则引擎兜底 | 数据覆盖率 < 90% |
| 推理延迟超标 | 第2周做延迟压测 | 模型蒸馏+缓存 | P99 > 150ms |
| CTR下降 | A/B测试延长至4周 | 保留规则引擎并行 | CTR 下降 > 5% |
| 老系统无法回滚 | 设计灰度切换机制 | 双系统并行1个月 | 回滚成功率 |
| 运营无法干预 | 开发运营干预面板 | 紧急人工干预流程 | 运营投诉数 |

### 结果

- 发现了 2 个被原方案遗漏的关键风险（数据质量、运营干预）
- 调整了项目计划：增加了 2 周数据准备期和运营干预功能
- 灰度方案从"上线后灰度"改为"双系统并行灰度"
- 项目最终延期 2 周上线，但避免了可能的重大事故

## 与其他工具的结合

### 预-mortem + 决策矩阵

```
决策矩阵选出最优方案
        ↓
预-mortem 验证最优方案的隐藏风险
        ↓
风险过高 → 回到决策矩阵重评
风险可控 → 执行 + 风险应对计划
```

### 预-mortem + 风险评估

```
风险评估识别已知风险
        ↓
预-mortem 发现未知风险（盲点）
        ↓
合并两份风险清单
        ↓
统一制定应对策略
```

### 预-mortem + ADR

```
ADR 记录决策及依据
        ↓
预-mortem 补充决策风险和应对
        ↓
ADR 的"风险"和"备选方案"段更完整
```

## 常见错误

### 1. 流于形式

**问题**：团队敷衍了事，只列出显而易见的风险
**后果**：预-mortem 变成走过场，未发现真正盲点
**解决**：
- 强制"独立生成"环节，避免从众
- 鼓励"荒谬但可能"的原因
- 邀请项目外部人员参与（新鲜视角）

### 2. 只识别不应对

**问题**：列出失败原因后没有制定应对策略
**后果**：团队知道风险但没有任何行动改变
**解决**：
- 每个高优先级风险必须指定责任人
- 应对措施纳入项目计划，而非单独存在
- 设定预警指标和检查点

### 3. 过度使用

**问题**：所有决策都做预-mortem
**后果**：团队疲劳，分析瘫痪
**解决**：
- 只在重大决策（影响 > 1 个月工作量或 > 50 万预算）时使用
- 小决策用简版：只做步骤 2（独立生成）+ 步骤 4（评估）

### 4. 否认假设

**问题**：参与者说"这不可能失败"
**后果**：乐观偏误未被克服
**解决**：
- 主持人必须坚持"假设已经失败"的前提
- 用历史案例证明"好项目也会失败"
- 反问："如果你是事后分析者，你会怎么看？"

## 进阶技巧

### 1. 角色化预-mortem

让不同角色从各自视角想象失败：
- **用户视角**："用户为什么不用/讨厌这个功能？"
- **竞争者视角**："竞争者会怎么利用我们的弱点？"
- **监管者视角**："监管机构会关注什么问题？"
- **黑客视角**："系统会被怎么攻击或滥用？"

### 2. 时间序列预-mortem

不只在决策前做一次，而是在关键里程碑前重复：

```
项目启动 → 预-mortem 1（战略风险）
MVP交付  → 预-mortem 2（执行风险）
规模化   → 预-mortem 3（运营风险）
```

### 3. 红队对抗

指定 1-2 人组成"红队"，专门寻找方案弱点。红队不受社交压力影响，可以自由表达反对意见。

### 4. AI 辅助预-mortem

利用 AI 生成失败场景：
- 将决策方案和上下文输入 AI
- 让 AI 从不同行业/历史案例中检索类似失败案例
- AI 生成 10-20 个可能的失败原因
- 团队在此基础上补充和筛选

---

**记住**：
- 预-mortem 不是悲观，而是**理性的乐观**——预见了风险才能自信前行
- 最重要的洞察往往来自"只有一人提出"的原因
- 预-mortem 的价值在于**改变计划**，而不只是增加意识
- 好的预-mortem 会让团队更自信，而非更恐惧
