# LLM应用开发实践指南

> 从0到1搭建生产级LLM应用的完整路线图，避开常见坑。
> 更新时间：2026-06-29

---

## 一、LLM应用开发成熟度路线图

不要一上来就做复杂Agent，按成熟度逐步迭代：

| 阶段 | 架构 | 适合场景 | 难度 | 成功率 |
|---|---|---|---|---|
| **Level 0** | 基础Prompt工程 | 文案生成、总结、翻译、简单问答 | ★☆☆☆☆ | 90% |
| **Level 1** | RAG检索增强 | 知识库问答、文档查询、基于自有数据回答 | ★★☆☆☆ | 80% |
| **Level 2** | 工具调用/Function Calling | 查询数据、调用API、执行简单操作 | ★★★☆☆ | 60% |
| **Level 3** | 单Agent ReAct/Plan-Execute | 多步骤推理、复杂任务、数据分析 | ★★★★☆ | 40% |
| **Level 4** | 多Agent协作 | 跨领域复杂任务、生产流水线、专业分工 | ★★★★★ | 20% |

**核心建议**：能在低Level解决的问题，就不要升高Level。90%的业务场景Level 0-2就够了，不要盲目上复杂Agent。

## 二、Level 0：Prompt工程最佳实践

### 好Prompt的核心结构
```
[角色] 你是谁，要做什么
[背景] 业务背景信息
[任务] 具体要完成什么
[规则] 必须遵守的约束、输出格式
[示例] 1-3个Few-shot示例（最有效提升效果的手段）
[输入] 用户问题/数据
```

### 必用技巧
1. **写清楚输出格式**：用JSON/Markdown/固定模板，模型输出稳定性提升50%+
2. **给示例（Few-shot）**：比写一堆规则管用10倍，给1-2个正确示例比写100字说明效果好
3. **思维链（CoT）**：复杂推理要求"一步一步思考"，准确率提升明显
4. **明确边界**：告诉模型"不知道就说不知道，不要编造"
5. **拆分任务**：复杂任务拆成多个小Prompt，不要让一个Prompt干所有事
6. **温度参数调优**：
   - 创造性任务（写文案）：temperature=0.7-1.0
   - 事实性任务（问答/分析）：temperature=0.1-0.3
   - 代码/结构化输出：temperature=0-0.1

### Prompt反模式
- ❌ 模糊不清："帮我写个方案"（不说什么方案、给谁看、要多长）
- ❌ 信息过载：一次塞10万字上下文让模型总结（关键信息被淹没）
- ❌ 没有示例：格式要求写了一堆，结果模型还是输出错
- ❌ 要模型同时做矛盾的事：既要简洁又要详细，既要创新又要保守
- ❌ 过长System Prompt：超过2000字的System Prompt模型注意力会下降，重要内容放最后或者靠近用户问题

## 三、Level 1：RAG检索增强生成（最常用）

### RAG适合解决什么问题？
- LLM没有你的私有数据（内部文档、产品知识、业务数据）
- 需要回答有时效性的问题（最新资讯、实时数据）
- 需要引用来源、避免幻觉
- 不需要微调模型，成本低上线快

### RAG标准流程
```
文档加载 → 分块 → 向量化 → 存入向量库 
                                      → 查询→ 检索TopK相关块 → 拼入Prompt → LLM生成答案
用户问题 → 查询改写/扩展 → 向量化 →
```

### 分块策略（最影响RAG效果的环节）
| 分块方法 | 适合场景 | 块大小建议 | 重叠长度 |
|---|---|---|---|
| 固定长度分块 | 通用文本、结构化不强的内容 | 256-1024 token | 10-20%块大小 |
| 按语义分块 | 文章、文档、语义连贯的内容 | 512-2048 token | 10-20% |
| 按结构分块 | Markdown/HTML/代码，有标题层级 | 按标题/段落分，一块一个主题 | 不需要重叠 |
| 父子分块（Parent-Document） | 检索小块精准，返回大块给LLM看上下文 | 子块256，父块2048 | - |

**经验值**：通用业务知识库用512-1024token，重叠100-200token效果最好。

### 检索优化手段
1. **混合检索**：BM25关键词检索 + 向量语义检索，RRF融合，比单一向量检索准确率提升20-30%
2. **检索前查询改写**：用LLM把用户问题改写成更适合检索的查询，扩展同义词
3. **Rerank重排序**：检索Top20，用Reranker模型重排序取Top5给LLM，准确率提升明显
4. **元数据过滤**：按文档类型、时间、部门等元数据先过滤再检索，减少噪音
5. **多查询扩展**：生成3-5个不同角度的查询，检索结果合并去重

### RAG常见坑
- ❌ 块太大：一个块几千token，检索不精准，LLM抓不住重点
- ❌ 块太小：一块只有一句话，上下文缺失，答案不完整
- ❌ 只依赖向量检索：关键词匹配效果差，很多专有名词/编号检索不到
- ❌ 不做Rerank：Top10里很多不相关内容，干扰LLM
- ❌ 不做评测：凭感觉调参数，不知道效果到底好不好
- ❌ 不引用来源：生成答案不标来自哪里，用户不知道对不对

### RAG评测指标
- **检索阶段**：Recall@K（前K个块包含正确答案的比例）、MRR
- **生成阶段**：答案相关性、忠实度（不编造）、上下文利用率（有用到检索内容）
- **工具**：RAGAS、TruLens，或者自己构建黄金问答集

## 四、Level 2：工具调用/Function Calling

### 适合场景
- 需要查询实时数据（数据库、API、天气、股票）
- 需要执行操作（发消息、建文档、改配置）
- 需要计算/运行代码（数学计算、数据处理）

### 工具设计最佳实践
1. **工具要小而专一**：一个工具只干一件事，不要做万能工具。比如不要做一个`query_data`工具既查SQL又查CSV又查API，拆成`execute_sql`/`read_csv`/`call_api`三个工具
2. **参数描述要清晰**：每个参数的含义、类型、取值范围、示例写清楚，模型才会正确传参
3. **错误信息要友好**：工具调用失败返回自然语言错误信息和修复建议，模型可以自动纠错重试
4. **危险工具加确认**：写操作、删数据、发消息这类有副作用的工具，要让用户确认后再执行
5. **工具数量控制**：一次给模型的工具不要超过10个，太多模型会选错

### 工具调用错误处理
- 参数缺失/格式错误 → 返回错误说明，模型自动修正重试（最多重试2次）
- 工具执行失败 → 返回错误原因，模型换方法或者告诉用户
- 结果太长 → 截断返回，或者总结后返回，不要把几MB的结果塞回Prompt

## 五、Level 3：单Agent开发

### 什么时候需要Agent？
任务满足以下全部特征才需要：
1. 任务需要**多步推理**，一步Prompt做不完
2. 任务有**多种路径**，需要动态选择下一步做什么
3. 任务需要**使用工具**获取外部信息/执行操作
4. 任务有**反馈循环**，可以根据上一步结果调整下一步

如果只是问个问题、总结个文档，不需要Agent，RAG就够了。

### Agent核心循环（ReAct）
```
Thought: 我现在需要做什么，为什么，下一步行动是什么
Action: 调用什么工具，传什么参数
Observation: 工具返回结果
...（重复思考-行动-观察）
Thought: 我已经得到答案了
Final Answer: 最终答案
```

### Agent开发核心原则（来自Anthropic实践）
1. **保持简单**：能不用Agent就不用，能用单轮调用就不要做多轮
2. **先做透工具，再做Agent**：工具的可靠性比Agent逻辑重要10倍，工具都调不对Agent不可能好用
3. **SOP优先于自由发挥**：专业领域Agent写死SOP流程，比让模型自己"思考"可靠得多（参考我们的AB实验/灰度发布Agent）
4. **限制循环次数**：设置最大步数（比如10步），防止无限循环
5. **人工介入点（HITL）**：关键决策点让用户确认，不要让Agent自动做不可逆操作
6. **状态持久化**：每一步都保存状态，崩溃了可以断点续跑
7. **可观测性**：每一步的Thought、Action、Observation都要记录日志，方便调试

### Agent开发常见坑
- ❌ 一开始就做"通用人工智能"：什么都能干的Agent什么都干不好，先做垂直领域单一场景的Agent
- ❌ 不给边界：让Agent自由发挥，结果跑偏、死循环、乱调用工具
- ❌ 没有SOP：完全靠模型自己规划流程，结果每次走的路径都不一样，效果不稳定
- ❌ 不做错误处理：工具调用失败就卡住，不会重试也不会告诉用户
- ❌ 不做截断：上下文越聊越长，最后爆窗口或者成本爆炸

## 六、Level 4：多Agent系统

### 什么时候需要多Agent？
- 任务确实需要多个不同领域的专业能力（比如一个做数据分析、一个做可视化、一个写报告）
- 任务可以并行拆分，多个Agent同时干活提高效率
- 需要角色分工：比如一个主Agent调度，一个执行，一个审核验证

### 多Agent协作模式
| 模式 | 说明 | 适合场景 |
|---|---|---|
| **主从模式（Router）** | 一个主Agent负责路由，把任务分给专业子Agent，汇总结果 | 最常用，专业分工场景 |
| **流水线模式（Pipeline）** | 任务按步骤流经多个Agent，上一个输出是下一个输入 | 固定流程任务，如：获取数据→分析→生成报告→发布 |
| **平等协作模式** | 多个Agent平等讨论，投票/辩论得出结论 | 需要多方视角的决策任务 |
| **分层模式** | 上层Agent做规划，中层Agent分解任务，下层Agent执行 | 非常复杂的长任务（代码开发、科研） |

### 多Agent坑比你想的多
- ❌ 通信开销大：Agent之间来回传消息，Token成本飙升，延迟很高
- ❌ 错误传播：一个Agent出错，后面全错，调试极其困难
- ❌ 协调成本高：主Agent分配任务不合理，子Agent互相等待
- ❌ 效果不一定比单Agent好：很多时候把多个专业能力做成单Agent的多个工具，效果比多Agent好，还简单

**核心建议**：能单Agent+多工具解决的，就不要做多Agent。我们的AB实验/灰度发布Agent都是单Agent+子Agent派单模式，主流程自己做，超出领域才派给子Agent，这是性价比最高的模式。

## 七、LLM应用工程化必做事项

### 1. 模型网关层
- 不要直接调用OpenAI/Anthropic/豆包等API，用一层网关统一接入
- 支持多模型 fallback：主模型挂了自动切备用模型
- 统一Token计量、成本统计、限流、重试
- 推荐工具：LiteLLM、Portkey，或者自己写一层很简单

### 2. 可观测性（必做！！！）
不做可观测性LLM应用就是黑盒，出了问题根本不知道为什么错：
- 记录每一次请求的：输入Prompt、输出、模型、耗时、Token消耗、成本
- 记录每一步工具调用：参数、返回结果、耗时
- 记录Agent每一步的Thought/Action/Observation
- 错误日志：失败请求、重试、模型报错
- 用户反馈：点赞/点踩，收集bad case
- 推荐工具：Langfuse、LangSmith、Helicone，或者自己打点存数据库

### 3. 成本控制
- Prompt缓存：相同System Prompt/固定前缀用Prompt Caching，成本降50-90%
- 模型选型：简单任务用小模型（7B/8B），复杂任务用大模型，不要什么都用GPT-4o/Claude 3.5
- 上下文压缩：长对话自动摘要，不要每次都把全部历史塞进去
- 结果缓存：相同问题/相同工具调用结果缓存，重复请求直接返回
- Token限制：设置单请求最大Token、单用户日Token上限，防止被刷

### 4. 延迟优化
- 流式输出：必须做，用户感知延迟差10倍
- 并行调用：多个工具调用可以并行就不要串行（比如同时查多个SQL）
- 异步处理：长任务后台跑，完成了通知用户，不要让用户等几分钟
- 就近部署：模型API选近的区域，减少网络延迟
- 小模型路由：简单问题用快速小模型回答，复杂问题再升大模型（路由分类准确率90%以上就划算）

### 5. 安全与防护
- Prompt注入防护：用户输入不要直接拼到System Prompt，做输入过滤
- 工具权限控制：危险工具加白名单、加用户确认、加审计日志
- PII脱敏：敏感信息（手机号、身份证）不要传给模型
- 输出审核：生成内容过一遍内容审核，防止生成违规内容
- 速率限制：单用户/单IP限流，防止被恶意调用

### 6. 评测体系
没有评测你根本不知道改完是变好还是变坏：
- **离线黄金集**：构建100-500个典型问题，有标准答案，每次改完Prompt/模型/参数跑一遍，算准确率
- **LLM-as-Judge**：用大模型自动打分，评估答案相关性、忠实度、完整性
- **在线AB测试**：新版本上线做AB，看业务指标（解决率、用户满意度、任务完成率）
- **Bad Case收集**：用户反馈的错误案例加到黄金集，持续优化

## 八、快速上线Checklist

上线前检查这10项，避免90%的问题：
- [ ] Prompt经过测试，覆盖常见场景和边缘case
- [ ] 结构化输出有校验，格式错误自动重试
- [ ] 工具调用参数有校验，错误有处理和重试
- [ ] 有Token限制和成本告警
- [ ] 全链路日志可追溯（输入/输出/工具调用/耗时）
- [ ] 有错误兜底：模型调用失败有友好提示
- [ ] 长任务有状态提示，用户知道在处理中
- [ ] 敏感词/注入/输出审核已配置
- [ ] 有基本的黄金集评测，核心场景准确率达标
- [ ] 用户可以反馈结果好坏（点赞/点踩）

## 九、LLM应用开发反模式总结

1. **上来就做Agent**：90%场景RAG+工具调用就够了，Agent是复杂度最高的选项
2. **盲目追新**：今天学Multi-Agent，明天学GraphRAG，后天学Self-RAG，基础RAG都没做好
3. **只调Prompt不做评测**：凭感觉改Prompt，不知道改完效果变好还是变差
4. **忽略工程化**：只关注算法，不做可观测性、缓存、降级、错误处理，上线就崩
5. **追求完美架构**：搞一堆复杂抽象、框架，结果核心功能做的一塌糊涂，先跑起来再优化
6. **过度依赖大模型**：能用代码/规则解决的事情（比如计算、格式校验、路由）不要让LLM做，代码100%可靠，LLM有幻觉
7. **没有反馈闭环**：上线就不管了，不收集bad case，不持续优化，效果越来越差

---

**核心心法**：LLM应用开发是工程问题不是算法问题，80%的效果来自于细节打磨：清晰的Prompt、干净的数据、可靠的工具、完善的错误处理、持续的bad case优化，而不是用了多么新的算法或框架。
