# 【教程】 高品质产品原型的落地：Design Token构造

<callout emoji="👍">
核心洞察：**AI 越强大，约束体系越重要。** 不是给 AI 更多自由度，而是用更精确的约束让 AI 的输出稳定可控。设计师的专业价值从「执行层」上移到了「规则层」——你定义的 Token 和规范，决定了 AI 生成的每一个页面的视觉品质。
</callout>

> 从设计视角讲清楚：Design Token 怎么定义、怎么分层、怎么被 AI 消费，设计在哪些环节介入，以及为什么这套体系能让原型质量稳定可控。

---

## 这篇教程讲什么

我们在小米智学项目中，用 AI 生成了 5 个版本的高保真可交互原型（V1.0 登录与对话、V1.1 对话卡片、V1.2 任务页、V1.3 移动端增强、V2.0 拍题与批改），覆盖了从登录引导到 Agent 任务卡片系统的完整产品形态。

这不是一篇讲「什么是 Design Token」的概念文。这篇教程从设计的角度，讲清楚四件事：

1. **设计在哪里介入**：AI Native 流程中，设计的 6 个关键介入点和具体动作
2. **design-system.md 的角色**：一份 3000+ 行的视觉宪法，怎么同时服务于产品、设计、研发的AI工具
3. **Design Token 怎么落地**：从变量定义到组件消费到版本隔离的完整链路
4. **AI 怎么被约束**：为什么 AI 越强大，约束体系越重要

## 一、设计在 AI Native 流程中的介入点

传统模式下，设计是流水线的中间环节：收到 PRD → 出设计稿 → 交付研发。AI Native 模式下，流程反转了：

```Plain Text
旧模式：PRD →  原型 → 设计稿 → 评审 → 开发
新模式：原型 Spec → AI 生成 HTML 原型 → Demo 演示 → 内部试用 → PRD → 开发
```

设计不再是串行流水线上的一站，而是贯穿全流程的 6 个关键介入点：

### 介入点 1：审查原型 Spec（AI 生成之前）

产品写完原型 Spec 后、AI 生成原型之前，设计第一次介入。

**做什么**：

- 检查 Spec 中的页面布局是否符合设计规范（对话流是否单栏、间距是否引用了正确的 Token 名）
- 检查交互原则是否与已有版本一致（动画参数是否沿用统一规范的 150ms/200ms/250ms 三档）
- 检查状态覆盖是否完整（异常态、空态、加载态有没有遗漏）
- 如果涉及新组件，确认命名是否遵循 `ds-*` 共享组件规范

**不做什么**：不定义精确像素值（那是 Token 的事），不画设计稿（还没到 Figma 环节）

**对齐物**：原型 Spec + design-system.md

### 介入点 2：审查 AI 生成的 HTML 原型（最核心）

AI 生成原型后、Demo 演示前，设计打开原型页面逐帧检查。**这是设计最核心的介入点。**

**做什么**：

- **品牌调性**：整体色调是否保持冷白雾面 + 青绿品牌色？有没有违反禁止规则（高饱和蓝紫、纯黑块面、儿童风格）？
- **Token 一致性**：颜色、字号、间距、圆角是否引用了正确的 Token？有没有出现 13px、17px 这种无规则间距？
- **组件一致性**：导航栏、输入框、Tab 栏等共享组件是否与其他版本保持一致？有没有组件漂移？
- **交互一致性**：动画时长和缓动曲线是否符合统一规范？
- **信息层级**：字号、字重、颜色、留白是否正确拉开层级？
- **色彩比例**：品牌色是否控制在 3-8%？

**如果有问题**：反馈给产品，产品修改 Spec，AI 重新生成。循环直到通过。

### 介入点 3 & 4：Demo 演示 + 内部试用

原型审查通过后，设计参与 Demo 演示和内部试用。

**做什么**：从用户体验角度评估信息层级、操作路径、等待体验。发现问题时判断是 Spec 层面的（改交互规则）还是生成层面的（调样式）。关注跨版本一致性：新版本的组件和旧版本是否和谐共存。

### 介入点 5：Figma 精修（唯一需要「画像素」的环节）

PRD 确认后，设计基于已通过审查的 HTML 原型在 Figma 中精修。

**关键变化**：传统模式下 Figma 是「从零设计」，AI Native 模式下 Figma 是「精修已验证的原型」。HTML 原型已经确定了页面结构、交互逻辑、状态覆盖，设计在 Figma 中精修图标精度、微妙渐变、动效曲线。Figma 的 Variables 必须与 theme.css 的 Token 一一对应。

设计的创造性工作前移到了介入点 1 和 2，Figma 环节更多是精修和交付。

### 介入点 6：设计走查（研发提测后）

测试通过后，设计打开测试环境，逐页对照 Figma 标注检查颜色、字号、间距、圆角、阴影、动效的还原度。不通过则在飞书 PRD 对应章节下评论反馈。

## 二、design-system.md —— 设计师写给 AI 的视觉宪法

### 它是什么

`design-system.md` 是一份 3000+ 行的生产级设计系统规范文档。它不是给人翻阅的参考手册，而是同时服务于三个消费方的机读标准：

| 消费方 | 怎么用 design-system.md |
|-|-|
| AI 生成原型 | 读取 Token 定义、组件规范、禁止规则，生成符合规范的 HTML |
| Figma 设计稿 | 对齐 Variables、Auto Layout 规则、组件状态定义 |
| 前端研发 | 对齐 Android/iOS 的颜色常量、字号常量、组件结构 |

三条链路的所有颜色、字体、间距、圆角、阴影、组件结构、交互状态必须 1:1 对齐，不允许任何一端自行发挥。design-system.md 就是这个对齐的唯一基准。

### 它和代码文件的关系

design-system.md 是「规范文档」，theme.css / components.css / shell.js 是「代码实现」。两者的关系是：

```Plain Text
design-system.md（视觉宪法，定义语义和规则）
  │
  ├──→ theme.css（Token 的 CSS 实现）
  │      第 4-8 章的变量定义 → 落地为 :root 下的 ~150 个 CSS 变量
  │
  ├──→ components.css（组件的 CSS 实现）
  │      第 10 章组件规范 → 落地为 16 章组件样式，消费 theme.css 的变量
  │
  ├──→ shell.js（组件的 HTML 骨架实现）
  │      第 10 章组件规范 → 落地为 create* 函数，生成标准 HTML 结构
  │
  ├──→ tailwind-config.js（Tailwind 映射）
  │      把 CSS 变量映射为 Tailwind class，内容区快速布局用
  │
  └──→ 各版本 Spec 文档
         每个 Spec 的「设计遵循」字段引用 design-system.md 的版本号
         如：「设计遵循：小米智学移动端视觉规范 V2.5」
```

### design-system.md 的核心章节

| 章节 | 内容 | 设计关注度 |
|-|-|-|
| 第 0 章 当前运行基线 | 冻结版本号、编辑入口 vs 真实引用 | ★★★ 优先级最高 |
| 第 2 章 产品分析 | 情感设计方向、设计总原则、9 条禁止规则 | ★★★ 设计核心 |
| 第 4 章 颜色系统 | 背景/文字/品牌/语义/边框/渐变，全部 Token 化 | ★★★ 设计核心 |
| 第 5 章 字体系统 | MiSans 字族、4 级字重、13 级字号行高 | ★★★ 设计核心 |
| 第 6-8 章 间距/圆角/阴影 | 4px 栅格间距、13 级圆角、15 级阴影 | ★★☆ |
| 第 10 章 组件规范 | 每个组件的结构、状态、交互 | ★★★ 设计核心 |
| 第 12 章 动效规范 | 过渡时长、缓动曲线、触发条件 | ★★★ 设计核心 |
| 第 14 章 Figma 标准 | Auto Layout、Variables 规则 | ★★★ 设计核心 |
| 第 17 章 质量检查清单 | 逐条可打勾的检查项 | ★★☆ |

### 设计在 design-system.md 上的持续责任

design-system.md 不是写完就不动的。设计对它有 5 项持续维护责任：

1. **定义和更新 Token**：品牌色调整、新增组件需要新 Token 时，设计在 design-system.md 中定义语义和色值，然后同步到 theme.css
2. **定义组件规范**：新组件的结构、状态、交互规则由设计定义，写入第 10 章，然后落地到 components.css 和 shell.js
3. **维护禁止规则**：发现 AI 生成中反复出现某类问题时，在第 2.7 章追加禁止规则
4. **审查三端一致性**：确保 design-system.md 的定义在 CSS 变量、Figma Variables、Android/iOS 常量中保持一致
5. **版本控制**：Token 层冻结后仅允许 bugfix 级微调，小版本更新不得删除已有 Token，只允许新增

## 三、Design Token 怎么落地 —— 从定义到消费的完整链路

### 3.1 Token 定义：theme.css

theme.css 大约 200 行，定义了整个产品的视觉基因。所有设计决策都编码成 CSS 变量：

```CSS
:root {
  /* 背景：冷白雾面基底，不是纯白 */
  --c-bg: #F9FAF9;
  
  /* 品牌主色：青绿，用于 CTA、链接、强调 */
  --c-brand: #00D1B2;
  
  /* 文字：三级灰度递减，冷绿灰色调 */
  --c-text:   #1A3C34;   /* 一级：标题、正文 */
  --c-text-2: #3D5A52;   /* 二级：副标题 */
  --c-text-3: #5A7A70;   /* 三级：占位符、注释 */
  
  /* 间距：严格 4px 栅格 */
  --sp-4:  8px;   --sp-8:  16px;   --sp-10: 24px;
  
  /* 圆角：克制的圆润感 */
  --r-md: 12px;   --r-xl: 20px;   --r-full: 9999px;
  
  /* 阴影：品牌色调阴影，不是黑色 */
  --sh-sm: 0 0 16px rgba(0, 150, 137, 0.07);
}
```

几个关键设计决策：

- 背景 `#F9FAF9` 带冷绿，不是纯白——这是 HyperOS 3 Liquid Glass 的基底
- 阴影用 `rgba(0, 150, 137, ...)` 品牌色，不是黑色——让阴影融入整体色调
- 文字 `#1A3C34` 是冷绿灰，不是纯黑——在青绿渐变底上更和谐

Token 分 10 大类，约 150 个变量：背景与表面（12 个）、文字色（11 个）、品牌色（7 个）、语义功能色（8 个）、边框与分割（11 个）、渐变（14 个）、字号（13 个）、间距（15 个）、圆角（13 个）、阴影（15 个），加上布局尺寸、z-index、过渡动画、液态玻璃模糊值。

**语义别名**是一个容易被忽略但很重要的设计：同一个色值 `#90A1B9`，在不同组件里有不同的语义名（`--c-text-hint`、`--c-tab-inactive`、`--c-composer-placeholder`……）。将来要单独调整某个组件的颜色时，只改对应的语义别名，不影响其他组件。

### 3.2 第一层消费：components.css —— 共享组件样式

components.css 是所有手机帧内组件样式的唯一来源，分 16 个章节。所有属性值必须通过 `var(--xxx)` 引用 Token，禁止裸色值。

```CSS
/* 导航栏标题——每个属性都引用 Token */
.ds-shell-nav__title {
  font-size: var(--fs-heading);      /* 17px */
  line-height: var(--lh-snug);       /* 1.35 */
  font-weight: var(--fw-semibold);   /* 600 */
  color: var(--c-control-icon);      /* #1A3C34 */
}

/* 输入框液态玻璃——复杂视觉效果封装成 Token */
.ds-input-shell {
  border-radius: var(--r-xl);        /* 20px */
  box-shadow: var(--sh-glass-md);    /* 4 层复合阴影 */
  backdrop-filter: blur(var(--blur-lg)) saturate(1.8);
}
```

这里的 `--sh-glass-md` 展开后是 4 层复合阴影（外层投影 + 0.5px 描边模拟 + 顶部内高光 + 底部内阴影）。如果每个页面自己写，必然不一致。封装成 Token 后，所有输入框的玻璃感都是同一个。

### 3.3 第二层消费：shell.js —— 用函数生成组件骨架

光有样式不够，组件的 HTML 结构也需要统一。shell.js 用函数生成骨架：

```Julia
const createStatusBar = ({ time = '9:41' } = {}) => `...`;
const createNav = ({ title = 'AI老师', subtitle = '内容由AI生成', ... } = {}) => `...`;
const createComposer = ({ interaction = 'static', ... } = {}) => `...`;
const createTabBar = ({ items = DEFAULT_TAB_ITEMS, activeKey = 'home', ... } = {}) => `...`;
```

关键设计：参数有默认值（AI 不需要记住每个细节）、版本化选择器（`data-shell-version` 确保后续迭代不影响已有页面）、函数签名只增不删（向后兼容）。

### 3.4 第三层消费：版本 styles.css —— 业务差异覆盖

每个版本目录有自己的 styles.css，只处理该版本的私有差异：

```CSS
/* V1.1 对话卡片：定义版本私有动画参数 */
:root {
  --anim-micro: 150ms;
  --anim-panel: 250ms;
  --anim-expand: 200ms;
}

/* 覆盖共享组件的版本差异 */
.ds-thought-text { color: #5F6675; }

/* 新增版本私有组件，仍然引用共享 Token */
.ds-task-card {
  padding: var(--sp-6);
  border-radius: var(--r-lg);
  border: 1px solid var(--c-border);
}
```

三层消费的分层关系：**Token（theme.css）→ 组件（components.css + shell.js）→ 版本差异（Vx.x/styles.css）**。这就是为什么 V1.0 的登录页和 V2.0 的拍题页风格统一——它们共享同一套 Token 和组件基础。

## 四、版本隔离 —— 解决 AI 生成原型的两大故障

### 故障一：组件漂移

同一个输入框，在 V1.0 是 20px 圆角 + 玻璃阴影，到 V2.0 变成 12px 圆角 + 普通阴影。根因：每个页面各自定义组件样式。

**解决**：组件样式唯一来源（components.css）+ 骨架函数生成（shell.js）+ 三张速查表快速定位（组件名称、字段名称、页面名称）。页面不允许在 `<style>` 中重复定义 `ds-*` 组件样式。

### 故障二：版本污染

改了共享样式后，历史版本原型跟着变形。根因：所有版本引用同一份文件。

**解决**：物理隔离——exports（live runtime）和 versions（冻结快照）两条路径：

| 谁引用 | 引用哪里 | 特点 |
|-|-|-|
| `mobile.html` 主壳 | `exports/mobile-shell/` | 随时更新，影响线上 |
| V1.0 / V1.1 / V2.0 版本页 | `versions/v1.0.0/` | 冻结不动，历史安全 |

为什么不用选择器隔离？因为选择器隔离依赖「只追加不修改」的纪律，一旦有人改了旧规则，历史页面就坏了。物理隔离是硬保障——快照文件冻结后不再修改。

### 版本冻结流程

```Plain Text
新功能在 Vx.x/styles.css 里孵化
  → 评审确认后，稳定内容并入 exports/mobile-shell/
  → 需要新基线时，复制 exports/ 为 versions/vX.Y.Z/
  → 更新 route-registry.json 的 sharedBaseline
  → 新孵化页开始引用新的 vX.Y.Z
```

迭代硬规则：theme.css 变量只增不删、shell.js 函数签名只增不删。

## 五、AI 怎么被约束

### 原型 Spec —— AI 的输入指令

AI 不凭空生成。它的输入是结构化的原型 Spec，定义了设计目标、交互原则、页面级规范（布局、元素、交互、状态覆盖）、动画参数、字段协议、验收清单。Spec 不定义像素值（那是 Token 的事），定义的是页面结构和交互规则。

### AI 生成的硬约束

1. 必须引用 `_shared/` 的 CSS 变量和组件样式，不能自己发明色值
2. 组件样式只从 components.css 获取，页面 `<style>` 中不得重复定义 `ds-*` 类名
3. HTML `<head>` 必须按标准顺序引用 10 个资源文件
4. 每个 HTML 文件对应 Spec 的一个页面章节
5. 每个文件不超过 5 个状态帧
6. 步骤 1 必须是可交互的主帧

### 色彩比例的量化约束

| 色系 | 占比 | 说明 |
|-|-|-|
| 中性冷白 | 72-82% | 背景、表面、分割线 |
| 文字结构灰 | 10-18% | 文字、图标、边框 |
| 品牌主色 | 3-8% | CTA、链接、活跃状态 |
| 冷光辅助色 | 1-4% | 玻璃反射、数据高光 |
| 功能色 | ≤3% | 仅在语义明确时使用 |

即使 AI 生成 100 个页面，品牌色永远只占 3-8%，不会出现「满屏绿色」。

## 六、设计角色的本质变化

| 维度 | 传统模式 | AI Native 模式 |
|-|-|-|
| 核心产出 | Figma 设计稿 | design-system.md + Token 体系 + 审查意见 |
| 工作起点 | 收到 PRD | 审查原型 Spec（介入点 1） |
| 创造性工作 | 画每一个页面 | 定义规则体系 + 审查 AI 输出（介入点 1-2） |
| Figma 的角色 | 从零设计 | 精修已验证的原型（介入点 5） |
| 质量保障 | 逐像素交付 | 审查 AI 输出 + 设计走查（介入点 2 + 6） |
| 迭代周期 | 天级 | 小时级（改 Token 或 Spec，AI 重新生成） |
| 影响范围 | 单个页面 | 全局（改一个 Token 影响所有页面） |
