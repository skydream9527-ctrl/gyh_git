# 共识背景

目前的AI Native开发方案存在一些问题：

- **信息损耗严重**：HTML 原型无法无损转化为 Flutter 代码，产品调一遍界面、客户端再调一遍，冗余浪费。
- **瀑布流分工与 AI 赋能矛盾**：虽然目前已是spec驱动，但是仍无法逃离产品spec→设计spec→研发spec→测试spec的串行分工，没有发挥并行工作优势。
- **仓库体积膨胀**：随着功能增多，每次用 AI 工作时上下文很快占满，缺乏高效的索引机制。

<callout emoji="◽">
**每个团队成员 = 人（经验、审美、架构思维、产品思维、市场判断）+ AI 能力 = 全栈工程师**
</callout>

一个人使用这些能力，在统一的仓库架构上，完成一个需求从提出到落地的全流程。

## Git仓库工作流程（基础设置）

### 需求产出流程

所有需求走同一条路径，需求产出单位：个人或产研结对。

<whiteboard token="JRMqwUiO9hiTj0bBuIVcIDKPnVf"></whiteboard>

**快捷通道 1：线上 bugfix / 小需求改动**

- 准入标准（满足全部才可走快捷通道）：

| 条件 | 说明 |
|-|-|
| 改动文件 ≤ 5 个 | 超过 5 个文件说明影响范围较大，应走正式流程 |
| 不涉及新增接口 / 新增数据库字段 / 新增埋点 | 新增这些内容意味着需要更新注册表，应走正式流程 |
| 不涉及跨模块改动 | 如果同时改 server + client + agent，说明是联动需求，应走正式流程 |

- 归档义务：快捷通道不免除归档，只免除 Spec/Demo/评审。CI 在合并到 online 时自动生成轻量归档记录：

```Java
{
  "type": "hotfix",
  "timestamp": "2026-04-14T10:00:00+08:00",
  "author": "张三",
  "summary": "修复错题详情页在空数据时的崩溃",
  "affected_modules": ["client"],
  "commit_hash": "abc1234",
  "files_changed": 2
}
```

**快捷通道 2：提示词修改 / 配置修改**

- CI 自动更新事实仓库中的提示词版本记录（`archive/registry/prompts/`）
- 如果提示词修改影响了评测基线，自动触发评测集回归测试
- 配置修改自动记录变更日志（`archive/registry/configs/`）

<callout emoji="⚠️">
连续走多次快捷通道后，事实仓库的轻量记录可能与线上实际状态产生漂移。
防护机制：**每个迭代周期结束时（如每周），运行一次"全量对账"脚本**，对比 online 分支的实际代码与事实仓库的注册表，输出差异报告。发现漂移时，由管理员决定是否补充正式归档。
</callout>

### 多仓库 + 权限隔离架构

团队统一采用“多仓库 + 权限隔离 + 同仓库双分支”的组织方式：

> 产研在 `develop` 分支内完成 Spec、Demo、评审和增量实现；合并到 `online` 后，再把当次确认版本的 Spec、Demo 快照和评审结论沉淀到事实库。

<whiteboard token="SFplwPyNrhP7LMb4xAocf3B8nce"></whiteboard>

## 多视角 Spec （能力覆盖）

因为一个人要完成多工种的工作，Spec 不再只是各自为阵，而是包含**6个视角**的综合规格文档：

| 视角 | 核心内容 | 立场 |
|-|-|-|
| **产品视角** | 需求范围、功能定义、功能点、交互效果、数据逻辑 | 业务标准、用户体验 |
| **测试视角** | 用例设计大纲、代码质量、功能测试、覆盖率要求、边界条件清单、回归范围 | 功能完善程度与代码质量 |
| **数据视角** | 埋点需求、指标定义、数据分析计划、实验设计 | 业务指标定义与观测 |
| **评测视角** | 提示词设计、工具调用、模型参数、AI 输出质量评测标准、降级策略 | AI 产出质量 |
| **技术视角** | 架构设计、接口定义、组件设计、数据模型、性能、安全 | 功能实现路径 |
| **运维视角** | 监控方案、回滚方案、依赖服务 SLA | 服务稳定性 |

这些视角的具体定义和撰写要求统一存储在事实库的 Spec 撰写规范中，AI 每次生成 Spec 时必须先读取该规范；具体工作稿则放在对应需求工作仓库的 `develop` 分支。评审结论直接回写到同一需求的 Spec 章节中，再驱动 `develop → online` 的后续合并与归档。

<callout emoji="💡">
核心原则：  
**6 个视角全部由 Agent 自动生成，人只在产品视角定稿时审阅一次，在全部spec产出后审阅一次。**
</callout>

| 目的 | 为所有仓库提供统一的规范、索引和归档，**缩减上下文** |
| 维护者 | 少数维护者写入，其他角色按需只读，系统与工具按协议回写 |

**权限**

| **角色** | **权限** |
|-|-|
| 普通成员 | 只读 — 不需要手动维护，只需读取 |
| 系统/CI | 自动写入 — Git Hook 联动、归档沉淀、数据回收回流 |
| 管理员 | 手动修正 — 更新设计规范、代码规范等规范性内容 |

**自动归档机制**

| 触发时机 | 自动归档内容 | 写入位置 |
|-|-|-|
| develop → online 合并时 | 完整 Spec 文件、Demo 快照、评审结论 | 事实库 `archive/specs/{version}/` |
| develop → online 合并时 | 从 Spec 中提取的新增字段、接口、埋点、组件信息 | 事实库 `archive/registry/`（注册表） |
| 快捷通道完成时 | 轻量归档记录（变更摘要、影响范围、commit hash） | 事实库 `archive/hotfix-log/` |

<callout emoji="💡">
**注册表更新从 Spec 提取，不从代码反向解析**。Spec 中已有结构化的字段清单、接口清单、埋点清单，直接搬运即可，自动化成本远低于代码解析。人只维护规范文件（`standards/` 目录），归档文件全部由 CI/Hook 自动写入。
</callout>

**注册表双向校验**  
"从 Spec 提取注册表"是主路径，但实现过程中接口可能发生变更。为防止 Spec 与代码不一致：

1. **PR 级门禁**：代码 PR 中如果修改了接口签名、数据库 DDL、协议字段，CI 自动检测并要求同步更新 Spec
2. **合并时双向校验**：合并到 online 时，不仅从 Spec 提取注册表，还从代码中提取接口签名（OpenAPI/Swagger 自动生成），对比两者是否一致，不一致时阻塞合并
3. **每次合并后自动对账**：全量对账从"每个迭代周期"改为"每次 online 合并后自动触发"  
**Design Token 引用强制规则**

<callout emoji="❌">
Spec 中的视觉规格**必须引用 Design Token 名称**（如 `AppColorTokens.kBrand`），**禁止硬编码数值**（如 `#00D1B2`）。Token 值修改后全局自动生效，实现一键替换。
</callout>

事实库的 Design Token 文件提供 CSS 变量映射表，AI 生成 Spec 和 Demo 时强制使用 Token 引用。Token 变更时 CI 自动扫描所有引用了该 Token 的 Spec，生成变更影响清单。

**事实仓库的引用方式**

<callout emoji="❌">
**禁止**将事实仓库的内容复制到其他仓库。事实仓库是独立 Git 仓库，其他仓库只读。
</callout>

所有仓库 clone 到同一个工作区目录下并排放置：

```Plain Text
workspace/
├── fact-repo/          # 事实仓库（独立 Git 仓库）
├── ai-tutor/           # 需求仓库（独立 Git 仓库）
├── tutorclaw/          # 服务仓库（独立 Git 仓库）
└── autoeval/           # 工具仓库（独立 Git 仓库）
```

各仓库的 Steering 文件通过**相对路径**引用事实仓库的规范，事实仓库更新后各仓库开发者只需 `git pull` 即可获取最新规范，不存在跨仓库文件冲突。新人 onboard 时提供一键脚本把所有仓库 clone 到同一个 workspace 下。

**AI 读取策略（三级索引，强制）**

1. 先读本仓库本地索引（每次 commit 自动更新，最快最新）
2. 本地索引覆盖不了的 → 读事实库摘要索引（跨仓库全局索引）
3. 事实库也覆盖不了的 → 才读源代码文件
4. 上下文超过 70% → 降级低优先级内容为摘要引用

**上下文管理落地方案：分层 Steering 文件**

原"70% 降级"规则在当前 AI 工具中不可操作（工具无法感知自身上下文占用率）。

替代方案为分层 Steering + 按需展开：

| 层级 | 文件 | 加载策略 | 内容 |
|-|-|-|-|
| L1 必读 | `L1-must-read.md` | 始终自动加载 | 架构红线、命名规范、分层约束、禁止项。控制在 2000 字以内 |
| L2 按需 | `L2-{module}.md` | 条件加载（编辑对应模块时触发） | 当前模块的接口契约、数据模型、组件规范 |
| L3 索引 | `L3-index.md` | 手动引用 | 事实仓库的摘要索引（每个模块一句话 + 文件路径），需要时通过引用加载 |

<whiteboard token="BUcywUckEhRqIJbvhKic24MKnXb"></whiteboard>

> 优势：上下文占用可预测（L1 约 2000 字 + L2 约 3000 字），不依赖 AI 工具的运行时感知能力，与 Kiro/Cursor/Claude Code 的 steering 机制天然兼容。

## 需求仓库

| 分支 | 说明 |
|-|-|
| 线上分支`online` | 多视角 `Spec`、可运行 Demo、评审记录、`handoff.json`、增量代码与联调脚本。 |

### 需求工作仓库（开发分支）

**定位**

| 属性 | 说明 |
|-|-|
| 定义 | 同一业务仓库中的 `develop` 分支，承接需求从 0 到 1 的孵化、评审以及增量实现 |
| 目的 | 在一个分支内完成 `Spec → Demo → 评审 → 正式实现 → 测试`  <br/>避免产品方案和代码实现长期分仓，有信息差 |
| 维护者 | 当前需求成员共同读写，按模块边界协作 |

**承载**

**原则**

1. Demo 是评审资产，也是合并后需要归档到事实库的证据。
2. `handoff.json` 记录同一需求内的协作改动，不再承担跨仓交接含义。
3. 未合并到 `online` 前，相关资料都视为工作稿，不写入事实库正式归档。

**develop 分支策略**

<callout emoji="💡">
**每个需求点一个 develop 分支**，不是所有需求共用一个 develop。
</callout>

**分支命名规范**：`develop/{version}-{feature-name}`  
示例：`develop/V3.1-whiteboard-enhance`、`develop/V3.2-photo-batch-import`  
**分支生命周期**：

<whiteboard token="T3towlksth9GXpb7GQtcz2xMnmf"></whiteboard>

**并行规则**：多个 develop 分支可以同时存在互不干扰，每个 develop 分支内只有一份 Spec（该需求的 6 视角 Spec），合并到 online 前需要解决与其他已合并分支的代码冲突。  
**分支冲突预防机制**

1. **DDL migration 编号段预分配**：每个 develop 分支创建时预分配 migration 编号段（如 V8-V10），避免多分支 DDL 编号冲突
2. **定期 rebase**：develop 分支存活超过 5 个工作日，必须从 online rebase 一次，CI 自动检测偏移量并预警
3. **冲突预检**：develop 分支合并前自动 rebase online，冲突超过 5 个文件时强制要求人工介入
4. **hotfix 反向同步**：hotfix 合入 online 后，CI 自动向所有活跃 develop 分支发起 merge PR，无冲突自动合并，有冲突通知分支负责人 24 小时内处理

**handoff.json 定义**

<callout emoji="⚡">
类比：handoff.json 相当于同一个 develop 分支上所有 AI Coding 使用者的**共享聊天窗口**。
</callout>

handoff.json 是同一个 develop 分支内、多个 AI Coding 使用者之间的**变更通知媒介**。当一个人（通过 AI）修改了页面结构、协议字段、接口定义等，其他人的 AI 工具读取 handoff.json 就能知道"同伴改了什么，我需要同步改什么"。

**Schema**：

```Java
{
  "version": "V3.1",
  "feature": "whiteboard-enhance",
  "changes": [
    {
      "id": "chg-001",
      "timestamp": "2026-04-14T10:00:00+08:00",
      "author": "产品视角 / 张三",
      "scope": "页面结构",
      "summary": "新增错题详情页底部操作栏",

    }
  ]
}
```

**字段说明**：

| 字段 | 类型 | 说明 |
|-|-|-|
| `id` | string | 变更唯一标识，格式 `chg-{序号}` |
| `timestamp` | ISO 8601 | 变更时间 |
| `author` | string | 变更发起者（视角 + 人名） |
| `scope` | string | 变更范围：页面结构 / 接口定义 / 数据模型 / 协议字段 / 提示词 / 配置 |
| `summary` | string | 一句话描述改了什么 |
| `affected_perspectives` | string[] | 哪些视角需要跟进同步 |
| `details` | object | 结构化的变更详情（新增/修改的页面、字段、接口等） |
| `action_required` | string | 需要其他视角执行的具体动作 |

**使用规则**：

1. 每次通过 AI Coding 修改了 Spec 或代码中的关键结构，AI 工具自动追加一条 change 记录
2. 其他人的 AI 工具在开始工作前，先读取 handoff.json 检查是否有未处理的变更
3. 处理完变更后不删除记录（保留完整历史），可在 change 上追加 `resolved_by` 字段标记已处

**handoff.json 格式与并发策略：**

- handoff.json 是多人共写的公共上下文（类似共享聊天窗），必须保持单文件语义。为解决 JSON 数组追加的 Git 合并冲突，**推荐格式：NDJSON（每行一条 change）**

```Plain Text
{"id":"zhangsan-2026-04-14T10-00-00","timestamp":"2026-04-14T10:00:00+08:00","author":"产品视角/张三","scope":"页面结构","summary":"新增底部操作栏","affected_perspectives":["技术视角"],"action_required":"补充组件设计"}
{"id":"lisi-2026-04-14T10-05-00","timestamp":"2026-04-14T10:05:00+08:00","author":"技术视角/李四","scope":"接口定义","summary":"新增详情API","affected_perspectives":["测试视角"],"action_required":"补充接口测试"}
```

- 保持单文件、共享上下文的语义
- Git 行级合并可自动处理大部分追加场景
- `id` 使用 `{author}-{timestamp}` 格式，彻底避免序号冲突

**状态管理**  
将 `resolved_by` 改为 `resolutions` 数组，每个受影响视角独立追踪处理状态：

```Java
"resolutions": [
  {"perspective": "技术视角", "status": "resolved", "resolved_by": "李四", "resolved_at": "2026-04-14T11:00:00+08:00"},
  {"perspective": "测试视角", "status": "pending"}
]
```

**归档机制**  
当一条 change 的所有 `affected_perspectives` 都标记为 resolved 后，下次 CI 运行时将该条 change 移入 `handoff-archive.ndjson`。AI 工具日常只读取 `handoff.json`（未完全处理的变更），需要查历史时才读 archive。

3. 线上 hotfix 处理完成后，必须反向同步回 `develop`。

# 多视角Spec介绍

视角 Spec 是**需求孵化的核心准则**。每个需求只有一份 Spec，包含 6 个视角。

> 每个 Spec 开头声明改动范围：纯前端 / 纯后端 / 前后端联动 / 纯提示词 / 纯配置，便于控制评审范围、测试范围等

## 产品视角 — 做什么、用户怎么用

| **产出项** | **具体内容** | **输出标准** |
|-|-|-|
| 设计目标 | 3-5 条核心目标，说明要达成什么用户心智或业务效果 | 每条目标可验证 |
| 用户画像与场景 | 目标用户（年龄段/角色/技术熟练度）  <br/>使用场景（什么时间/什么情境触发）  <br/>User Story 列表（作为 X，我想要 Y，以便 Z） | 每个 Story 标注优先级 P0/P1/P2 |
| 适用范围与排除范围 | 本次覆盖的页面/功能清单  <br/>明确排除的内容（避免范围蔓延） | 两个列表，无歧义 |
| 页面清单 | 分模块、分步骤地列出Demo要展示的界面的名称、状态、功能范围 | 每个步骤做一个手机帧展示 |
| 组件清单 | 涉及到的组件（包括新组件以及复用组件）的名称、字段、结构、样式、交互、约束等 | 组件支持跨页面复用 |
| 边界场景处理 | 风控逻辑、网络异常、空数据、超长内容、连续操作、降级方案、未上线能力的处理方式 | 每个场景有明确的用户可见行为 |

**CI 归档容错机制**：事实仓库写入失败时，归档内容暂存到业务仓库 `.archive-pending/` 目录，定时补偿 Job 每 30 分钟扫描重试，3 次失败后告警人工介入。归档 CI 建议采用 PR 模式（向事实仓库提交 PR 而非直接 push），天然串行化且有审计记录。
