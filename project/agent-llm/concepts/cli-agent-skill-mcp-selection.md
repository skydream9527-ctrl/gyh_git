# AI 产品四件套：CLI、Agent、Skill、MCP 的职责边界与选型

> 在 AI 产品（尤其 Coding Agent、助手类产品）的设计中，**CLI、Agent、Skill、MCP** 经常被并列提及，也经常被混用。本文用最短篇幅讲清四者**各自负责什么、各自的优劣势、以及产品经理在选型时该怎么决策**。

## 一、一句话定位

> **CLI 是入口，Agent 是主体，Skill 是方法论，MCP 是协议。**

| 概念 | 一句话 | 类比 |
|---|---|---|
| **CLI** | 人与 AI 系统交互的命令行入口 | 收银台 |
| **Agent** | 能自主决策、多步执行的智能体主体 | 店长 |
| **Skill** | 封装"某类问题怎么做"的方法论 | SOP 手册 |
| **MCP** | 标准化连接外部工具/数据的协议 | USB-C 接口 |

它们解决的是**四个不同层面**的问题，不存在替代关系，只存在协作关系。

## 二、四者各负责什么

### 2.1 CLI —— 负责交互入口

- **职责**：接收用户输入、渲染输出、管理会话状态、处理权限确认
- **不负责**：决策"做什么"、调用什么工具
- **典型形态**：`claude`、`cursor`、`codex` 命令行；Trae/Cursor 的对话框；Slack bot 入口

### 2.2 Agent —— 负责决策与执行

- **职责**：理解意图、规划步骤、选择工具、调用工具、观察结果、循环直到完成
- **不负责**：具体某一步"怎么做"（这是 Skill 的职责）；怎么连到外部系统（这是 MCP 的职责）
- **典型形态**：ReAct loop、Plan-Execute loop、Multi-Agent 协作

### 2.3 Skill —— 负责方法论封装

- **职责**：把"某类问题应该怎么解决"的工作流、规则、检查清单封装成可加载的指令
- **不负责**：执行具体原子操作（这是 Tool 的职责）；连接外部数据源（这是 MCP 的职责）
- **典型形态**：`code-review-checklist.md`、`debug-protocol.md`、`research-methodology.md`

### 2.4 MCP —— 负责工具/数据的标准化连接

- **职责**：定义"如何发现工具、如何调用工具、如何共享资源"的标准协议
- **不负责**：决定何时调用工具（这是 Agent 的职责）；工具内部怎么做（这是 Tool 实现的职责）
- **典型形态**：GitHub MCP Server、Postgres MCP Server、Slack MCP Server

## 三、核心对比表

| 维度 | CLI | Agent | Skill | MCP |
|---|---|---|---|---|
| **本质** | 交互界面 | 决策主体 | 方法论文档 | 连接协议 |
| **抽象层次** | 表现层 | 控制层 | 知识层 | 协议层 |
| **形态** | 可执行程序 | 循环 + LLM | Markdown 指令 | JSON-RPC 协议 |
| **状态** | 有状态（会话） | 有状态（规划/记忆） | 无状态 | 无状态（连接有状态） |
| **谁调用它** | 终端用户 | CLI / 上层 Agent | Agent 加载 | Agent / Tool 调用 |
| **改动成本** | 高（需重新发版） | 中（改 prompt） | 低（改 md） | 中（改 server） |
| **复用范围** | 单产品 | 项目内 | 项目内 / 跨项目 | 跨产品（生态级） |

## 四、各自优劣势

### 4.1 CLI

**优势**
- 用户体验最直接，延迟最低
- 权限控制、审计、流式输出都在这一层
- 适合需要人在回路（HITL）的场景

**劣势**
- 必须有人盯着，无法后台自主运行
- 交互带宽有限，复杂任务表达成本高
- 单点入口，挂了就全挂

**适用场景**
- 开发者日常编码、调试
- 需要实时反馈和权限确认的任务
- Demo、教学、交互式探索

---

### 4.2 Agent

**优势**
- 能处理开放式、多步骤、需要规划的复杂任务
- 可以自主组合工具、迭代修正
- 是"长任务"的唯一可行形态

**劣势**
- 成本高（多轮 LLM 调用、token 消耗大）
- 不可控性强，容易跑偏、陷入循环
- 评估难（结果好坏难以自动化判定）
- 延迟高，不适合简单确定性任务

**适用场景**
- 长任务（写 PRD、做调研、改大型代码库）
- 需要跨多个工具组合的任务
- 目标明确但路径不确定的任务

---

### 4.3 Skill

**优势**
- **成本极低**：本质是 Markdown，改一行字就能上线
- **可解释**：人能读、能 review、能审计
- **可组合**：一个 Agent 可加载多个 Skill
- **按需加载**：description 在 context，body 按需注入，省 token

**劣势**
- 依赖 Agent 的指令遵循能力，模型弱时形同虚设
- 不能直接执行操作，必须借 Tool 落地
- Skill 之间可能冲突（两个都触发时优先级难定）
- 难以评估单个 Skill 的边际贡献

**适用场景**
- 沉淀团队方法论（PRD 模板、决策框架、调试 SOP）
- 给 Agent 注入领域知识（业务术语、合规规则）
- 跨项目复用工作流

---

### 4.4 MCP

**优势**
- **生态级复用**：写一次 server，所有支持 MCP 的客户端都能用
- **标准化**：工具发现、调用、鉴权协议统一
- **解耦**：工具开发者和 Agent 开发者可以独立迭代
- **能力扩展性强**：从 M×N 降到 M+N

**劣势**
- 协议开销：简单的工具用 MCP 是过度工程
- 调试复杂：跨进程、跨网络，排查链路长
- 生态早期，质量参差
- 性能：相比进程内函数调用有额外开销

**适用场景**
- 对接外部系统（GitHub、Jira、数据库、SaaS API）
- 需要跨多个 AI 客户端复用的工具
- 工具需要独立鉴权、独立部署的场景

## 五、四者如何协同：一个完整例子

任务：**"帮我调研竞品 X 的最新功能，整理成飞书文档"**

```
┌─────────────────────────────────────────────────────────────┐
│  用户在 CLI 输入："帮我调研竞品 X 的最新功能，整理成飞书文档" │
└─────────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  Agent 接管：                                                 │
│  1. 理解意图：调研 + 整理 + 发布                              │
│  2. 规划步骤：搜索 → 阅读 → 提炼 → 写文档                    │
└─────────────────────────────────────────────────────────────┘
                          │
        ┌─────────────────┼─────────────────┐
        ▼                 ▼                 ▼
┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│  Skill 加载  │  │  Skill 加载  │  │  MCP 调用    │
│              │  │              │  │              │
│ research-    │  │ feishu-doc-  │  │ web-search   │
│ methodology  │  │ writer       │  │ server       │
│              │  │              │  │              │
│ "按这5步调研"│  │ "用金字塔结构"│  │ fetch 网页   │
└──────────────┘  └──────────────┘  └──────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  Agent 多轮执行：搜索→读→提炼→写→用飞书 MCP 发布            │
└─────────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  CLI 渲染结果，流式输出，等待用户确认是否发布                │
└─────────────────────────────────────────────────────────────┘
```

**职责切分**：
- **CLI**：负责把用户的话接进来、把过程渲染出来、最后等用户确认
- **Agent**：负责"先搜再读再写"的整体规划
- **Skill**：告诉 Agent"调研应该按这 5 步""文档应该用金字塔结构"
- **MCP**：让 Agent 能调到 web-search、能写入飞书，不需要自己写集成代码

## 六、选型决策框架

### 6.1 决策树

```
你要解决的问题是什么？
│
├─ 用户怎么和系统交互？
│   └─→ CLI（这是入口，不是选项）
│
├─ 任务是否需要多步规划/自主决策？
│   ├─ 是 → 需要 Agent
│   └─ 否（单次确定性调用） → 直接用 Tool / MCP
│
├─ 是否有"方法论"需要复用？
│   ├─ 是 → 写成 Skill
│   └─ 否 → 不需要 Skill
│
└─ 要连接的外部系统是否需要跨产品复用？
    ├─ 是 → 做 MCP Server
    └─ 否（只在当前产品用） → 直接写 Tool 函数
```

### 6.2 选型矩阵

| 你的目标 | 优先选 | 次选 | 不选 |
|---|---|---|---|
| 让用户用得爽 | CLI 优化 | Agent 交互设计 | MCP |
| 让任务能自主完成 | Agent | Skill（教 Agent 怎么做） | CLI |
| 沉淀团队经验 | Skill | — | MCP |
| 对接外部系统 | MCP | Tool 函数 | Skill |
| 降低 token 成本 | Skill（按需加载） | 简化 Agent prompt | 多 Agent |
| 提升可控性 | Skill（强约束） + HITL CLI | 限制 Agent 自由度 | — |

### 6.3 常见误区

| 误区 | 正确理解 |
|---|---|
| "有了 Agent 就不需要 Skill" | Agent 决定"做什么"，Skill 告诉它"怎么做才对"。没有 Skill 的 Agent 是新手 |
| "MCP 是 Agent 的替代" | MCP 是工具协议，Agent 是决策主体，二者是协作不是替代 |
| "CLI 越强大越好" | CLI 应该薄。重逻辑放 Agent / Skill，CLI 只管交互 |
| "所有工具都做成 MCP" | 简单、单产品用的工具直接写成函数。MCP 是为跨产品复用而生 |
| "Skill 越多越好" | Skill 多了会冲突、会占 context。按场景裁剪 |
| "Agent 能做一切" | Agent 适合开放式任务。确定性任务用脚本更快更稳 |

## 七、产品视角的取舍清单

做产品决策时，依次问自己：

1. **这个功能用户会直接用吗？** → 是 → 优化 CLI 体验
2. **任务路径是否需要 LLM 来规划？** → 是 → 上 Agent
3. **有没有"我们团队的做法"需要固化？** → 是 → 写 Skill
4. **要连的系统是否多个 AI 产品都会用？** → 是 → 做 MCP Server
5. **延迟和成本能不能接受？** → 不能 → 砍 Agent 轮数、精简 Skill、用确定性函数替代

---

## 八、总结

| 概念 | 产品定位 | 何时投入 |
|---|---|---|
| **CLI** | 用户体验层 | 永远要投入，但保持薄 |
| **Agent** | 能力核心层 | 任务复杂、需要自主性时投入 |
| **Skill** | 知识沉淀层 | 有可复用方法论时投入，成本低收益高 |
| **MCP** | 生态连接层 | 要跨产品复用工具时投入 |

**一句话收尾**：CLI 决定用户喜不喜欢用，Agent 决定能不能完成任务，Skill 决定做得专不专业，MCP 决定能连多大生态。四者各司其职，缺一不可。
