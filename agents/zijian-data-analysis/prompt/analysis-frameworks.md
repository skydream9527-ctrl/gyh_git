# 7 大分析框架（按优先级）

> 来自都江堰自建信息流历史分析报告沉淀。每个分析任务**必须命中且仅命中一个框架**，作为 Phase 1 拆解的 tag。

---

## P0 · 下钻维度分析（最高频）

**思路**：整体指标 → 一级维度 → 二级维度 → 根因定位

**SQL 能力**：分组聚合、维度下钻、占比计算

**典型场景**：
- 指标拆解（消费时长大盘 → 实验组 → 频道 → 体裁）
- 用户分层（设备年龄 → 起售价 → 性别）
- 体裁对比（图文 vs 短视频 vs 小视频 vs 短故事 vs 小说）

**子任务模板**：
```
T1. 主指标复核（实验大组维度）
T2. 按 feed_channel 一级下钻
T3. 按 item_type 二级下钻（限定 T2 的主导频道）
T4. 占比计算 + 贡献度排序
```

**SQL 范式**：CTE → 外层 GROUP BY 单维度 + SUM/COUNT DISTINCT

**关键陷阱**：
- 外层 GROUP BY 字段必须先在 model.sql 中已定义；要新维度就改 CTE，不要在外层 CASE WHEN
- 占比要算两层（行内占比 vs 总量占比），别混用

## P1 · 用户行为转化分析

**思路**：定义起始行为 → 定义转化目标 → 行为序列 → 分层对比 → 效果评估

**SQL 能力**：窗口函数、时间窗口、转化率计算

**典型场景**：
- Push 点击转化（push 曝光 → 点击 → 落地页消费时长）
- 全屏行为（列表页曝光 → 点击 → 内流下滑 → 完播）
- 列表页 CTR（双频道列表页曝光 PV → 点击 PV）

**子任务模板**：
```
T1. 起始事件量（曝光 PV / 曝光 UV）
T2. 中间转化（点击 PV / 点击 UV）
T3. 终点转化（消费 PV / 完播完读 PV）
T4. 各层转化率 + 漏斗图
```

**SQL 范式**：CTE → 外层多个 SUM(IF(...)) + 外层做除法

**关键陷阱**：
- 转化率分母必须是同一时间窗、同一用户群（否则就是错的）
- push 渠道单独算（feed_channel='push'），不要混进推荐渠道

---

## P1 · 对比分析

**思路**：明确对比对象 → 定义评估指标 → 分层对比 → 效果排序

**SQL 能力**：多表 JOIN、分组聚合、UNION ALL、排序

**典型场景**：
- 实验组对比（自建 vs 火山，36% vs 2%反转 vs 2%纯净）
- 召回队列效果（不同 djy_rec_expid 的渗透率）
- A/B 实验复盘

**子任务模板**：
```
T1. 整体指标对比（按 exp_group_6 分组）
T2. 关键维度对比（按 item_type / feed_channel 拆 × exp_group_6）
T3. 显著性判断（人均指标 + 差值百分比）
T4. 综合排序 + 推全建议
```

**SQL 范式**：CTE → 外层 GROUP BY 多维度（含 exp_group_6） → 同一行用多个聚合函数

**关键陷阱**：
- 自建组和火山组的曝光 PV 不能直接对比（流量盘子不同），必须用 `djy_expose_pv / total_djy_expose_pv` 这种"组内占比"
- 对比 SQL 用 UNION ALL 时各分支的口径必须相同

## P2 · 指标异动分析

**思路**：定位异动 → 时间定位 → 分层下钻 → 趋势对比 → 归因

**SQL 能力**：时间序列对比、分布分析、趋势分析、CycleDiff

**典型场景**：
- 核心指标突然变化（DAU 跌、消费时长涨/跌）
- 渠道异动（某个 feed_channel 流量异常）
- 版本灰度后的大盘波动

**子任务模板**：
```
T1. 异动事实复核（按 date 看每日趋势，确认起跳点）
T2. 维度下钻（异动起跳日 vs 基线日，按 feed_channel / item_type 拆）
T3. 实验组归因（按 exp_group_6 看是不是某组贡献的）
T4. 版本/事件关联（叠加 push 推送、灰度版本时间线）
```

**SQL 范式**：CTE → 外层 GROUP BY date + 维度 → ORDER BY date

**关键陷阱**：
- 同比/环比阈值：BF 业务线 |Δ| > 15% 才必须下钻；BM > 10%；CC > 8%
- 节假日效应：跨春节/五一/十一的趋势必须先剥离
- 不要拿单日数据下结论；至少 3~7 天滑动平均

## P2 · 样本筛选分析

**思路**：定义筛选条件 → 构建筛选集 → 对比验证

**SQL 能力**：WHERE 条件、子查询、集合操作（IN / NOT IN / EXCEPT）

**典型场景**：
- 特定用户筛选（深度用户、新用户、有效用户）
- 特定内容筛选（冷启内容、优质作者、高时效内容）
- 反向筛选（流失用户、负反馈用户）

**子任务模板**：
```
T1. 筛选规则定义（输出筛选条件 SQL 与命中率）
T2. 筛选集 vs 全量大盘对比（同一指标）
T3. 筛选集内部分层（再下钻 1~2 维）
```

**SQL 范式**：CTE → 外层 WHERE + GROUP BY；或两个子查询 EXCEPT

**关键陷阱**：
- 筛选条件用 fieldType=5 维度时，必须从 dimensions.csv 还原完整 CASE WHEN（详见 sop.md「关键约束」）
- 筛选集太小（< 1k UV）的结论慎用，告知用户置信度低

## P3 · 假设检验分析

**思路**：提出假设 → 设计验证方案 → 数据验证 → 结论

**SQL 能力**：分组对比、统计检验（差值/p-value）

**典型场景**：
- 验证某策略是否提升留存
- 验证某体裁是否对女性用户更友好
- 验证发布时效对消费的影响

**子任务模板**：
```
T1. 假设陈述（H0/H1）+ 评估指标
T2. 控制变量后的对照查询（用 UNION ALL 同时跑 H0/H1）
T3. 差值 + 显著性判断（如有 AB 实验，引用 ab-experiment Agent 的 p 值）
T4. 结论：拒绝 / 不拒绝 / 数据不足
```

**SQL 范式**：CTE → UNION ALL（H0 / H1 各一段）→ 外层差值

**关键陷阱**：
- 不要"先看数据再编假设"，那是过拟合，不是分析
- 假设若涉及 AB 实验，**优先用 ab-experiment Agent**（有 p 值表 + 有效用户率，更严谨）

## P3 · 分发路径排查

**思路**：定义排查对象 → 追踪分发链路 → 定位关键节点

**SQL 能力**：多表 JOIN、链路追踪、占比计算

**典型场景**：
- 内容冷启（新内容从入库到首次曝光的延迟与命中率）
- 召回来源分析（item_alg_source 各召回的曝光占比）
- 作者分发（item_author_id 维度的曝光分布）

**子任务模板**：
```
T1. 排查对象总量（item / author / 召回队列）
T2. 链路各节点命中量（曝光 → 点击 → 消费）
T3. 漏出节点定位（哪一层流失最大）
T4. 异常样本抽查（item_id 列表给用户挑刺）
```

**SQL 范式**：CTE → 外层 GROUP BY 链路节点字段 → COUNT DISTINCT item_id

**关键陷阱**：
- 冷启场景关注 `is_cold` + `ruku_delay` 字段
- 作者分发用 `item_author_id`（id）而不是 `item_author_name`（重名）

## 框架命中速查

| 用户原话片段 | 推荐框架 |
|---|---|
| "为什么…在跌"、"…突然不对"、"复盘…变化" | P2 异动 |
| "对比 A 和 B"、"自建 vs 火山"、"哪个组好" | P1 对比 |
| "拆解一下…的来源"、"哪些…贡献最大" | P0 下钻 |
| "Push 转化"、"列表页 CTR"、"漏斗" | P1 转化 |
| "新用户/老用户/深度用户的…"、"筛出…的用户" | P2 筛选 |
| "我猜…是不是因为…" | P3 假设 |
| "冷启"、"新内容多久能曝光" | P3 路径 |

如果用户的命题同时命中多个框架，**T1 走 P0 下钻**（确认事实），后续 T 再分别套对应框架。
