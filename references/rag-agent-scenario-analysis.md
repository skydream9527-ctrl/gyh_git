## 一、核心问题：Claude Code 为什么不用 RAG？

小林的文章通过拆解 Claude Code 源码，回答了一个尖锐问题：**为什么 Claude Code 这个业界公认的顶级 AI 编程工具，反其道而行不用 RAG，只靠 grep + 读文件？**

### 1.1 RAG 在代码场景的 5 大"水土不服"

| 痛点 | 具体问题 | 举例 |
|-|-|-|
| **切不动** | 代码有严格结构，粗暴切片会断语义 | 200 行函数按 100 行切，上半段 if 开头，下半段 else 结尾 |
| **匹配不准** | 向量擅长"模糊相似"，但代码要"精确命中" | 找 getUserById，返回 getUserByName 等一堆相似函数 |
| **索引过期** | 代码每天 commit 都在变，索引重建成本高 | 不重建查到旧版本，增量更新边界情况极多 |
| **冷启动慢** | 百万行代码建索引要十几分钟 | 用户打开工具等几分钟才能用 |
| **黑盒不可解释** | 向量召回结果无法追溯原因 | 答错了不知是召回错还是理解错 |

### 1.2 Claude Code 的反直觉方案

Claude Code 的设计哲学：**让模型像程序员一样，自己去找代码。**

不预处理、不建库、不算向量，每次需要代码就实时去现场查，工具就三个：

- **Grep**：基于 ripgrep，按内容关键字精确搜索，三种输出模式（content/files/count）
- **Glob**：按文件名模式匹配，结果按修改时间倒序，100 文件上限
- **Read**：按需读取，默认 2000 行，支持 offset/limit 分段读，永远读磁盘最新内容

**组合工作流**：Glob 找候选文件 → Grep 精确定位 → Read 读取上下文，每一步基于上一步结果调整方向。

<callout emoji="🔥">
**关键设计**：复杂探索任务派出**子 Agent**，子 Agent 在独立上下文里多轮 grep/read，只把最终结论返回主 Agent，保护主 Agent 上下文不被"污染"。这就是**上下文压缩**的精髓。
</callout>

\## 二、RAG 和 Agent 的本质关系

<callout emoji="⭐">
\*\*业界共识：RAG 和 AI Agent 不是竞争关系，而是分层协同关系。\*\*
</callout>

## 二、RAG 和 Agent 的本质关系

### 2.1 四层黄金架构

| 层级 | 技术 | 解决的问题 |
|-|-|-|
| Layer 4 | A2A（多智能体协同） | 多个 Agent 怎么配合 |
| Layer 3 | MCP（工具连接协议） | Agent 怎么调用工具 |
| Layer 2 | AI Agent（执行中枢） | 把事做完 |
| Layer 1 | RAG（知识地基） | 回答得准 |

**RAG 解决"AI 知道什么"，Agent 解决"AI 能做什么"。**

### 2.2 智能体真正需要的是"接地"

智能体的核心循环：感知环境 → 推理规划 → 执行动作 → 观察结果 → 修正下一步

接地的通道有多种：RAG 上下文注入、工具调用（API/DB/文件）、长上下文、持久记忆、数据库直连、MCP 协议。

RAG 只是接地的一种方式，不应被当成智能体架构的唯一核心。应按任务本质选择组合路径。

## 三、场景选择指南

### 3.1 适合纯 RAG 知识库的场景

**特征**：用户问一个问题，系统检索文档，给出答案。流程是线性的、单次的。

| 场景 | 举例 | 为什么 RAG 够用 |
|-|-|-|
| 企业制度/政策问答 | "年假怎么算？""报销流程？" | 答案在固定文档里，不需要执行动作 |
| 产品文档检索 | "API 接口参数有哪些？" | 结构化文档 + 精确问答 |
| 合规/制度查询 | "这个操作合规吗？" | 需要可追溯、有出处的回答 |
| 客服 FAQ | "怎么退货？""发票怎么开？" | 高频重复问题，答案标准化 |
| 技术文档检索 | "组件怎么用？""报错怎么解决？" | 文档量大但结构清晰 |

**判断标准**：用户需求是"**我知道一件事**"，不需要系统做任何操作，纯 RAG 就够了。

### 3.2 适合纯 AI Agent 的场景

**特征**：用户给一个目标，系统自主规划、调用多个工具、多步执行、验证结果。

| 场景 | 举例 | 为什么需要 Agent |
|-|-|-|
| 代码开发/修改 | "给登录功能加验证码" | 搜索代码、理解上下文、写代码、测试 |
| 运维根因分析 | "QPS 下降原因是什么？" | 查指标、查日志、查变更、综合推理 |
| 数据查询+分析 | "上周 DAU 为什么下降？" | 写 SQL、查数据、做归因、生成报告 |
| 自动化操作 | "帮我建个 Jira 工单" | 调 API、填参数、创建资源 |
| 多步骤工作流 | "帮我部署这个服务" | 构建、测试、发布、验证多步串联 |

**判断标准**：用户需求是"**帮我做一件事**"，需要系统执行动作、调用工具、观察结果。

### 3.3 适合 RAG + Agent 结合的场景

**特征**：任务执行过程中，Agent 需要查阅知识库来辅助决策或生成内容。

| 场景 | RAG 提供什么 | Agent 做什么 | 结合方式 |
|-|-|-|-|
| 智能客服（复杂问题） | 产品文档、FAQ、政策 | 理解意图→查知识库→判断转人工/退款 | Agent 主控，RAG 作知识弹药库 |
| 数据分析+业务知识 | 指标定义、口径、术语 | 写SQL→查数据→理解指标→生成报告 | Agent 按需检索知识库理解业务语义 |
| 代码 Review+规范 | 编码规范、安全规则 | 拉MR diff→对照规范→逐条检查 | Agent 执行Review，RAG提供规范依据 |
| 运维告警处理 | 故障手册、历史案例、SOP | 收到告警→查知识库→查指标→按SOP处理 | Agent分析执行，RAG提供处理经验 |
| 合规审查 | 合同模板、法规条文 | 读取待审文件→逐条对照法规→标注风险 | Agent编排审查，RAG提供法规依据 |
| BI 报告生成 | 指标体系、分析框架、模板 | 查数据→按框架组织→套模板生成 | Agent做分析编排，RAG提供方法论 |

## 四、一句话判断标准

| 问题 | 用什么 |
|-|-|
| 用户只是**问问题**，答案在文档里？ | **纯 RAG** |
| 用户要系统**做事情**，不需要查文档？ | **纯 Agent** |
| 用户要系统**做事情**，过程中需要**查阅知识**辅助决策？ | **RAG + Agent 结合** |

## 五、RAG 落地的 5 个避坑要点

<callout emoji="⚠️">
RAG 不是"切片→向量化→检索"三步就完事。失败的 RAG 项目通常是这条链路断裂：资料治理差 → 切片不合理 → 检索不准 → 生成无约束 → 缺乏评估。
</callout>

1. **先治理资料**：文档质量决定检索质量，过时、矛盾、格式混乱的文档切了也没用
2. **切片不是越细越好**：过细丢失上下文，过粗召回不精准，要按语义单元切
3. **检索不要只靠向量相似度**：混合检索（向量+BM25+重排序）效果远好于纯向量
4. **生成阶段要有约束**：Prompt 中明确"只基于提供的上下文回答，不确定就说不知道"
5. **一定要做评估**：建立评估数据集，定期测 Recall@K、Answer Accuracy，用数据驱动优化

## 六、对数据产品 AI 平台的启示

<callout emoji="🚀">
结合数据产品知识库 + RAG 规划及 AI 平台的 Agent 建设，最适合的架构就是 **RAG + Agent 结合**。
</callout>

\*\*RAG 知识库层\*\*（让 AI 懂业务）：灌入指标定义与口径说明、埋点文档与参数说明、看板使用手册、分析案例与预测场景、数据产品 SOP。  
  
\*\*Agent 执行层\*\*（帮 AI 做分析）：用户问"上周信息流 CTR 为什么下降"→ Agent 写 SQL 查数据 → 从 RAG 检索指标定义和历史案例 → 综合推理生成报告。  
  
\*\*核心原则\*\*：RAG 负责"让 AI 懂业务"，Agent 负责"帮 AI 做分析"，两者分工明确。

\## 七、RAG 知识库详解

<callout emoji="📚">
RAG（Retrieval-Augmented Generation，检索增强生成）是让大模型"查资料再回答"的核心技术。本章从原理到落地，系统介绍 RAG 知识库的全貌。
</callout>

## 七、RAG 知识库详解

### 7.1 什么是 RAG 知识库

RAG 的本质：**在大模型生成回答之前，先从外部知识库中检索相关内容，把检索结果注入到 Prompt 中，让模型基于真实资料生成回答。**

- **没有 RAG 的大模型**：像一个记忆力超强但爱面子的学霸，遇到不知道的问题不会说"我不知道"，而是当场编一个听起来很靠谱的答案——这就是"幻觉"。
- **有 RAG 的大模型**：像一个严谨的研究员，遇到问题先去查资料库，找到相关文献后再结合自己的理解给出有据可查的回答。

RAG 解决了大模型的三大核心痛点：

| 痛点 | 说明 | RAG 如何解决 |
|-|-|-|
| 幻觉问题 | 模型会编造不存在的信息 | 检索真实文档作为回答依据，有据可查 |
| 知识过时 | 训练数据有截止日期 | 外部知识库可随时更新，不受训练时间限制 |
| 私有数据缺失 | 模型无法访问企业内部数据 | 将企业文档灌入知识库，模型即可引用 |

### 7.2 实现原理：6 步闭环

<whiteboard token="AkbVwGlb8hxcXJbXMvUcSdainPf"></whiteboard>

| 步骤 | 名称 | 详细说明 |
|-|-|-|
| 1 | 数据预处理 | 将原始文档（PDF/Word/Wiki/网页等）解析为纯文本，按语义单元切片，生成文档片段 |
| 2 | 向量化嵌入 | 每个文档片段通过 Embedding 模型转为高维向量（语义指纹），存入向量数据库 |
| 3 | 查询优化 | 用户提问时，先对原始 Query 做优化：歧义消解、关键词补充、多 Query 扩展 |
| 4 | Top-K 召回 | 将 Query 向量与知识库向量做相似度计算，召回最相关的 Top-K 个文档片段 |
| 5 | 重排序和过滤 | 对初筛结果用 Cross-Encoder 精排，过滤无关内容、敏感信息和过期数据 |
| 6 | 生成回答 | 组装 Prompt（Query + 上下文 + 规则 + 格式），交给大模型生成回答，标注来源引用 |

### 7.3 三种检索范式

- **Naive RAG（基础版）**：Query → 向量检索 → Top-K → 生成。流程最简单，适合简单问答
- **Advanced RAG（进阶版）**：增加查询优化、混合检索（向量+BM25）、重排序、答案溯源。显著提升精度
- **Agentic RAG（智能版）**：让 Agent 自主规划检索策略：多轮检索、跨文档整合、工具调用、自我纠错。适合复杂多跳推理

**进阶优化技巧：**

| 技巧 | 说明 | 效果 |
|-|-|-|
| 混合检索 | 向量语义检索 + BM25 关键词检索组合 | 兼顾语义理解和精确匹配 |
| 重排序（Reranking） | 用 Cross-Encoder 模型对召回结果精排 | 检索精度提升 15%-30% |
| 查询改写 | 对用户原始问题做扩展、消歧、分解 | 召回率显著提升 |
| 分层索引 | 摘要层 + 细节层两级检索 | 先粗筛再精读，效率更高 |
| 父子块关联 | 检索命中小块时，自动扩展返回所属大块上下文 | 避免上下文碎片化 |

### 7.4 适用场景全景

| 场景类型 | 典型案例 | 知识库内容 | RAG 优势 |
|-|-|-|-|
| 企业内部知识问答 | "报销流程""年假政策" | 制度文档、员工手册、Wiki | 答案有出处，可追溯 |
| 智能客服 | "怎么退货""产品参数" | 产品文档、FAQ、售后政策 | 7x24小时，回答一致 |
| 合规/法务审查 | "条款合规吗""法规依据" | 法规条文、合同模板、审查标准 | 可追溯到具体法条 |
| 技术文档检索 | "API怎么调""报错怎么解决" | API文档、错误码手册 | 比全文搜索更智能 |
| 医疗/科研辅助 | "症状可能是什么病" | 医学文献、临床指南 | 减少幻觉，引用来源 |
| 数据分析辅助 | "指标口径是什么""分析框架" | 指标定义、分析案例、SOP | 让AI理解业务语义 |
| 代码/运维辅助 | "服务架构""故障怎么处理" | 架构文档、故障手册、SOP | 结合Agent做根因分析 |

## 八、Agent 设计：工具分工与选型指南

<callout emoji="🧩">
一个成熟的 AI Agent 系统，核心问题是"工具箱里该放什么工具"以及"什么时候用什么工具"。本章系统梳理 Skill、MCP、RAG、Function Calling 等工具类型的定位、分工与选型策略。
</callout>

### 8.1 Agent 工具体系全景

Agent 的工具体系可以分为四大类，每类解决不同层面的问题：

| 工具类型 | 一句话定位 | 核心价值 | 类比 |
|-|-|-|-|
| RAG（检索增强） | 让 Agent "查资料" | 注入外部知识，减少幻觉 | 图书馆 |
| MCP（工具连接） | 让 Agent "动手做事" | 标准化连接外部系统和 API | 万能插座 |
| Skill（技能封装） | 让 Agent "掌握套路" | 封装复杂工作流为可复用能力 | 操作手册 |
| Function Calling | 让 Agent "调函数" | 模型原生的结构化工具调用 | 螺丝刀 |

### 8.2 四种工具的详细对比

| 维度 | RAG | MCP | Skill | Function Calling |
|-|-|-|-|-|
| 本质 | 检索增强生成 | 工具连接协议 | 能力封装 | 结构化函数调用 |
| 输入 | 用户 Query | 工具描述 + 参数 | 触发条件 + 上下文 | 函数签名 + 参数 |
| 输出 | 相关文档片段 | 工具执行结果 | 结构化结果或报告 | 函数返回值 |
| 是否有状态 | 无状态，每次检索独立 | 无状态，每次调用独立 | 可有状态，支持多步骤 | 无状态，单次调用 |
| 复杂度 | 中（需要知识库维护） | 低（标准化协议） | 高（需要工作流设计） | 低（定义函数即可） |
| 典型延迟 | 100ms-2s | 100ms-10s | 1s-60s | 50ms-1s |
| 适合场景 | 知识问答、文档检索 | 系统操作、数据查询 | 复杂流程、多步编排 | 简单计算、格式转换 |

### 8.3 RAG 的定位与设计

**RAG 解决什么问题**：Agent 不可能把所有业务知识都塞进系统提示词，需要按需检索。

**RAG 在 Agent 中的三种接入模式：**

| 模式 | 说明 | 适用场景 | 示例 |
|-|-|-|-|
| 被动注入 | Agent 每次回答前自动检索相关知识 | 通用问答、客服 | 用户问"报销流程"，系统自动检索制度文档 |
| 主动检索 | Agent 判断需要时才调用 RAG 工具 | 复杂分析、多步骤任务 | Agent 分析数据时，主动检索指标定义和历史案例 |
| 多轮增强 | Agent 根据初步结果，多次检索不同知识 | 深度研究、多跳推理 | 先检索行业报告，再检索竞品数据，最后检索分析框架 |

**RAG 的优势：**

- 让 Agent 拥有实时更新的业务知识，不受训练数据截止日期限制
- 回答可追溯到具体文档来源，增强可信度
- 知识库与 Agent 解耦，可以独立更新和维护

**RAG 的局限：**

- 检索质量直接影响回答质量，"垃圾进垃圾出"
- 无法执行任何操作，只能提供信息
- 对知识库的维护成本较高，需要持续治理

### 8.4 MCP 的定位与设计

**MCP（Model Context Protocol）解决什么问题**：Agent 需要连接各种外部系统（数据库、API、文件系统、SaaS 工具），MCP 提供了标准化的连接协议，避免为每个工具写适配代码。

**MCP 在 Agent 中的三种使用模式：**

| 模式 | 说明 | 适用场景 | 示例 |
|-|-|-|-|
| 数据读取 | 通过 MCP 连接数据源查询信息 | 数据查询、监控 | 查询数据库获取指标数据、读取日志系统 |
| 操作执行 | 通过 MCP 调用外部系统执行操作 | 自动化、运维 | 创建 Jira 工单、发送飞书消息、部署服务 |
| 双向交互 | 读取+写入组合，完成复杂操作 | 多系统联动 | 读取告警信息 → 查知识库 → 创建工单 → 通知相关人 |

**MCP 的优势：**

- 标准化协议，一次适配，多处复用
- 工具描述自动注入 Agent，无需手动配置
- 支持动态发现和加载工具

**MCP 的局限：**

- 只解决"连接"问题，不解决"编排"问题
- 每个 MCP Server 需要独立开发和维护
- 错误处理和超时控制需要额外设计

### 8.5 Skill 的定位与设计

**Skill 解决什么问题**：某些任务需要一套固定的流程（多步骤、有判断逻辑、有输出格式），Skill 把这些流程封装为可复用的能力模块。

**Skill 的三种设计模式：**

| 模式 | 说明 | 适用场景 | 示例 |
|-|-|-|-|
| 流程型 Skill | 固定步骤的流水线 | 标准化分析、报告生成 | SQL 查询 → 数据分析 → 图表生成 → 报告输出 |
| 决策型 Skill | 有分支判断的流程 | 异常处理、根因分析 | 检查指标 → 判断是否异常 → 选择分析路径 → 输出结论 |
| 协作型 Skill | 需要多工具配合的复杂任务 | 端到端自动化 | 触发告警 → 检索知识库 → 查数据 → 生成报告 → 发送通知 |

**Skill 的优势：**

- 将复杂工作流封装为简单调用，降低 Agent 决策负担
- 可复用、可测试、可版本管理
- 输出格式可控，适合对接下游系统

**Skill 的局限：**

- 灵活性不如 Agent 自主规划，流程固化后难以应对新场景
- 设计和维护成本高，需要深入理解业务流程
- 过度封装可能导致 Agent 变成"脚本执行器"，丧失自主性

### 8.6 Function Calling 的定位与设计

**Function Calling 解决什么问题**：最基础的工具调用方式，让模型以结构化方式调用单个函数。

**Function Calling 的典型应用：**

| 应用类型 | 说明 | 示例 |
|-|-|-|
| 数据查询 | 单次数据获取 | 查询某个指标的最新值 |
| 格式转换 | 数据格式化处理 | JSON 转 CSV、日期格式转换 |
| 简单计算 | 数学或逻辑运算 | 计算环比增长率、求平均值 |
| 信息提取 | 从文本中提取结构化信息 | 从日志中提取错误码和时间戳 |

**Function Calling 的优势：**

- 模型原生支持，调用最直接
- 定义简单，一个 JSON Schema 就够
- 延迟最低，适合实时场景

**Function Calling 的局限：**

- 只能完成单步操作，无法编排复杂流程
- 没有标准化协议，不同模型的实现不兼容
- 不适合需要多步协作的场景

### 8.7 场景选型决策矩阵

| 场景 | RAG | MCP | Skill | Function Calling | 推荐组合 |
|-|-|-|-|-|-|
| 企业知识问答 | 必选 | 不需要 | 可选 | 不需要 | RAG 为主 |
| 数据查询分析 | 可选 | 必选 | 推荐 | 可选 | MCP + Skill |
| 智能客服 | 必选 | 可选 | 推荐 | 可选 | RAG + Skill |
| 运维告警处理 | 推荐 | 必选 | 推荐 | 可选 | RAG + MCP + Skill |
| 代码开发辅助 | 不需要 | 可选 | 不需要 | 推荐 | Function Calling 为主 |
| 自动化工作流 | 不需要 | 必选 | 推荐 | 可选 | MCP + Skill |
| 合规审查 | 必选 | 可选 | 推荐 | 不需要 | RAG + Skill |
| BI 报告生成 | 推荐 | 必选 | 必选 | 可选 | RAG + MCP + Skill |

### 8.8 设计原则与最佳实践

**原则一：按需选择，不要过度设计**

不是所有 Agent 都需要 RAG + MCP + Skill 全套武装。简单场景用 Function Calling 就够了，复杂场景再逐步引入更多工具。

**原则二：分层解耦，独立演进**

| 层级 | 职责 | 技术选型 |
|-|-|-|
| 知识层 | 提供业务知识和上下文 | RAG 知识库 |
| 连接层 | 对接外部系统和数据源 | MCP 协议 |
| 能力层 | 封装可复用的工作流 | Skill |
| 执行层 | 单步操作和工具调用 | Function Calling |
| 编排层 | 理解意图、规划路径、调度工具 | Agent Core |

**原则三：Agent 核心的职责边界**

Agent Core 应该专注于三件事：

1. 理解用户意图：准确解析用户到底想做什么
2. 规划执行路径：决定用哪些工具、按什么顺序
3. 整合输出结果：把多个工具的结果整合为用户可理解的回答

不要让 Agent Core 做以下事情：

- 不要让 Agent 自己写复杂的 SQL（交给 Skill）
- 不要让 Agent 自己维护知识（交给 RAG）
- 不要让 Agent 自己对接 API（交给 MCP）

**原则四：从简单开始，逐步增强**

推荐的建设路径：

Step 1: 先用 Function Calling 实现基础能力（数据查询、简单计算）  
Step 2: 引入 RAG 知识库，让 Agent 懂业务  
Step 3: 通过 MCP 连接更多外部系统  
Step 4: 将高频流程封装为 Skill  
Step 5: 用 Agent Core 编排所有工具，实现端到端自动化

<callout emoji="💡">
核心理念：Agent 是指挥官，RAG 是情报官，MCP 是通信兵，Skill 是特种部队，Function Calling 是随身武器。各司其职，才能打赢每一场仗。
</callout>

### 8.9 对数据产品 AI 平台的具体建议

结合数据产品的实际情况，建议的 Agent 工具架构：

| 工具 | 数据产品中的应用 | 优先级 |
|-|-|-|
| RAG 知识库 | 灌入指标定义、埋点文档、分析案例、产品 SOP | P0 - 最先建设 |
| MCP - 数据查询 | 连接数据仓库，支持自然语言转 SQL 查询 | P0 - 核心能力 |
| MCP - 通知推送 | 查询结果推送到飞书群、邮件 | P1 - 提升体验 |
| Skill - 数据分析 | 封装"查数据 → 做归因 → 生成报告"流程 | P1 - 高频场景 |
| Skill - 异常检测 | 封装"监控指标 → 发现异常 → 根因分析"流程 | P2 - 进阶能力 |
| Function Calling | 日期转换、单位换算、简单计算 | P0 - 基础能力 |
