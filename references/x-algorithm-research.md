# X (Twitter) 推荐算法 x-algorithm 核心思路提炼

> 来源：https://github.com/xai-org/x-algorithm
> 分析时间：2026-05-26
> 定位：X（Twitter）"为你推荐"信息流的完整推荐系统开源代码

---

## 一、一句话概括

X 的推荐系统用 **Grok 大模型替代了所有手工特征和启发式规则**，通过两阶段管线（检索→排序）为用户生成"为你推荐"信息流。

---

## 二、核心架构：两阶段管线

```
用户请求
  │
  ▼
┌─────────────────────────────────────────────────┐
│              HOME MIXER（编排层）                  │
│                                                   │
│  ① Query Hydration  → 加载用户上下文               │
│  ② Candidate Source → 两条线并行召回候选           │
│     ├── Thunder（关注账号的站内内容）               │
│     └── Phoenix Retrieval（全局内容的向量检索）     │
│  ③ Hydration        → 补充帖子元数据               │
│  ④ Filtering        → 去重/屏蔽/过期过滤           │
│  ⑤ Scoring          → Phoenix Transformer 打分    │
│  ⑥ Selection        → 按分数排序取 Top-K           │
│  ⑦ Post-Filter      → 最终可见性检查               │
└─────────────────────────────────────────────────┘
  │
  ▼
排序后的信息流
```

### 各组件职责

| 组件 | 位置 | 职责 |
|------|------|------|
| **Home Mixer** | home-mixer/ | 编排层，组装 For You 信息流，暴露 gRPC 接口 |
| **Thunder** | thunder/ | 站内帖子的内存实时索引，消费 Kafka 事件流 |
| **Phoenix** | phoenix/ | ML 组件，负责检索（双塔）+ 排序（Transformer） |
| **Candidate Pipeline** | candidate-pipeline/ | 可复用的推荐管线框架，定义 Source/Hydrator/Filter/Scorer/Selector 等 Trait |
| **Grox** | grox/ | 内容理解管线，负责垃圾检测、帖子分类、政策执行 |

## 三、两条候选来源

### 3.1 Thunder：站内内容（In-Network）

- **原理**：纯内存的实时帖子存储，消费 Kafka 的帖子创建/删除事件
- **存储结构**：按用户维护三个存储——原创帖子、回复/转发、视频帖子
- **查询性能**：亚毫秒级，不走外部数据库
- **数据生命周期**：自动清理超过保留期的旧帖子

### 3.2 Phoenix Retrieval：站外内容（Out-of-Network）

- **原理**：双塔模型（Two-Tower），用 ANN（近似最近邻）从全局语料中检索
- **User Tower**：将用户特征和行为历史编码为归一化嵌入向量 [B, D]
- **Candidate Tower**：将所有帖子编码为归一化嵌入向量 [N, D]
- **相似度计算**：点积相似度，取 Top-K

**关键洞察**：Thunder 保证站内内容的低延迟；Phoenix Retrieval 用向量相似度发现用户未关注但可能感兴趣的内容。

## 四、Phoenix 排序模型（核心）

### 4.1 整体架构

```
Phoenix Ranking Model
├── 输入层
│   ├── User Embedding [B, 1]           ← 用户哈希特征
│   ├── History Embeddings [B, S, D]    ← 用户历史行为序列
│   └── Candidate Embeddings [B, C, D]  ← 候选帖子嵌入
│
├── Transformer 层（来自 Grok-1 架构）
│   └── 关键设计：Candidate Isolation Attention Mask
│
└── 输出层
    └── [B, num_candidates, num_actions]
        ├── P(like)      P(reply)     P(repost)
        ├── P(click)     P(dwell)     P(video_view)
        └── ... 共 19 种行为预测
```

### 4.2 Candidate Isolation（候选隔离）—— 最关键的设计

```
Attention Mask：
           Keys →
           User  History(×S)  Candidates(×C)
Query ↓
User      ✓     ✓             ✗
History   ✓     ✓             ✗
Candidate ✓     ✓             对角线(仅自注意)
```

**设计意图**：

| 特性 | 说明 |
|------|------|
| **排序稳定性** | 每个候选的分数只取决于用户和历史，不取决于同批次的其他候选 |
| **并行计算** | 所有候选可以并行计算分数，提高推理效率 |
| **可复现** | 同一用户+同一候选集 = 同一结果，不受候选排序影响 |

**与传统方案的对比**：

| 方案 | 分数是否互相影响 | 能否并行 | 稳定性 |
|------|-----------------|---------|--------|
| 传统 Listwise 排序 | 是 | 低 | 受候选集影响 |
| Pointwise 排序 | 否 | 高 | 稳定但缺少上下文 |
| Phoenix（候选隔离） | 否（候选之间） | 高 | 稳定且有用户上下文 |

### 4.3 多行为预测

模型不是预测一个单一的"相关性分数"，而是**同时预测 19 种用户行为的概率**：

```
最终分数 = Σ (weight_i × P(action_i))

示例：
Score = 0.3×P(like) + 0.2×P(reply) + 0.15×P(repost)
      + 0.2×P(click) + 0.1×P(dwell) + 0.05×P(video_view)
```

**优势**：
- **可调**：通过调整权重改变推荐策略（鼓励互动 → 提高 reply 权重）
- **可解释**：可看到每个帖子在不同行为维度上的预测分数
- **多目标**：同时优化多种用户参与度指标

### 4.4 Mini 模型配置（开源版）

| 参数 | 值 |
|------|-----|
| 嵌入维度 | 128 |
| Transformer 层数 | 4 |
| 注意力头数 | 4 |
| Key Size | 32 |
| Widening Factor | 2 |
| 历史序列长度 | 127 |
| 候选序列长度 | 64 |
| User/Item/Author 词汇表 | 各 1,000,000 |
| 每个实体的哈希数 | 2 |

> 生产环境使用更大的模型（更多层、更宽的嵌入），但架构相同。

## 五、特征设计：Hash-Based Embeddings

```
用户/帖子/作者 → 多个哈希函数 → 多个嵌入表查找 → 拼接/平均 → 最终嵌入
```

| 实体 | 词汇表大小 | 哈希数 |
|------|-----------|--------|
| User | 1,000,000 | 2 |
| Item | 1,000,000 | 2 |
| Author | 1,000,000 | 2 |

**为什么用哈希而不是直接查表？**
- **节省内存**：不需要为每个可能的 ID 预留位置
- **处理新实体**：未见过的 ID 也能通过哈希得到嵌入
- **冲突缓解**：多哈希函数提供一定的冲突缓解能力

## 六、最激进的设计决策：消除所有手工特征

README 原文：

> "We have eliminated every single hand-engineered feature and most heuristics from the system. The Grok-based transformer does all the heavy lifting by understanding your engagement history and using that to determine what content is relevant to you."

这意味着：
- **没有** "帖子热度分"、"作者权威度"、"时间衰减因子" 等传统推荐特征
- **没有** "如果用户是新注册就推热门内容" 等启发式规则
- **所有判断**都交给 Grok Transformer 从用户行为序列中自动学习

**这是与传统推荐系统最大的区别**——把特征工程的活儿交给了模型本身。

## 七、内容理解：Grox 管线

新增的 Grox 模块负责帖子级别的内容理解：

| 能力 | 说明 |
|------|------|
| 垃圾检测 | 识别 spam 内容 |
| 帖子分类 | 自动分类帖子主题（体育、科技、政治等） |
| 政策执行 | 检测违反平台政策的内容（PTOS） |
| 嵌入生成 | 为帖子生成语义嵌入 |

这些分类结果作为特征输入到 Phoenix 排序模型中。

## 十一、对浏览器信息流推荐的启发

| 启发 | x-algorithm 的做法 | 浏览器信息流的借鉴 |
|------|-------------------|-------------------|
| **两阶段管线是标配** | 检索（双塔 ANN）+ 排序（Transformer） | 信息流同样需要先粗筛再精排 |
| **候选隔离注意力** | 候选之间不互相影响 | 保证推荐结果稳定，避免"刷到一条好内容后全变了" |
| **多行为预测** | 19 种行为概率加权 | 阅读时长、点击、分享、收藏等分别预测，灵活调权 |
| **消除手工特征** | 模型自动从行为序列学习 | 减少特征工程维护成本，但需要更多训练数据 |
| **内容理解管线** | Grox 做分类/检测/嵌入 | 浏览器需要对网页内容做分类、质量评估、敏感内容检测 |
| **实时事件流** | Kafka 驱动的帖子索引 | 新闻/内容的实时性要求同样高 |
| **Hash-Based Embeddings** | 多哈希函数查表 | 节省内存，处理新实体，适合海量用户和内容 |
| **纯内存存储** | Thunder 不走数据库 | 高频访问的推荐结果需要缓存层 |
