# ICE Workbench v6 阶段性设计思考与进度总结

本文档旨在对 v6 版本的初期构思、核心功能方向以及 UI/UX 架构重构的阶段性成果进行归档，以便后续团队能在此基础上继续讨论和迭代。

## 一、 核心定位演进
从 v5 到 v6，ICE Workbench 的产品形态正在经历一次重要跨越：
**从「带插件的 AI 对话窗口」进化为「基于多智能体协同的自动化数据工作舱 (Data Workspace)」。**

过去，用户心智停留在“我问 AI 答”；v6 开始，用户心智应转变为“我配置数据源，我下发目标，我监控执行流，我在关键节点做决策”。

## 二、 确定的四大核心能力方向

1. **多智能体协同 (Multi-Agent Workflows)**：打破 Agent 执行黑盒，将主 Agent (Planner) 拆解任务、调度子 Agent (Data Analyst, Report Agent 等) 的过程显性化。
2. **深度 Human-in-the-loop (HITL)**：复杂数据流不可完全脱离人工。支持任务“挂起 (Paused)”，通过富交互卡片让用户审批高危操作或直接在 UI 上修正中间态脏数据。
3. **外部数据源“活挂载”**：深度整合飞书生态。支持将多维表格 (Bitable)、文档目录作为**活数据/知识库**直接绑定在任务侧边栏，Agent 随取随用。
4. **长耗时与异步任务接管**：解绑 WebSocket。后台离线执行跑批任务，辅以完善的全局通知 (Notification) 和重连回捞机制。

## 三、 界面交互范式重构：全新「三栏式」Workspace

为了承载上述复杂能力，我们将传统的双栏对话界面重构为高信息密度的**渐进式三栏布局**：

*   **左栏 (Context & Data)**：任务上下文停靠区。展示绑定的数据文件、活飞书链接、RAG 知识源。并在底部提供主控 Agent 的配置入口。
*   **中栏 (Chat & Action)**：核心交互与事件流。
    *   保留对话气泡，但引入**系统级事件轴**（Task Started / Finished）。
    *   引入 **HITL 审批卡片**（如原型中的“异常数据检测表单”，支持用户 Input 输入补充原因并点击“确认修正并继续”）。
*   **右栏 (Execution Plan)**：动态执行时间线。
    *   以树状节点展示任务当前的流转状态（Completed, Paused, Pending）。
    *   实时标明是哪个 Sub-agent 正在执行，或卡在哪一步。

## 四、 顶部导航与全局入口优化 (结合定时功能)

在 `workspace_v6_v2.html` 原型中，我们确立了以下导航栏操作规范：

1. **最高优的全局操作**：在 TopNav 右侧常驻 `+ New Task`，满足用户随时开启新数据分析链路的诉求。
2. **基于上下文的任务内操作 (Dropdown)**：
   * 采用点击任务标题展开下拉菜单的交互模式。
   * **定时能力融合**：将 `配置定时调度 (Cron)` 融入任务菜单。理念是：**任何一次手动跑通的 Task，都可以一键转化为 Cron 定时任务**。
   * 菜单内还包含重命名、导出结果、删除等次级操作，保持面板清爽。

## 六、 下一步讨论与工作方向建议

在下一次会议/开发中，我们可以从以下几个节点继续：

1. **Dashboard 与 定时大盘的 v6 设计**：当前只是在单个 Task 页面加了定时的“入口”，如何在中控台 (Dashboard) 和 Scheduled 页面展示这些后台跑批的任务卡片？
2. **富交互卡片 (Rich Cards) 协议定义**：前端如何通过 WebSocket 接收并渲染这些带有表格、Input、Button 的审批卡片？(需定义一套内部 JSON Schema)。
3. **后端 Agent 调度层 (Runtime) 适配**：后端如何真正把执行树 (Execution Plan) 吐给前端，并支持任务状态持久化（Pause & Resume）。
