Section 01

主循环:一圈一圈
把活干完

调模型 → 冒出工具调用 → 执行 → 结果回填 → 再来一圈

一句话:Agent = 一个带工具的 while 循环

模型自己不能读文件、跑命令。它只能"说"出「我要调 Read」,由运行时真正执行,再把结果塞回去,让它接着想下一步。
🧠
模型只负责"想"

输出文字,或输出一个 tool_use 块表示"我要用某工具"。

🔧
运行时负责"做"

执行工具,把结果包成一条 user 消息,回填进对话。

🔁
循环负责"续"

只要这轮还有 tool_use,就再调一次模型;没有了就收工。

Section 02

流式执行:别等模型
说完才动手

把只读工具"藏"进模型流式输出的那几秒里

一边听模型说,一边把工具先跑起来

模型流式吐字要好几秒。与其干等它说完再批量跑工具,不如边流边跑 —— 哪个 tool_use 块先说全,就先执行哪个。
🌊
StreamingToolExecutor

addTool() 边流边排队执行,getCompletedResults() 边产出结果。

🔀
并发安全闸

isConcurrencySafe:只读工具(Read/Grep)可并行;写工具默认串行,避免踩踏。

✋
中断安全

中途取消,给排队/执行中的工具补齐合成 tool_result,保证块配对。

Section 03

工具与权限:
能力要给,闸门也要有

60+ 工具 · 三态判定 · 五种模式 · 让模型给自己当安检

每个工具跑之前,都要过一道权限关

canUseTool 在执行前判定三态:allow 直接放行 · deny 拒绝 · ask 弹给用户确认。每个决策都带 decisionReason,可解释可审计。
📜
Tool 契约 fail-closed

buildTool 默认"会写、不可并行" —— 除非工具明确声明安全。

🎚️
五种权限模式

default / plan / acceptEdits / bypassPermissions / auto。

🤖
auto = 模型当安检

Bash 命令交给 haiku 分类器判"该不该拦"。

子 Agent:一个能
生出很多个的 Agent

派生 · Fork 自我分身 · Coordinator 编排 · depth 追踪防套娃

主 Agent 能派生子 Agent,甚至 fork 自己

AgentTool 派生一个带独立上下文的子 Agent(depth+1)去干活,跑完只把一条总结交回来 —— 中间的工具噪音不污染主上下文。
🌱
派生 worker

结果以 <task-notification> 回流;可并行 fan-out。

🍴
Fork 自我分身

省略 subagent_type:继承上下文 + 共享 prompt 缓存(前缀逐字节一致)。

🛡️
depth 追踪 + 防递归

depth+1 可观测;isInForkChild 拦"fork 里再 fork"。

Section 05

上下文治理:
越聊越长怎么办

多级压缩 · compact_boundary · 缓存失效 · 续命不是回春

桌子快满了,系统分级收拾

长会话的上下文会不断膨胀。Claude Code 在每圈循环开头叠一条压缩流水线,从"轻剪枝"到"全量摘要"层层递进 —— 尽量保住细粒度,实在不行才总结。
✂️
snip / microcompact

剪僵尸消息、按 tool_use_id 删旧调用,细粒度腾空间。

📝
autocompact

到阈值把历史摘成一段,打出 compact_boundary 边界。

🚑
reactive compact

只有真的 413 撑爆了才兜底触发,摘要后重试一次。

⚠️ 压缩 = 重写历史 = 开头变了 → 前缀缓存失效;摘要还会丢细节 —— 续命,不是回春。

一页速查 五个机制,一句话记住

① 主循环
needsFollowUp 唯一决定:有工具就续圈,没有就收工。
② 流式执行
只读工具藏进模型输出里并行跑,写工具串行 —— 省时间。
③ 权限
三态 allow/deny/ask × 五模式,默认 fail-closed。
④ 子 Agent
派生 depth+1 回流 notification;Fork 共享前缀缓存。
⑤ 上下文压缩
多级管线;autocompact 一压 → 缓存失效、细节回不去。
⚠️ 一条底线
有界一切:maxTurns / 预算 / token 上限 / depth,没有无界自动行为。
🏁 一个循环转起来 · 工具边流边跑 · 权限先过闸 · 活多就派 Agent · 满了就压缩
