Agent 框架设计解读

一份面向工程读者的运行时解读：这套框架不是"一个大模型 + 几个工具"，而是一套多参与者、受治理、文件优先的 Agent 协作运行时。核心是同一个"单跳"执行内核，靠不同的外层循环叠加能力，靠治理闸门横切安全。

User→Twin（唯一 leader）→Agents
1单跳内核 _run_single_hop
3 / 6A2A 软预算 / 硬顶跳数
3 / 20自驱：每回合 / 每日上限
6Twin 仲裁拒绝闸
本文所有结论均来自 backend/app/services/ 的真实实现（agent_runtime · orchestrator_svc · context_assembler_svc · plan_svc · proactivity_svc · proposal_svc · tools/registry）。注意：agents/_shared/context-protocol.md 描述的是迁移来源 ice-workbench 的运行时，与本项目实际实现不是同一套，勿混淆。
01

设计全景：User → Twin → Agents

协作的顶层模型是三层：你只跟自己的数字分身 Twin 对话；Twin 作为唯一 leader 去编排专业 Agent 干活；所有高风险动作回流给你确认（HITL）。Agent 之间不做对等自由对话——一切经 Twin 仲裁。

用户发言→ Twin 接管/编排→ 专业 Agent 执行→ （可）Agent 间接力→ 高风险 → 回你确认
02

六层分层架构

框架自上而下分为六层，安全治理作为横切面贯穿每一层。理解这套分层，是理解后面 A2A 与自驱的前提。

层落点职责
定义层agents/{id}/agent.json + agent.md声明式契约：工具白名单 / 派单目标 / 权限模式 / 模型 / 推理强度
运行时agent_runtime.py有界 ReAct + speaker 路由 + 事件流；单跳是最小执行单元
编排层orchestrator_svc.py有界多跳 A2A 的纯函数状态机：防环 + 预算 + Twin 仲裁
上下文层context_assembler_svc.py分层注入 + 访问控制内嵌在读取点（先过滤后检索）
工具层tools/registry.py统一执行入口 + fail-fast 治理门链 + 统一错误信封
治理层core/permissions · hitl · proactivity · control三轴权限 / HITL / 主动性护栏 / 一键暂停 / 全程审计（横切）
03

Agent 声明式契约

每个 Agent 是一个目录，用 agent.json 声明能力边界，prompt/ 合成 agent.md 作系统提示词来源。关键字段本身就是"契约"：

能力与边界字段

  • tools — 工具白名单（这个 Agent 能看到哪些工具）
  • spawn_targets — 可派单子 Agent 白名单（["*"] 全部 / [] 不可派）
  • permission_mode — default / read_only / confirm_write / confirm_network
  • features — 细粒度开关（todo_write / exit_plan_mode / spawn_subagent）
  • model / effort — 模型分级与推理强度提示

一个关键设计哲学 D-50

"先兑现契约，再扩能力"。这 11 个 Agent 从 ice-workbench 迁入，富字段一开始只是"无害透传"；随后在 M19 里把 tools / permission_mode / effort / model 在运行时真正强制生效——避免"声明了却不执行"的安全错觉。

04

运行时内核：单跳 _run_single_hop

整个框架只有一个执行内核。A2A 的每一次接力、autostep 的每一步推进，走的都是它。理解这个内核，就抓住了两条增强路径的共同基础。

# _run_single_hop：一个 speaker 的一跳
1. 月度预算硬门  llm_available() and is_over_budget() → yield budget_exceeded，不发付费调用
2. yield turn_start（若来自转交，带 handoff）
3. assemble_context(task, speaker, handoff)   # 分层组装 + 访问控制内嵌
4. select_model(speaker)                       # 按角色/预算选实际模型（M19-T02）
5. for evt in stream_chat(...):  yield text / 记 error / done 记账
6. raw = 原始输出（含指令）;  content = strip_directives(raw)  # raw/content 分离
7. append_turn(content, handoff)  +  audit  +  post_tool notify hook
8. return {raw, content, errored}                # raw 交给编排器解析转交
raw / content 分离是贯穿 A2A 的关键：raw（含 [[handoff...]] 指令）交给编排器解析；content（已清洗）用于落盘和展示。用户永远看不到指令标记，但编排器能读到。
05

A2A 完整多跳循环

M7 把协作从"单跳"升级为"多个 Agent 一轮内接力"。入口和出口都在 run_agent_turn 的一个 while True 里；编排的"脑子"抽到 orchestrator_svc 的纯函数状态机，运行时只负责驱动。

落盘用户发言 · 任务置 doing
初始 speaker = resolve_speaker()：@定向 > assignee > Twin > Agent > 兜底 general
state = new_hop_state(speaker)
初始 speaker 计入序列，但不算一跳
进入 while → _run_single_hop(speaker, handoff)
执行一跳，产出 turn_start / text / turn_done 事件，返回 {raw, content, errored}
若 A2A 开关关 or 本跳出错 → break
开关默认关时退化为单跳，行为与 v1 完全一致
req = parse_directive(raw)
解析输出末尾的转交指令；无指令 → break
decision = arbitrate_handoff(...) ← Twin 仲裁
刷新参与者后交给 Twin 把关（防环/预算/目标校验）
yield handoff 事件（honored=?）
无论批不批都发事件，前端可渲染"A→B 转交卡"；不 honor → break
honor：auto_join → add_participant(role=tool, L2)
把"平台已知但未参与"的 Agent 拉进来，不放宽其权限
record_hop · speaker = target
hops+1、记转交对/序列、切换发言人
↺ 回到 while 顶部，target 成为新 speaker

循环结束后，才进入独立的 回合后自驱 _autostep。

06

转交指令协议

这是个务实取舍：LLM 网关不产原生 function-calling 的 tool_use，所以转交靠解析 speaker 输出末尾的文本标记实现。

[[handoff:<agent_id>|<理由>]]   # 请求接力（交出控制权）
[[ask:<agent_id>|<问题>]]        # 一次性咨询
  • parse_directive 取最后一条——Agent 可能先推理再给最终决定。
  • strip_directives 把标记从展示文本里清掉。
  • 另有 handoff_to_agent / ask_agent 两个工具作等价入口。
07

HopState：防环 + 预算的载体

编排状态机是纯函数 + dataclass，刻意不 import agent_runtime，既避免循环依赖，也让防环/预算逻辑能脱离运行环境独立单测。

@dataclass
class HopState:
    hops: int = 0
    max_hops: int = 3        # IDW_A2A_MAX_HOPS，软预算
    hard_cap: int = 6        # IDW_A2A_HARD_CAP，硬顶
    speaker_seq:   list[(type,id)]   # 出现过的发言序列
    handoff_pairs: list[(from,to)]   # 已发生的转交对

    effective_max = max(1, min(max_hops, hard_cap))   # 软硬取小，至少 1
    budget_exhausted = hops >= effective_max
  • 初始 speaker 计入 speaker_seq 但不算一跳 → 默认 max_hops=3 允许 3 次转交、共 4 个 speaker 发言。
  • hard_cap 是"即使把 max_hops 配到 99 也顶死在 6"的保险丝。
  • handoff_pairs 是防环记忆：记 (from,to) 查重复，反查 (to,from) 查乒乓。
08

Twin 仲裁六道闸 D-15

Twin 是唯一 leader，不盲从 Agent 的转交请求。arbitrate_handoff 任何一条命中即拒绝。每条都有独立单测（纯函数，脱离环境可验证）。

拒绝条件判据防的是
目标为空not to指令残缺
自转交to == cur_idAgent 转给自己空转
目标不存在不在参与者且不在平台已知转给幽灵 Agent
预算耗尽budget_exhausted()无限接力烧钱
A↔B 乒乓(to, cur) in handoff_pairs两个 Agent 互踢皮球
重复转交(cur, to) in handoff_pairs同一条边走两遍

通过后返回 honor=True + target，并算出 auto_join = 目标不在当前参与者里。集成测试固化了闭环：test_pingpong_blocked（A→B 成功、B→A 被拦）、test_budget_caps_hops（max_hops=1 时第二次转交被预算拦）。

A2A 接手上下文

切换 speaker 后，incoming_handoff={from,to,reason} 传进下一跳，触发 context_assembler 注入一个专门的 a2a 层："谁因何转交 + 前序产出摘要"（last_content_by 取来）。

# 协作接力（A2A）
- 你是接手方：data-analysis 因「写报告」把任务转交给你。
- 请基于前序产出继续推进，不要重复已完成的工作。

## data-analysis 的前序产出（摘要）
DAU 环比下降 8%，主因新增渠道质量下滑。   # ← last_content_by()

① 转交上下文永不丢

_budget_clamp 显式跳过 task 与 a2a 层——哪怕 char_budget=50 也保留。转交上下文丢了，接手方就抓瞎。

② 访问控制不放宽

接手 Agent 拿到"前序产出摘要"，但读记忆仍按 by-user/{自己} 走，不会因为接力就能读发起方的私有经验。

回合后自驱：触发与主循环

_autostep 在主循环（含 A2A 多跳）跑完之后才被调用，与 A2A 正交独立。要同时满足三道前置门：

IDW_AUTOSTEP_ENABLED 开& 本回合未出错& 非 Plan（只读）模式 
_autostep:
  plan = get_plan(task_id)
  if 无 plan 或无 steps: return          # ← 自驱严格绑定"计划"，没计划不自驱

  steps_run = 0
  while True:
    nxt = next_ready_step()              # 第一个 todo 且依赖全 done 的步
    if nxt is None: break                # 计划完成 or 被依赖阻塞 → 停

    gate = proactivity_svc.check()       # 护栏三道闸
    if (not gate.allowed) or steps_run >= AUTOSTEP_PER_TURN(=3):
        prop = create_proposal(KIND_NEXT_STEP, payload={step_id, owner_agent})
        yield {"type":"proposal", "proposal":prop, "speaker": Twin}   # ← 拦成提议
        break                            # 不越权，交还用户裁决

    proactivity_svc.record(kind="autostep")   # 计入每日预算 + 审计
    start_next()                              # nxt: todo → doing
    speaker = _step_speaker(task, nxt)        # 步骤的 owner_agent

    hop = _run_single_hop(speaker, handoff=None)   # ← 复用同一单跳
    if hop.errored:  update_step(error); break
    update_step(done); steps_run += 1
13

如何复用单跳（核心）

自驱的每一步和用户驱动的一跳走的是完全同一个执行单元 _run_single_hop：同样的上下文组装、流式 LLM、落盘、审计、用量记账、月度预算硬门、pre/post hook 全部继承。差异只有三处：

 用户回合A2A 接力跳autostep 步
speaker 来源resolve_speaker（默认 Twin）转交 target_step_speaker（步骤 owner_agent）
handoffNone{from,to,reason}None
外层循环—HopState 多跳plan 步骤推进
一个重要边界：_autostep 直接调 _run_single_hop（不是 run_agent_turn）。所以自驱推进的每一步是单跳的——步骤内部不会再触发 A2A 多 Agent 接力。横向接力与纵向推进不会互相嵌套失控。

计划是自驱的唯一真相源 D-17。没有 plan.steps 直接返回——自驱不是"随便再多说几句"，而是严格"推进下一就绪步"。next_ready_step 返回 None 的两种情况（全做完 / 被 depends_on 阻塞）都直接停，不强推被阻塞的步骤；且每轮重新求就绪步，不预取，保证依赖与状态变化被重新评估。

14

护栏与两级预算 D-16

proactivity_svc.check 在每一步之前问一次，顺序判定三道闸：

① 一键暂停 control_svc.is_paused()
全局治理开关，与运行时开头那道暂停门是同一个
② 静默时段 IDW_QUIET_HOURS="22-8"
防打扰，支持跨午夜；深夜不主动打扰用户
③ 每日预算 IDW_AUTOSTEP_DAILY_CAP=20
按任务按 UTC 日累计，超限拒绝
两级预算，务必分清：
  • AUTOSTEP_PER_TURN = 3 — 硬编码常量，单个用户回合内最多自驱 3 步。防"一次对话把整个计划一口气跑完"。
  • IDW_AUTOSTEP_DAILY_CAP = 20 — 单任务按 UTC 日累积的持久计数（读 proactivity.jsonl 日期前缀）。防"长期跨回合烧钱/刷屏"。
两者叠加。record() 每次自驱都 append + 审计，所以主动行为全程可追溯、可一键停。
15

拦成"提议"：不越权的降级路径

护栏不通过（或本回合已推 3 步）时，_autostep 不静默停止，而是降级成一条待确认提议。这就是 agent.md「不可越权边界」的落地——主动性到"请求确认"为止，绝不替用户做最终决策。

自驱受阻→ create_proposal(next_step, step_id)→ Twin Dock 待确认→ 用户 accept→ 上层据 step_id 重新驱动
  • 落 proposals.jsonl，status=pending，payload 带 step_id——记录"卡在哪一步、由谁负责"。
  • 只提议"下一就绪步"这一条，不为剩余所有步批量生成提议（后续步可能依赖这一步）。
  • proposal_svc 不直接执行；decide(accepted) 也只记录裁决，真实动作由调用方在 accept 后执行——队列与执行解耦。
  • list_pending_for_user 把用户所有任务的 pending 提议聚合到 Workbench / Twin Dock，形成统一"待你确认"入口。
16

A2A vs autostep：一对镜像

两条增强路径共享同一个单跳内核，各自用不同的外层循环 + 不同的治理闸包起来。这是这套框架最一致的设计手法。

 A2A 多跳autostep 自驱
时机用户回合内用户回合后
方向横向：Agent → Agent 接力纵向：推进 plan 步骤
真相源转交指令 + HopStateplan.json 步骤
约束防环 + 多跳预算（Twin 仲裁）三道护栏 + 两级预算
受阻时break（可能带未 honor 的 handoff 事件）降级成 proposal，请求确认
复用_run_single_hop_run_single_hop
开关IDW_A2A_ENABLEDIDW_AUTOSTEP_ENABLED
一句话总结这套手法：内核稳定单一，能力靠外层循环叠加，安全靠闸门横切。两条路径都默认关、都以"不越权 / 可暂停 / 可审计"收口。
17

贯穿的设计原则

① 文件优先（G3）

文件系统是唯一真相源；SQLite / pgvector 只是可重建派生索引。plan.json / conversation.jsonl / proposals.jsonl 都走 G3。

② 访问控制内嵌读取点

不是事后过滤，而是"先过滤（访问边界）再排序（top-k 检索）"，越权数据根本不进检索池。

③ 有界一切

多跳有预算、自驱有上限、上下文有 budget、成本有月度硬顶。没有无界的自动行为。

④ 安全先于智能

能力默认关灰度、高风险恒 HITL、先兑现契约再扩能力。关时零行为变化。

⑤ 可解释 / 可审计

上下文每层带来源标注（_layer / _source）；每跳、每次主动动作都落审计与 jsonl，事后可完整追溯。

18

待确认的实现细节

以下是读代码时发现、值得团队确认"是有意简化还是待补 gap"的点，非缺陷断言。

① ask 目前与 handoff 同路径。文档说 ask 是"咨询后控制权回到发起方"，但主循环拿到 req 后不区分 action，ask 也会 record_hop + speaker=target。即"问完返回发起方"的语义未在编排循环里落地。若需真正的咨询-返回，需补一个分支：ask 被 honor 时执行一跳拿回答，但不切换主 speaker。
② 自驱步失败会连带把任务置 error。_run_single_hop 内部 errored 就会 set_status(error)，自驱步失败与用户回合失败被同等对待——是否符合预期值得斟酌。
③ 提议的 speaker 是 Twin（即便被卡步骤 owner 是具体 Agent）。语义上是"Twin 替你把关、代为请示"，与"Twin 唯一 leader"一致。
④ 静默时段走进程级 env（os.environ.get("IDW_QUIET_HOURS")），不走 get_settings() 缓存——改环境变量即时生效，但未纳入 settings 统一管理，测试时需注意。
