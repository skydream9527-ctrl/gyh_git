Claude Code · Agent 机制解读

一份面向工程读者的运行时解读：Claude Code 不是"一个大模型 + 一堆工具"，而是一个流式、可中断、受权限治理、可无限自我派生的 Agent 运行时。核心是一个 async generator 主循环，围绕它叠出工具执行、子 Agent 编排、上下文压缩三套能力。

User / SDK→QueryEngine（会话）→queryLoop（回合）→Tools / Sub-agents
1async generator 主循环
60+内置工具（可延迟加载）
5权限模式 · 三态判定
5+上下文压缩层级
本文所有结论均来自 参考内容/cc-recovered-main/src/ 的真实实现（重点文件：QueryEngine.ts · query.ts · Tool.ts · tools.ts · tools/AgentTool/* · coordinator/coordinatorMode.ts · types/permissions.ts · Task.ts）。该源码是从 Claude Code 2.1.88 的 sourcemap 逆向恢复并重整为可构建的 npm 工程，含少量兼容 shim/stub，行为不保证与官方 bundle 完全一致，作机制研究足够可靠。
01

这套源码是什么

Claude Code 是一个跑在终端里的编码 Agent。它的运行时是纯 TypeScript，UI 用 ink（终端里的 React），模型侧通过 @anthropic-ai/sdk 与 claude-agent-sdk 走流式接口。理解它的价值在于：它是目前工业级 Agent Loop 里少见的、把"模型-工具-子Agent-上下文治理"四件事在一份代码里同时做扎实的实现。

入口形态

交互式 REPL（终端 TUI）、无头 -p / SDK 单发、远程 bridge（手机 / 桌面 / cowork）。三种入口共享同一套 QueryEngine 与 query() 内核。

技术栈

React 19 + ink 渲染，zod 定义工具入参，esbuild 打包。全流程基于 async generator，天然可流式、可中断。

阅读地图

看懂三个文件就抓住主干：QueryEngine.ts（会话）、query.ts（回合循环）、Tool.ts（工具契约）。其余是围绕它们的治理与编排。

02

全景：六层运行时

自上而下六层，一次用户提问由上层驱动到下层执行；权限、Hooks、预算作为横切治理贯穿始终。理解这套分层，是后面主循环与子 Agent 的前提。

层落点职责
入口层entrypoints / bridge / cliREPL / SDK / 远程；把用户输入变成一次 submitMessage
会话层QueryEngine.ts一个会话一个实例，持有 messages / usage / 文件缓存；管理"回合"边界与结果封装
循环层query.ts · queryLoop真正的 agentic loop：调模型 → 解析 tool_use → 执行工具 → 回填 → 再调，async generator 流式产出
工具层Tool.ts · tools.ts · tools/*60+ 工具的统一契约、装配、并发执行、结果落盘
子 Agent 层tools/AgentTool/* · coordinator/派生专业子 Agent、Fork 自我分身、Coordinator 编排 workers
治理层（横切）permissions · hooks · 预算 / compaction权限四态、Pre/Post/Stop Hooks、maxTurns / 成本 / token 上限、多级压缩
一句话抓住主干：上面五层都在给最下面那个 while(true) 循环"喂料"和"设闸"——喂的是消息与工具，设的是权限与预算。
03

会话层：QueryEngine

QueryEngine 是"一个会话一个实例"。它不做 agentic 推理，只负责回合的进出：装配系统提示、处理用户输入（含斜杠命令）、驱动 query() 循环、把流式事件翻译成 SDK 消息、最后封装 result。跨回合的状态（消息、用量、文件缓存）都挂在它身上。

// QueryEngine.submitMessage(prompt)：一个用户回合的骨架
1. fetchSystemPromptParts(tools, model, mcp)   # 系统提示 + userContext + systemContext
   systemPrompt = [customSystemPrompt ?? default, memoryPrompt?, appendSystemPrompt?]
2. processUserInput(prompt)                    # 解析斜杠命令 / 附件；决定 shouldQuery
3. recordTranscript(messages)                     # 用户消息先落盘 → 崩溃也可 --resume
4. yield buildSystemInitMessage(...)           # 首条 system.init：模型 / 工具 / 权限模式
5. for await (msg of query({messages, systemPrompt, canUseTool, ...})) {
     记录 assistant / user / compact_boundary；累计 usage；yield* normalizeMessage(msg)
     检查 maxBudgetUsd / maxTurns / 结构化输出重试上限 → 命中即 yield result 并 return
   }
6. yield {type:"result", subtype:"success", result: textResult, usage, cost, ...}

回合 vs 会话

每次 submitMessage() 是一个回合；同一 QueryEngine 跨多个回合复用 mutableMessages。ask() 是它的一次性包装，供无头 / SDK 单发场景。

可中断

interrupt() 直接 abortController.abort()。因为全链路是 generator，中断能干净地传播到模型流与工具执行，并补齐缺失的 tool_result 块。

04

主循环：queryLoop 的一次回合

这是整个 Agent 的心脏。query.ts 里的 queryLoop 是一个 async function*，用一个 while(true) 承载经典的 ReAct 循环。每一圈：先治理上下文，再流式调模型，边流边跑工具，把结果作为 user 消息回填，然后决定"再来一圈"还是"收工"。

① 上下文治理 applyToolResultBudget → snip → microcompact → autocompact → collapse
先把历史修剪 / 压缩到安全区间；工具结果超预算的先做内容替换（见第 15 节）
② token 硬门 isAtBlockingLimit
自动压缩关闭时，预留空间让用户还能手动 /compact
③ 组装 fullSystemPrompt = systemPrompt + systemContext · 选模型
getRuntimeMainLoopModel：plan 模式且历史超 200k 时切更大上下文模型
④ 流式调模型 queryModelWithStreaming
逐块产出 assistant 消息；yield 给上层实时渲染
⑤ 检测 tool_use 块 → needsFollowUp = true
流里只要出现工具调用块，就登记进 toolUseBlocks，标记"本轮还得续"
⑥ 执行工具（流式或批量）→ 结果 normalizeMessagesForAPI 成 user 消息
toolResults 累积；期间还会 drain 排队命令、注入附件（技能 / 记忆）
⑦ 若 !needsFollowUp → 收工
没有工具调用了 → 跑 Stop Hooks，若无异议则 return Terminal
⑧ 否则把 assistant + toolResults 并入 messages
↺ state = {messages: [...]}; continue —— 回到 ① 再调模型
循环的唯一"继续信号"是 needsFollowUp。它只由"模型输出里是否含 tool_use 块"决定——模型想调工具就续圈，模型只说话就收工。这比依赖 stop_reason === 'tool_use' 更可靠（源码注释明确指出后者"不总是被正确设置"）。
05

流式工具执行：不等模型说完就开跑

最能体现工程功力的一处优化。默认路径下工具不是"等模型整段输出完再批量执行"，而是用 StreamingToolExecutor 在模型还在流式吐字的时候，就把已经完整到达的 tool_use 块塞进执行器提前跑起来。

两条执行路径

  • 流式（默认，gate 开）：streamingToolExecutor.addTool(block) 边流边执行，getCompletedResults() 边产出
  • 批量（回退）：流结束后 runTools(toolUseBlocks, ...) 统一跑

并发安全闸

工具是否能并行，看 isConcurrencySafe(input)。默认 false（假设会写）；只读工具（Read/Grep/Glob）返回 true 才允许与同批工具并跑。写操作串行，避免互相踩踏。

// 流式循环内：一边收模型消息，一边喂执行器、收结果
if (message.type === 'assistant') {
  assistantMessages.push(message)
  const toolUses = message.content.filter(c => c.type === 'tool_use')
  if (toolUses.length > 0) { toolUseBlocks.push(...toolUses); needsFollowUp = true }
  for (const block of toolUses) streamingToolExecutor.addTool(block, message)  # 立即排队执行
}
for (const r of streamingToolExecutor.getCompletedResults()) {
  yield r.message
  toolResults.push(...normalizeMessagesForAPI([r.message]).filter(m => m.type==='user'))
}
中断安全：一旦 abortController 触发，执行器通过 getRemainingResults() 为排队中 / 执行中的工具补齐合成的 tool_result 块——保证每个 tool_use 都有配对结果，否则下一次 API 调用会因为块不配对而报错。
06

恢复、压缩与有界：循环里的安全网

主循环不是"顺风顺水地转圈"，它内建了多套失败恢复与硬性边界。这些机制让长会话既不会因为上下文爆掉而崩，也不会无限烧钱。

触发机制动作
模型过载FallbackTriggeredError切 fallbackModel，清空本轮消息与工具，continue 重试；提示"因高需求已切换模型"
上下文超限 (413)prompt_too_long先 context-collapse 排空折叠，再 reactive compact 全量摘要，重试一次
输出超长max_output_tokens同一请求把上限从 8k 升到 64k 重试；再不行走多轮恢复
图片过大ImageSizeError产出友好错误消息，return {reason:'image_error'}
回合上限maxTurns产出 max_turns_reached 附件 → result error_max_turns
成本上限maxBudgetUsd每圈查 getTotalCost()，超则 error_max_budget_usd
用户中断abort补齐 tool_result，return {reason:'aborted_streaming' | 'aborted_tools'}

Query 链追踪 depth

每次进循环带 queryTracking = {chainId, depth}。主线程 depth=0；子 Agent 继承并 depth+1。这让"谁在第几层调了谁"可观测，也是防止子 Agent 无限套娃的基础。

终止即 Terminal

queryLoop 的返回值是一个 Terminal = {reason}：model_error / prompt_too_long / aborted_* / hook_stopped / image_error 或正常结束。上层据此决定 result 形态。

07

Tool 声明式契约

每个工具都是一个实现 Tool 接口的对象。它不止是"一个函数"，而是一份把执行、鉴权、渲染、序列化、安全分级全部打包的契约。buildTool() 为常被省略的方法填 fail-closed（保守）默认值，让所有工具都拿到一致的完整形态。

成员作用默认（buildTool）
call()真正执行，返回 ToolResult（可带 newMessages / contextModifier）—
inputSchemazod 定义入参，供模型生成结构化调用—
validateInput()执行前校验，失败把原因回给模型（不弹 UI）放行
checkPermissions()工具专属鉴权逻辑，返回 allow/deny/askallow 交给通用权限系统
isConcurrencySafe()能否与同批工具并行false（假设会写）
isReadOnly() / isDestructive()只读 / 不可逆（删除、覆盖、发送）false / false
maxResultSizeChars结果超此长度即落盘，回给模型一个文件路径预览各工具自定（Read=∞）
interruptBehavior()运行中来了新消息：cancel 还是 blockblock
render* / mapToolResult*TUI 渲染 + 转成 API 的 tool_result 块—
Fail-closed 是刻意的：isConcurrencySafe → false、isReadOnly → false，意味着"除非工具明确声明安全，否则一律当作会写、不可并行"。安全默认优先于性能默认。

09

ToolSearch：工具延迟加载

工具一多，把所有工具的完整 JSON Schema 都塞进初始 prompt 会又贵又占上下文。Claude Code 的解法是"延迟加载"：一部分工具以 defer_loading: true 发送，模型看不到完整 schema，需要先用 ToolSearch 按关键字把它"搜"出来才能调用。

初始只带 alwaysLoad 工具→ 模型用 ToolSearch(关键字)→ 命中 searchHint 的工具展开 schema→ 模型正常调用

shouldDefer

标记该工具默认延迟，不进初始 schema。

alwaysLoad

强制第一轮就带完整 schema——模型必须在 turn 1 就能看到的工具（MCP 侧用 anthropic/alwaysLoad）。

searchHint

3–10 词的能力短语，供关键字匹配。刻意用工具名里没有的词（如给 NotebookEdit 配 "jupyter"）。

效果是把"工具很多"与"初始 prompt 精简、缓存稳定"这对矛盾解开：常用工具直发，长尾工具按需检索。
10

权限系统：模式 × 规则 × 分类器

工具执行前必过权限关。canUseTool 先调工具自己的 checkPermissions，再叠加会话的权限模式与规则，产出 allow / deny / ask 三态（外加 passthrough 透传）。每个决策都带 decisionReason，可解释可审计。

五种权限模式（用户可寻址）
模式含义
default标准：按规则判定，未知的写操作弹确认
plan只读计划模式：不落地改动，只读与调研
acceptEdits自动接受文件编辑，减少打断
dontAsk不主动询问（在允许范围内直行）
bypassPermissions绕过权限（需环境显式可用），高危

另有两个内部模式：auto（下面的分类器模式）与 bubble（Fork 子 Agent 把权限提示"冒泡"回父终端）。

规则的六个来源

allow / deny / ask 三类规则可来自 userSettings · projectSettings · localSettings · policySettings · cliArg · session 等。deny 规则在装配阶段就把工具从池里滤掉（连模型都看不到），不只是调用时拦。

auto 模式：模型当安检

开启后，Bash 命令交给一个 haiku 分类器（YoloClassifier，两阶段 fast → thinking）判"是否该拦"。敏感路径（.git/ .claude/、shell 配置）标 classifierApprovable 让分类器带上下文裁决，而非一律弹窗。

// 权限判定的优先级链（简化）
canUseTool(tool, input, ctx):
  tool.validateInput()            # 入参非法 → 直接回错，不进权限
  tool.checkPermissions()         # 工具专属逻辑
  → deny 规则命中?   deny       # decisionReason: {type:'rule'}
  → allow 规则命中?  allow
  → 按 mode 决策           # plan/acceptEdits/... {type:'mode'}
  → Hook / classifier 介入      # {type:'hook' | 'classifier' | 'safetyCheck'}
  → 否则 ask 用户            # 弹确认，可记为持久规则
11

AgentTool：派生专业子 Agent

Claude Code 最有辨识度的能力：主 Agent 可以用 Agent 工具派生子 Agent 去自主完成复杂多步任务。子 Agent 是一个带独立上下文的 query() 实例（depth+1），跑完把一条总结消息交回主 Agent——中间的工具噪音不污染主上下文。

AgentDefinition：一个子 Agent 就是一份声明

子 Agent 定义可以是内置的，也可以来自 .md（frontmatter）或 .json，落在 user / project / plugin 等来源。关键字段本身就是能力边界：

字段含义
tools / disallowedTools工具白名单 / 黑名单（子 Agent 能看到哪些工具）
model / effort模型分级与推理强度（inherit = 跟随父级）
permissionMode子 Agent 的权限模式
maxTurns子 Agent 自身的回合上限
skills / mcpServers / hooks预载技能、专属 MCP、会话级 Hooks
memory持久记忆范围 user / project / local
isolationworktree（临时 git worktree）/ remote（远程 CCR 沙箱）
background是否总以后台任务方式运行
omitClaudeMd只读 Agent（Explore/Plan）省略 CLAUDE.md 以省 token

内置子 Agent

general-purpose · explore · plan · statusline-setup · code-guide · verification。CLAUDE_AGENT_SDK_DISABLE_BUILTIN_AGENTS 可让 SDK 用户拿到"空白画布"。

前台 vs 后台

run_in_background：后台子 Agent 完成时以 <task-notification> 主动通知——提示词明确要求"别 sleep、别轮询"，去干别的活。前台则用于"结果决定下一步"的调研。

续接而非重启：用 SendMessage 以子 Agent 的 id / name 作 to，可让它带着完整上下文继续；相对地，每次全新 Agent(subagent_type) 都是零上下文起步，必须在 prompt 里交代清楚。
12

Fork：Agent 的自我分身

当 FORK_SUBAGENT 特性开启时，调用 Agent 工具却省略 subagent_type，就触发一次 fork——子进程继承父级完整对话上下文与系统提示，而不是零起步。它专治"这段工具输出我不想留在自己上下文里"的场景（大范围调研、多处实现）。

为什么 fork 很便宜

所有 fork 子进程共享父级的 prompt 缓存：buildForkedMessages 让每个子进程的 API 请求前缀逐字节一致（相同占位 tool_result + 仅末尾指令不同），最大化缓存命中。所以提示词说"别在 fork 上设 model"——换模型就无法复用父级缓存。

Fork 的合成定义

FORK_AGENT：tools:['*']（继承父级全部工具，保前缀一致）、model:'inherit'、permissionMode:'bubble'（权限提示冒泡回父终端）、maxTurns:200。

// 注入给 fork 子进程的强指令（buildChildMessage 摘要）
<fork-boilerplate>
  你是一个 fork 出来的 worker 进程，不是主 Agent。
  1. 系统提示说"默认 fork"—— 忽略它，你就是那个 fork，直接执行，别再派子 Agent。
  2. 别对话、别问问题、别加元评论；静默用工具，最后一次性汇报。
  3. 改了文件就先 commit，汇报里带 commit hash。
  4. 严格待在 directive 范围内；回复必须以 "Scope:" 开头。
</fork-boilerplate>
<fork-directive>{父级下达的具体指令}</fork-directive>
防递归 fork：fork 子进程为了缓存一致仍保留 Agent 工具，因此靠 isInForkChild() 扫历史里的 fork 标记，在调用时拒绝"fork 里再 fork"。此外 fork 与 coordinator 模式互斥——两者都想当编排者。worktree 隔离下还会注入路径翻译提示，告知子进程它在独立工作副本里干活。
13

Coordinator：主 Agent 当编排者

COORDINATOR_MODE 开启后，主 Agent 的人设从"亲自干活"变成"协调者"：它把研究 / 实现 / 验证派给 workers（子 Agent），自己负责综合结论、跟用户沟通。worker 的结果以 user-role 消息里的 <task-notification> XML 回流——看着像用户消息，实为内部信号。

Research（并行 workers）→ Synthesis（coordinator 亲自消化）→ Implementation（workers 按 spec 改 + commit）→ Verification（workers 独立验证）

并行是超能力

提示词直言"parallelism is your superpower"：独立的活并发起（一条消息里多个 Agent 调用）。只读调研随便并行；写操作按文件集串行；验证可与实现在不同文件区并行。

绝不"懒委派"

禁止 "based on your findings, fix it" 这类把理解甩给 worker 的话。coordinator 必须亲自综合，写出带文件路径、行号、具体改动的 spec——用"证明你读懂了"的方式派活。

Continue vs Spawn 决策

派活前先判断上下文重叠度：worker 刚好探过要改的文件 → Continue（SendMessage 复用其上下文）；调研宽泛但实现很窄、或要"换个新脑子"验证别人写的代码 → Spawn fresh（干净上下文避免锚定）。没有万能默认，看重叠度。

14

任务类型与生命周期

Task.ts 把所有"可派发的活儿"抽象成统一的任务模型：一个 TaskType + 一套状态机 + 一个落盘输出文件。子 Agent、后台 shell、远程执行、定时监控都收敛到这里。

TaskType（7 类）

  • local_agent / remote_agent — 本地 / 远程子 Agent
  • local_bash — 后台 shell 任务
  • in_process_teammate — 进程内队友
  • local_workflow — 工作流脚本
  • monitor_mcp — MCP 监控
  • dream — 空闲自驱

状态机

pending→ running→ completed

终态还有 failed / killed。isTerminalTaskStatus() 守护：不往死队友里注消息、终态任务从 AppState 逐出。

任务 ID 是安全设计：generateTaskId 用"类型前缀 + 8 位随机（36 进制，约 2.8 万亿组合）"，注释点明这是为了抵御符号链接爆破攻击——任务输出落盘在以 ID 命名的文件里，ID 可猜就有风险。每个任务有独立 outputFile 与读偏移，通知已读位置可增量。
15

多级压缩管线：把上下文按住

长会话最大的敌人是上下文膨胀。Claude Code 不靠单一手段，而是在每圈循环开头叠了一条压缩流水线，各管一段，从"轻剪枝"到"全量摘要"层层递进。目标：尽量保留细粒度上下文，实在不行才总结。

① applyToolResultBudget — 工具结果预算
聚合工具结果超预算时做内容替换（按 tool_use_id，对缓存透明）；maxResultSizeChars=∞ 的工具（如 Read）豁免
② snip — 剪僵尸
移除僵尸消息与陈旧标记，释放的 token 数还会喂给下游阈值判断
③ microcompact — 微压缩
按 tool_use_id 删除旧工具调用（缓存编辑），细粒度腾空间
④ context collapse — 折叠视图
读时投影的折叠：跑在 autocompact 前，若折叠后已达标就不必全量摘要
⑤ autocompact — 自动摘要
到阈值把历史总结成摘要，产出 compact_boundary 边界消息
⑥ reactive compact — 反应式（兜底）
仅当真的 413 prompt-too-long 才触发，全量摘要后重试一次
compact_boundary 是分水岭：压缩后 getMessagesAfterCompactBoundary() 只取边界之后的消息进 API，边界之前的原始消息可被 GC 释放。taskBudget 还会跨压缩边界记账——压缩前的最终上下文窗口被"减去"，让服务端不会因为只看到摘要而少算花费。
16

系统提示与记忆

系统提示不是一个大字符串，而是分件组装（fetchSystemPromptParts）再拼装的：默认提示 + 用户上下文 + 系统上下文，可被 custom 替换或 append 追加。记忆分两条线：项目级 CLAUDE.md 与 Agent 持久记忆。

系统提示三段

  • defaultSystemPrompt — 内核人设与规则（或被 customSystemPrompt 整段替换）
  • userContext — 工作目录、coordinator worker 工具说明等
  • systemContext — 环境 / 会话态，循环内 appendSystemContext 合入
  • appendSystemPrompt — 调用方追加的策略文本

两条记忆线

  • CLAUDE.md（nested memory）：分层注入为附件，loadedNestedMemoryPaths 去重，避免同一文件被重复注入几十次
  • Agent memory：MEMORY.md，范围 user / project / local，跨会话持久；SDK 用 CLAUDE_COWORK_MEMORY_PATH_OVERRIDE 显式接入时才注入"记忆使用说明"
省 token 的小心思：只读子 Agent（Explore / Plan）带 omitClaudeMd——它们不需要提交 / PR / lint 规范，主 Agent 有完整 CLAUDE.md 并负责解读其产出。源码注释称这跨海量 Explore 派生每周省下 5–15 Gtok。
17

Hooks：横切进循环的扩展点

Hooks 让外部逻辑在关键节点插进 Agent 生命周期——做校验、注入上下文、或直接拦停。它既是权限判定链的一环，也是回合收尾的把关者。

时机作用
PreToolUse工具执行前：可 ask 二次确认，或直接拦（权限判定链里的 {type:'hook'}）
PostToolUse工具执行后：审计、通知、后处理
postSampling模型响应完成后异步执行（executePostSamplingHooks）
Stop回合收工时把关：可注入延续或 hook_stopped_continuation 阻止收工
SessionStart / UserPromptSubmit会话开始 / 用户提交时注入上下文
Stop Hook 的死循环防护：源码特意指出——prompt-too-long 时不走 Stop Hooks。因为模型根本没产出有效响应，跑 Stop Hook 会形成"错误 → hook 拦停 → 重试 → 又错误"的死亡螺旋（hook 每轮还往里灌 token）。这类"别在 X 情况下触发 Y"的边界，正是工业级 Loop 与玩具 Loop 的差距所在。
18

Skills：可发现的能力包

Skill 是把"一段可复用的能力 / 工作流"打包成可被模型发现和调用的单元。它通过 Skill 工具触发，来源有内置 bundled 与用户 / 项目目录（loadSkillsDir）。

发现机制

循环里有 skill discovery 预取（startSkillDiscoveryPrefetch），在模型流式期间并行找相关技能；discoveredSkillNames 记录本回合已浮现的技能（喂遥测 was_discovered）。

预载与派发

子 Agent 定义里的 skills 字段可预载技能；coordinator 模式则被要求把技能调用（如 /commit、/verify）下派给 worker 执行。

与斜杠命令

getSlashCommandToolSkills 把斜杠命令也纳入技能体系，在 system.init 时一并声明给模型。

19

贯穿的设计原则

把散落在各处的实现抽象出来，Claude Code 的 Agent 运行时其实反复在贯彻同一组原则。

① 流式即一等公民

全链路 async generator：模型输出、工具执行、子 Agent 结果都能边产边渲染、边产边中断。工具甚至在模型没说完时就开跑（第 5 节）。

② 有界一切

maxTurns / maxBudgetUsd / taskBudget / token 上限 / 子 Agent depth。没有无界的自动行为，每条路径都能收口。

③ Fail-closed 安全默认

工具默认"会写、不可并行"；权限未明即 ask；deny 规则在装配期就滤掉工具。安全默认优先于性能与便利。

④ 上下文是稀缺资源

延迟加载工具、多级压缩、子 Agent 隔离噪音、只读 Agent 省 CLAUDE.md——处处在为上下文窗口"省地方"。

⑤ Prompt 缓存驱动架构

工具排序保连续前缀、Fork 逐字节对齐、Agent 列表改走附件——很多结构决策的第一动机就是"别把缓存打穿"。

⑥ 可恢复 / 可观测

transcript 提前落盘（崩溃可 --resume）、任务输出独立文件、query 链带 chainId/depth、每个权限决策带 decisionReason。

与 ICE-DATA-WORK 的对照

两套系统解决的问题不同——Claude Code 是单用户、本地、编码的通用 Agent；ICE-DATA-WORK 是多参与者、受治理、数据工作的协作运行时。但骨架上有不少可互相印证的设计。此处仅作机制对照，不评优劣。

维度Claude CodeICE-DATA-WORK
执行内核queryLoop 有界 ReAct（async generator）_run_single_hop 单跳内核
多 AgentAgentTool 派生 / Fork / Coordinator 编排Twin 唯一 leader + A2A 有界多跳
Agent 声明AgentDefinition（tools / model / permissionMode / maxTurns）agent.json（tools / spawn_targets / permission_mode / effort）
防失控depth 追踪 · fork 防递归 · maxTurns / 预算HopState 防环 + 软硬跳数预算 + Twin 六道仲裁
权限五模式 × allow/deny/ask × 分类器三轴权限 / HITL / 主动性护栏
上下文多级压缩 + 延迟加载 + 记忆分层分层注入 + 访问控制内嵌读取点
结果回流子 Agent → <task-notification> user 消息A2A handoff 事件 + 前序产出摘要
真相源transcript / task output 文件 + session文件优先（G3）：jsonl / plan.json
共通的手法：两者都是"单一稳定内核 + 外层循环叠能力 + 治理闸门横切"。差异在治理重心——Claude Code 侧重本地安全与成本 / 缓存，ICE-DATA-WORK 侧重多方协作的越权防护与可审计。看懂一套，另一套的设计取舍也就有了参照系。
