## 一、产品概述
### 1.1 产品定位
ICE-DATA-WORK 2.0是**人 + 数字分身（Twin Agent）+ 工具Agent团队**在同一块任务看板上协作的AI数据工作台，核心解决非数据专业角色在跨岗协作和专家咨询场景下的效率问题，实现「用户布置完任务就走开，回来直接拿结论」的AI协作体验。

### 1.2 产品slogan
> 你的数字分身，帮你搞定数据工作

### 1.3 背景与问题
当前AI工具（Claude Code、Codex等）在数据工作场景存在三大核心痛点：
| 痛点 | 具体表现 | 影响 |
|------|----------|------|
| **认知盲区** | AI默认关键业务信息（口径、规则、坑点），用户难以发现隐含错误 | 结论可信度低，需要人工反复核对 |
| **确认效率低** | AI每一步都需要用户确认，提问完备性存疑，用户必须全程盯守 | 用户成为协作瓶颈，效率提升有限 |
| **速度不匹配** | AI产出速度远超人工确认速度，大量时间浪费在等待用户确认上 | AI能力无法充分发挥，整体效率提升打折扣 |

现有单Agent「用户问→AI答」模式，本质上是用户直接对接初级分析师，所有判断都需要用户自己做，没有发挥AI的代理价值。

### 1.4 产品目标
#### 短期目标（Q3 2026）
- 核心链路验证：跑通Twin+工具Agent协作闭环，覆盖80%日常数据分析场景
- 效率提升：简单数据分析任务（取数、报表、基础分析）人力投入减少70%，用户无需全程盯守
- 内部 adoption：数据组3人以上日常使用，沉淀20+高频场景经验包
- 质量达标：Twin低风险确认项代答准确率≥90%，SQL查询准确率≥95%

#### 中期目标（Q4 2026）
- 团队推广：覆盖浏览器/内容中心数据产品、运营等5+非数据岗位用户
- 能力扩展：支持AB实验分析、灰度版本分析、日报自动生成等10+工具Agent
- 平台化准备：输出可复用的Agent设计规范和经验包标准，评估向公司Mika/Chatflow平台迁移

#### 长期目标（2027）
- 成为团队级AI协作入口，沉淀团队共享经验包库
- 实现「人人都有自己的数据分析师」，非数据岗无需写SQL即可完成80%数据需求
- 形成可对外输出的多Agent协作产品方案

### 1.5 核心价值
| 用户角色 | 价值 |
|----------|------|
| **数据分析师/产品** | 从重复取数、基础分析中解放出来，聚焦高价值判断和复杂问题 |
| **非数据岗（产品/运营）** | 无需学习SQL、无需等待数据排期，自己就能快速获取可信的数据分析结论 |
| **团队管理者** | 所有分析过程可回溯、经验可沉淀，避免知识流失，新人快速上手 |

## 二、用户场景与需求
### 2.1 目标用户
| 用户分层 | 特征 | 核心诉求 |
|----------|------|----------|
| **核心用户（数据产品/分析师）** | 熟练使用SQL，日常有大量取数、分析、报告需求 | 减少重复劳动，自动处理低风险确认，提升产出效率 |
| **次要用户（产品/运营）** | 不会SQL或不熟练，依赖数据团队排期 | 快速获取数据，减少沟通等待，自主完成简单分析 |
| **潜在用户（团队管理者）** | 需要看数据结论，关注团队效率和知识沉淀 | 过程可追溯，经验可复用，降低人员流动风险 |

### 2.2 典型用户故事
#### US1：日常数据波动归因（核心场景）
> **作为** 数据产品经理
> **我想要** 告诉Twin「看看昨天信息流消费时长环比降了7%，帮我定位原因」然后去开会
> **这样** 我开完会回来就能看到完整的归因结论、关键数据和可视化报告，而不用全程盯着AI一步步确认

#### US2：固定报表自动生成
> **作为** 运营人员
> **我想要** 每周一让Twin自动生成上周核心业务指标周报
> **这样** 我不用每次找数据团队排期，也不用自己复制粘贴做表，Twin会自动取数、排版、发飞书给我

#### US3：AB实验结果分析
> **作为** 产品经理
> **我想要** 给Twin一个实验链接，让它自动帮我完成实验显著性检验、结论总结、报告生成
> **这样** 我不用自己去查数据、算p值、写报告，节省半天时间

#### US4：跨团队数据需求交付
> **作为** 数据分析师
> **我想要** 把业务方提的需求交给Twin去做初步取数和口径确认
> **这样** 我只需要处理复杂的分析部分，低风险的口径确认和取数让Twin代劳，提升交付效率

### 2.3 五类协作模式覆盖
ICE 2.0重点覆盖「跨岗协作」和「专家咨询」高价值场景：
| 协作模式 | 场景示例 | ICE 2.0支持度 |
|----------|----------|--------------|
| 上下级同岗 | 带新人做熟悉领域 | P2，后续扩展 |
| 平级同岗 | 同事分工协作 | P2，后续扩展 |
| **上下级跨岗** | 安排不熟悉领域工作 | ✅ P0核心支持 |
| **平级跨岗** | 找其他岗位专家帮忙 | ✅ P0核心支持 |
| **专家咨询** | 请教专业问题 | ✅ P0核心支持 |

## 三、核心创新点
### 3.1 核心创新：Twin Agent（数字分身）理念与设计
ICE 2.0最核心的创新是**引入「数字分身」作为用户唯一代理人**，区别于现有所有单Agent或多Agent协作模式：

| 模式 | 用户角色 | 协作关系 | 核心问题 |
|------|----------|----------|----------|
| **传统工具模式（BI/SQL客户端）** | 操作者 | 用户→工具，人驱动工具 | 门槛高，效率低，用户必须全程操作 |
| **单Agent模式（ChatGPT/Claude Code）** | 监工 | 用户↔Agent，直接对话 | 用户必须全程盯守，每步确认，认知盲区大，速度不匹配 |
| **多Agent平权模式（AutoGPT类）** | 裁判 | 用户→Agent1→Agent2→...，链式传递 | 上下文丢失，无人对最终结果负责，错误层层放大 |
| **ICE Twin模式** | 老板/目标设定者 | 用户→Twin（唯一代理人）→工具Agent团队，星型拓扑 | ✅ Twin替用户盯守执行、代答低风险确认、验收产出<br>✅ 用户布置完任务就走开，回来拿结果<br>✅ Twin以用户画像为基础，决策符合用户偏好，不越权 |

**Twin设计哲学：**
- **代理而非工具**：Twin不是工具，是你的数字员工，有明确的职责边界、决策权限和汇报线
- **代答不越权**：基于风险分级矩阵，低风险确认自动代答留痕，高风险必升级用户决策，所有决策可追溯可纠正
- **验收不妥协**：Twin作为甲方验收工具Agent产出，不合格直接打回重做，直到符合交付标准，对最终结果负责
- **进化不停止**：从每次任务、每次用户反馈中学习，持续校准画像和决策规则，越用越像你

### 3.2 次级创新点
#### 创新点2.1：多Agent协作的空间设计
区别于其他多Agent系统，ICE设计了**四类空间严格分离**的架构，解决多Agent上下文污染、记忆混乱、不可追溯问题：
1. **用户空间（私有）**：每个用户独立的Twin画像、知识库、记忆、决策规则，其他人不可见
2. **Agent空间（共享）**：工具Agent的SOP、领域知识、技能包团队共享，所有用户使用同一套标准Agent能力
3. **任务空间（临时）**：每个任务独立的运行时空间，对话/SQL/数据/决策/思考过程完整落盘，任务结束归档
4. **技能空间（公共）**：通用工具能力（取数/飞书/模型网关）标准化封装，所有Agent按需调用

这种空间隔离设计确保：
- 不同用户的个性化偏好不会互相干扰
- 工具Agent专注于专业能力，不掺杂用户个性化信息
- 每个任务完整可追溯，上下文边界清晰
- 技能可复用、可升级，不影响核心逻辑

#### 创新点2.2：工作空间沉淀为经验包的设计
区别于传统RAG方案和固定Skill，ICE设计了**轻量级经验包机制**：
- **经验包vs Skill**：Skill是固定的能力封装，重而全，注入上下文成本高；经验包是从历史任务中蒸馏的细粒度知识单元（一个口径选择、一个分析范式、一个踩坑教训），轻而准，按需检索注入
- **自动沉淀流程**：任务完成→Twin自动提炼关键要素（需求描述/SQL模板/分析过程/结论/坑点）→生成经验包→人工审核后入库
- **按需检索注入**：新任务启动时，Twin和工具Agent根据任务场景检索Top N相关经验包，精准注入上下文，避免无效信息冗余
- **经验包流动**：个人经验包可升级为团队共享经验包，团队优质经验可反哺个人，形成组织知识沉淀飞轮

#### 创新点2.3：多人多Twin协作的理念与设计
支持**团队级多用户多Twin网状协作**，是真正的生产级协作系统：
- 每个用户有自己的专属Twin，Twin之间可以互相派发任务、沟通协作，模拟真实组织架构下的协作关系
- 示例：产品经理给运营提需求→产品的Twin把任务派发给运营的Twin→运营的Twin调度自己的工具Agent完成→结果返回给产品Twin→交付给产品经理
- 权限边界清晰：每个Twin只能访问自己用户的私有空间，协作过程可审计、可追溯
- 未来可支持部门级Twin、团队级Twin，形成层级化数字组织

### 3.3 底层技术逻辑
#### 3.3.1 Twin记忆与决策信息来源：Git工作空间自动蒸馏
Twin之所以能「像你一样做决策」，核心是**基于Git版本化的个人工作空间自动蒸馏用户画像和决策规则**，不需要用户手动配置：
```
┌─────────────────────────────────────────────────────────┐
│                 Git工作空间（用户沉淀）                  │
│  AGENTS.md · USER.md · CURRENT.md · WORK-PLAN.md        │
│  MEMORY.md · DECISIONS.md · 历史报告 · 历史SQL · 会议纪要 │
└───────────────────────┬─────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────────────┐
│               Twin画像蒸馏引擎（每日定时运行）            │
│  1. Git diff提取每日变更                               │
│  2. LLM结构化提取关键信息：                            │
│     → 用户决策风格（保守/激进）                        │
│     → 常用指标口径默认值                                │
│     → 业务坑点与硬性规则                                │
│     → 说话风格与交付偏好                                │
│     → 当前重点工作与优先级                              │
│  3. 更新profile.md和知识库，自动版本管理                │
└───────────────────────┬─────────────────────────────────┘
                        │
                        ▼
┌─────────────────────────────────────────────────────────┐
│                 Twin运行时决策                          │
│  低风险确认项 → 基于画像自动代答 → 记录决策理由         │
│  高风险问题   → 升级用户决策 → 决策结果回流更新画像     │
└─────────────────────────────────────────────────────────┘
```
- **零配置启动**：只要用户使用标准化Git工作空间记录工作（对齐现有ICE工作区规范），Twin自动学习，不需要手动填任何配置
- **动态更新**：每日定时蒸馏工作空间变更，Twin画像随用户工作动态更新，保持同步
- **人工可干预**：用户可以直接编辑工作空间文件纠正Twin认知，也可以对代答决策点踩反馈，快速校准
- **可解释**：Twin所有决策都可以追溯到具体的工作空间文件来源，黑盒变白盒

#### 3.3.2 同空间内多Agent上下文设计、隔离与对话机制
解决多Agent协作最核心的上下文污染问题，ICE设计了**三级上下文隔离+精准注入+受控对话**机制：

##### 三级上下文隔离机制
| 上下文层级 | 内容 | 注入对象 | 注入时机 |
|------------|------|----------|----------|
| **全局上下文** | 任务目标、用户身份、基础规则、安全约束 | 所有Agent | 任务启动时一次性注入 |
| **角色上下文** | Twin：用户画像、代答规则、验收标准<br>工具Agent：领域SOP、指标口径、工具使用方法 | 对应角色Agent | 每次对话时注入 |
| **任务上下文** | 当前对话历史、已获取的数据、中间结论、经验包 | 按需注入 | 根据当前Agent和任务阶段精准检索注入 |

- **严格隔离**：工具Agent看不到其他Agent的思考过程、看不到用户私有记忆、看不到无关经验包，只拿到完成当前步骤必要的最小上下文
- **避免冗余**：不做全量上下文拼接，采用检索增强注入，只有相关经验包、相关历史对话才会注入，无效信息不进入上下文，降低token消耗，提升准确率
- **记忆隔离**：Twin的进化记忆和工具Agent的进化记忆物理分离，工具Agent不记忆用户个性化偏好，保持通用性；Twin记忆用户个性化信息，不污染工具Agent能力

##### Agent间受控对话机制
- **星型拓扑，不允许对等对话**：所有工具Agent只和Twin对话，Agent之间不直接通信，所有信息都经过Twin转发和过滤，避免信息混乱和责任不清
- **标准化通信协议**：Agent之间通过结构化JSON动作通信（assign/ask/report/deliver/escalate等），不允许自由自然语言对话，确保指令清晰无歧义
- **对话留痕**：所有Agent间通信完整记录在conversation.jsonl中，可审计可回溯，Twin转发时可以补充要求、打回不合格产出
- **步数预算控制**：每个Agent每次任务有最大对话步数限制（默认28步），用户插话后重置，防死循环防跑飞

### 3.4 多Agent协作的交互与生产关系设定
ICE的多Agent协作不是简单的任务调度，而是**模拟真实职场的生产关系设计**，每个角色有明确的汇报线、职责、考核标准：

```
                        ┌──────────────┐
                        │    用户      │  老板/目标设定者
                        │  (最终甲方)   │
                        └──────┬───────┘
                               │  汇报/升级
                               ▼
                        ┌──────────────┐
                        │  Twin Agent  │  项目经理/用户代理人
                        │  (唯一接口)   │  对最终结果负责
                        └──────┬───────┘
           ┌───────────┬───────┴───────┬───────────┐
           │           │               │           │
           ▼           ▼               ▼           ▼
      ┌─────────┐ ┌─────────┐     ┌─────────┐ ┌─────────┐
      │数据分析 │ │AB实验   │ ... │样式优化 │ │报告撰写 │  各领域专家
      │ Agent   │ │Agent    │     │Agent    │ │Agent    │  对Twin负责
      └─────────┘ └─────────┘     └─────────┘ └─────────┘
```

##### 生产关系规则：
1. **汇报线唯一**：工具Agent的甲方是Twin，不是用户；工具Agent只对Twin汇报，不直接响应用户，避免多头指挥
2. **职责边界清晰**：
   - 用户：只做高价值决策（目标设定、高风险取舍、最终验收）
   - Twin：做项目管理工作（理解需求、派发任务、低风险决策、质量验收、进度把控、风险升级）
   - 工具Agent：做专业执行工作（按SOP完成自己领域的任务，不做决策，遇到问题抛给Twin）
3. **验收机制**：工具Agent交付产出后，Twin按照预设验收标准（从用户偏好蒸馏）自动验收，不合格直接打回重做，不需要用户介入，直到符合标准
4. **权责对等**：Twin有代答决策的权力，同时对最终结果负责；工具Agent有执行权力，同时对产出质量负责；用户拥有最高决策权，同时对高风险决策负责
5. **冲突解决**：如果Twin和工具Agent对产出有分歧，由Twin裁决；如果Twin无法裁决，升级用户决策
6. **可扩展性**：新增工具Agent相当于招聘新的领域专家，只需要编写其SOP和领域知识，加入Agent花名册即可，不需要修改核心协作逻辑

## 四、产品架构与核心概念
### 4.1 整体架构
```
┌─────────────────────────────────────────────────────────┐
│                     用户交互层                           │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐  │
│  │ 任务看板 │  │ 对话协作 │  │ 过程产物 │  │ Agent管理│  │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘  │
├─────────────────────────────────────────────────────────┤
│                     Twin编排层                           │
│  ┌───────────────────────────────────────────────────┐  │
│  │ 意图理解 · 任务派发 · 确认项代答 · 验收 · 升级交付 │  │
│  └───────────────────────────────────────────────────┘  │
├─────────────────────────────────────────────────────────┤

### 4.2 核心角色定义
多方角色在**同一个对话区**里协作，星型拓扑，Twin是唯一编排者：

| 角色 | 定位 | 核心职责 | 边界 |
|------|------|----------|------|
| **用户（User）** | 目标设定者&最终验收人 | 给目标、审阅最终结果、拍板高风险决策；可随时@插话/私聊Twin | 不盯每一步执行 |
| **Twin Agent（数字分身）** | 用户代理人/唯一编排者 | 理解意图、任务派发、替用户回答低风险确认项、验收产出、打回重做、升级高风险问题、最终交付 | 只提议不越权；不做具体执行（不写SQL/不排版/不查数）；高风险必问用户 |
| **工具Agent（Tools）** | 各领域专家 | 按SOP完成专业工作：<br>- 数据分析Agent：取数+分析+归因<br>- AB实验Agent：实验设计+显著性检验<br>- 样式优化Agent：报告排版美化<br>- 灰度分析Agent：版本灰度对比<br>- nl-sql Agent：自然语言转SQL<br>- 报告撰写Agent：结构化文档输出<br>...可扩展 | 甲方是Twin不是用户；只对Twin汇报；不替用户拍板；只读操作不修改数据 |

### 4.3 Twin五大核心能力
1. **角色定义**：基于用户工作空间文件（AGENTS.md/USER.md/CURRENT.md/MEMORY.md/DECISIONS.md等）蒸馏用户画像，理解用户偏好、决策风格、常用口径、业务坑点
2. **工作契约**：基于风险分级矩阵明确什么级别的问题可以代答、什么必须升级，不越权
3. **反馈闭环**：每个代答决策都记录理由，用户可事后纠正，持续优化判断准确率
4. **记忆管理**：跨任务记忆用户习惯、业务规则、坑点教训，越用越懂用户
5. **定时校准**：定期回顾代答决策，根据用户反馈调整代答策略，避免偏差累积

## 六、非功能需求
### 6.1 性能需求
- 简单取数任务端到端响应时间≤3分钟
- LLM调用超时时间支持配置（默认30分钟）
- kyuubi单条查询超时默认2分钟，可配置
- SSE消息推送延迟≤1秒
- 支持最长28步Agent协作，上下文压缩后支持长任务运行

### 6.2 安全需求
- **数据安全**：kyuubi查询白名单仅允许SELECT/WITH，硬禁DDL/DML/多语句，从工具层杜绝写操作
- **密钥安全**：LLM API密钥仅在服务端保存使用，不下发到浏览器
- **注入防护**：SQL通过参数数组传递，杜绝shell注入；文件读取做路径穿越校验
- **降级容错**：缺密钥/缺kyuubi/网络错误时优雅降级，返回友好提示而非崩溃
- **本地优先**：v1版本为单机单用户设计，默认仅本地访问，无对外端口

### 6.3 可用性需求
- 零依赖启动：无需npm install，Node.js≥18即可npm start启动
- 零构建前端：原生HTML/CSS/JS，无需前端构建工具
- 断点续跑：服务重启后自动恢复执行中的任务，不丢失进度
- 操作简单：输入目标点「交给Twin」即可使用，学习成本≤10分钟

### 6.4 可扩展性
- Agent扩展：新增工具Agent只需新增agent.md+知识文件+花名册登记，无需修改核心代码
- 工具扩展：集成新工具（飞书/Python/MCP）只需新增集成模块，不影响核心编排
- 部署扩展：核心逻辑与平台无关，后续可平滑迁移到Mika/Chatflow等公司级平台

## 七、交互设计
### 7.1 核心页面结构
```
┌────────────────────────────────────────────────────────────┐

├─────────────────────────────┬──────────────────────────────┤

│  - 用户目标                  │  - 参与Agent列表+实时状态     │
│  - Twin交付汇报             │  - ❓确认项 → ✅Twin如何代答    │
│  - 「我替你做的决定」清单     │  - 🔧 真实工具调用记录+耗时    │
│  - ⬆️ 高风险待确认问题       │  - 🔁 打回重做记录            │
│  - @插话输入框               │  - 实时进度提示               │
├─────────────────────────────┴──────────────────────────────┤
│  底部产物区：SQL / 数据表 / 图表 / 报告（可切换、可下载）     │
└────────────────────────────────────────────────────────────┘
```

### 7.2 关键交互流程
#### 任务创建流程
1. 用户访问首页，输入数据分析目标
2. 选择Twin和各Agent使用的模型（默认值已预选）
3. 点击「交给Twin」，自动跳转到任务协作页，开始执行

#### 任务执行流程
1. Twin解析目标，派发给对应工具Agent
2. 工具Agent执行，遇到确认门抛给Twin
3. Twin判断风险：低风险自动代答→记录决策清单；高风险升级→通知用户
4. Agent完成产出→Twin验收→不合格打回重做，合格流转到下一个Agent
5. 所有步骤完成→Twin打包交付（结论+决策清单+产物）→通知用户
6. 用户回来查看结果，可回溯全过程，可对代答决策反馈

#### 用户插话流程
1. 用户在输入框@某个Agent输入补充要求
2. 消息即时显示在对话区，编排引擎自动将消息路由到目标Agent
3. 步数预算重置，Agent根据新指令调整执行方向
4. 继续执行直到完成或下一次插话

### 8.3 编排状态机（核心）
多个Agent交替发言，每次只输出一个JSON动作，`turn`表示下一个发言者：
```
┌─────────── turn = twin ───────────┐
│  assign/answer/rework → 工具Agent  │
│  beautify → 样式Agent             │
│  deliver → 用户（已交付，空闲）     │
│  escalate → 用户（待确认，空闲）    │
└────────────────────────────────────┘
┌─────── turn = <toolKey> ───────────┐

## 十、风险与应对
| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| Twin代答准确率不足，用户不信任 | 中 | 高 | 1. 初期保守策略，宁可多升级也不错误代答；2. 所有代答留痕，用户可反馈纠正；3. 持续迭代风险分级矩阵，目标准确率≥90% |
| 人力不足，埋点重写窗口期冲突 | 高 | 中 | 聚焦P0核心功能，非核心功能延后二期，不追求大而全，先跑通最小闭环 |
| 外部平台对接优先级低，后续迁移困难 | 中 | 中 | 先内部验证价值，拿出实际效率提升数据后再争取资源，核心逻辑与平台解耦，不依赖外部平台也能独立使用 |
| 信息孤岛，团队工作内容不统一沉淀 | 中 | 高 | 从数据组试点开始推动标准化GitLab工作空间，赵益达负责团队规范推广，成熟后再全团队推广 |
| LLM长上下文性能问题，长任务失败 | 低 | 中 | 落地四级上下文自动压缩机制，控制token预算，支持断点续跑 |
| SQL生成准确率不足，取数错误 | 中 | 高 | 基于现有nl-mapping-table-sql资产，用真实业务指标/表模板生成SQL，Twin做SQL校对，沉淀正确SQL作为经验包 |
