## 一、项目背景与解决痛点

### 1.1 当前AI协作的三大痛点
当前使用Claude Code、Codex等AI工具进行数据工作时，存在三个核心问题：
1. **认知盲区**：AI会默认一些关键信息（口径、业务规则），用户难以发现隐含错误
2. **确认效率低**：AI每步都要用户确认，提问完备性存疑，用户需要全程盯守
3. **速度不匹配**：AI产出速度远超人工确认速度，用户成为协作瓶颈

### 1.2 协作模式分类
根据「检查能力强弱」和「质量风险高低」，AI协作分为五类模式：
| 模式 | 场景 | 痛点 |
|------|------|------|
| 上下级同岗 | 带新人做熟悉领域 | 培养成本高 |
| 平级同岗 | 同事分工协作 | 沟通成本 |
| 上下级跨岗 | 安排不熟悉领域工作 | 认知盲区、验收难 |
| 平级跨岗 | 找其他岗位专家帮忙 | 沟通门槛、响应慢 |
| 专家咨询 | 请教专业问题 | 信息不对称、确认成本高 |

**ICE 2.0核心解决的是「跨岗协作」和「专家咨询」场景**：让产品、运营等非数据专业角色，也能高质量完成数据分析、报告产出等专业工作，无需全程盯守。

### 1.3 平台演进方向
- ✅ **v1.0**：nl-sql问数能力，单Agent直连用户
- ⭐ **v2.0（本方案）**：Twin Agent数字分身，多Agent协作，「布置完就走开」
- 🔮 **v3.0**：团队级协作平台，经验包共享，向公司Mika/Chatflow迁移

## 二、核心理念：Twin Agent 数字分身架构

### 2.1 一句话定位
**人 + 数字分身（Twin Agent）+ 工具Agent团队** 在同一块任务看板上协作的AI数据工作台，核心链路：
```
User → Twin（代理人）→ 专业化工具Agent团队 → Twin验收交付 → User
```

用户只给目标和最终验收，中间低风险确认项由Twin基于用户画像、工作习惯、业务规则**自动代答**，只有真正高风险决策才升级回来问用户。目标是让用户「布置完任务就走开」，回来直接看结论 + 一份「我替你做的决定」清单。

### 2.2 设计哲学（对齐v1沉淀）
| 原则 | 说明 |
|------|------|
| **统一Agent抽象** | Twin与工具Agent是同一抽象的两种kind，都以`agent.md`为操作手册，共用运行时 |
| **文件为唯一真相源** | 所有对话、SQL、数据、决策都落文件，可diff/grep/回溯，不依赖数据库黑盒 |
| **有界多轮协作** | Twin是唯一编排者，星型拓扑，工具Agent之间不直接对话，防跑飞/防环 |
| **主动性有边界** | Agent只提议不越权，高风险走HITL（人在回路），人保留最终控制权 |
| **降级不阻塞** | 任何依赖不可用时优雅降级，不崩溃、不丢失进度 |
| **全程可回溯** | 每一步都留痕，包括Agent思考过程、Twin代答理由，建立信任 |

## 三、角色模型与职责边界

| 角色 | kind | 定位 | 核心职责 | 边界 |
|------|------|------|----------|------|
| **用户（User）** | - | 目标设定者&最终验收人 | 给目标、审阅最终结果、拍板高风险决策；可随时@插话/私聊Twin | 不盯每一步执行 |
| **Twin Agent（数字分身）** | twin（特权） | 用户代理人/唯一编排者 | 理解意图、任务派发、替用户回答低风险确认项、验收产出、打回重做、升级高风险问题、最终交付 | 只提议不越权；不做具体执行（不写SQL/不排版/不查数）；高风险必问用户 |
| **工具Agent（Tools）** | tool | 各领域专家 | 按SOP完成专业工作：<br>- 数据分析Agent：取数+分析+归因<br>- 样式优化Agent：报告排版美化<br>- AB实验Agent：实验设计+显著性检验<br>- 灰度分析Agent：版本灰度对比<br>- nl-sql Agent：自然语言转SQL<br>- 报告撰写Agent：结构化文档输出<br>...可扩展 | 甲方是Twin不是用户；只对Twin汇报；不替用户拍板；只读操作不修改数据 |

### 3.1 Twin的五个核心能力
1. **角色定义**：基于用户工作空间文件（AGENTS.md/USER.md/CURRENT.md/MEMORY.md等）蒸馏用户画像，理解用户偏好、决策风格、常用口径
2. **工作契约**：明确什么级别的问题可以代答、什么必须升级，遵循风险分级矩阵
3. **反馈闭环**：每个代答决策都记录理由，用户可事后纠正，持续优化判断准确率
4. **记忆管理**：跨任务记忆用户习惯、业务规则、坑点教训，越用越懂用户
5. **定时校准**：定期回顾代答决策，根据用户反馈调整代答策略，避免偏差累积

### 3.2 工具Agent扩展方式
新增工具Agent只需两步：
1. 在`agents/{key}/`目录下放置`agent.md`（操作手册）+ `knowledge/`（领域知识）+ `reference/`（人设档案）
2. 在Agent花名册中登记元信息（key/名称/能力标签/所需工具权限）

运行时自动被Twin识别为可调度团队成员，无需修改核心代码。

## 四、核心机制

### 4.1 确认项代答机制（项目灵魂）
工具Agent按SOP执行时，每个阶段之间天然存在「确认门」（命题澄清、口径确认、是否下钻、SQL校对等），这些确认项抛给Twin后：
- **✅ 低风险确认**（口径选择、维度下钻、SELECT语句校对等）→ Twin基于用户画像+业务常识+口径缺省**直接代答**，任务不中断，并记录到「我替你做的决定」清单
- **⚠️ 高风险/拿不准**（超出目标范围、全表扫描、涉及写操作/外发等）→ Twin**升级回用户**，任务挂起待确认，用户回复后自动续跑

所有代答都留痕可查，交付时完整呈现给用户，用户可以纠正并让Twin学习。

### 4.2 经验包（Experience Package）机制
针对现有Skill过于臃肿、上下文负担重的问题，设计比Skill更细粒度的知识单元——**经验包**：
- **来源**：从历史成功任务（SQL、分析过程、报告、决策）中提取关键要素
- **结构**：需求描述→使用数据→分析过程→结论→坑点
- **调用**：Twin/工具Agent根据任务场景按需检索并注入当前任务空间，不需要把所有知识都塞进system prompt
- **优势**：与固定Skill解耦，灵活轻量，减少上下文浪费，可跨团队共享复用
- **一期落地**：先从周报自动化、常见分析场景（波动归因/对比分析/漏斗分析）开始沉淀

### 4.3 自动进化机制
- **每日进化**：定时归纳当天所有任务中的对话、错误、用户反馈，提炼可复用经验和易错点，写入Agent的`memory/LEARNINGS.md`，自动注入后续任务的system prompt
- **记忆晋升**：自动更新的LEARNINGS.md是附加记忆，核心`agent.md`保持稳定；经过验证的经验经人工审阅后可晋升进核心手册
- **跨任务镜像**：每个Agent的对话和思考过程自动镜像到自身空间，形成跨任务历史，作为进化原料

### 4.4 上下文自动压缩
参考Claude Code的五级级联压缩思想，实现四级渐进式压缩，避免长任务撑爆上下文：
| 级别 | 方法 | 说明 |
|------|------|------|
| L0 | 大结果落盘 | 查询结果自动存文件，上下文只留预览+路径 |
| L1 | 归档旧结果 | 保留最近N条查询结果原文，更旧的替换为文件指针 |
| L2 | LLM摘要中间段 | 用LLM压缩历史对话，保留关键信息 |
| 兜底 | 紧急裁剪 | 上下文溢出时强制压缩后重试 |
- **原则**：system prompt永远不压缩（稳定缓存前缀）；压缩不丢失真实数据，SQL/结果文件完整保留可回溯。

### 4.5 全程可回溯机制
每个任务所有过程完整落盘到任务空间：
```
tasks/{tid}/
  meta.json           # 任务元信息（目标/状态/模型/时间）
  conversation.jsonl  # 全量事件流（所有对话/工具调用/代答/交付）
  decisions.jsonl     # 「我替你做的决定」清单（问题/答案/理由）
  thinking.jsonl      # 每次LLM调用的思考过程（reasoning+原始输出）
  sql/T{n}_*.sql      # 每条真实执行的SQL原文
  data/T{n}_*.json    # kyuubi返回的真实结果
  STATE.md            # 人类可读的状态快照
```
- 所有日志append-only不修改，支持断点续跑、审计复盘
- 前端提供过程产物页，可逐条回看Agent思考过程，把黑盒变白盒

### 4.6 用户随时接管
三种交互通道保证用户「布置完就走开」同时可随时介入：
1. **@插话**：主对话区@Twin或任意工具Agent补充要求/纠偏，插话后自动重置步数预算
2. **回复升级**：Twin升级高风险问题时，用户直接回复即可继续执行
3. **侧栏私聊Twin**：独立side频道，不打断主流程，Twin用口语回答进度/问题，不影响Agent协作

## 五、技术架构（基于Demo验证）
Demo已在`/Users/mi/Desktop/agent-agent-demo`跑通核心链路，技术栈验证可行：

```
浏览器（原生HTML/CSS/JS，零构建，状态优先UI）
     │  SSE流式实时推送对话/工具调用/状态
     ▼
本地Node.js服务（纯内置模块，零第三方依赖）
     ├── HTTP层：路由/静态服务/SSE运行时/插话队列
     ├── 编排引擎：Twin⇄工具Agent有界状态机/断点续跑/自动压缩
     ├── 进化引擎：每日经验归纳/记忆更新
     ├── 集成层：Mify LLM网关/kyuubi真实取数（只读安全）
     ├── 领域层：Agent花名册/任务存储/详情组装
     └── Prompt层：按Agent类型组装system prompt（注入画像/知识/记忆）
     ▼
Mify LLM网关（xiaomi/mimo-v2.5等） + kyuubi网关（真实数据，只读SELECT）
```

### 5.1 编排状态机
核心是一个「谁下一个发言」的有界状态机：
```
┌─────────── turn = twin ───────────┐
│  assign/answer/rework → 工具Agent  │
│  beautify → 样式Agent             │
│  deliver → 用户（已交付，空闲）     │
│  escalate → 用户（待确认，空闲）    │
└────────────────────────────────────┘
┌─────── turn = <toolKey> ───────────┐
│  query → 真实取数，结果回喂继续    │
│  ask → 抛确认项给Twin             │
│  report → 交回结论给Twin验收       │
│  styled → 排版稿交回Twin（样式Agent）│
└────────────────────────────────────┘
turn = null：空闲，等待用户插话/回复
```

### 5.2 安全设计
| 层面 | 措施 |
|------|------|
| LLM密钥 | 只在服务端解析使用，不下发浏览器 |
| 数据查询 | kyuubi白名单只允许SELECT/WITH，硬禁DDL/DML/多语句，从工具层杜绝写操作 |
| 命令执行 | 用execFile+参数数组传SQL，杜绝shell注入 |
| 路径安全 | 静态服务和文件读取做前缀校验，防路径穿越 |
| 降级容错 | 缺key/缺kyuubi返回配置错误而非崩溃，断点续跑不丢进度 |

### 5.3 四层空间设计
源码与工作空间数据分离，对齐现有ICE目录规范：
```
workspace/
├── users/{uid}/twin/    # 用户空间：Twin私有（agent.md/用户画像/代答规则/交付模板/记忆）
├── agents/{key}/        # Agent空间：工具Agent共享（agent.md/领域知识/人设/进化记忆）
├── skills/              # 共享技能包（kyuubi/mify/feishu/nl-mapping等）
└── tasks/{tid}/         # 任务空间：运行时产物（对话/SQL/数据/决策/思考日志）
```

## 八、风险与应对
| 风险 | 影响 | 应对措施 |
|------|------|----------|
| Twin代答准确率不足，用户不信任 | 核心价值受损 | 1. 初期保守策略，宁可多升级也不错误代答；2. 所有代答留痕，用户可纠正反馈；3. 持续迭代风险分级矩阵 |
| 人力不足，埋点重写窗口期冲突 | 进度延迟 | 聚焦核心链路验证，非核心功能延后二期，不追求大而全 |
| 外部平台对接优先级低，迁移困难 | 平台化受阻 | 先内部验证价值，拿出实际效果后再争取资源，不依赖外部平台也能独立使用 |
| 信息孤岛，工作内容不统一沉淀 | Twin效果差 | 从团队规范入手，推动GitLab标准化工作空间，先从数据组试点 |
| LLM长上下文性能问题 | 长任务失败 | 落地四级上下文自动压缩机制，控制token预算 |
