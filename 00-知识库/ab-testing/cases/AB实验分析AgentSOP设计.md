## 一、Agent定位与核心设计原则

### 定位
资深数据分析专家，专注互联网产品AB实验数据分析与放量决策，严格遵循统计学方法给出科学可靠结论。

### 10条核心原则（按优先级排序）
1. **实事求是（最高）**：严禁编造数据，宁可报告不完整，也不可报告不真实
2. **主动建议而非被动等问**：异常时主动推荐下钻，完成后主动提补充分析方向
3. **SOP对齐**：严格对齐团队SOP流程、报告结构、决策标准
4. **数据驱动**：所有结论有真实数据支撑
5. **结论先行**：先给结论，再展开细节
6. **智能引导**：默认自动执行，必要时才暂停等人工
7. **统计严谨**：正确选检验方法，p值仅供参考，结合效应量和业务价值综合判断
8. **可追溯**：每个结论都能追溯到数据源和计算方法
9. **自我进化**：通过反馈闭环持续优化知识库
10. **上下文精简**：SQL原始数据提取关键值后结构化总结，不保留冗余输出

## 二、六阶段SOP标准流程（严格顺序，不可跳步）

### Phase 0 · 信息收集 📝
**目标**：收集分析所需全部输入，8项必填缺一不可

| # | 信息 | 说明 | 示例 |
|---|---|---|---|
| 1 | 实验ID | 实验平台唯一标识 | `12345` |
| 2 | 关键变量 | 实验组与对照组核心差异 | "首页新增视频入口" |
| 3 | 直接指标 | 被变量直接影响的指标 | "视频入口渗透率/点击率" |
| 4 | 因果链路 | 变量→直接指标→核心指标的预期路径 | "新增入口→渗透率↑→人均VV↑→人均时长↑" |
| 5 | 放量阶段 | 当前实验流量阶段 | "10%阶段" |
| 6 | AB数据日期 | 实验期起止 | "4/20~4/25" |
| 7 | AA数据日期 | 可留空（=无AA） | "4/13~4/19" |
| 8 | AA实验状态 | 必须明确确认 | "有AA"/"无AA" |

**选填增强信息**（提供后解锁AI自主下钻）：
- 变量所在页面/模块 → 匹配页面结构知识库
- 变量影响机制 → 预生成下钻假设

**交互模式**：
- 默认生成`tmp_experiment_input.md`模板让用户填写
- 支持直接对话输入，仅追问缺失项
- 收到增强信息立即反馈匹配结果

### Phase 1 · 数据获取 🔄
**目标**：多数据源获取数据，输出标准化`ExperimentData`结构

#### 数据获取优先级
| 优先级 | 数据源 | 说明 |
|---|---|---|
| 1（默认） | Kyuubi SQL查询 | 服务端已配置，直接传SELECT SQL |
| 2 | 文件上传 | CSV/Excel解析 |
| 3 | 飞书表格 | Feishu Skill读取 |

#### SQL查询规范
1. **枚举字段先探查**：`SELECT x, COUNT(*) GROUP BY x ORDER BY cnt DESC LIMIT 20`，不猜测枚举值
2. **实验过滤条件**：从埋点配置读取实验ID字段（eid/exp_id/new_eid），不要硬编码
3. **单日验证→全量执行**：先用最后一天验证SQL逻辑，再跑全量范围
4. **日均口径强制**：先按天聚合→再对天取AVG，**禁止**多天直接聚合（跨天去重会偏差）
   ```sql
   WITH daily AS (
     SELECT date, ... FROM ... WHERE ... GROUP BY date
   )
   SELECT AVG(metric) FROM daily
   ```
5. **SQL展示确认**：向用户展示最终SQL，标准模板直接执行，自定义SQL需用户确认

#### 需获取的数据类别
| 类别 | 日期范围 | 说明 |
|---|---|---|
| 核心指标 | AB期间 | 规模/消费/收入三类护栏指标 |
| 直接指标 | AB期间 | 验证变量是否生效 |
| AA基线 | AB前7天 | 扣除基线波动 |
| SRM校验 | AB期间 | 样本比例偏差检验 |

#### 标准化数据结构
```
experiment_id, phase, date_range{start,end}, aa_date_range?{start,end}
groups: { control{name,sample_size}, treatment{name,sample_size} }
core_metrics: [{metric_name, category, type, control_value, treatment_value, ...}]
direct_metrics: [{metric_name, control_value, treatment_value, ...}]
aa_metrics?: [{metric_name, control_value, treatment_value}]
```

### Phase 2 · 数据校验 ✅
**目标**：4项数据质量门控，保证结论可信

| # | 校验项 | 方法 | 通过标准 | 不通过处理 |
|---|---|---|---|---|
| 1 | 数据完整性 | 核心字段缺失/空值检查 | 无关键缺失 | ⚠️ 暂停，提示补充 |
| 2 | SRM样本比例偏差 | 卡方检验 | p > 0.05 | ⚠️ 暂停，警告分流异常 |
| 3 | AA波动 | 对照AA期阈值校验 | 在阈值内 | 按规则扣除基线偏移 |
| 4 | 统计功效 | 样本量计算 | ≥ 80% | ⚠️ 提示功效不足，建议继续积累 |

**暂停规则**：仅数据质量可能导致结论不可信时暂停，明确说明问题/影响/解决方案，用户确认后继续。

---

### Phase 3 · 四步核心分析 🔥
**严格按Step1→2→3→4顺序执行**

#### Step 1 · 核心指标分析（护栏指标）
遍历核心指标配置，逐个指标计算：
- 变化率 = (实验组 - 对照组) / 对照组 × 100%
- 检验方法按指标类型选型：
  - `ratio`（比率型，如渗透率/留存率）→ 双样本Z检验
  - `continuous`（连续型，如时长/VV）→ Welch's t检验
- 计算：p值、95%置信区间、效应量（Cohen's d）
- 判定每个指标方向，组合判定整体类型（全部正向/部分正向/此消彼长/出现下降）

#### Step 2 · 直接指标分析
验证关键变量是否真的影响了预期的直接指标：
- 计算变化方向和幅度
- 判断是否符合因果链路预期：方向一致/相反/无显著变化

#### Step 3 · 因果链路验证（关键）
验证「关键变量→直接指标变化→核心指标变化」逻辑链：
1. ≥1个直接指标显著正向响应 → ✅
2. ≥60%核心指标正向/平稳 → ✅
3. AA超阈值则按公式扣除基线后重新判定 → ✅
- 任一步失败 → 因果链断裂，需排查埋点/外部干扰/延长观察期

#### Step 4 · 异常下钻分析（条件触发）
**触发条件**：核心指标下降 OR 直接指标不符合预期

**下钻流程（必须用户确认假设）**：
1. **假设生成**：匹配下钻知识库，结合页面结构生成Top3假设
2. **假设确认**：向用户展示假设列表，用户确认/补充后才执行，**不得擅自跑SQL**
3. **维度选择**：优先使用有现成SQL模板的下钻维度
4. **验证执行**：按优先级逐个验证，**必须验证≥3个竞争假设，不得单一归因**
5. **结果输出**：量化归因（每个假设解释总变化的X%）+ 根因定位 + 业务建议

---

### Phase 4 · 决策建议 🎯
匹配决策矩阵给出放量/暂停/回滚建议：

| 场景 | 核心指标 | 直接指标 | AI建议 | 决策权 |
|---|---|---|---|---|
| S1 | 全部正向 | 符合预期 | ✅ 建议继续放量/全量 | 人工确认 |
| S2 | 部分正向+平稳 | 符合预期 | ✅ 建议继续放量 | 人工确认 |
| **S3** | **此消彼长** ⚠️ | 任意 | ⚠️ 需人工评估权衡 | **必须人工决策** |
| S4 | 出现下降 | 任意 | ❌ 暂不放量，先下钻 | 人工确认 |
| S5 | 任意 | 不符合预期 | ❌ 暂不放量，变量可能未生效 | 人工确认 |

**S3此消彼长强制暂停**：
输出`status='PAUSED_WAITING_HUMAN_DECISION'`，展示评估维度等待用户拍板。

同时根据当前放量阶段，给出下一阶段流量/预期DAU/跳过/终止条件建议。

---

### Phase 5 · 报告生成 📊
严格按模板生成5节结构报告：
1. **结论先行**：发布建议+核心理由+关键数据摘要+风险提示
2. **实验概览**：实验ID/关键变量/因果链路/阶段/时间范围
3. **核心数据表现**：三类指标热力图+显著性+护栏状态+组合判定
4. **直接指标&因果链路**：符合性表+三步验证面板
5. **下钻分析**（条件显示）：触发原因+假设列表+根因+建议
6. **附录**：4项校验结果/SRM/功效/AA基线/检验方法说明

**角色自适应**：
- PM视图（默认）：折叠统计细节，突出业务价值与决策
- 数据分析师视图：完整展开p值/置信区间/效应量/计算细节

---

### 反馈闭环（自我进化）
报告输出后询问满意度：
1. ✅ 满意 → 写入good_cases，提炼经验回写下钻/规则/模板知识库（用户确认后回写）
2. ⚠️/❌ 不满意 → 追问原因修正，写入bad_cases，提炼避坑要点回写知识库

**回写规则**：每条标注来源，回写前必须向用户展示修改内容并获确认。

## 三、知识库驱动设计
Agent行为完全由结构化知识库驱动：

| 知识资源 | 用途 |
|---|---|
| 数据字典 | 表别名、字段定义、查询tips、实验ID字段配置 |
| 核心指标配置 | 指标名称/类型/分类/维度/显著性规则 |
| 埋点库（SQLite） | 事件埋点查询，支持直接指标动态生成SQL |
| AA规则 | AA时长/阈值/免AA条件/异常基线扣除公式 |
| 放量阶段规则 | 各阶段流量/最短时长/跳过/终止条件 |
| 决策矩阵 | 指标组合→决策映射、人工决策标记 |
| 下钻知识库 | 场景模式→假设→下钻维度映射 |
| 页面结构模型 | 产品页面元素/卡片/流转关系，辅助下钻 |
| SQL模板库 | 各业务线指标查询模板、下钻维度模板 |
| 案例经验库 | good/bad历史案例，持续积累 |
| 报告模板 | 固定报告结构+角色自适应规则 |

## 四、子Agent派单规则
| 触发场景 | 派给Agent | Prompt要点 |
|---|---|---|
| 跨业务线/大盘对照取数 | data-analysis | 业务线/口径/时间窗/期望产物 |
| 异动多假设量化归因 | wave-attribution | 异常点/初步下钻/剩余假设 |
| 报告沉淀到飞书文档 | know | 标题/位置/是否建索引 |
| 火山平台实验对照 | volcano-abtest | media/exp_id/日期 |

**派单约束**：
- 子Agent无对话通道，prompt必须自包含所有上下文
- Phase0信息收集、Phase4决策（尤其S3此消彼长）不能派单，主Agent必须亲自完成
- 子Agent不能再spawn，>2分钟任务走后台运行
- 最终统计判定和决策由主Agent拍板，子Agent输出只作为证据

## 五、与灰度发布Agent的区别
| 维度 | AB实验Agent | 灰度发布Agent |
|---|---|---|
| 分组依据 | 实验桶（随机分流） | app_version（版本号） |
| AA校验 | **必须AA校验**通过才进AB | 大DAU产品通常免AA |
| 核心目标 | 验证变量因果效应，显著正向才上线 | 护栏监控，核心指标不跌即可放量 |
| 信息收集项 | 8项（含实验ID/因果链路） | 5项（版本号/变量/直接指标/阶段） |
| 决策重点 | 实验变量是否真的有效 | 新版本是否引入负向问题 |
| 下钻触发 | 指标下跌/直接指标不符合预期 | 同左+此消彼长必须人工 |

## 六、设计亮点复用要点
1. **SOP分阶段门控**：每个阶段有明确的进入条件和输出，质量不达标不进入下一阶段
2. **知识库完全外置**：业务规则、指标定义、SQL模板、决策逻辑全在配置文件，不需要改Prompt就能调整
3. **人机边界清晰**：什么场景AI可以自动决策，什么场景必须暂停等人工，规则明确
4. **假设验证不武断**：下钻必须验证≥3个竞争假设，禁止单一归因
5. **SQL口径强制规范**：日均聚合口径、枚举值探查、单日验证这些工程细节写进SOP，避免常见错误
6. **反馈闭环持续进化**：每次分析的经验教训自动沉淀回知识库，越用越准
7. **三层记忆分离**：用户偏好→Agent记忆→任务状态，跨任务复用经验但不冗余存储单次数据
