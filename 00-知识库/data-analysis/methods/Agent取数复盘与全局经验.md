## 一句话结论

Agent 取数不能只理解为“写 SQL 并执行”，而要固定成一条闭环：**需求契约 → 口径校验 → 探针验证 → 分批执行 → 合并校验 → 业务解释 → 资产沉淀**。这条闭环跑通后，一次取数才会变成后续 `nl-sql` 可复用的范式。

## 两类取数模式

### 1. 探索型取数：先搭地图，再下钻

适合“帮我看一下某业务现状”“做流量地图”“找结构性问题”这类需求。

执行顺序：

1. **收敛问题边界**：确认业务域、时间范围、数据源、是否只看某端、是否需要过滤实验组。
2. **搭一级框架**：先产出大盘、模块、入口、核心行为、留存等骨架，不要一开始陷入单点明细。
3. **小窗口探查**：优先 1 天或 7 天，避免还没校验口径就跑大查询。
4. **模块化 SQL**：每个问题单独成 SQL，命名按 `01-主题.sql`，结果按同名 JSON 归档。
5. **失败分级**：区分阻塞失败、可重跑失败、非核心透视失败；不要把“部分查询失败”包装成全量成功。
6. **业务解释优先**：结果汇总要先回答结构、机会、风险，再附表格。

### 2. 口径型取数：先复现，再扩展

适合“按官方口径取某指标”“给我 3 个月分天数据”“用于汇报/看板”的需求。

执行顺序：

1. **锁定口径来源**：飞书文档、数仓口径、历史 SQL、看板截图必须记录链接与时间。
2. **单日探针校验**：先取文档中的校验日，要求关键指标完全一致。
3. **方言等价改写**：如 `rlike` 改 `regexp_like`，改写后必须再次探针校验。
4. **性能优化**：确认子查询已按 `(date,did)` 去重后，才允许 `count(distinct did)` 改 `count(*)`。
5. **分批执行**：季度级千万 did 查询默认按月分批；半年以上按周或半月。
6. **完整性校验**：合并后检查行数、日期连续性、空值、异常峰谷，再输出 CSV/XLSX。

## Agent 取数交付物标准

每次取数默认形成一个独立目录，例如 `03-数据分析/数据需求/MMDD需求名/`：

| 文件 | 是否必需 | 作用 |
|---|---|---|
| `00-需求与口径说明.md` | 口径型必需，探索型建议 | 记录需求、数据源、时间范围、口径来源、过滤条件 |
| `NN-主题.sql` | 必需 | 保留可复跑 SQL，按分析模块拆分 |
| `查询结果/*.json` | 必需 | 保存原始查询结果，避免只留下二次加工结论 |
| `_execution_summary.json` | 建议 | 记录每条 SQL 的成功/失败、耗时、行列数、错误信息 |
| `03-结果汇总.md` 或 `查询结果汇总.md` | 必需 | 面向业务的结论、表格、风险说明 |
| `.csv` / `.xlsx` | 交付型必需 | 面向业务方、看板或后续加工的数据明细 |
| `.html` / 飞书文档 | 正式汇报时必需 | 可视化展示与协作传播 |
| 知识库模板/踩坑 | 可复用时必需 | 回写到 `00-知识库/data-analysis/` 或 `00-知识库/数据资产/` |

## 全局经验

### 1. 先让 Agent 交付“取数计划”，再跑数

取数前必须先生成简短计划：业务问题、指标口径、时间范围、数据源、SQL 模块、校验方式、输出格式。用户只需要确认关键边界，不需要审完整 SQL。

### 2. 探针是高质量取数的分水岭

没有探针直接跑全量，容易出现两类浪费：一是口径错了导致全量结果不可用，二是 SQL 性能差导致反复 OOM。官方指标尤其必须先复现一个已知校验值。

### 3. 执行失败要分类，不要笼统重试

| 失败类型 | 判断方式 | 处理策略 |
|---|---|---|
| SQL 方言/语法 | 报错指向 `rlike`、中文别名、`HAVING` 别名等 | 修改 SQL 后重新探针 |
| 资源超限 | `exceeded distributed user memory limit` | 等价优化、缩小时间窗、按月/周分批 |
| 临时网络/worker | `Error fetching`、`Connection aborted` | SQL 不变，降低并发后重跑 |
| 权限/workspace | `Token belongs to...` | 按报错切换 workspace，不改 SQL |
| 业务异常 | 人均时长极高、UV 异常低、趋势断崖 | 回查埋点覆盖、事件选择、过滤条件 |

### 4. Presto 默认优先，但要带着约束使用

Presto 适合 ad-hoc 和大多数 iceberg/hive 查询，但季度级千万 did 聚合容易碰到 240GB 内存上限。默认策略是：**presto → 等价优化 → 按时间分批 → spark 兜底**，不要一遇到慢/错就直接切 spark。

### 5. 宽口径探索和精确指标复现不能混用验收标准

流量地图这类探索型任务，可以接受非核心透视表失败后先交付主体结构，但必须标注失败项；官方 DAU/时长这类口径型任务，任何关键指标未校验一致都不能进入全量取数。

### 6. 每次取数都要变成下一次的资产

完成业务交付后至少沉淀三类资产之一：SQL 模板、口径说明、踩坑记录。若能覆盖高频需求，再补成 `nl-sql` few-shot：**业务问题 → 口径 → SQL → 校验值 → 输出格式 → badcase**。

## 给用户的高质量提问模板

下次让 Agent 取数时，可以直接这样描述：

```text
请按 Agent 取数协议执行：
1. 业务问题：我要回答什么决策/分析问题？
2. 数据范围：端、版本、用户范围、时间范围。
3. 指标口径：口径来源链接/历史 SQL/看板截图；如果没有，请先探查并标注推断。
4. 验收标准：是否需要和某个校验日/看板值一致？允许误差是多少？
5. 输出格式：只要 SQL、结果表、CSV/XLSX、Markdown 结论、HTML/飞书文档中的哪些？
6. 沉淀要求：是否需要同步沉淀到 SQL 模板、口径字典、badcase/few-shot？
```

## Agent 自检清单

- [ ] 已读取工作区入口文件和对应知识域 README
- [ ] 已确认需求类型：探索型 / 口径型 / 交付型 / 看板型
- [ ] 已记录数据源、时间范围、过滤条件和口径来源
- [ ] 已先跑单日/小窗口探针
- [ ] 已对方言改写和性能优化做等价性说明
- [ ] 已保存原始 JSON，不只保存加工后的表格
- [ ] 已校验日期连续性、行数、空值和异常值
- [ ] 已将失败项、风险和未覆盖范围写入结果汇总
- [ ] 已输出业务结论，而不是只输出 SQL/数据表
- [ ] 可复用内容已回写知识库或数据资产层
