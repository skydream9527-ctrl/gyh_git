## 一、为什么你做的分析没人看（一段扎心的论述）

数据分析师/数据PM最常遇到的灵魂拷问：

> "我跑了这么多数据，为什么业务还是不按我说的做？"
> "这个指标涨了5%，然后呢？能说明什么？"
> "你这分析报告写了20页，结论到底是什么？"

这些问题的本质不是你SQL写得不好、图表做得不漂亮，而是**你给的是数据，不是答案；你给的是现象，不是行动建议**。

业务方真正想知道的永远是三个问题：
1. **发生了什么？**（指标异动、问题定位）
2. **为什么会发生？**（根因分析、驱动因素）
3. **我该怎么做？**（行动建议、预期效果）

绝大多数分析停留在第一层，偶尔到第二层，极少能给出第三层的可落地建议。数据分析要解决的就是：**把原始数据转化为业务决策可直接使用的洞察和行动建议。**

> 当你能做AB实验验证因果关系时，请优先做。详见 [../ab-testing/](../ab-testing/)。当实验做不了需要从观察数据推导因果时，用 [../causal-inference/](../causal-inference/) 的方法。数据分析是这些方法的基础载体。

## 二、数据分析的三层能力模型

把整个数据分析能力简化成三层，绝大多数人停留在第一层：

```
┌───────────────────────────────────────────────────────────┐
│  3. 洞察层（Insight）    为什么？怎么做？                  │ ← 根因定位、因果推断、行动建议
│  2. 分析层（Analysis）   拆维度、做对比、找异常            │ ← 漏斗/留存/分群/归因/对比
│  1. 数据层（Data）       取数、清洗、算指标、做报表        │ ← SQL/Python/可视化工具
└───────────────────────────────────────────────────────────┘
            业务价值↑                           可替代性↓
```

**重点**：
- 第1层是基础，必须扎实，但价值有限，容易被AI/自动化工具替代
- 第2层是核心，掌握各种分析框架和方法，能把数据拆出信息量
- 第3层是壁垒，理解业务、能判断因果、能给出可落地的建议，这才是不可替代的

**记一条铁律**：用数据层的勤奋掩盖分析层和洞察层的懒惰，是新手最容易犯的错误。跑100个SQL不如想清楚1个业务问题。

## 三、核心分析框架（八板斧）

80%的业务问题都能用这八个框架解决：

| 框架 | 解决什么问题 | 核心逻辑 |
|---|---|---|
| **对比分析** | 这个数算高还是低？是好还是坏？ | 和自己比（环比/同比/目标）、和别人比（行业/竞品/同类） |
| **漏斗分析** | 用户在哪一步流失了？ | 拆解用户路径，每一步的转化率，找到流失瓶颈 |
| **留存分析** | 用户为什么不来了？ | 次日/7日/30日留存，分群看不同用户的留存差异 |
| **分群分析** | 是整体问题还是部分用户问题？ | 按新老/渠道/活跃度/行为特征切分，找到问题核心群体 |
| **归因分析** | 指标变动是哪些因素导致的？ | 按维度拆解贡献度，量化每个因素的影响大小 |
| **趋势分析** | 是短期波动还是长期趋势？ | 看时间序列，排除噪音，识别拐点和规律 |
| **相关性分析** | 哪些因素和结果相关？ | 找相关变量，但记住相关≠因果 |
| **A/B测试** | 这个改动真的有效吗？ | 随机对照，量化因果效应（详见ab-testing域） |

详见 [methods/数据分析方法论.md](./methods/数据分析方法论.md)。

## 四、指标异动排查SOP（最常用的实战流程）

老板突然说"今天DAU跌了10%，怎么回事？"，按这个流程走，不会慌：

```
1. 确认数据准确性
   ├─ 数据有没有延迟？
   ├─ 埋点有没有问题？
   └─ 统计口径有没有变？
   
2. 确认问题真实性
   ├─ 是真跌了还是正常波动？（和过去7天/30天同比）
   ├─ 是核心指标还是护栏指标？
   └─ 影响范围多大？（绝对值影响多少用户/收入）

3. 维度拆解定位
   ├─ 按平台拆：iOS/Android/Web？
   ├─ 按渠道拆：自然/推广/哪个渠道跌了？
   ├─ 按新老用户拆：新用户少了还是老用户不来了？
   ├─ 按地域拆：是全量还是某个地区？
   └─ 按版本拆：是不是新版本发布带来的问题？

4. 根因假设验证
   ├─ 是外部因素吗？（节假日/竞品大促/舆情/政策）
   ├─ 是产品改动吗？（刚上线什么功能？哪个模块改了？）
   ├─ 是运营活动吗？（活动结束了？推送停了？）
   └─ 是技术问题吗？（崩溃/卡顿/加载慢/服务故障）

5. 给出结论和行动建议
   ├─ 原因是什么？影响多大？会持续多久？
   ├─ 建议做什么？不做会怎么样？
   └─ 后续怎么监控？怎么避免再发生？
```

## 五、最常见的五个分析陷阱（先记住，后细学）

1. **辛普森悖论**
   - 现象：整体涨了，但每个细分群体都跌了；或者反过来
   - 后果：结论方向性错误
   - 解法：一定要拆维度看，不能只看总体

2. **相关当因果**
   - 现象："用了A功能的用户留存高，所以我们要推A功能"
   - 后果：推了之后发现留存根本没涨，因为本来就是高活用户才用A功能
   - 解法：要么做AB实验，要么用因果推断方法（见causal-inference域）验证

3. **幸存者偏差**
   - 现象："调研了我们的付费用户，他们都喜欢XX功能"
   - 后果：你没看到因为XX功能走了的用户
   - 解法：看全量用户，不要只看留下来的/成功的样本

4. **忽略基数**
   - 现象："这个功能转化率涨了100%！" （从0.1%涨到0.2%）
   - 后果：高估重要性，资源投入错地方
   - 解法：永远同时看相对变化和绝对影响

5. **过度拟合噪音**
   - 现象：对着一天的波动找了半天原因，结果第二天自己回去了
   - 后果：浪费时间在随机波动上
   - 解法：先看是不是统计显著，有没有持续几天，再找原因

## 六、AI时代的数据分析：怎么和AI协作

AI已经能写SQL、做图表、甚至写初步分析了，但现在还替代不了好的分析师。核心协作模式：

| 环节 | 人做 | AI做 |
|---|---|---|
| 问题定义 | ✅ 明确业务问题，确定分析目标 | ❌ |
| 指标设计 | ✅ 定义指标口径，选择分析框架 | ❌ |
| 取数清洗 | ❌ （SQL都可以让AI写） | ✅ 写SQL、清洗数据、算指标 |
| 初步分析 | ⚠️ 定拆解维度 | ✅ 跑维度拆解，做初步可视化 |
| 根因判断 | ✅ 结合业务知识判断真正原因 | ❌ （AI只会给可能的原因） |
| 洞察建议 | ✅ 给出可落地的行动建议 | ⚠️ 辅助生成建议框架 |
| 报告撰写 | ✅ 把控结论和逻辑 | ✅ 辅助生成图表、整理文字 |

用好 [templates/数据分析Prompt模板.md](./templates/数据分析Prompt模板.md)，能让你的分析效率提升3-5倍。

## 七、推荐学习路径

### 入门（1-2周可上手）
1. 本OVERVIEW建立分析心智
2. [methods/数据分析方法论.md](./methods/数据分析方法论.md) 掌握八大分析框架
3. 学会SQL：能写多表关联、窗口函数、分组聚合
4. 练手：拿一个真实指标异动问题，按上面的SOP走一遍

### 进阶（1-2个月）
5. 学会Python Pandas做更灵活的分析
6. 掌握常见业务指标体系：DAU/留存/时长/转化/ARPU
7. 学习AB测试基础（见ab-testing域）
8. 复盘3-5个你做过的分析，看哪里停在了数据层没到洞察层

### 高阶（按需）
9. 因果推断方法（见causal-inference域）：解决相关性不等于因果的问题
10. 指标体系设计：怎么从0到1搭建一套业务监控指标
11. 自动化分析：怎么用脚本/Agent自动做异动归因和报告生成
12. 数据可视化：怎么用图表讲故事，让业务方一眼看懂

## 九、心法

如果只能记三句话：

1. **先问为什么做，再想怎么做**——接到需求先问清楚业务问题是什么，要做什么决策，别上来就跑SQL。
2. **对比才有意义**——单个数字没有任何意义，所有判断都来自对比（和目标比、和过去比、和对照组比）。
3. **结论先行，用数据支撑**——报告第一页就写结论和建议，后面再放数据和论证过程，没人愿意翻20页找结论。

下次做分析前，Checklist：
1. 我要回答的业务问题是什么？这个分析支撑什么决策？
2. 核心指标是什么？对比基准是什么？
3. 我有没有拆到最小维度定位问题？
4. 相关性能不能支撑因果结论？要不要做实验？
5. 我的行动建议是什么？业务方看完知道该做什么吗？

回答不出来，先别急着取数。
