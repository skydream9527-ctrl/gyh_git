# nl-sql 取数的交互设计：从"一句话出 SQL"到"协作式取数"

> 创建时间：2026-06-30
> 标签：nl-sql、AI 交互设计、数据产品、ICE Workbench
> 关联：[AI-agent时代交互方式变化思考](2026-06-30-AI-agent时代交互方式变化思考.md) · [AI交互设计的反模式与正确方向](2026-06-30-AI交互设计的反模式与正确方向.md)

**核心判断**：「一句话出 SQL」是 nl-sql 最诱人的反模式。取数的真实难点不是把自然语言翻译成 SQL，而是消解意图歧义、约束数据范围、验证结果可信——这三件事都需要多轮协作，压成单次输入输出必然失败。

---

## 一、为什么「一句话出 SQL」是反模式

Q3 目标是 nl-sql 覆盖 80% 日常取数、准确率 90%+（来源：Q3 OKR，见 [data-product-workspace/MEMORY.md](../../project/data-product-workspace/MEMORY.md)）。直觉的产品形态是：一个输入框，用户敲「上周浏览器推荐流人均消费时长」，系统出 SQL 出数。

这个形态诱人但危险，它和 [AI 交互设计的反模式](2026-06-30-AI交互设计的反模式与正确方向.md) 同构：把一个本质上是**多轮协作**的过程，压成**单次输入 → 单次输出**。

| 取数的真实步骤 | 「一句话出 SQL」的假设 | 实际发生的事 |
|---|---|---|
| 意图消歧 | 用户说清楚了 | 「上周」=自然周还是滚动 7 天？「推荐流」=首页推荐还是含二级？「消费时长」=前台时长还是有效时长？ |
| 范围约束 | 系统知道查哪些表 | 同名指标跨表存在、口径不同；不约束就靠猜 |
| 结果验证 | 出数即对 | SQL 跑通 ≠ 口径正确；数字异常时用户分不清是数据问题还是口径问题 |

单次交互里这三步都被跳过，错误被推迟到「用户看到数字觉得不对」才暴露——而那时用户已经不知道错在哪。

## 二、最贵的代价：错误后置 + 不可归因

最坏的不是「答错」，而是「答错了但看起来合理」。把「有效时长」算成「前台时长」的 SQL，产出一个看起来正常的数字，用户拿去汇报，错到下游才被发现。这比「系统说它不会」伤害大一个量级——它透支的是信任，而 nl-sql 这种场景信任一旦透支就回不来（用户会退回手写 SQL）。

> **推断**：nl-sql 的 90% 准确率目标，若靠「一句话出 SQL」形态，大概率卡在 60-70%——不是模型不行，是交互形态把消歧成本全推给了用户事后核验。

## 三、正确方向：协作式取数工作流

把取数当成 Agent 与用户的**协作流程**，每一步可确认、可回退：

```
1. 意图澄清   Agent 反问关键歧义（时间口径 / 指标定义 / 范围），用户点选而非打字
2. 范围确认   Agent 给出候选表 + 字段 + 口径，用户确认或修正
3. SQL 生成   生成可读 SQL，附口径说明
4. 结果自检   Agent 主动跑合理性校验（趋势 / 分布 / 与历史对比），异常先标红
5. 反馈沉淀   用户确认 → 口径入库；纠正 → 写回调优记录
```

关键设计原则（呼应 [AI-agent时代交互方式变化](2026-06-30-AI-agent时代交互方式变化思考.md)）：

- **不要二元反馈（√/×），要具体指向**：用户不该只点「不对」，而要点「错在时间口径」或「错在指标定义」。
- **Agent 主动暴露不确定性**：与其猜对，不如明说「这里我用了有效时长口径，如果不是请纠正」。
- **把口径沉淀成资产**：每次确认的口径进知识库，下次同类问题直接复用——这是准确率从 70% 到 90% 的真正杠杆，不是模型变强，是歧义被历史决策消解。

## 四、落到 ICE Workbench 的具体设计

| 模块 | 单次交互形态 | 协作式改造 |
|---|---|---|
| 输入 | 单输入框 | 输入框 + Agent 实时反问卡片（歧义点可点选） |
| 表选择 | 系统自动猜 | 候选表列表 + 口径预览，用户一键确认 |
| SQL | 直接出 | SQL + 口径说明 +「我用了这些假设」清单 |
| 结果 | 出数即止 | 结果 + 合理性校验（与上周 / 同期对比）+ 异常标红 |
| 反馈 | 无 | 确认 → 口径入库；纠正 → 写回 [knowledge-base/](../../knowledge-base/) 调优记录 |

> 这套设计与 ICE Workbench 已有的「任务工坊 3-Step」「经验卡片审批闭环」同构——任务不是单次执行，是「创建 → 确认 → 沉淀」的流程。nl-sql 应复用这套骨架，而不是另起一个「输入框 → SQL」的单页。

## 五、待验证假设

- [ ] **推断**：协作式取数的前置澄清会让单次取数变慢，但准确率提升带来的返工减少使总耗时更低——需用 ICE Workbench 用量数据验证。
- [ ] **推断**：80% 日常取数的歧义集中在「时间口径」和「指标定义」两类，先解决这两类的自动反问，准确率提升最大——需采样日常取数需求归类。
- [ ] **风险**：反问过多会让用户嫌烦——反问必须限定在「高歧义 + 高频」点，低歧义直接出 + 事后可纠。

## 六、一句话结论

nl-sql 的产品壁垒不在「翻译 SQL」的模型能力，在「把取数歧义变成可协作流程」的交互设计——这是数据产品经理能赢的牌，因为模型厂商不会替你理解你的指标口径。
