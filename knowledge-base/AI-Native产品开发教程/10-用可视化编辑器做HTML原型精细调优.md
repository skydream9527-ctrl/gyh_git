# 【教程】用可视化编辑器做HTML原型精细调优

## 做这件事的目的

- 把 AI 先生成出来的 HTML 原型，从“能看、能点”继续打磨到“更像正式演示稿”。
- 用 `Cursor Browser Visual Editor` 或 `Codex 内置浏览器` 缩短“看效果 → 提修改 → 落代码 → 再看效果”的往返时间。
- 在不脱离现有 `Spec → HTML 原型 → PRD` 流程的前提下，完成最后一轮视觉和交互精修。

## 先说清楚：这一步适合改什么，不适合改什么

### 适合在可视化编辑器里精修的内容

- 间距、字号、圆角、颜色、阴影、对齐
- 区块顺序、按钮位置、栅格和布局
- 文案长度带来的视觉拥挤问题
- 某个状态帧里某一块 UI 的细节表现

### 不适合直接在这一步拍脑袋改掉的内容

- 新增业务流程
- 修改页面职责
- 修改字段协议、状态枚举、验收标准
- 明显会影响 Spec 和 PRD 结论的交互变化

<callout emoji="🎁">
如果你改的是后面这类内容，请先回到原型 Spec，再同步 HTML 原型。项目里已经明确要求修改顺序是 `原型 Spec → HTML 原型 → PRD`，不要只改下游展示稿。
</callout>

## 你需要准备什么

- 已经生成好的 HTML 原型
- 已安装并可正常使用的 Cursor 或 Codex
- 如果你用的是 Cursor，版本建议为 `2.2` 或更新版本
- 当前项目仓库：`/Users/mi/Documents/AI_tutor`
- 一个你准备精修的页面 URL



## 背景说明

Cursor 在 `2025-12-10` 的 `2.2` 更新里正式加入了 Browser 的布局和样式编辑能力，`2025-12-11` 又发布了单独的 Visual Editor 说明。

现在这套方式不只适用于 Cursor，Codex 的内置浏览器也可以承接同一类“打开页面 → 点选元素 → 提修改 → 落代码”的精修流程。  
无论你用哪个工具，这里重点都是四件事：

- 可以直接拖拽重排页面结构
- 可以在侧栏里调布局和样式
- 可以点选元素后直接用自然语言描述修改
- 改完后可以让 Agent 把视觉修改真正落实到代码里  
下面这篇教程，就是把这些能力套进当前这套 HTML 原型工作流。

## 操作步骤

### 第一步：先把本地原型跑起来

你现在在做：启动一个本地静态服务，让 `Cursor Browser` 或 `Codex 内置浏览器` 都能稳定打开 HTML 原型。

```Bash
cd /Users/mi/Documents/AI_tutor
python3 -m http.server 8000
```

成功判定：

- 终端里出现 `Serving HTTP on ... port 8000`
- 浏览器里能打开 `http://localhost:8000/index.html`  
如果 `8000` 端口被占用，可以换一个端口，例如：

```Bash
cd /Users/mi/Documents/AI_tutor
python3 -m http.server 8080
```

成功判定：

- 终端正常启动，没有报端口占用错误
- 你后续打开的地址改成 `http://localhost:8080/...`

### 第二步：确认这次要精修的是哪一个页面

你现在在做：缩小本轮修改范围，避免一上来就在总入口里到处点，最后不知道改了什么。  
建议先只选一个页面章节，比如：

- `http://localhost:8000/V1.2任务页/3.1今日视图默认态.html`
- `http://localhost:8000/V1.0登陆与对话/3.2主页面构成与账号管理.html`  
成功判定：
- 你能明确说出“这次只改哪一页”
- 你知道这页对应哪份 Spec 章节

### 第三步：在 Cursor 或 Codex 里打开页面，并进入可视化编辑视角

你现在在做：把页面放进你当前使用的工具里，这样后面的视觉调优和代码修改会在同一个工作区完成。  
操作方式：

1. 在 Cursor 或 Codex 里打开当前仓库 `/Users/mi/Documents/AI_tutor`
2. 打开 Browser 页签、内置浏览器预览，或 Codex 内置浏览器
3. 输入你刚才确认好的本地页面地址，例如 `http://localhost:8000/V1.2任务页/3.1今日视图默认态.html`
4. 切到可视化编辑相关界面  
补充说明：

- 如果你用的是 Cursor，不同小版本的按钮名字和位置可能略有差异
- 如果你用的是 Codex，重点也是先进入“页面预览 + 元素可选中 + 侧边样式或结构面板”的工作区
- 只要看到“页面预览 + 可选中元素 + 右侧布局/样式侧栏或组件树”这一类界面，就说明已经进入可用的可视化编辑工作区  
成功判定：
- 页面能在 Cursor Browser 或 Codex 内置浏览器中正常打开
- 你移动鼠标时，页面元素可以被高亮或选中
- 侧边区域能看到布局、样式或元素结构相关面板

### 第四步：先做一轮“只看不改”的走查

你现在在做：先把问题记清楚，再动手改，避免边看边改把问题越改越散。

建议按下面顺序过一遍：

- 第一眼看层级：标题、主按钮、信息卡片、辅助说明，谁最显眼
- 第二眼看节奏：上下间距是否均匀，是否有区域过挤或过空
- 第三眼看对齐：左边线、按钮线、卡片边缘是否整齐
- 第四眼看状态：默认、加载、空态、异常态有没有明显跳样
- 第五眼看演示感：这页拿给别人演示时，第一屏是否够清楚

建议先写一个 3 到 5 条的小清单，例如：

- 顶部标题离状态栏太近
- 主按钮颜色够亮，但按钮高度偏矮
- 卡片之间间距不统一
- 空态说明文案偏长，视觉重心太低

成功判定：

- 你已经列出本轮要处理的问题，不再边看边临时起意

### 第五步：优先用可视化编辑器做“小而确定”的修改

你现在在做：先改最容易确认的视觉问题，快速建立一版更顺眼的页面。

推荐优先级：

1. 间距和对齐
2. 字号和字重
3. 按钮尺寸、圆角和颜色
4. 卡片内边距和块间距
5. 区块顺序或栅格关系

操作建议：

- 先选中一个元素，只改一个维度，比如只改 `padding` 或只改 `font-size`
- 每改完一个点，立刻看整体，不要一口气同时改 7 个参数
- 优先让页面“更整齐”，再追求“更惊艳”

成功判定：

- 页面明显更规整
- 你能说清每个改动解决的是哪一个具体问题

### 第六步：碰到说不清参数的地方，用“点选 + 自然语言”提要求

你现在在做：把模糊的审美判断，变成能被 Agent 执行的可视化修改请求。

可以直接对着页面元素说这类话：

- “把这个卡片标题和正文的层级拉开一点，但不要更花”
- “让这个主按钮更像主操作，宽度再稳一点”
- “把这里的留白调成和上面卡片一致”
- “这块空态太散了，往上收紧一点”
- “把这里三个操作的视觉重心对齐”

更容易成功的说法是：

- 指出对象：改哪一块
- 指出问题：现在哪里别扭
- 指出方向：更紧一点、更轻一点、更稳定一点
- 指出边界：不要改文案、不要改页面结构、不要动其他卡片

成功判定：

- Agent 返回的修改范围和你的意图一致
- 改动没有波及无关区域

### 第七步：改完一轮后，马上让代码落地，不要只停在视觉层

你现在在做：把 Browser 里的视觉修改真正回写到 HTML/CSS，而不是只留在预览状态。

执行原则：

- 每完成一轮确认，就让 Agent 应用到代码
- 一次只提交一类变化，比如“顶部区优化”或“卡片间距统一”
- 不要把整页所有问题堆成一次超长提示

在当前项目里，重点检查这些文件是否被正确更新：

- 页面 HTML 文件，例如 `/Users/mi/Documents/AI_tutor/V1.2任务页/3.1今日视图默认态.html`
- 页面级样式文件，例如 `/Users/mi/Documents/AI_tutor/V1.2任务页/styles.css`
- 如涉及共享层，请确认是否真的需要升级 `_shared/`，不要因为局部页面微调就贸然改共享骨架

成功判定：

- 相关 HTML 或 CSS 文件确实发生了更新
- 重新刷新页面后，效果还在，不是一次性预览假象

### 第八步：每改完一轮，都回到“演示视角”再看一次

你现在在做：避免只盯着一个局部元素，结果把整页演示感做碎了。

至少检查这 4 件事：

- 第一屏有没有更清楚
- 主操作是不是更突出
- 卡片和区块的节奏是不是更一致
- 切换到其他状态帧时，有没有突然变形

如果这页属于多状态原型，至少把默认态、空态、异常态各看一遍。

成功判定：

- 页面不是“局部精致、整体失衡”
- 多个状态之间的风格保持一致

### 第九步：把本轮视觉改动反写成产品可复用结论

你现在在做：把“这次为什么这样改”沉淀下来，避免下次又从头试。

建议至少补一份简短记录，内容包括：

- 本轮改了哪一页
- 改了哪 3 到 5 个关键点
- 哪些属于纯视觉微调
- 哪些已经影响 Spec，需要后续同步

你可以直接用下面这个模板：

```Markdown
本轮精修页面：
- 3.1 今日视图默认态

本轮确认的有效修改：
- 顶部标题区下移，避免压迫状态栏
- 主按钮高度统一到更稳的视觉尺寸
- 卡片上下间距统一，列表节奏更顺
- 空态文案缩短，插画与文字整体上提

需要同步上游的内容：
- 无 / 有（请写明）
```

成功判定：

- 下次再看这页时，你知道为什么会变成现在这样
- 如果有人问“这个改动是视觉决定还是需求决定”，你答得出来

## 常见错误与排查

### 1. 页面在 Cursor Browser 或 Codex 内置浏览器里打不开

排查方式：

- 先确认本地服务是否还在运行
- 确认 URL 是否写成了 `http://localhost:8000/...`
- 如果你换过端口，记得把地址一并改掉

### 2. 打开了页面，但没有看到可视化编辑能力

排查方式：

- 如果你用的是 Cursor，先确认版本是否升级到 `2.2` 或更新版本
- 重新打开 Browser 页签或 Codex 内置浏览器页面
- 观察是否有元素选择、高亮、样式侧栏、组件树之类的入口  
补充说明：
- Cursor 官方在 `2025-12-10` 和 `2025-12-11` 已经发布 Browser Visual Editor 相关更新
- Codex 现在也可以用内置浏览器承接同类页面精修流程
- 具体按钮位置可能随工具版本变化，这里不要死记某个图标长什么样，重点看“是否进入可选元素 + 可改样式”的工作区

### 3. 改完当下看着不错，一刷新又没了

排查方式：

- 你可能只在视觉层试了效果，还没有真正应用到代码
- 检查 HTML/CSS 文件有没有被写入
- 重新刷新页面确认效果是否持久

### 4. 一轮提示改太多，结果页面越改越乱

排查方式：

- 把任务拆小
- 每次只改一个区域或一类问题
- 先做“更整齐”，再做“更有设计感”

### 5. 改着改着把需求本身改掉了

排查方式：

- 回看这次变更到底是“视觉微调”还是“交互/需求变化”
- 如果影响页面职责、状态定义、验收标准，就回到 Spec 先改

## 你最少要记住什么

1. `Cursor Browser Visual Editor` 或 `Codex 内置浏览器` 都适合做精修，不适合绕过 Spec 直接改需求。
2. 先启动本地页面，再进 Cursor Browser 或 Codex 内置浏览器，否则很难形成稳定的“看效果 → 落代码”闭环。
3. 一次只改一个区域或一类问题，效果最好。
4. 每轮视觉修改都要真正应用到 HTML/CSS，再刷新验证。
5. 如果改动已经影响业务逻辑或状态定义，要同步回 Spec，再考虑 PRD。
