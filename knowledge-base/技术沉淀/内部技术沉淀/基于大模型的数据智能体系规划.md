# 基于大模型的数据智能体系规划

<文档概述>

<callout emoji="💡">
本文档系统规划了基于大模型的数据智能体系建设方案，围绕 **Agent-Skill 体系**、**机器人与 NL-SQL 优化**、**RAG 知识库**三大核心板块展开，旨在构建"对话即分析、知识即能力、数据即决策"的智能化数据工作范式，全面提升内容生态数据团队的分析效率与决策支撑能力。
</callout>

## 一、Agent-Skill 规划

### 1.1 总体架构

Agent 体系的核心思路是：将数据团队不同角色的专业经验与能力，蒸馏压缩到大模型可调用的 Agent 中，使每个 Agent 成为该领域的"虚拟专家"。通过 Skill 机制将分析动作标准化、可复用化，实现"经验即代码、对话即执行"。

### 1.2 数据分析 Agent

<callout emoji="📊">
**定位**：面向数据分析师的智能助手，能够自如对话，完成从数据获取到洞察输出的全流程分析工作。
</callout>

**核心能力**：

<grid>
<column width-ratio="0.330000">
**🗣️ 自由对话式分析**
用户以自然语言描述分析需求，Agent 自动拆解为可执行的分析步骤，逐步推进并输出结论。支持多轮对话中的上下文理解与追问，分析师无需切换工具即可完成从"我想看看最近用户留存的变化"到"帮我归因一下留存下降的原因"的全链路分析。
</column>
<column width-ratio="0.330000">
**🧠 经验蒸馏与 Skill 封装**
在 Agent.md 中蒸馏压缩数据分析师的核心经验，包括分析框架（如 AARRR、OSM、人货场）、常见分析套路（趋势归因、漏斗拆解、同期群分析）、异常检测策略、数据解读原则等。每类经验封装为可调用的 Skill，确保分析过程有方法论支撑。
</column>
<column width-ratio="0.330000">
**✅ 分析质量保障**
内置分析质量校验机制，包括样本量检查、统计显著性判断、数据口径一致性校验、结论可复现性验证等，避免低质量分析输出误导决策。
</column>
</grid>

**Agent.md 核心内容规划**：

<callout emoji="📝">
Agent.md 是数据分析 Agent 的"大脑"，承载了从资深分析师经验中提炼的方法论与规则体系，是 Agent 输出高质量分析的前提保障。
</callout>

- **分析框架库**：预置 AARRR、OSM、人货场、RFM 等经典分析框架，Agent 根据问题类型自动匹配框架
- **分析套路库**：趋势分析→归因拆解、漏斗分析→瓶颈定位、同期群分析→留存分层等标准化分析路径
- **数据解读规则**：同比环比判断标准、异常波动阈值设定、数据质量检查清单
- **输出规范**：分析报告结构模板、结论表述规范、可视化建议规则

### 1.3 数据产品 Agent

<callout emoji="🔍">
**定位**：面向数据产品工作的专业 Agent，对埋点体系、指标口径、指标体系有深入理解，能够辅助完成数据产品全生命周期工作。
</callout>

**核心能力**：

<grid>
<column width-ratio="0.330000">
**📍 埋点体系管理**
理解埋点设计原则与规范，能够根据业务需求生成埋点方案、校验埋点完整性、识别埋点覆盖盲区。对埋点命名规范、参数设计规范、上报逻辑等有体系化认知，确保埋点方案的规范性与可维护性。
</column>
<column width-ratio="0.330000">
**📏 指标口径治理**
深入理解每个指标的业务定义、计算公式、数据来源、适用场景，能够精准回答"这个指标怎么算的""为什么两个报表数据不一致"等口径问题。支持指标口径的自动校验与冲突检测，消除"同名异义"和"异名同义"问题。
</column>
<column width-ratio="0.330000">
**🏗️ 指标体系构建**
掌握 OSM 模型、指标分层等体系构建方法论，能够根据业务目标自动推导指标体系结构，识别缺失指标与冗余指标，确保指标体系的完整性与精简性。
</column>
</grid>

**Skill 规划**：

| **Skill 名称** | **功能描述** | **触发场景** |
|-|-|-|
| 埋点方案生成 | 根据业务需求自动生成标准化埋点方案 | 新功能上线、埋点补全 |
| 埋点覆盖检查 | 检查业务流程的埋点覆盖完整性 | 埋点验收、季度审计 |
| 口径查询与解释 | 精准回答指标口径问题 | 口径争议、数据对齐 |
| 口径冲突检测 | 自动识别同名异义/异名同义指标 | 指标治理、新指标上线 |
| 指标体系推导 | 基于业务目标推导指标体系结构 | 新业务线搭建、体系优化 |
| 指标健康评估 | 评估指标体系的完整性与可用性 | 季度复盘、体系迭代 |

### 1.4 经营分析 Agent

<callout emoji="📈">
**定位**：面向业务经营的专业 Agent，对业务策略和动作有深入理解，能够将业务动作与数据变化建立因果关联，实现精准归因。
</callout>

**核心能力**：

<grid>
<column width-ratio="0.330000">
**🎯 业务动作追踪**
持续跟踪业务侧的策略调整与运营动作（如新功能上线、活动投放、定价调整等），建立"动作—数据"的时序关联。对每个业务动作的执行时间、影响范围、预期效果有结构化记录。
</column>
<column width-ratio="0.330000">
**🔍 数据变化归因**
当核心指标发生显著变化时，Agent 能够自动关联同期业务动作，结合统计方法（如差分法、CausalImpact 等）进行归因分析，区分"业务动作驱动"与"自然波动"，输出可解释的归因结论。
</column>
<column width-ratio="0.330000">
**📊 策略效果评估**
对每个业务动作的执行效果进行闭环评估，对比预期与实际差异，沉淀策略经验，为后续决策提供参考。

## 二、机器人 Skill 规划与 NL-SQL 优化

<callout emoji="⚡">
**核心理念**：机器人定位为"数据获取与简单分析的效率工具"，而非"复杂分析的替代者"。通过中间表标准化、NL-SQL 精准化、推送自动化，让机器人在擅长的领域做到极致可靠。
</callout>

### 2.1 中间表完善：所有分析基于中间表

**设计思路**：

当前 NL-SQL 的核心痛点在于直接查询原始表时，SQL 生成复杂度高、准确率低、执行耗时长。解决方案是构建完善的中间表体系，将复杂的数据加工逻辑前置到中间表层，使 NL-SQL 只需查询"宽表"即可满足大部分分析需求。

**中间表建设规划**：

<grid>
<column width-ratio="0.500000">
**👤 用户域中间表**
用户基础属性表、用户行为聚合表、用户生命周期表、用户价值分层表，覆盖用户画像与行为分析的核心需求。
**📄 内容域中间表**
内容基础信息表、内容消费聚合表、内容互动聚合表、内容质量评分表，覆盖内容运营分析的核心需求。
</column>
<column width-ratio="0.500000">
**💰 商业域中间表**
广告收入聚合表、ARPU 明细表、客户结构表、竞价效率表，覆盖商业化分析的核心需求。
**🌐 流量域中间表**
页面流量聚合表、渠道质量表、来源归因表，覆盖流量分析的核心需求。
</column>
</grid>

**中间表设计规范**：

| **规范项** | **要求** |
|-|-|
| 命名规范 | `dws_{业务域}_{聚合粒度}_{业务描述}` |
| 时间粒度 | 统一支持日/周/月三种粒度 |
| 维度对齐 | 同域中间表共享核心维度，确保交叉分析可行 |
| 数据时效 | T+1 日凌晨产出，核心表支持小时级延迟 |
| 质量保障 | 每张中间表配套数据质量监控规则 |

### 2.2 机器人只做数据获取和简单分析

**能力边界定义**：

明确机器人的能力边界，避免"什么都能做但什么都做不好"的困境。机器人聚焦以下三类场景：

<grid>
<column width-ratio="0.330000">
**🔎 数据查询**
用户用自然语言描述数据需求，机器人通过 NL-SQL 获取数据并返回结果。例如"昨天 DAU 是多少""本周新增用户对比上周变化了多少"。
</column>
<column width-ratio="0.330000">
**📈 简单分析**
基于查询结果进行趋势判断、同比环比计算、Top-N 排序等简单分析。例如"最近 7 天留存率的趋势""各渠道新增用户排名"。
</column>
<column width-ratio="0.330000">
**💬 数据解释**
对查询结果进行自然语言解读，帮助用户快速理解数据含义。例如"DAU 环比下降 5%，主要受周末效应影响"。
</column>
</grid>

<callout emoji="⚠️">
**不做的场景**：复杂归因分析、多变量因果推断、策略方案设计等需要深度业务理解的工作，交由 Agent 处理。
</callout>

---

### 2.3 机器人囊括日报周报等常规推送

**推送体系规划**：

- **日报**：每日定时推送核心指标概览，包括 DAU、新增用户、留存率、人均时长、收入等关键指标及日环比变化，异常指标自动高亮并附简要说明
- **周报**：每周推送周度经营分析摘要，包括核心指标周趋势、本周关键业务动作回顾、下周重点关注指标，帮助管理层快速掌握经营全局
- **专项推送**：针对特定场景的定制化推送，如 AB 实验结果通知、指标阈值告警、数据质量异常提醒等

**推送机制设计**：

| **推送类型** | **频率** | **推送渠道** | **生成方式** |
|-|-|-|-|
| 核心日报 | 每日 10:00 | 飞书群/个人 | 模板 + 数据自动填充 |
| 经营周报 | 每周一 10:00 | 飞书群/邮件 | 模板 + 数据 + 简要分析 |
| 指标告警 | 实时 | 飞书群/个人 | 阈值触发 + 归因摘要 |
| 实验通知 | 实验结束时 | 飞书群 | 实验结论 + 策略建议 |

## 三、RAG 知识库

### 3.1 知识库搭建的原理和方案设计

**RAG 原理简述**：

RAG（Retrieval-Augmented Generation）的核心思路是：在用户提问时，先从知识库中检索相关文档片段，再将检索结果作为上下文注入大模型的 Prompt，使模型基于真实知识生成回答，从而解决大模型幻觉问题并确保回答的准确性与可追溯性。

**方案架构**：

<whiteboard token="Zxr3w6viMhsOKzbDnn5cBmjunbd"></whiteboard>

**关键设计决策**：

- **文档切分策略**：按语义单元切分（而非固定字数），每个 Chunk 保留完整的知识上下文，Chunk 大小控制在 500-1000 tokens，重叠 100 tokens 避免语义截断
- **Embedding 模型选择**：采用中文优化的 Embedding 模型（如 BGE-M3），确保中文语义检索的准确率
- **混合检索策略**：向量检索（语义匹配）+ 关键词检索（精确匹配）双路召回，通过 RRF（Reciprocal Rank Fusion）算法融合排序，兼顾语义相似性与关键词精确性
- **检索评估体系**：建立检索准确率、召回率、MRR 等指标的评估基准，持续优化检索效果

### 3.2 内容生态数据产品知识库搭建

知识库是 Agent 与机器人的"记忆"，其内容质量直接决定输出质量。内容生态数据产品知识库按以下结构组织：

| **知识域** | **内容类型** | **具体内容** |
|-|-|-|
| 埋点知识 | 规范文档 | 埋点命名规范、参数设计规范、上报逻辑规范 |
| 埋点知识 | 事件字典 | 全量事件定义、参数说明、触发时机 |
| 埋点知识 | 常见问题 | 埋点丢失排查、重复上报处理、参数缺失修复 |
| 指标知识 | 指标字典 | 全量指标定义、计算公式、口径说明 |
| 指标知识 | 指标体系 | OSM 拆解关系、指标分层结构、指标依赖关系 |
| 指标知识 | 口径 FAQ | 常见口径争议及统一结论 |
| 分析知识 | 分析方法论 | 分析框架、分析套路、归因方法 |
| 分析知识 | 历史分析 | 经典分析案例、复盘报告、策略效果记录 |
| 业务知识 | 业务架构 | 业务模式、产品功能、运营策略 |
| 业务知识 | 动作日历 | 业务动作时间线、策略变更记录 |

**知识库更新机制**：

- **自动更新**：指标字典、事件字典等结构化知识，通过数据血缘与元数据系统自动同步更新
- **半自动更新**：分析案例、口径 FAQ 等，由分析师提交草稿，经审核后入库
- **定期巡检**：每月对知识库进行全量巡检，识别过期内容并标记更新

### 3.3 知识库通过 MCP 接入 Agent 的方案

**MCP 协议概述**：

MCP（Model Context Protocol）是大模型与外部工具/数据源交互的标准化协议。通过 MCP，Agent 可以在对话过程中动态调用知识库的检索能力，实现"按需检索、实时注入"的知识供给模式。

**接入架构**：

**MCP Server 功能设计**：

- **knowledge_search**：根据用户问题检索相关知识片段，支持按知识域过滤，返回 Top-K 结果及来源引用
- **knowledge_query**：精确查询特定知识条目（如查询某指标的定义），适用于结构化知识的精准获取
- **knowledge_update**：向知识库新增或更新知识条目，支持 Agent 在对话中学习并沉淀新知识
